# NX-83 — Wiring scheduler/cron pentru joburi (compose + entrypoint)
**Owner:** S · **Faza:** P1-O1 · **Zi/Ord:** după F2-3 (rollup_usage) + NX-51 (cleanup_dedupe) · **Branch:** `feat/NX-83-scheduler` · **Complexitate:** S/M · **Estimare:** 4h

## Goal
Cele 3 joburi de mentenanță (`rollup_usage`, `cleanup_dedupe`, `embed_products`)
există dar sunt `__main__`-only: rulează doar manual de la terminal. În prod nu le
pornește nimic → **rollup-ul nocturn de facturare nu rulează** (dashboard + facturare
citesc din `usage_daily`, care rămâne gol) și **purja `inbound_dedupe` nu rulează**
(tabelul crește la infinit). Taskul scrie un mini-scheduler intern (`src/jobs/scheduler.py`),
îl pune ca serviciu compose cu restart + healthcheck, și îl pune să cheme funcțiile `run()`
existente la intervalele corecte. Zero logică de job duplicată — schedulerul e doar
orchestrare temporală peste funcțiile deja scrise.

## Business Context
Fără rollup nocturn, `usage_daily` rămâne gol → nu putem factura clienții (nu știm
conversații/tokeni/cost/comenzi atribuite per zi) și dashboardul e mort. Fără purja
`inbound_dedupe`, tabelul de dedupe crește nelimitat (un marker per mesaj inbound, pe vecie)
→ degradare lentă a DB-ului. `embed_products` incremental ține `search_products` semantic
la zi când catalogul se schimbă, fără re-embed scump (content_hash → re-rulare la zi = $0).
Astea sunt joburi de infrastructură: nu aduc bani direct, dar fără ele bucla de bani
(facturare) și retrieval-ul (vânzare) se rup tăcut.

## Technical Description

### Decizia: mini-scheduler intern, NU pg_cron, NU cron de sistem
Avem 3 opțiuni; alegem **mini-scheduler intern în Python** ca serviciu compose separat:
- **pg_cron** — bun pentru DDL pur (drop partition, NX-84), dar `rollup_usage` și `embed_products`
  rulează cod Python (OpenAI pentru embed, agregare în query-uri Python) → pg_cron nu le poate chema.
- **cron de sistem (crontab pe VPS)** — fragil: depinde de host, nu e în compose, nu apare în
  `docker compose ps`, healthcheck manual. Respins.
- **mini-scheduler intern** — un proces `python -m src.jobs.scheduler` care ține o buclă asyncio,
  cheamă funcțiile `run()` existente la intervale fixe. E în compose (restart, healthcheck,
  loguri unde sunt toate). Simplu, fără dependență nouă (folosim `asyncio.sleep` + calcul de delay).

Nu adăugăm `apscheduler` sau `celery` — overhead inutil pentru 3-4 joburi periodice.
`asyncio.sleep` până la următoarea rulare e suficient (precizia de „rulează după miezul nopții"
e ok la ±câteva minute).

### Scheduler-ul (`src/jobs/scheduler.py`)
Buclă unică asyncio cu un registru de joburi `(nume, funcție, planificare)`. Fiecare job e
descris declarativ; bucla calculează care e următorul de rulat și doarme până atunci.

```python
"""Mini-scheduler intern pentru joburile de mentenanță (NX-83).

Un singur proces, buclă asyncio. NU dublează logica de job — cheamă funcțiile
`run()` existente din src/jobs/*. Toate joburile rulează pe admin_conn (mentenanță
cross-tenant / DDL), exact ca acum în __main__. Un job care crapă e logat și sărit
— NU oprește bucla (principiul 6: degradare, nu cădere tăcută).

    python -m src.jobs.scheduler
"""
from __future__ import annotations
import asyncio, logging, time
from dataclasses import dataclass
from datetime import UTC, datetime, timedelta

from src.config import get_settings
from src.db.connection import close_pool
from src.jobs import cleanup_dedupe, embed_products, rollup_usage

log = logging.getLogger(__name__)


@dataclass
class Job:
    name: str
    run: callable                  # async () -> Any ; cheamă funcția existentă
    interval_seconds: int          # cadență (ex. 24h pt rollup)
    at_hour_utc: int | None = None # opțional: ancorat la o oră UTC (rollup nocturn)
    next_run: float = 0.0          # epoch al următoarei rulări (monotonic-ish)


async def _safe_run(job: Job) -> None:
    """Rulează un job, prinde orice excepție (un job nu trebuie să oprească restul)."""
    start = time.monotonic()
    try:
        await job.run()
        log.info("job %s ok (%.1fs)", job.name, time.monotonic() - start)
    except Exception:  # noqa: BLE001 — un job picat nu rupe schedulerul (P6)
        log.exception("job %s a eșuat — sărit, reîncerc la următorul interval", job.name)


def _compute_next(job: Job, now: datetime) -> float:
    """Următorul moment de rulare (epoch). Dacă e ancorat la oră UTC, prima rulare
    e la acea oră azi/mâine; altfel now + interval."""
    if job.at_hour_utc is not None:
        candidate = now.replace(hour=job.at_hour_utc, minute=10, second=0, microsecond=0)
        if candidate <= now:
            candidate += timedelta(days=1)
        return candidate.timestamp()
    return now.timestamp() + job.interval_seconds


def _build_jobs() -> list[Job]:
    s = get_settings()
    jobs = [
        Job("rollup_usage", rollup_usage_run, interval_seconds=86400, at_hour_utc=0),
        Job("cleanup_dedupe", cleanup_dedupe.run, interval_seconds=21600),  # la 6h
    ]
    if s.embed_job_enabled and s.openai_api_key:   # embed cere cheie OpenAI
        jobs.append(Job("embed_products", embed_products_run, interval_seconds=3600))  # orar, incremental
    return jobs
```

`rollup_usage_run` și `embed_products_run` sunt thin wrappers care deschid pool-ul/llm
și cheamă `run_rollup`/`embed_pending` exact ca `_main`-urile lor (refolosim, nu rescriem):

```python
async def rollup_usage_run() -> None:
    from src.db.connection import admin_conn, get_pool
    pool = await get_pool()
    async with admin_conn(pool) as conn:
        await rollup_usage.run_rollup(conn, day=rollup_usage.yesterday_utc())

async def embed_products_run() -> None:
    from src.agent.llm import get_llm
    from src.db.connection import admin_conn, get_pool
    llm = get_llm()
    if llm is None:
        log.warning("embed_products sărit — OPENAI_API_KEY lipsește"); return
    pool = await get_pool()
    async with admin_conn(pool) as conn:
        await embed_products.embed_pending(conn, llm)
```

`cleanup_dedupe.run` are deja semnătura `() -> int` (deschide pool intern) → se cheamă direct.

### Bucla principală + healthcheck
```python
async def _loop(jobs: list[Job], *, run_on_start: bool = True) -> None:
    now = datetime.now(UTC)
    for j in jobs:
        j.next_run = now.timestamp() if run_on_start else _compute_next(j, now)
    while True:
        now_ts = datetime.now(UTC).timestamp()
        due = [j for j in jobs if j.next_run <= now_ts]
        for j in due:
            await _safe_run(j)
            j.next_run = _compute_next(j, datetime.now(UTC))
        _touch_heartbeat()                       # scrie /tmp/scheduler_alive (healthcheck)
        sleep_for = max(5.0, min(j.next_run for j in jobs) - datetime.now(UTC).timestamp())
        await asyncio.sleep(min(sleep_for, 300)) # cap 5 min: reverificăm heartbeat-ul des
```

**Healthcheck:** schedulerul scrie periodic un fișier heartbeat (`_touch_heartbeat`); compose
verifică vechimea lui. Nu deschidem port HTTP doar pentru asta.
```python
HEARTBEAT = "/tmp/scheduler_alive"
def _touch_heartbeat() -> None:
    with open(HEARTBEAT, "w") as f:
        f.write(str(int(time.time())))
```
Comanda de healthcheck (în compose) verifică că fișierul nu e mai vechi de ~6 min:
`python -c "import time,sys; sys.exit(0 if time.time()-float(open('/tmp/scheduler_alive').read())<360 else 1)"`.

### Serviciu compose
Adăugăm serviciul `scheduler` în `docker-compose.yml`. NU depinde de redis (joburile
folosesc DOAR DB-ul, prin `admin_conn`), dar îl ținem aliniat la restul (env_file, volume src):
```yaml
  # Mini-scheduler intern (NX-83): rulează joburile de mentenanță la intervale fixe.
  #  • rollup_usage zilnic (facturare/dashboard din usage_daily)
  #  • cleanup_dedupe la 6h (purjă inbound_dedupe >48h)
  #  • embed_products orar incremental (doar dacă OPENAI_API_KEY e setat)
  # Healthcheck pe heartbeat file; restart automat.
  scheduler:
    build: .
    command: python -m src.jobs.scheduler
    volumes:
      - ./src:/app/src
    env_file: .env
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "python", "-c", "import time,sys; sys.exit(0 if time.time()-float(open('/tmp/scheduler_alive').read())<360 else 1)"]
      interval: 60s
      timeout: 5s
      retries: 3
      start_period: 30s
```
`restart: unless-stopped` îl punem și pe scheduler (e singurul serviciu pentru care un
restart e curat — joburile sunt idempotente: `rollup_usage_day` e UPSERT pe `(business_id, day)`,
`cleanup_dedupe` e DELETE pe vârstă, `embed_products` sare ce e la zi prin `content_hash`).

### Idempotență la restart / dublă rulare
Toate cele 3 joburi sunt deja idempotente (vezi mai sus) → un restart al schedulerului care
re-rulează un job la pornire NU strică nimic. `run_on_start=True` (rulăm o dată la boot) e sigur
și util: după un deploy, rollup-ul prinde imediat ziua de ieri dacă fereastra a fost pierdută.

### Câmpuri TurnContext scrise aici
**Niciunul.** Schedulerul rulează în afara pipeline-ului, fără `TurnContext` (ca joburile pe care
le orchestrează). Nu atinge un mesaj inbound, nu produce reply.

## Principii CLAUDE.md aplicabile
- **P6 (niciodată tăcere / degradare):** un job picat e logat și sărit, bucla continuă; restart
  automat la cădere de proces. Rollup-ul nu se mai poate „pierde tăcut" — fie rulează, fie
  healthcheck-ul pică zgomotos.
- **P7 (business_id pe tot):** schedulerul nu adaugă query-uri noi — refolosește funcțiile
  existente. `rollup_usage_day($1=business_id, ...)` și restul filtrează deja pe `business_id`;
  `cleanup_dedupe` e mentenanță cross-tenant pe `admin_conn` (markere non-PII, excepție documentată).
- **P10 (observabilitate din runner):** schedulerul (orchestratorul) logă start/ok/fail per job
  + durată; joburile nu știu că sunt măsurate.
- **P12 (PII):** logurile schedulerului conțin DOAR nume de job + nr. rânduri + durată. Niciun
  telefon / id de canal / corp de mesaj (joburile oricum nu le ating, mai puțin embed pe text de produs).

## Out of Scope
- **NX-84** (cleanup partiții vechi `messages`/`analytics_events` + purjă `semantic_cache` expirat) —
  jobul `cleanup.py` nu există încă; când e scris, se adaugă în registrul schedulerului ca un `Job`
  nou (DDL pe `admin_conn`). Aici doar lăsăm cârligul, NU scriem jobul.
- **Lock distribuit / leader election** între replici — schedulerul rulează ca **o singură replică**
  (joburile sunt idempotente, dar nu pornim 2 instanțe). Coordonarea multi-replică = task separat dacă
  vreodată scalăm schedulerul.
- **Retry per-job cu backoff fin** — v1: un job picat se reîncearcă la următorul interval (suficient
  pentru joburi periodice). Backoff exponențial intra-interval = peste nevoie acum.
- **Dashboard / alertă pe job picat** (Slack via `operator_alert_webhook`) — v1 = log + healthcheck;
  alerta proactivă pe eșec de rollup poate veni separat.
- **pg_cron pentru DDL** — decizie luată (Python scheduler); pg_cron rămâne o opțiune doar pentru NX-84
  dacă vreodată preferăm DDL-ul pur în DB.

## Test Cases
**Happy Path:**
1. `_loop` cu un singur job mock cu `next_run` în trecut → jobul e chemat exact o dată într-un ciclu,
   apoi `next_run` recalculat în viitor (verificat cu un mock care numără apelurile + `_loop` rulat
   un ciclu prin monkeypatch pe `asyncio.sleep`/break).
2. `_build_jobs()` cu cheie OpenAI setată (settings monkeypatch) → conține `rollup_usage`,
   `cleanup_dedupe` ȘI `embed_products`, cu intervalele din config.

**Edge Cases:**
1. `_compute_next` la `23:55 UTC` cu `at_hour_utc=0` → următoarea rulare = mâine `00:10 UTC`
   (nu azi, deja trecut).
2. Două joburi „due" în același ciclu → ambele rulate în ordine, fiecare cu `next_run` propriu
   recalculat (un job lent nu fură slotul celuilalt).
3. `run_on_start=False` → niciun job nu rulează la boot; primul tick e la `_compute_next`.

**Failure Cases:**
1. `rollup_usage_run` aruncă (mock pe `run_rollup` → `RuntimeError`) → `_safe_run` îl prinde,
   logă `job rollup_usage a eșuat`, bucla NU se oprește, restul joburilor rulează.
2. `embed_products_run` cu `get_llm()` → `None` (fără cheie) → logă „sărit", iese curat, fără excepție.
3. Heartbeat file ne-scriibil (mock `open` → `OSError`) → eroarea NU oprește bucla (prins în `_safe_run`
   sau try local pe `_touch_heartbeat`); healthcheck va eșua corect dacă fișierul lipsește.

> Testele de scheduler folosesc DOAR mock-uri: funcțiile `run` sunt înlocuite cu async mocks,
> `asyncio.sleep` monkeypatched ca să nu blocheze, settings prin `get_settings.cache_clear()` +
> env override. ZERO conexiuni DB sau apeluri OpenAI în CI.

## Cost
Schedulerul în sine nu cheamă LLM. Singurul cost LLM derivat e **`embed_products`**, dar acela e
**incremental prin `content_hash`**: re-rulare orară când catalogul NU s-a schimbat = **$0**
(query-ul găsește hash-uri identice → `todo` gol → 0 apeluri embed). Costul apare DOAR la produse
noi/modificate: `text-embedding-3-small` ~ $0.00002 / 1K tokeni; un produs ≈ 50-150 tokeni →
~$0.000002-0.000004 / produs. La 500 produse re-embed complet (rar, doar la `--force` sau prima
populare): ~$0.001-0.002 total. **Ce limitează costul:** (1) `content_hash` (nu re-embed ce e la zi),
(2) gate pe `embed_job_enabled` + prezența cheii, (3) cadența incrementală (orar, nu continuu).
Raportat la „per 1000 mesaje", embed-ul e decuplat de volumul de mesaje (depinde de catalog, nu de trafic)
→ contribuție ~$0 pe mesaj. Rollup și cleanup sunt cod determinist (cost $0 LLM).
