# NX-72 — Strat GDPR Python — erase wrapper + export/access
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după 003 (rol bot_runtime) · **Branch:** `feat/NX-72-gdpr-layer` · **Complexitate:** M · **Estimare:** 4h

## Goal
`src/gdpr/` e 0 octeți. Funcția SQL `gdpr_erase_contact(p_contact)` există în schema_v2
dar nimeni n-o cheamă, `gdpr_requests` e netins, iar rolul `gdpr_svc` din CLAUDE.md nu
există în nicio migrare. Taskul scrie stratul Python care leagă cele trei piese: un
**wrapper de erase** (rulează funcția `security definer` printr-o conexiune dedicată),
un **handler de export** (toate datele unui contact într-un JSON) și unul de **access**
(subset citibil), fiecare **înregistrat în `gdpr_requests`** (kind + status) și **auditat
în `audit_log`**. Fără asta nu putem onora o cerere GDPR (drept de ștergere / portabilitate /
acces) — obligație legală pentru orice client din UE.

## Business Context
Clienții agenției sunt retaileri din România → GDPR e obligatoriu, nu opțional. O cerere de
ștergere („vreau să-mi ștergeți datele") sau de export trebuie onorată în 30 de zile, cu
urmă de audit. Fără acest strat răspunsul e manual în SQL pe producție (risc + zero audit).
Cardul livrează drumul determinist: o cerere → o linie în `gdpr_requests` → execuție →
PII distrus / exportat → `audit_log`. E infrastructură de conformitate, nu pe hot path.

## Technical Description

### Stare curentă vs țintă
- **Curent:** `gdpr_erase_contact(p_contact uuid)` (schema_v2, linia 754) face deja
  anonimizarea: `contacts` (display_name=NULL, profile='{}', rfm=NULL, erased_at=now()),
  `delete channel_identities`, `messages.body=NULL/payload='{}'/media_ref=NULL`, + un INSERT
  în `audit_log` cu `actor='gdpr_svc'`. **Dar:** rulează `security definer` ca proprietarul
  funcției, nu cere `business_id`, și NU atinge `gdpr_requests`. Rolul `gdpr_svc` e doar un
  string literal în acel INSERT — nu există ca rol DB.
- **Țintă:** un strat Python care (1) creează cererea în `gdpr_requests`, (2) o trece pe
  `processing`, (3) execută (erase → SQL existent; export/access → SELECT-uri read-only),
  (4) o trece pe `done`/`failed`, (5) lasă urmă în `audit_log`. Plus migrarea `009` care
  creează rolul `gdpr_svc` + grant EXECUTE pe funcție (aliniere cu CLAUDE.md).

### Conexiune dedicată — NU bot_runtime, NU hot path
`gdpr_erase_contact` e `security definer` și șterge `channel_identities` (PII) — `bot_runtime`
are DELETE pe ea, dar erase-ul cross-tabel cu audit e o operație de **control plane**, nu de
tenant runtime. Rulăm prin `admin_conn` (pool privilegiat, `db/connection.py`) — singura cale
care precede/transcende scope-ul de tenant. Migrarea `009` creează rolul `gdpr_svc` + EXECUTE
pe funcție pentru aliniere cu modelul documentat; în cod, până la provisioning separat de
login, folosim `admin_conn` (la fel ca joburile admin cleanup/embed). Justificare în docstring,
ca la `admin_conn`.

### API intern (src/gdpr/erase.py)
```python
async def erase_contact(business_id: str, contact_id: str, *, requested_by: str) -> str:
    """Onorează o cerere de ștergere GDPR. Întoarce gdpr_requests.id.
    Rulează pe admin_conn (control plane): funcția e security definer + atinge PII
    cross-tabel. Idempotent pe re-rulare (erase pe contact deja șters = no-op anonimă)."""
    pool = await get_pool()
    async with admin_conn(pool) as conn:
        req_id = await create_request(conn, business_id, contact_id, "erase", requested_by)
        try:
            await mark_processing(conn, business_id, req_id)
            async with conn.transaction():
                await conn.execute("select gdpr_erase_contact($1)", contact_id)
                # audit suplimentar cu business_id + req_id (funcția scrie un audit fără ele)
                await write_audit(conn, business_id, "gdpr_erase", "contact", contact_id,
                                  {"request_id": req_id, "requested_by": requested_by})
            await mark_done(conn, business_id, req_id, result_ref=None)
        except Exception:
            await mark_failed(conn, business_id, req_id)
            raise
        return req_id
```

```python
async def export_contact(business_id: str, contact_id: str, *, requested_by: str) -> dict:

    Read-only. business_id în FIECARE select (P7). Întoarce structura, NU o scrie în storage
    (result_ref rămâne None în v1; upload în storage cu TTL = task separat)."""
```

`access_contact` = subset al export-ului (kind='access'): ce ținem despre el, fără dump-ul
integral de mesaje (sumar de volum). Toate trei: cerere în `gdpr_requests` + audit.

### Query-uri (src/db/queries/gdpr.py) — business_id în fiecare
```python
async def create_request(conn, business_id, contact_id, kind, requested_by) -> str:
    return await conn.fetchval(
        """insert into gdpr_requests (business_id, contact_id, kind, requested_by, status)
           values ($1, $2, $3, $4, 'pending') returning id::text""",
        business_id, contact_id, kind, requested_by)

async def mark_processing(conn, business_id, req_id) -> None:
    await conn.execute(
        "update gdpr_requests set status='processing' where business_id=$1 and id=$2",
        business_id, req_id)

async def mark_done(conn, business_id, req_id, *, result_ref) -> None:
    await conn.execute(
        """update gdpr_requests set status='done', completed_at=now(), result_ref=$3
           where business_id=$1 and id=$2""", business_id, req_id, result_ref)

async def mark_failed(conn, business_id, req_id) -> None:
    await conn.execute(
        """update gdpr_requests set status='failed', completed_at=now()
           where business_id=$1 and id=$2""", business_id, req_id)

async def write_audit(conn, business_id, action, entity, entity_id, details) -> None:
    await conn.execute(
        """insert into audit_log (business_id, actor, action, entity, entity_id, details)
           values ($1, 'gdpr_svc', $2, $3, $4, $5::jsonb)""",
        business_id, action, entity, entity_id, json.dumps(details))
```
Export SELECT-uri (toate cu `where business_id=$1`):
```python
# contact:        select display_name, locale, profile, lead_score, lifecycle, consent
#                 from contacts where business_id=$1 and id=$2
# identities:     select channel_kind, external_id, verified, created_at
#                 from channel_identities where business_id=$1 and contact_id=$2
# conversations:  select id, status, locale, created_at from conversations
#                 where business_id=$1 and contact_id=$2
# messages:       select conversation_id, direction, author, content_type, body, created_at
#                 from messages where business_id=$1 and contact_id=$2 order by created_at
# orders:         select external_id, status, total, created_at from orders
#                 where business_id=$1 and contact_id=$2
```

### PII și loguri (P12)
- Export-ul **conține** PII de canal (external_id) — e dreptul persoanei la propriile date,
  livrat către operatorul autorizat. NU se loghează: logăm doar `gdpr_request_id`, `kind`,
  `contact_id`, `business_id` — niciodată `display_name`/`external_id`/`body`.
- După erase: `channel_identities` ȘTERS (telefonul dispare), `messages.body=NULL`,
  `contacts.display_name/profile/rfm` golite, `erased_at` setat. Structura rămâne pentru
  agregate (P12: anonimizare, nu ștergere fizică).

### Câmpuri TurnContext scrise
**Niciunul.** Stratul GDPR rulează în afara pipeline-ului (nu primește `TurnContext`) —
e invocat dintr-un job admin / CLI / endpoint de dashboard, nu dintr-un stagiu. Nu atinge
`ctx.*`.

## Test Cases
> Conexiune integration (Supabase) într-o tranzacție rollback-uită, ca `tests/test_tenant_isolation.py`
> — seed contact + identity + 1 conversație + 2 mesaje + 1 order; ZERO apeluri LLM (taskul nu
> cheamă niciun LLM/embeddings → fără mock de model).

**Happy Path:**
1. `erase_contact` pe contact seedat → re-SELECT: `channel_identities` gol, `messages.body`
   NULL, `contacts.erased_at` not null; `gdpr_requests` done; `audit_log` are linia.
2. `export_contact` pe același contact → dict cu toate secțiunile populate; `identities` conține
   `external_id` (e dreptul lui); `gdpr_requests` are linie `kind='export'`, `status='done'`.

**Edge Cases:**
1. `erase_contact` rulat de două ori pe același contact → a doua oară no-op anonimă (deja șters),
   tot `status='done'`, fără excepție (idempotent).
2. `access_contact` pe contact fără mesaje/comenzi → dict cu liste goale, nu eroare; request done.
3. `export_contact` cu un contact care are mesaje în 2 luni (partiții diferite) → toate mesajele
   apar (SELECT-ul nu e legat de o singură partiție).

**Failure Cases:**
1. `contact_id` inexistent în tenant → erase nu găsește nimic de anonimizat; request marcat
   `failed` (sau `done` no-op, conform deciziei din implementare — testul fixează comportamentul),
   fără excepție scursă în afara funcției.
2. `business_id` al unui contact din ALT tenant → SELECT-urile de export întorc 0 rânduri pe
   secțiunile tenant-scoped; erase nu atinge nimic; NICIO dată din alt tenant în rezultat.
3. Eșec DB la mijloc (ex. tranzacția de erase aruncă) → `mark_failed` lasă `status='failed'`,
   `completed_at` setat, excepția se propagă controlat la apelant (test cu monkeypatch pe
   `conn.execute` care aruncă la `gdpr_erase_contact`).
