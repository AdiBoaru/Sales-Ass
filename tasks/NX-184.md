# NX-184 — Planner: response_shape + obligations (mixed-intent)

**Owner:** Claude (build) / Codex (verify) · **Faza:** P1 · **Track:** Response Quality · **Branch:** `feat/NX-184-planner-shapes` · **Complexitate:** M · **Estimare:** 2-3 zile
**Parent plan:** [docs/RESPONSE-QUALITY-EPIC.md](../docs/RESPONSE-QUALITY-EPIC.md) · **Depinde de:** NX-183 (unidirecțional — NX-183 e self-contained cu `response_shape` opțional; acest card îl populează complet)

## Goal
Planner-ul emite `response_shape` (INDICIU de compoziție, nu dispatch nou în renderer) + `obligations` pentru intențiile secundare. Modelul NU alege singur forma. **Mixed-intent nu se pierde:** „vreau o cremă lejeră **și cât durează livrarea?**" primește AMBELE răspunsuri — inclusiv când FAQ-ul ar fi interceptat mesajul.

## Business Context
`ResponsePlan.mode` (`planner.py:74`) e doar observabilitate; dispatch-ul real e prin condiționale în `render`. Deci `response_shape` = **hint cu fallback**, nu enum care rescrie renderer-ul. **Blocant confirmat în cod:** `faq_stage` rulează ÎNAINTE de triaj și face early-exit pe hit (`runner.py:256-259`) → pe un mesaj mixed-intent planner-ul nici nu rulează. Un prompt nu poate repara asta; e nevoie de mecanism în pipeline.

## Contract
**Vocabular unic peste tot** (JSON, cod, telemetrie): `response_shape` — NU `shape`, NU `response_hint`.
```json
{ "response_shape": "recommendation", "obligations": ["answer_delivery_question"] }
```
- `response_shape`: `recommendation | direct_followup | detail` (hint; semnale slabe → fallback `recommendation`; niciodată card gol). Câmp intern: `ResponsePlan.response_shape`.
- `obligations`: intenții secundare care TREBUIE onorate în același tur (ex. livrare/retur/plată → `faq_lookup` + inclus în răspuns).

## Mecanismul mixed-intent (DECIS: detector TRI-STATE pre-FAQ)
FAQ rulează înaintea triajului, deci decizia de early-exit nu poate aștepta semnalul triajului. Detectorul e o funcție PURĂ, ieftină, FĂRĂ LLM, cu rezultat **TRI-STATE** (un bool nu poate reprezenta „incert"):

```python
MixedIntentDecision = PURE_FAQ | POSSIBLE_MIXED | UNKNOWN
```

1. **`mixed_intent_decision(body, domain_pack)`** — semnale din **DomainPack/categorii** (termeni de produs/categorie/concern/brand prezenți) + segmentarea mesajului (două clauze: conjuncție SAU propoziții separate + interogativ de politică „cât/cum/ce…"). Generic pe vertical, zero hardcodare per-caz.
2. **`PURE_FAQ`** → early-exit ca azi (SINGURUL rezultat care permite early-exit; calea comună neschimbată).
3. **`POSSIBLE_MIXED`** → **NU early-exit**: atașează rezultatul FAQ grounded în context (`ctx`, pentru agent) și lasă pipeline-ul să continue la triaj/agent; planner-ul creează `obligations` din el.
4. **`UNKNOWN`** (incert) → tratat ca POSSIBLE_MIXED: pipeline complet, conservator — costul unui fals-pozitiv e un tur de agent în plus, nu un răspuns pierdut.
5. **Verificare deterministă:** renderer-ul verifică că fiecare `obligation` a fost satisfăcută (nu „modelul ar trebui"); obligație nesatisfăcută → completare deterministă din rezultatul FAQ atașat.

## Technical Description
- Derivare DETERMINISTĂ a `response_shape` din semnalele existente (attr_query/detail/rehidratare/cheaper) + rută.
- `obligations` construite de planner din rezultatul FAQ atașat + semnalele de multi-intent.
- `response_shape` slab → `recommendation`; `detail` pe `displayed_products` gol → fallback recommendation/clarify (NU „detaliu pe nimic"); REFINE (constrângere nouă) NU se tratează ca `repeat` — altfel se pierde constrângerea.
