# NX-133 — Stiva de constrângeri multi-tur: rafinarea NU resetează căutarea

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** IZI-parity R3 · **Branch:** `feat/NX-133-constraint-stack` · **Complexitate:** M · **Estimare:** 4h

## Goal
Constrângerile exprimate de client de-a lungul conversației (buget, ingredient/feature, tip de
produs, brand, „fără parfum") se ACUMULEAZĂ într-o stivă deterministă în `conversations.state` și
se aplică la fiecare re-căutare, în loc să se piardă la primul mesaj de rafinare. Runda 2 a
arătat bug-ul concret: „ser cu vitamina C sub 150 lei" + „am tenul mixt" → am pierdut complet
„vitamina C" și am întors un TONER; iZi păstrează stiva și re-rankează același set.

## Business Context
Diferența dintre „bot care caută" și „consultant care ține minte" — patternul P5 din analiza iZi
(runda 3). Fiecare constrângere pierdută înseamnă rezultate off-target, un client care repetă
ce-a spus deja și încredere pierdută exact în mijlocul funnel-ului de vânzare. Zero apeluri LLM
noi: nano extrage deja sloturile (NX-116), codul doar le persistă și le re-injectează.

## Technical Description

### 1. Gaura structurală (constatată în auditul de prompturi)
Lanțul de azi pierde constrângerile prin design:
- Triajul extrage sloturi „DOAR ce spune clientul EXPLICIT" în mesajul CURENT
  (`src/worker/stages/triage.py:123-125`) → pe un tur de rafinare, bugetul din turul 1 nu e în slots.
- `_filters_hint` (`src/worker/stages/agent.py:697-714`) seedează search-ul DOAR din
  `route.filters` ale turului curent.
- Promptul buclei de tool-uri nu spune nicăieri „păstrează constrângerile din conversație" —
  istoricul e în context, dar modelul nu e instruit să le care în argumentele search.

### 2. Stiva în state (`state.constraints`)
Dict compact în `ConversationState` (ref-uri, nu obiecte — P8):
```json
{"budget_max": 150, "concerns": ["vitamina c", "ten mixt"], "brand": null,
 "category_key": "seruri", "suitable_for": null}
```
Mărginit în cod: max 6 chei, `concerns` max 5 termeni (P4). Scrierea urmează fluxul existent de
state: stagiul agent actualizează obiectul `ctx.state` (owner unic al câmpului `constraints` —
P3), iar persistarea se face ca azi, tranzacțional în Sender prin `state_patch` (un singur punct
de scriere în DB, nemodificat).

### 3. Merge determinist per tur (funcție pură, testabilă)
`merge_constraints(stored, slots, category_key) -> dict`:
- slot NOU din turul curent → SUPRASCRIE cheia respectivă (bugetul nou bate bugetul vechi);
- slot absent în turul curent → rămâne valoarea din stivă;
- `concerns` se UNESC (dedupe, cap 5, cele mai recente primele);
- RESET total când triajul dă un `category_key` DIFERIT de cel stocat (subiect nou: „acum caut
  șampon") sau la route ≠ sales (order/handoff nu ating stiva). `category_key` null (follow-up
  neancorat) → stiva se PĂSTREAZĂ — exact cazul „am tenul mixt".

### 4. Aplicare: hint + prompt
- `_filters_hint` primește dict-ul DEJA merged (stored + slots curente) în loc de doar slots →
  hint-ul determinist către primul `search_products` conține toată stiva.
- O regulă nouă în `_TOOLS_BLOCK` (`src/agent/prompt_builder.py`): „La un mesaj care RAFINEAZĂ o
  căutare anterioară, PĂSTREAZĂ constrângerile deja spuse în conversație (buget, ingredient, tip
  de produs) în noul apel search_products — constrângerile detectate le primești deja în hint."
  Promptul e întărire, hint-ul e mecanismul (CALM: codul decide, modelul formulează).

### 5. Gate de onestitate SIMETRIC (fix-ul „gate inversat" din runda 2)
Cu stiva aplicată, căutarea de la turul 2 include din nou „vitamina C" → dacă catalogul n-o
satisface, semnalul de relaxare EXISTENT (`«relaxat»` în tool result, prompt_builder.py:82-83)
se aprinde pe turul CORECT. Nu construim gate nou — repararea inputului repară gate-ul.

## Principii CLAUDE.md aplicabile
- **P3 (single-owner-per-field):** `state.constraints` are UN scriitor — stagiul agent (după
  triaj, înainte de search). Triajul rămâne owner pe `route.filters`; Sender rămâne singurul
  care persistă state-ul.
- **P4 (buget în cod):** capul de chei/termeni impus în `merge_constraints`, nu în prompt;
  state-ul rămâne sub 8KB.
- **P8 (state = ref-uri):** doar valori scalare + liste scurte de termeni, zero obiecte produs.
- **P2 (LLM doar la triaj+agent):** merge-ul e funcție pură; nano nu se re-apelează.
- **P10 (observabilitate):** emit `constraints_merged` cu cheile (nu valorile PII-free oricum).

## Test Cases
**Happy Path:**
- stored={budget:150, concerns:[vit C]}, slots={suitable_for: ten mixt}, category null →
  merged conține toate 3; hint-ul le enumeră.
- slots={budget:80} peste stored={budget:150} → bugetul devine 80 (recent bate vechi).

**Edge Cases:**
- category_key nou diferit de stocat → reset total, doar sloturile curente rămân.
- concerns duplicate („vitamina c" în ambele) → dedupe, o singură apariție.
- stored gol + slots goale (follow-up pur conversațional) → merged gol, hint absent, zero crash.

**Failure Cases:**
- `state.constraints` corupt (string în loc de dict, state vechi) → tratat ca `{}`, pipeline
  continuă (back-compat).
- Triaj fără slots (nano vechi / JSON fără câmp) → stiva se păstrează neatinsă, nu se șterge.
