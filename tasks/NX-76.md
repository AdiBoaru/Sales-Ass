# NX-76 — Media routing: Vision imagine→match catalog în Gates
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G5a (gates) + job embed_products (search semantic) · **Branch:** `feat/NX-76-vision-image-match` · **Complexitate:** L · **Estimare:** 7h **[PROPUNERE — split, vezi mai jos]**

## Goal
Când un client trimite o POZĂ (content_type=image), botul nu mai tace prostește. În Gates
(stagiul 3, ramura de media routing din docstring-ul modulului) descărcăm media de la canal,
o trecem prin Vision (prin adaptorul unic `src.agent.llm`, ca excepția deja stabilită pentru
embed/moderate — clasificator/extractor de siguranță, NU generare), obținem o DESCRIERE +
atribute textuale, le injectăm în `ctx.message.body` ca text de căutare ("[poză client] ser
hidratant Cerave, ambalaj alb/albastru") + un marker în state, iar pipeline-ul continuă normal:
triajul rutează SALES, agentul cheamă `search_products` pe textul derivat din poză. Imagine →
text → search. NU avem image-embedding în catalog (vezi mai jos), deci ăsta e singurul match
posibil acum, fără schema nouă.

## Business Context
„Vreau ăsta, mai aveți?" + poza unui produs e fluxul #1 de vânzare ratat azi: poza ajunge
`body=None`, triajul n-are text, agentul nu pornește, iese fallback-ul de clarificare —
clientul percepe botul ca rupt. Match-ul poză→produs e exact momentul de cumpărare (intenție
maximă). Costă un apel Vision per poză (ieftin, bornat în cod), dar deblochează conversii pe
care altfel le pierdem. Verticalul beauty (clientul demo) trăiește pe poze de ambalaj/etichetă.

## Technical Description

### Starea actuală (verificat în cod)
- `webhook/meta.py` parsează deja `image/audio/...` și pune `media_id` în `InboundEvent.media_id`
  (`_extract_body` → caption + media id). Telegram poller IGNORĂ media (`_to_event`: fără `text` →
  `None`) — în afara scopului acestui task (TEST channel; WhatsApp e canalul de poze).
- `processor.handle_turn` mapează `event.media_id → ctx.message.media_ref` și `content_type` în
  `InboundMessage`. Mesajul inbound se scrie cu `media_ref` setat.
- `gates.py` NU are NICIUN cod de media: o poză (`body=None`) trece prin toate porțile (nimic nu
  prinde), `language_stage`/`cache`/`triage` n-au text, agentul nu pornește (route ≠ SALES/ORDER),
  iese `fallback_stage`. Marker-ul din audit (gap): „body=None → pipeline mai târziu decide
  STT/Vision" — niciun stagiu nu decide nimic.
- `agent.py:_card_products` are un câmp `image` — e poza DE IEȘIRE a unui produs (cardul W1/R2),
  NU media inbound. Confirmat: nicio legătură cu poza primită.
- `LLMClient` (adaptorul unic) are deja `embed` + `moderate` ca excepții non-generare. Adăugăm
  `describe_image` în ACELAȘI loc (singurul care vorbește OpenAI).
- `product_embeddings.embedding` = `vector(1536)` din `text-embedding-3-small` (TEXT, pe
  `ai_summary`/content_hash). NU există image-embedding în schemă. Deci match-ul vizual real
  (CLIP/image-vector) e imposibil azi fără schema nouă → match-ul e imagine→TEXT→`search_products`.

### Plasare: o nouă poartă în Gates, ÎNAINTE de language (text-keyed)
`gates_stage` capătă o ramură de media routing, după rate-limit/moderation (nu cheltui Vision pe
un contact throttled/blocat), înainte de a returna. Vision rulează DOAR pe `content_type=image`:
```python
# 7. media routing (NX-76): poză → Vision → descriere ca text de căutare, apoi pipeline normal.
if ctx.message.content_type == "image":
    await _route_image(ctx, deps)   # setează ctx.message.body + state marker; NU early-exit
```
Voce (`audio`) = task separat (STT), document/sticker = out of scope. `_route_image` NU setează
`reply` și NU oprește pipeline-ul (nu e early-exit ca moderation): doar ÎMBOGĂȚEȘTE `ctx.message.body`
ca triajul + agentul să curgă normal pe textul derivat.

### Descărcarea media (margine de canal — NU în pipeline)
WhatsApp media e în 2 pași: GET `/{media_id}` → `url` (semnat, scurt) → GET pe `url` cu Bearer →
bytes. Asta e cod de TRANSPORT (principiul: cuplajul de canal stă la margini), deci trăiește pe
`MetaClient`, nu în gates:
```python
# src/meta_client.py — MetaClient
async def download_media(self, media_id: str) -> tuple[bytes, str]:
    """media_id → (bytes, mime). 2 hop-uri Graph API. Ridică la HTTP error."""
    meta = (await self._http.get(f"{self._base}/{media_id}",
            headers={"Authorization": f"Bearer {self._token}"})).raise_for_status().json()
    url, mime = meta["url"], meta.get("mime_type", "image/jpeg")
    blob = await self._http.get(url, headers={"Authorization": f"Bearer {self._token}"})
    blob.raise_for_status()
    return blob.content, mime
```
Gates primește un downloader prin `deps`, NU instanțiază clientul (gates rămâne agnostic): adăugăm
`media_downloader: MediaDownloader | None` în `PipelineDeps` (Protocol cu o metodă
`download(channel_kind, account_id, media_id) -> tuple[bytes,str] | None`). Implementarea WhatsApp
delegă la `MetaClient.download_media`; Telegram → `None` (nu rutăm media pe TEST). `None` (fără
downloader / canal nesuportat) → fail-soft (vezi mai jos).

### Vision prin adaptorul unic (principiul 2, excepție ca embed/moderate)
```python
# src/agent/llm.py — LLMClient
async def describe_image(self, image_b64: str, mime: str, *, model: str | None = None) -> str:
    """Descrie o poză de produs ca TEXT de căutare în catalog (NU conversație: extracție,
    ca embed/moderate). Întoarce o frază scurtă cu tip de produs + brand vizibil + atribute
    (culoare/ambalaj/text de pe etichetă). Ridică la eroare — caller-ul (gate) degradează."""
    resp = await self._client.chat.completions.create(
        model=model or self.model_vision,
        messages=[
            {"role": "system", "content": _VISION_SYSTEM},
            {"role": "user", "content": [
                {"type": "text", "text": "Descrie produsul din poză ca interogare de căutare."},
                {"type": "image_url",
                 "image_url": {"url": f"data:{mime};base64,{image_b64}", "detail": "low"}},
            ]},
        ],
        max_tokens=120,
    )
    return (resp.choices[0].message.content or "").strip()
```
`model_vision` în settings (default `gpt-5.4-mini` — modelul agentului are vedere; nano nu).
`detail: "low"` + `max_tokens=120` = costul tăiat în cod (un tile, nu high-res). `_VISION_SYSTEM`
cere STRICT atribute observabile (tip produs, brand vizibil, culoare, text de pe ambalaj), interzice
inventarea preț/disponibilitate (groundarea rămâne treaba search-ului + validatorului din agent).

### Orchestrarea în gate (`_route_image`)
```python
async def _route_image(ctx: TurnContext, deps: PipelineDeps) -> None:
    s = get_settings()
    if not s.vision_enabled or deps.llm is None or deps.media_downloader is None:
        ctx.message.body = _IMG_FALLBACK_BODY; return         # fail-soft: triaj→clarify cu text
    media_id = ctx.message.media_ref
    if not media_id:
        ctx.message.body = _IMG_FALLBACK_BODY; return
    try:
        blob = await deps.media_downloader.download(
            ctx_channel_kind, ctx_account_id, media_id)        # canal din conv/event
        if blob is None or len(blob[0]) > s.vision_max_bytes:  # cap de dimensiune în cod
            raise ValueError("media indisponibilă / prea mare")
        desc = await deps.llm.describe_image(b64encode(blob[0]).decode(), blob[1])
    except Exception as e:   # noqa: BLE001 — fail-soft, NU tăcere (P6)
        log.warning("vision: %s → fallback clarificare imagine", type(e).__name__)
        ctx.message.body = _IMG_FALLBACK_BODY; return
    caption = (ctx.message.body or "").strip()                # caption-ul WA, dacă există
    derived = f"[poză client] {desc}" + (f" — text client: {caption}" if caption else "")
    ctx.message.body = derived
    ctx.state.active_search = {**ctx.state.active_search, "from_image": True}  # marker pt agent
    ctx.emit("image_routed", chars=len(desc))                 # FĂRĂ bytes/base64 în analytics (P12)
```
Cost guard (G2c): apelul Vision intră în contorul zilnic ca un apel de agent. `_route_image`
adaugă `cost_vision_usd` în Redis prin `cost_add` (best-effort, ca summarizer-ul), ca să nu
scape plafonului. Peste buget (`deps.llm` deja `None` din cost guard) → fail-soft automat.

### Câmpuri TurnContext scrise aici
- `ctx.message.body` — ÎMBOGĂȚIT cu descrierea Vision (gate-ul e proprietarul când rutează media;
  niciun alt stagiu nu scrie `message.body`). Triajul/agentul îl CITESC.
- `ctx.state.active_search["from_image"]` — marker (ref, nu obiect, P8); agentul îl poate folosi
  ca să formuleze „din poza ta caut...". State rămâne ≤8KB.
- `ctx.events` — `image_routed` (+ `image_route_failed`). Owner Gates, ca acum.
NU scrie `reply` (nu e early-exit), NU scrie `route`/`retrieval`.

## Out of Scope
- **STT / voce** (`content_type=audio` → Whisper) — task frate separat, același pattern de adaptor.
- **Image-embedding real (CLIP / vector vizual)** — cere coloană nouă + reindex + job de embed;
  doar pentru un vertical care chiar cere match vizual, nu text-derivat.
- **Media pe Telegram** — poller-ul ignoră media azi (canal de TEST); rutarea pe TG = follow-up.
- **Document / sticker / video** — doar `image` acum; restul rămân `body=None` → fallback.
- **OCR dedicat / cod de bare** — Vision dă text de pe etichetă best-effort, fără pipeline OCR separat.
- **Persistarea pozei în storage** — descărcăm în memorie, nu salvăm binary (doar `media_ref` id-ul).

## Test Cases
**Happy Path:**
1. Poză + caption „mai aveți?" → `describe_image` mock = „ser hidratant Cerave ambalaj alb/albastru"
   → `ctx.message.body` = „[poză client] ser hidratant ... — text client: mai aveți?", `image_routed`

**Edge Cases:**
1. `content_type="text"` → `_route_image` nu e apelat (fără download/Vision), comportament neschimbat.

3. Bytes exact peste `vision_max_bytes` → fail-soft (nu trimitem la Vision o poză uriașă).

**Failure Cases:**
1. `download` aruncă (Graph API 401/timeout) → fail-soft: body=clarificare, fără excepție propagată,
   `image_route_failed` {reason:"vision_error"|"no_media"}.
2. `describe_image` aruncă (OpenAI down) → fail-soft identic; pipeline-ul produce clarificare, NU tăcere.
3. Cost guard a setat `deps.llm=None` (peste plafon zilnic) → ramura de imagine nu cheltuie Vision,
   fail-soft direct.

## Cost (per 1000 mesaje)
Vision rulează DOAR pe poze (≈o fracțiune din trafic; estimăm ≤10% = 100 poze/1000 mesaje). Per poză:
1 apel `gpt-5.4-mini` cu `detail:"low"` (un singur tile, ~85 image-tokens) + `max_tokens=120` output
≈ `cost_vision_usd` (default $0.003, ca un apel de agent). **100 poze/1000 mesaje ≈ $0.30/1000.**
Ce LIMITEAZĂ costul, structural în cod: (1) `content_type=image` — text-ul nu atinge Vision;
(2) `detail:"low"` + `max_tokens=120` — fără high-res, fără răspuns lung; (3) `vision_max_bytes` —
poze uriașe respinse înainte de apel; (4) cost guard G2c — peste plafonul zilnic `deps.llm=None`
→ Vision sărit automat; (5) rate-limit/moderation rulează ÎNAINTE → un contact throttled/blocat nu
declanșează Vision.
