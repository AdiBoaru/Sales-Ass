# NX-24 — Web Widget W5 — context pagină
**Owner:** S · **Faza:** P2 (E26 · V1.5) · **Zi/Ord:** după NX-20 + NX-22 (W3) · **Branch:** `feat/NX-24-page-context` · **Complexitate:** S · **Estimare:** 3h

## Goal
Widget-ul web știe pe ce pagină stă vizitatorul (`page_url`, opțional `product_id`).
Taskul duce acel semnal prin envelope-ul web (NX-20) în pipeline și îl transformă într-un
**hint de retrieval PRIORITAR** în context builder (stagiul 6): produsul paginii e pre-încărcat
determinist din catalog și injectat în contextul agentului, astfel încât un mesaj deictic
(„cât costă **asta**?", „e bun pentru ten gras?") să se rezolve pe ACEL produs, nu pe o căutare
oarbă. E un câmp nou în `TurnContext` (`page_context`) cu un singur proprietar (context builder
pentru blocul derivat; processor pentru câmpul brut din envelope).

## Business Context
Pe web, jumătate din întrebări sunt deictice — clientul se uită la o pagină de produs și întreabă
despre „asta", fără să numească produsul. Pe WhatsApp nu există acest semnal; pe web e gratis și
îl irosim dacă nu-l folosim. Cu el, botul răspunde corect din prima (conversie mai mare, zero
„despre care produs?"); fără el, ratează intenția sau forțează o clarificare inutilă pe traficul
cel mai cald (vizitator pe pagina de produs = aproape de cumpărare).

## Technical Description

### De unde vine semnalul — envelope-ul web (NX-20)
Gateway-ul web (NX-20) primește pe `POST /web/messages` un payload care, pe lângă textul
mesajului, poartă contextul paginii din widget (NX-21):
```jsonc
{ "session_token": "...", "text": "cât costă asta?",
  "page": { "url": "https://shop.ro/p/ser-vitamina-c", "product_id": "<uuid catalog>" } }
```
NX-20 pune asta în envelope-ul NEUTRU pe stream-ul `inbound`, sub o cheie de canal dedicată
(NU în câmpurile core, ca să rămână agnostic):
```jsonc
{ "channel_kind": "web", "channel_account_id": "...", "sender_external_id": "<visitor_id>",
  "provider_msg_id": "...", "content_type": "text", "body": "cât costă asta?",
  "page_context": { "url": "https://shop.ro/p/...", "product_id": "<uuid>" } }
```
`product_id` e **trimis de widget** (citit din `data-product-id` pe `<script>` sau din dataLayer),
deci e **NETRUSTED**: îl tratăm ca un hint, NU ca o sursă de adevăr. Singura validare reală e
lookup-ul în catalog scoped pe `business_id` (vezi mai jos) — dacă id-ul nu aparține tenantului,
hint-ul pică în gol (zero rezultate), nu poate scurge produsul altui client.

### Cum curge prin pipeline
1. **Processor** (`handle_turn`) citește `event.get("page_context")` și îl pune pe
   `InboundMessage.payload["page_context"]` (câmpul `payload` există deja) + setează câmpul nou
   `ctx.page_context: PageContext | None`. Doar copiere de date, fără I/O. Owner: processor.
2. **Context builder** (stagiul 6, `src/worker/context.py`) e singurul care DERIVĂ blocul:
   dacă `ctx.page_context.product_id` e prezent, pre-încarcă produsul determinist din catalog
   (`get_products_by_ids`, scoped pe `ctx.business.id`) și formează un bloc PRIORITAR pus
   PRIMUL în `context_blocks(ctx)` (înaintea profilului/state-ului):
   ```
   Clientul se uită ACUM la pagina acestui produs: [<id>] Ser Vitamina C (Brand) — 89.00 lei.
   Dacă întrebarea se referă la „acesta/asta", răspunde despre ACEST produs.
   ```
   Blocul intră în mesajul USER (dinamic) — promptul static rămâne byte-identic (prompt caching
   neatins, ca tot ce e în `context_blocks`).

### De ce pre-fetch determinist și NU „lasă agentul să cheme get_product_details"
- **Determinism + cost**: produsul paginii e cunoscut fără LLM. Îl injectăm direct → agentul nu
  arde un tool call (din cele 3) ca să afle ce e deja sub ochii clientului. Pe rutele `simple`
  (triajul nano compune răspunsul) și pe cache, agentul nici nu rulează — dar blocul de context
  e tot acolo dacă vreun stagiu îl folosește. Lookup-ul produsului = o interogare SQL ieftină pe
  PK, fără embedding.
- **Grounding**: produsul pre-încărcat intră în `ctx.retrieval` (vezi mai jos) → validatorul
  (stagiul 8) acceptă prețul/linkul lui ca grounded. Fără asta, dacă agentul vorbește despre
  prețul paginii fără să-l fi retrievat, validatorul l-ar respinge (corect) ca preț neîntemeiat.

### Injectarea în retrieval (grounding pentru validator)
Pentru ca „cât costă asta? → 89 lei" să treacă validatorul, produsul paginii trebuie să fie în
`ctx.retrieval`. `ctx.retrieval` e scris DOAR de stagiul agent (regula 1 proprietar/câmp). Soluție
fără a încălca proprietatea: context builder-ul **nu** scrie `ctx.retrieval`; expune produsul
pre-încărcat pe `ctx.page_context.product` (câmpul brut, owner context builder), iar `agent_stage`
îl **prepend-uiește în `retrieved`** la început (seed determinist al buclei), exact ca pe un tool
result implicit. Astfel un singur stagiu (agent) scrie `ctx.retrieval`, dar îl alimentează cu
produsul paginii. Triajul vede blocul de context la fel ca agentul, deci poate ruta `simple` →
`sales` corect când întrebarea e despre „asta".

### Modelul nou
```python
@dataclass
class PageContext:
    """Context de pagină din widget-ul web (NX-24). Owner: processor (din envelope).
    `product` e pre-încărcat de context builder (owner pe acel sub-câmp), NETRUSTED până la
    lookup scoped pe business_id."""
    url: str | None = None
    product_id: str | None = None
    product: dict[str, Any] | None = None   # pre-încărcat de context builder (compact)
```
`TurnContext.page_context: PageContext | None = None  # owner: processor (web envelope)`.

### Securitate / multi-tenant
- `product_id` e NETRUSTED (vine de la browser). NICIODATĂ folosit fără
  `WHERE business_id = $1` — refolosim `get_products_by_ids(conn, ctx.business.id, [pid], limit=1)`,
  care e deja scoped + RLS ca plasă. Un `product_id` străin → 0 rezultate → hint ignorat.
- `page_url` NU e PII, dar îl tratăm ca date externe: nu-l logăm întreg (poate conține query
  params cu token-uri), doar host+path scurt; nu-l injectăm verbatim în prompt (folosim numele de
  produs din catalog, nu URL-ul brut). În analytics punem doar `has_page_product` (bool), nu URL-ul.
- Canale non-web: `page_context` lipsește din envelope → `ctx.page_context is None` → tot fluxul e
  no-op. Zero impact pe WhatsApp/Telegram.

### Câmpuri TurnContext scrise aici
- `ctx.page_context` (câmpul brut url/product_id) — **owner: processor** (copiere din envelope).
- `ctx.page_context.product` (sub-câmpul pre-încărcat) — **owner: context builder**.
- NU scriem `ctx.retrieval` aici (rămâne al agentului); doar îl alimentăm cu produsul paginii.

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** semnalul curge înainte (envelope → processor → context builder →
  agent), fără săritură înapoi.
- **P2 (LLM doar triaj+agent):** lookup-ul produsului paginii e **cod determinist** (SQL pe PK),
  fără LLM/embedding. Nu adăugăm niciun apel de model.
- **P3 (un proprietar per câmp):** `page_context` brut e al processor-ului; sub-câmpul `product`
  e al context builder-ului; `retrieval` rămâne al agentului — nu-l scriem din alt stagiu.
- **P7 (business_id pe tot):** lookup-ul produsului netrusted = `business_id = $1` obligatoriu;
  RLS ca plasă.
- **P9 (cuplajul de canal doar la margini):** `page_context` intră prin parserul web (margine),
  pipeline-ul rămâne agnostic — citește un câmp opțional, nu „știe" de web.
- **P12 (PII / loguri):** `page_url` netrusted → log redactat (host+path), zero URL brut în
  analytics; folosim numele de catalog, nu URL-ul, în prompt.

## Implementation Steps
1. `src/models.py`: adaugă dataclass `PageContext` + câmpul `TurnContext.page_context` (default
   `None`), cu docstring de proprietar.
2. `src/worker/processor.py`: în `handle_turn`, citește `event.get("page_context")` → setează
   `ctx.page_context = PageContext(url=..., product_id=...)` (doar dacă există); pune-l și pe
   `InboundMessage.payload["page_context"]` pentru trasabilitate. Fără I/O nou.
3. `src/worker/context.py`: funcție nouă `page_context_block(ctx)` + integrare în
   `context_blocks(ctx)` ca PRIM bloc. Aici se face pre-fetch-ul determinist
   (`get_products_by_ids` pe `ctx.business.id`), cu cap de caractere ca celelalte blocuri;
   setează `ctx.page_context.product` (compact: id, name, price, url) la succes.
4. `src/worker/stages/agent.py`: dacă `ctx.page_context and ctx.page_context.product`,
   prepend-uiește produsul în `retrieved` ÎNAINTE de bucla de tool-calling (seed determinist) →
   intră în `_dedupe` + `ctx.retrieval` → grounding pentru validator. Emite `page_context_used`.
5. `src/db/queries/catalog.py`: NICIO modificare — refolosim `get_products_by_ids`.
6. Teste (`tests/test_page_context.py` + extensie `tests/test_context.py`): mapare envelope →
   `ctx.page_context`; pre-fetch (stub catalog) → bloc prioritar; product_id străin → 0 rezultate,
   bloc gol; canal non-web → no-op; agent: produsul paginii e în `ctx.retrieval`, validatorul
   acceptă prețul lui. ZERO apeluri reale (LLM scriptat, catalog stub-uit).
7. `ruff check . && ruff format . && pytest -x -q` verde.

## Test Cases
**Happy Path:**
1. Envelope `{channel_kind: web, page_context:{product_id: P1}}`, catalog stub întoarce P1
   (89.00 lei) → `context_blocks` începe cu „Clientul se uită ACUM la… Ser Vitamina C — 89.00 lei";
   `page_context_used {has_product:true, resolved:true}` emis.
2. „cât costă asta?" pe pagina P1, LLM scriptat răspunde „Costă 89 lei." → produsul P1 e în
   `ctx.retrieval`, `_valid` trece (preț grounded), reply servit fără tool call de căutare.

**Edge Cases:**
1. `page_context` cu `url` dar FĂRĂ `product_id` → fără pre-fetch; blocul e gol sau doar text
   scurt informativ; pipeline-ul rulează normal (căutare ca de obicei).
2. Canal `whatsapp` (fără `page_context`) → `ctx.page_context is None`; `context_blocks`
   identic cu baseline (regresie verde).
3. `product_id` prezent dar produsul `status != active` / inexistent în tenant → lookup 0 →
   `resolved:false`, bloc gol, niciun preț injectat.

**Failure Cases:**
1. `product_id` al ALTUI business → `get_products_by_ids` cu `business_id = $1` întoarce [] →
   zero leak, hint ignorat (test explicit de izolare cu stub care verifică argumentul business_id).
2. Catalog lookup aruncă (DB jos) → context builder prinde excepția, blocul de pagină e sărit
   (best-effort), restul contextului (profil/state) iese normal — niciodată tăcere (P6).
3. `page_context` malformat (`product_id` non-string / dict gol) → ignorat defensiv în processor
   (ca `ConversationState.from_jsonb`), `ctx.page_context` rămâne None, fără excepție propagată.

> Teste fără apeluri reale: catalog STUB-uit (monkeypatch pe `get_products_by_ids`), LLM SCRIPTAT
> (ScriptedLLM ca în `test_golden.py`). ZERO OpenAI / DB real în CI.

## Cost
**$0 LLM adăugat.** Pre-fetch-ul produsului paginii e SQL determinist pe PK (un round-trip ieftin,
fără embedding, fără apel de model). Per 1000 mesaje web cu context de pagină: +1000 interogări
PK pe `products` (sub-milisecundă fiecare, index existent), zero tokeni. Costul e mărginit de:
(a) lookup-ul rulează DOAR când `product_id` e prezent în envelope (web, pagină de produs);
(b) un singur produs per tur (PK lookup, `limit=1`); (c) blocul e bugetat la `max_chars` ca toate
blocurile de context, deci nu umflă promptul. Indirect ECONOMISEȘTE: agentul nu mai cheamă
`get_product_details` (un tool call ~ un apel mini) ca să afle produsul deja sub ochii clientului,
și evită clarificări inutile (un tur întreg) pe traficul cel mai cald.
