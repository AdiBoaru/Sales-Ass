# G7-3 — Faza 3 tool: check_order (status comandă) + ruta ORDER activă
**Owner:** S · **Faza:** P1 · **Branch:** `feat/G7-3-check-order` · **Complexitate:** M-L · **Estimare:** ~6h

## Goal
Agentul răspunde la „unde e comanda mea?" / „ce status are ORD-123?" cu status + tracking real
din `orders` + `shipments`. Necesită DOUĂ piese (designul verificat adversarial a arătat că tool-ul
singur ar fi cod mort): **(A)** ruta `ORDER` să ruleze agentul cu un toolset propriu; **(B)** tool-ul
`check_order` read-only, izolat pe contact, cu validator grounded pe totaluri.

## Business Context
Post-cumpărare e prima sursă de tichete de suport. Botul închide bucla: comanda atribuită (F2) →
clientul întreabă de status → răspuns instant grounded. Reduce volumul către om.

## Technical Description (corectat după review adversarial — 3 blocking reparate)

### (A) Ruta ORDER rulează agentul (altfel tool = cod mort)
- `agent_stage`: gate extins `route.route in {SALES, ORDER}` (azi doar SALES → pe ORDER agentul
  nu rula → check_order nu era chemat niciodată).
- `enabled_tools(business, route)` devine **route-aware**: SALES → catalog read + `checkout_link`;
  ORDER → `check_order`. Modelul primește DOAR toolset-ul rutei (nu oferi search pe ORDER, nici
  check_order pe SALES). `_TOOL_SYSTEM` descrie toate; oferta restrânge.

### (B) Tool `check_order` (`src/tools/orders_tools.py`)
- `CheckOrderArgs{order_ref: str|None}` (business_id NU e arg — din ctx). 
- Lookup: `order_ref` → după `external_id` **ȘI** `contact_id=ctx.contact.id` (izolare în SQL, nu doar
  în cod); fără `order_ref` → ultimele ≤3 comenzi ale contactului. O comandă a ALTUI contact →
  `not_found` (nu divulgă existența → anti-enumerare).
- `ToolResult(ok, products=[], prices=<totaluri grounded>, llm_view=<status/AWB/ETA/total, fără PII>)`.

### (C) Query `get_orders_status` (`commerce.py`)
- UN SELECT: `orders` + lateral `jsonb_agg(order_items)` + lateral „ultimul shipment" (updated_at desc).
  `where business_id=$1 and (external_id match) and (contact_id match)`. `order_items` n-are
  `business_id` → izolare tranzitivă prin orders (comentat). Fără N+1.

### (D) Validator grounded pe sume (oglindește allowed_links)
- `ToolResult.prices: list[float]` (sume citite din DB, grounded prin construcție).
- `agent_stage.execute`: acumulează `grounded_prices` (ca `generated_links`).
- `_prices_ok(reply, products, allowed_prices)` = preț ∈ produse ∪ grounded. Threading prin
  `_valid`/`_finalize` + lista de retry include sumele grounded. **Totalul comenzii e acceptat
  fiindcă vine din DB**, NU prin slăbirea validatorului.
- **Calea fără produse e validată** (bug-ul prins de review): azi `elif final` servea textul
  NEVALIDAT. Acum: `final` valid (preț/link grounded) → servește; invalid → retry order-shaped
  (`_ORDER_RECO_SYSTEM`) → fallback determinist sigur (fără numere, non-tăcere). Fallback-ul de
  produse (`_deterministic_reply`, „Îți recomand…") NU se folosește pe ORDER.

### (E) Gaură pre-existentă închisă
- `checkout_link` emitea totalul în `llm_view` dar nu-l returna grounded → adăugăm `prices=[total]`.

## Principii
P2 (tool determinist) · P7 (business_id + contact_id în SQL) · P8 (products=[], doar llm_view compact)
· P12 (contact din ctx/channel_identities; `orders` n-are telefon; llm_view fără PII) · P6 (not_found
+ fallback order-shaped, niciodată tăcere) · Validator (zero sume inventate, prin grounding nu relaxare).

## Definition of Done

## Test Cases
**Happy:** check_order al contactului → ok + prices; agent pe ORDER compune status validat.
**Edge:** fără shipments → fără AWB/ETA, fără crash; fără items → llm_view ok; fără order_ref → ultimele 3.
**Failure:** order_ref al altui contact → not_found; tool aruncă → ToolResult(ok=False) (turul continuă);
total inventat pe ORDER → retry → fallback sigur.

> ZERO OpenAI/DB real în unit: tool cu `get_orders_status` monkeypatch; agent cu FakeLLM scriptând
> check_order; SQL real DOAR în testul `@integration`.
