# NX-161 — DB conn-per-op + admission control (conexiunea DB aparține operației, nu turului)

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** Runtime/Scalare · **Branch:** `feat/NX-161-conn-per-op-admission` · **Complexitate:** L · **Estimare:** ~30h → **livrat feliat (0A→7, vezi Implementation Steps)**

## Goal
`handle_turn` ține o singură conexiune `bot_pool` (max=10) pe **tot** turul, inclusiv apelurile LLM.
Măsurat (`scripts/sim/pool_probe.py`): un tur de sales ține conexiunea ~18.9s din care doar ~4.0s e
muncă DB → **~79% idle-held** (așteaptă OpenAI). Sistemul eșuează prin **concurență, nu debit**: cu
`W=18.9s`, ~10 conversații concurente saturează poolul (Little). Taskul schimbă **proprietatea
resursei**: conexiunea DB devine efemeră (checkout scurt per operație), iar timpul de LLM/API
trăiește în AFARA poolului. Corolar dur, verificat în cod: poolul de 10 e azi **admission control
accidental** (debouncer-ul spawnează un task/expeditor NEMĂRGINIT — `debounce.py:65`), deci eliberarea
lui FĂRĂ o frână de înlocuire mută overload-ul în OpenAI/cost/memorie. De aceea epic-ul livrează ȘI o
**frână explicită minimă** (semafor de admission). Design canonic APROBAT (Codex):
[`docs/CONN-HOLD-ANALYSIS-2026.md`](../docs/CONN-HOLD-ANALYSIS-2026.md).

## Business Context
Agenția onboardează magazine cu trafic bursty pe WhatsApp (promo blast, peak de seară). Poolul
partajat pe tenanți se umple din DURATĂ, nu volum: la primul burst cu câțiva clienți, tururile
intră în coadă pe `pool.acquire()` → latență care crește → timeout → botul pică EXACT în vârful de
intenție (clientul tocmai a fost provocat să scrie). Fix-ul mută plafonul de la ~0.5 tururi/s la un
ordin de mărime mai sus cu ACELAȘI pool, și pune o frână configurabilă în locul uneia accidentale →
scalăm workerii și poolul independent, fără să omorâm OpenAI-ul.

## Technical Description

### Arhitectura pe 3 straturi (separă preocupări azi încurcate în poolul de 10)
1. **DB ownership fix** [acest epic] — conn scurtă per operație; LLM/API fără conn.
2. **Admission control minim** [acest epic, OBLIGATORIU] — semafor global de tururi concurente
   (+ opțional per-business). Înlocuiește frâna accidentală a poolului.
3. **Fair scheduling per-tenant complet** [EPIC SEPARAT] — cozi per tenant, fairness, circuit
   breakers, rate shaping. NU în acest epic.

### Modelul de acces (Opțiunea C → converge la A)
```python
# Curent                                   # Țintă
PipelineDeps(conn=conn, ...)               PipelineDeps(db=db_provider, ...)
                                           async with deps.db() as conn:   # checkout scurt
                                               ...                          # între operații: ZERO conn
```
`db_provider` = wrapper subțire peste `tenant_conn(business_id)` → fiecare checkout setează
`app.business_id`, rulează assert-ul de izolare (NX-04), resetează GUC la release. RLS rămâne plasă.

### Punte de compat (obligatorie — `PipelineDeps(conn=...)` × 114 în 33 fișiere teste)
`deps.db()` face fallback la `yield` la un conn static injectat (nu deschide checkout nou) când
`PipelineDeps` primește `conn=`. Astfel un stagiu migrat trece în testele legacy fără rescriere
big-bang. **Ieșire mecanică:** guard CI (grep) care PICĂ dacă `PipelineDeps(conn=` apare în `src/`
non-test după ce infra aterizează; testele NOI folosesc provider fake explicit.

### Frâna de admission (Faza 0C — plasă ÎNAINTE de a elibera poolul)
Semafor global `asyncio.Semaphore` (configurabil) achiziționat în `process_event`/`handle_turn`
ÎNAINTE de secțiunea LLM-heavy și **fără conn DB ținut**. Opțional semafor per-business când
`business_id` e cunoscut. Introdus la 0C (nu la Faza 6) fiindcă deja aftercare/cache/triage off-conn
încep să elibereze poolul → frâna trebuie să existe ca plasă mai devreme, chiar dacă e setată
conservator. **NU se ia în interiorul `tenant_conn`** (altfel repeți exact problema).

### Ce NU se schimbă (invariante păstrate)
Commit-ul Sender rămâne o singură TX atomică (outbox + patch state + `mark_inbound_completed`,
`processor.py:728-835`). `state_version` rămâne garda optimistic-lock (LLM rulează între load și
commit → state-ul se poate schimba). Idempotența (`claim_inbound` înainte de write-uri;
`mark_inbound_completed` doar la succes terminal). Pipeline-ul liniar (P1) — unit-of-work nu adaugă
loop de orchestrare.

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** unit-of-work boundaries (load/execute/commit/aftercare) rămân funcții
  coezive — FĂRĂ explozie de clase, FĂRĂ săritură înapoi.
- **P5 (un singur punct de ieșire):** commit-ul Sender rămâne atomic pe UN checkout dedicat.
- **P6 (niciodată tăcere):** admission deferral → re-queue cu backoff (ca NX-85), NU drop tăcut;
  timeout pe semafor → fallback grațios, nu tăcere.
- **P7 (business_id pe tot):** fiecare checkout de provider trece prin `tenant_conn` → `app.business_id`
  setat; zero `pool.acquire()` raw în hot path.
- **P10 (observabilitate din runner):** metricile (acquire_wait, admission_wait, inflight) emise din
  processor/runner; stagiile nu știu că sunt măsurate.
- **Câmpuri TurnContext scrise:** **NICIUNUL nou.** Taskul schimbă `PipelineDeps` (conn→db provider),
  NU contractul de scriere pe `ctx`. Owner-ii de câmpuri rămân neatinși (P3 intact).
- **LLM (P2):** NICIUN punct LLM nou — doar mutăm conexiunea de sub apelurile existente (triaj/agent/
  moderation/Vision/embed). Lista completă de metode peste care nu se ține conn în Technical/doc.

## Implementation Steps (feliat)

**Felia 0A — Instrumentare (~3h, PRIMA, independentă)**
1. Emite metrici prod (nu doar DB): `db_pool_acquire_wait_ms`, `db_conn_held_ms`, pool
   in-use/idle/waiters (din `asyncpg.Pool` stats), `turn_inflight`, LLM wait/stagiu. p50/p95/p99.
2. Rulează sub sarcină (sim/burst) → confirmă pragul real de trigger (p95 acquire wait > 100-250ms).

**Felia 0B — Infra `deps.db()` + compat bridge + guard CI (~4h, fără migrare stagii)**
3. `PipelineDeps` primește `db` (provider) alături de `conn` (compat). `deps.db()` = context manager:
   dacă `conn` static injectat → `yield` la el; altfel `async with tenant_conn(ctx.business.id)`.
4. Guard CI (`scripts/check_no_raw_conn.py` sau grep în workflow) — deocamdată WARN, hard-fail la Faza 7.
5. Teste mici pe provider (fake explicit + real tenant_conn).

**Felia 0C — Semafor de admission (~3-4h, plasă, OBLIGATORIE înainte de a elibera poolul)**
6. Semafor global configurabil (`ADMISSION_MAX_INFLIGHT`) achiziționat în `process_event` înainte de
   `handle_turn`, fără conn ținut; opțional per-business. Ocupat peste timeout → re-queue (P6).
7. Metrici: `turn_admission_wait_ms`, `turn_inflight`, `admission_rejected_or_deferred`.

**Felia 1 — Aftercare off-conn (~2-3h, risc MINIM)**
8. `_cache_writeback`/`_summarize_if_needed`/`_extract_profile_and_score` → checkout-uri scurte
   proaspete; LLM (embed/nano) FĂRĂ conn; fiecare write DB cu TX proprie. Reply-ul e deja commis.

**Felia 2 — Load batching (~2h, fără gather implicit)**
9. `history`+`summary`+`facts`(+data_version) pe UN checkout scurt (secvențial/SQL combinat). `gather`
   DOAR pentru operații lente/independente, DUPĂ metrici (în prod ~1ms/query → gather crește presiunea
   instantanee pe pool fără câștig).

**Felia 3 — Gates off-conn (~3h, mare pe voice/image)**
10. `moderate`/Vision/**download media** fără conn; re-acquire scurt DOAR pt `block_contact`/
    `request_human`. (`bot_active`/`handoff_until` sunt DEJA în `ctx` — gates nu citește flaguri din DB.)

**Felia 4 — Free layers off-conn (~4h, cache/FAQ/triaj)**
11. Cache: canonicalize în memorie → lookup exact conn scurt → release → `embed` fără conn → lookup
    semantic conn scurt → touch/evict conn scurt. FAQ analog. Triaj: slugs conn scurt → `classify_json`
    fără conn → (chips) frați conn scurt.

**Felia 5 — Agent prompt inputs off-conn (~2h)**
12. Încarcă category names + routing aliases → eliberează conn ÎNAINTE de `run_tool_loop`.

**Felia 6 — Agent tool loop conn-per-op (~4h, grosul; prerechizit: 0C livrată)**
13. Fiecare tool: `async with deps.db() as conn:` → DB → release. Thinking-ul între tool calls nu ține
    conn. `run_tool` (`src/tools/base.py`) primește provider, nu conn viu.

**Felia 7 — Curățare (~2h)**
14. Elimină `deps.conn` din pipeline/tool code; guard CI devine hard-fail; `PipelineDeps` = doar `db`.

## Test Cases
**Happy Path:**
- Tur de sales migrat: `pool_probe` v2 arată `held ≈ db_active` (conn eliberat pe durata `run_tool_loop`).
- Admission: N tururi concurente (N < prag) → toate rulează, `turn_inflight ≤ prag`, zero deferral.
- Aftercare off-conn: reply commis, apoi summarizer/cache/profil rulează pe checkout-uri scurte.

**Edge Cases:**
- Burst N > prag → surplusul re-queue cu backoff (P6), procesat după ce se eliberează sloturi; ZERO drop.
- Voice/image: gates face Vision/download FĂRĂ conn; doar block/request_human iau checkout scurt.
- State schimbat de alt tur între load și commit → `state_version` respinge write-ul stale (optimistic lock).

**Failure Cases:**
- Semafor timeout la acquire → fallback grațios / re-queue, NU tăcere (P6).
- Checkout provider pe tenant greșit (reuse murdar) → `IsolationError` (NX-04), nu date scurse.
- Redis jos (conv lock/admission) → fail-open documentat (nu blocăm tot traficul), metricile marchează.
- LLM replay/mock în CI — ZERO apeluri reale; probe-ul cu LLM real e out-of-CID.

## Cost
**Zero cost LLM nou** — taskul NU adaugă apeluri LLM/embed; doar mută conexiunea de sub apelurile
existente. Overhead: acquire/release provider (~µs pe pool cald) + achiziție semafor (~µs) —
neglijabil vs. LLM wait (secunde). Câștig indirect: throughput mai mare cu același pool → mai puțin
timp de conexiune facturat pe DB, headroom pt scalare fără a mări `bot_pool`.
