# NX-71 — Gating proactiv: consent + fereastră 24h + template approved
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7 (proactive_jobs există) · **Branch:** `feat/NX-71-proactive-gating` · **Complexitate:** S · **Estimare:** 3.5h

## Goal
Înainte de ORICE mesaj proactiv (AWB, back-in-stock, coș abandonat, follow-up),
sistemul trebuie să decidă determinist DACĂ are voie să trimită și CUM: mesaj
liber (în fereastra 24h Meta) sau DOAR un template aprobat (în afara ferestrei).
Astăzi NICIUN cod nu citește `in_24h_window()`, `wa_templates` sau
`contacts.consent` pentru decizii — proactivul ar trimite orb și ar arde quality
rating-ul / ar încălca politica Meta. Acest task scrie poarta pură (cod
determinist, fără LLM) pe care o consumă NX-70 (motorul proactiv) ca să producă
payload-ul de outbox sau să marcheze jobul `skipped_*`.

## Business Context
Mesajul proactiv în afara ferestrei de 24h pe WhatsApp e legal DOAR ca template
aprobat de Meta; un mesaj liber în afara ferestrei = strike de politică →
quality rating scade → numărul (activul comercial al clientului) e restricționat
sau banat. Trimiterea fără opt-in = reclamații + spam → idem. Poarta asta e frâna
care ține proactivul în regulile Meta, deci protejează direct livrabilitatea pe
care se bazează toți banii din proactiv (AWB, recuperare coș, restock).

### Contractul: o funcție pură de decizie
Poarta NU atinge pipeline-ul de inbound și NU scrie în `TurnContext` (proactivul
rulează în scheduler-ul separat, în afara celor 9 stagii — vezi CLAUDE.md

```python
# src/proactive/templates.py
from dataclasses import dataclass

@dataclass(frozen=True)
class ProactiveDecision:
    allowed: bool
    mode: str                      # 'free' | 'template' | 'blocked'
    reason: str                    # 'ok_free'|'ok_template'|'no_optin'|'no_window_no_template'
    rendered_text: str | None = None      # textul gata de trimis (free SAU template randat)
    template_id: str | None = None        # wa_templates.id (doar mode='template')
    provider_template_id: str | None = None  # pt API-ul Meta template (doar mode='template')

# mapare reason → proactive_jobs.status (pt NX-70):
#   ok_free / ok_template      → trimite (status devine 'sent' după dispatch)
#   no_optin                   → 'skipped_no_optin'
#   no_window_no_template      → 'skipped_no_window'
```

```python
async def decide_proactive(
    conn,                       # tenant_conn (bot_runtime), deja scoped pe business
    *,
    business_id: str,
    contact: Contact,           # are .consent (jsonb) + .id
    conversation: dict,         # rândul conv (are id, last_inbound_at) — pt fereastră
    channel_id: str,
    kind: str,                  # awb_update|back_in_stock|abandoned_cart|follow_up
    locale: str,                # din conversation/contact — parte din CHEIA de template
    template_name: str,         # ce template am vrea dacă suntem în afara ferestrei
    free_text: str,             # textul liber (folosit DOAR în fereastră)
    variables: dict[str, str],  # valori pt randarea template-ului
) -> ProactiveDecision:
    # 1. CONSENT — fără opt-in, nu trimite NIMIC (nici în fereastră)
    if not _has_optin(contact.consent, kind):
        return ProactiveDecision(False, "blocked", "no_optin")

    # 2. FEREASTRA 24h — decisă de DB (in_24h_window), nu recalculată în Python
    in_window = await is_in_24h_window(conn, business_id, conversation["id"])
    if in_window:
        return ProactiveDecision(True, "free", "ok_free", rendered_text=free_text)

    # 3. în afara ferestrei → DOAR template approved (business_id + channel + locale)
    tmpl = await get_approved_template(
        conn, business_id, channel_id=channel_id, name=template_name, locale=locale
    )
    if tmpl is None:
        return ProactiveDecision(False, "blocked", "no_window_no_template")
    text = render_template(tmpl["body"], tmpl["variables"], variables)
    return ProactiveDecision(
        True, "template", "ok_template",
        rendered_text=text, template_id=tmpl["id"],
        provider_template_id=tmpl["provider_template_id"],
    )
```

### Consent — formă în jsonb (cod determinist)
`contacts.consent` e jsonb liber. Convenție (documentată aici, citită de poartă):
`{"proactive": true, "marketing": false}` plus opțional override per-kind
`{"awb_update": true}`. Regula: marketing-ul (`abandoned_cart`, `follow_up`) cere
`marketing=true`; utilitarele tranzacționale (`awb_update`, `back_in_stock`) cer
`proactive=true`. Un kind explicit `false` în consent bate default-ul (opt-out fin).
```python
_MARKETING_KINDS = {"abandoned_cart", "follow_up"}

def _has_optin(consent: dict, kind: str) -> bool:
    if consent.get(kind) is False:          # opt-out explicit per kind
        return False
    if consent.get(kind) is True:           # opt-in explicit per kind
        return True
    key = "marketing" if kind in _MARKETING_KINDS else "proactive"
    return bool(consent.get(key, False))    # default: fără opt-in => NU
```

### Fereastra 24h — interogăm funcția SQL existentă (NU reimplementăm)
```python
# src/db/queries/conversations.py  (query NOU)
async def is_in_24h_window(conn, business_id: str, conversation_id: str) -> bool:
    val = await conn.fetchval(
        """
        select in_24h_window(c) from conversations c
         where c.business_id = $1 and c.id = $2
        """,
        business_id, conversation_id,
    )
    return bool(val)
```
Funcția SQL e sursa de adevăr (derivat, nu flag stocat — CLAUDE.md). `business_id = $1`
e prezent chiar dacă filtrăm pe PK: izolare multi-tenant fără excepție (P7).

### Lookup template aprobat (business_id + channel + locale + status)
```python
# src/db/queries/wa_templates.py  (fișier NOU)
async def get_approved_template(
    conn, business_id: str, *, channel_id: str, name: str, locale: str
) -> dict | None:
    row = await conn.fetchrow(
        """
        select id::text, name, language, body, variables, provider_template_id
        from wa_templates
        where business_id = $1
          and channel_id = $2
          and name = $3
          and language = $4          -- locale e PARTE din cheie (P11)
          and status = 'approved'    -- în afara ferestrei DOAR approved
        order by version desc
        limit 1
        """,
        business_id, channel_id, name, locale,
    )
    if row is None:
        return None
    d = dict(row)
    d["variables"] = json.loads(d["variables"]) if isinstance(d["variables"], str) else (d["variables"] or [])
    return d
```
`order by version desc limit 1` = cea mai nouă versiune aprobată. Dacă nu există
template aprobat în `locale`-ul cerut → `None` (NU cădem pe altă limbă: un template
în limba greșită e un bug, nu un fallback — P11).

### Randarea variabilelor (cod pur, fără LLM)
Meta folosește placeholders poziționali `{{1}}`, `{{2}}`. `wa_templates.variables`
e lista de nume așteptate (`["awb", "courier"]`); le mapăm pozițional la `{{n}}`.
```python
def render_template(body: str, var_names: list[str], values: dict[str, str]) -> str:
    out = body
    for i, name in enumerate(var_names, start=1):
        val = values.get(name, "")
        out = out.replace("{{" + str(i) + "}}", str(val))
    return out
```
Variabilă lipsă din `values` → string gol (nu crăpăm; jobul oricum se loghează).
Pentru API-ul Meta, NX-70 trimite `provider_template_id` + `values` ordonate;
`rendered_text` e oglinda lizibilă (loguri / canale fără template nativ ca Telegram).

### Error handling
- Conversație inexistentă / `last_inbound_at` NULL → `in_24h_window` = false →
  cădem corect pe ramura de template (nu trimitem liber din greșeală).
- Eroare DB la lookup template → propagă excepția spre NX-70 (jobul → `failed`,
  retry); NU întoarce „allowed" tăcut (proactivul nu trimite la incertitudine).

## Test Cases
Toate cu stub pe `conn` (`fetchval`/`fetchrow` returnează valori fabricate).
ZERO DB real, ZERO LLM/embeddings, ZERO rețea.

**Happy Path:**
1. opt-in (`consent={"proactive": true}`) + în fereastră → `mode='free'`,
   `rendered_text == free_text`, fără query de template apelat.
2. opt-in + în afara ferestrei + `get_approved_template` întoarce body
   `"AWB-ul tău {{1}} la {{2}}"` cu `variables=["awb","courier"]` și
   `values={"awb":"123","courier":"FAN"}` → `rendered_text == "AWB-ul tău 123 la FAN"`,
   `template_id` și `provider_template_id` setate, `reason='ok_template'`.

**Edge Cases:**
1. `consent={"abandoned_cart": true}` dar `marketing` absent + kind=`abandoned_cart`
   → opt-in (override per-kind bate default-ul). Iar `consent={"marketing": true,
   "abandoned_cart": false}` + kind=`abandoned_cart` → `no_optin` (opt-out fin).
2. `render_template` cu variabilă lipsă din `values` → `{{n}}` devine string gol,
   restul randat corect (nu crăpă).
3. `is_in_24h_window` False (conversație fără `last_inbound_at`) + există template →
   ramura template, NU `free` (nu scapă mesaj liber în afara ferestrei).

**Failure Cases:**
1. `get_approved_template` întoarce `None` (doar `submitted`/altă limbă) în afara
   ferestrei → `allowed=False, reason='no_window_no_template'` (nu trimite).
2. `conn.fetchrow` aruncă (DB jos) la lookup template → excepția se propagă (NX-70
   marchează jobul `failed`); poarta NU întoarce `allowed=True` tăcut.

## Cost
**$0 LLM** — poarta e cod determinist (if-uri + `str.replace` + 1-2 query-uri SQL
indexate). Niciun apel OpenAI/embeddings. Cost per 1000 mesaje proactive = doar
2 round-trip-uri DB ușoare per mesaj (fereastră + template), neglijabil. Ce
limitează costul real al proactivului (volum trimis) e NX-07 (pacing) + NX-05
(auto-pause), nu acest task.
