# G6-2 — Context builder bogat (felia 1): hidratare state + profil & state în prompt
**Owner:** S · **Faza:** P1 · **Branch:** `feat/G6-2-context-rich` · **Complexitate:** M · **Estimare:** ~4h

## Goal
Agentul vede acum doar istoricul brut (G6). Această felie adaugă, DETERMINIST (fără LLM):
**(1)** hidratarea `ctx.state` din `conversations.state` (azi NU se face — agentul nu știe ce
produse a afișat!), **(2)** un bloc compact de **profil client** (`contacts.profile`) și
**(3)** un bloc de **state references** (produse arătate + constrângeri) injectate în prompturile
de triaj + agent. Memoria scurtă a conversației → follow-up-uri coerente („al doilea", „ceva mai
ieftin decât ăla"). Summarizer-ul (>20 msg → `conversation_summaries`) = felia 2.

## Business Context
Botul „ține firul" mai bine: știe ce a recomandat și ce constrângeri are clientul, fără să
reîntrebe. Fundație pentru personalizare (profilul se umple de extractorul post-tur, felia 3).

## Technical Description

### Gap reparat: hidratarea `ctx.state` (processor)
- Azi `ctx.state` rămâne GOL (default) — `conv["state"]` se citește doar la SCRIERE. Adăugăm
  `ConversationState.from_jsonb(conv["state"])` la construcția `ctx` (loader = processor, ca
  pentru `history`/`language`). Read-only pt pipeline; calea de scriere (Sender) rămâne neatinsă.

### Builder (`src/worker/context.py`) — determinist, bugetat (principiul 4)
- `customer_profile_block(contact, *, max_chars=300)` — „Profil client: <k: v>; …" din
  `contact.profile` (+ `lifecycle` dacă ≠ new); sare cheile goale; tăiat la buget. Gol → "".
- `state_block(state, *, max_products=3, max_chars=400)` — „Produse arătate recent: Nume (preț
  lei), …" (ref-uri, principiul 8) + „Constrângeri știute: <k: v>". Gol → "".
- `context_blocks(ctx) -> str` — unește blocurile ne-goale (profil + state), fiecare pe linia lui.

### Wiring (triaj + agent)
- `triage_stage` și `agent_stage` prepend `context_blocks(ctx)` în mesajul user (după
  `history_block`). Promptul SYSTEM rămâne static byte-identic → prompt caching neatins
  (blocurile dinamice stau în user, ca istoricul).

## Principii CLAUDE.md aplicabile
- **P4 (buget în cod):** fiecare bloc are cap de caractere/produse; nu „prin disciplină".
- **P8 (ref-uri):** state_block folosește `ProductRef` (nume+preț), nu obiecte.
- **P2:** zero LLM aici (summarizer-ul, care ar genera, = felia 2).
- **Un proprietar/câmp:** `ctx.state` SEEDat de processor (loader), citit de builder; scrierea
  rămâne la Sender (neatins).
- **P12:** profilul nu conține PII de canal (telefon stă în channel_identities); blocul nu-l expune.

### Câmpuri TurnContext
`ctx.state` acum HIDRATAT de processor (înainte: gol). Niciun câmp nou.

## Implementation Steps
1. `src/models.py`: `ConversationState.from_jsonb(raw)` (defensiv: produse cu chei lipsă → skip).
2. `src/worker/context.py`: `customer_profile_block` + `state_block` + `context_blocks`.
3. `src/worker/processor.py`: `state=ConversationState.from_jsonb(conv["state"] or {})`.
4. `src/worker/stages/triage.py` + `stages/agent.py`: injectează `context_blocks(ctx)` în user.
5. Teste: `tests/test_context.py` (blocuri + hidratare + buget) + assert wiring.
6. `ruff check . && ruff format . && pytest -q -m "not integration"` verde.
