# NX-84 — Job cleanup — drop partiții vechi + expire semantic_cache
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după NX-83 (scheduler joburi) · **Branch:** `feat/NX-84-cleanup-job` · **Complexitate:** S · **Estimare:** 3h

## Goal
Două bucle de mentenanță care lipsesc azi: (1) partițiile lunare vechi din `messages` și
`analytics_events` nu se șterg NICIODATĂ → tabelele cresc la infinit (cost storage + index bloat);
(2) entry-urile expirate din `semantic_cache` (`expires_at < now()`) sunt filtrate la citire
(`semantic_cache.py:61,91`), dar nu sunt purjate niciodată → tabelul + indexul HNSW se umflă cu
rânduri moarte. Acest task scrie `src/jobs/cleanup.py`: `DROP PARTITION` pe partițiile peste
fereastra de retenție + `DELETE` pe cache-ul expirat. Rulează pe `admin_conn` (DDL + mentenanță
cross-tenant). Schema îl prevede explicit (`schema_v2_production.sql:822-827`, secțiunea „RETENȚIE
(pg_cron)") — exemplele comentate acolo sunt contractul pe care îl implementăm în Python ca să-l
poată chema scheduler-ul (NX-83), nu doar `pg_cron`.

## Business Context
Storage-ul Supabase și performanța query-urilor sunt costuri directe ale agenției. Fără cleanup,
partițiile vechi de mesaje și evenimentele de analytics se acumulează lună de lună (hot tables,
volum mare), iar `semantic_cache` se umple de rânduri expirate care încetinesc căutarea HNSW. E
mentenanță invizibilă pentru client, dar ține factura de infra plată și latența de cache mică.

## Technical Description

### Stare curentă vs țintă
- **Azi:** există `cleanup_dedupe.py` (purjă `inbound_dedupe`, model de urmat) și `rollup_usage.py`
  (iterare cross-tenant pe `admin_conn`). NU există `src/jobs/cleanup.py`. Partițiile se creează
  manual/pg_partman (`messages_2026_06`, `analytics_events_2026_07`, ...), dar nimic nu le șterge.
  `semantic_cache` are `delete_entry`/`purge_business`/`purge_by_product` (purje punctuale), dar
  NIMIC nu șterge bulk pe `expires_at < now()`.
- **Țintă:** un job idempotent, două operații, pe `admin_conn`.

### (1) Drop partiții vechi (DDL → `admin_conn`)
Partițiile au nume deterministe `{tabel}_{YYYY}_{MM}` (vezi schema `messages_2026_06`). Fereastra de
retenție implicită = **6 luni** (config `retention_months`, default 6). Algoritm: enumeră partițiile
existente din catalog, parsează luna din nume, dacă luna e mai veche decât `now() - retention_months`
→ `DROP TABLE`. NU atingem `*_default` (partiția catch-all rămâne mereu).

```python
# src/db/queries/maintenance.py  (query NOU — operație admin, NU tenant-scoped:
# DDL pe partiții fizice; nu are business_id pentru că partiția conține TOATE
# tenant-urile pe acel interval de timp, iar drop-ul e o decizie de retenție globală)
import re
from datetime import date

# pattern strict: doar partițiile lunare ale celor 2 tabele hot (NU *_default, NU altceva)
_PART_RE = re.compile(r"^(messages|analytics_events)_(\d{4})_(\d{2})$")

async def list_time_partitions(conn) -> list[tuple[str, date]]:
    """Partițiile lunare existente ale tabelelor hot + luna lor (din nume).
    Sursă: pg_inherits → relația părinte-copil; numele e contractul de retenție."""
    rows = await conn.fetch(
        """
        select c.relname as name
        from pg_inherits i
        join pg_class c   on c.oid = i.inhrelid
        join pg_class p   on p.oid = i.inhparent
        where p.relname in ('messages', 'analytics_events')
        """
    )
    out: list[tuple[str, date]] = []
    for r in rows:
        m = _PART_RE.match(r["name"])
        if not m:               # *_default sau orice altceva → skip (paranoia)
            continue
        out.append((r["name"], date(int(m.group(2)), int(m.group(3)), 1)))
    return out

async def drop_partition(conn, name: str) -> None:
    """DROP TABLE pe o partiție lunară. Numele e validat de _PART_RE în caller
    (allowlist) — nu se interpolează niciodată input liber în DDL."""
    # re-validare la marginea execuției: dublă plasă contra injection în identificator
    if not _PART_RE.match(name):
        raise ValueError(f"refuz drop pe partiție ne-validată: {name!r}")
    await conn.execute(f'drop table if exists "{name}"')
```

```python
# src/jobs/cleanup.py
async def drop_old_partitions(conn, *, retention_months: int) -> list[str]:
    """Șterge partițiile lunare mai vechi decât fereastra. Întoarce numele șterse.
    O partiție care crapă la drop e logată și sărită — restul continuă (idempotent
    oricum: DROP IF EXISTS + re-run inofensiv)."""
    today = datetime.now(UTC).date()
    cutoff = date(today.year, today.month, 1) - relativedelta(months=retention_months)
    dropped: list[str] = []
    for name, part_month in await list_time_partitions(conn):
        if part_month < cutoff:
            try:
                await drop_partition(conn, name)
                dropped.append(name)
            except Exception:  # noqa: BLE001 — o partiție nu trebuie să blocheze restul
                log.exception("drop partiție eșuat: %s", name)
    log.info("cleanup partiții: %d șterse (retenție %d luni, cutoff %s)",
             len(dropped), retention_months, cutoff)
    return dropped
```

### (2) Expire semantic_cache (DML cross-tenant → `admin_conn`)
Un singur `DELETE` global pe `expires_at < now()`. Aici NU filtrăm pe `business_id`: e mentenanță
cross-tenant (ca `cleanup_inbound_dedupe`), iar `expires_at` e ortogonal de tenant — un rând expirat
e gunoi indiferent de cine l-a scris. `bot_runtime` (RLS) ar vedea doar tenantul curent; de aceea
rulează pe `admin_conn`. (Purjele PER-tenant `business_id = $1` rămân în `semantic_cache.py` pentru
offboarding — alt drum.)

```python
# src/db/queries/maintenance.py
async def expire_semantic_cache(conn) -> int:
    """Purjă bulk a entry-urilor expirate. Cross-tenant intenționat (admin_conn):
    expires_at e independent de business_id, iar RLS ar limita la un tenant.
    Întoarce nr. de rânduri șterse."""
    res = await conn.execute("delete from semantic_cache where expires_at < now()")
    return int(res.split()[-1]) if res else 0
```

async def run() -> dict[str, int]:

## Principii CLAUDE.md aplicabile
- **P5 (un singur punct de ieșire):** jobul NU trimite niciun mesaj; e mentenanță pură. Nu atinge
  `outbox`/dispatcher.
- **P7 (business_id pe tot) — excepția documentată:** `expire_semantic_cache` și `drop_partition`
  sunt mentenanță CROSS-TENANT pe `admin_conn`, exact ca `cleanup_inbound_dedupe`/`rollup_usage`.
  Partițiile fizice și `expires_at` nu sunt date de un singur client; nu există filtru pe tenant.
  Purjele per-tenant (`purge_business`) rămân separate, cu `business_id = $1`.
- **P12 (PII într-un loc):** jobul nu citește niciun corp de mesaj/telefon — doar nume de partiție și
  `expires_at`. Logurile conțin DOAR nume de partiție + numărători, ZERO PII (numele partiției e
  `messages_2026_06`, nu conține date de client).
- **Securitate DDL:** numele de partiție e pus pe allowlist cu `_PART_RE` ÎN AMBELE locuri (caller +
  `drop_partition`) — niciun identificator liber nu ajunge în `DROP TABLE`. `IF EXISTS` → idempotent.

## Test Cases
**Happy Path:**
1. 4 partiții (`messages_2026_01..03`, `analytics_events_2026_01`) + data simulată 2026-08,
   `retention_months=6` → cele < 2026-02 sunt în `dropped`; `drop_partition` chemat exact pe ele.
2. `expire_semantic_cache` (integration, rollback): insert 1 entry cu `expires_at = now()-1h` + 1 cu
   `now()+1h` → întoarce 1; cel valid rămâne.

**Edge Cases:**
1. `messages_default` + `analytics_events_default` în listă → IGNORATE (nu sunt lunare, `_PART_RE`
   nu prinde) → niciun drop pe ele.
2. Zero partiții peste retenție → `dropped == []`, `run()` întoarce `partitions_dropped=0`, fără
   eroare.
3. Partiție chiar pe granița cutoff (luna == cutoff) → NU se șterge (`<` strict, nu `<=`).

**Failure Cases:**
1. `drop_partition` aruncă pe `messages_2026_02` (mock) → e logat, sărit; `messages_2026_03` tot se
   șterge; jobul nu propagă excepția.
2. Nume injectat (`'messages_2026_01"; drop table businesses; --'`) → `_PART_RE` nu prinde →
   `drop_partition` ridică `ValueError`, niciun DDL executat.
3. `expire_semantic_cache` pe `DELETE 0` → întoarce 0 (nu crapă pe parsarea „DELETE 0").

> Orchestrarea (`run`, `drop_old_partitions`) se testează cu query-urile din `maintenance`
> monkeypatch-uite (model `test_rollup_usage.py`) — ZERO DB și ZERO DDL real în CI. Doar
> `expire_semantic_cache` are un test marcat `@pytest.mark.integration` (DB real, rollback per test,
> exclus din gate-ul rapid). Niciun apel LLM/embeddings — jobul e cod pur determinist.
