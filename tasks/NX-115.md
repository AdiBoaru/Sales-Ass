# NX-115 — ChannelProfile capability matrix replacing scattered hasattr duck-typing

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** Wave0 · **Branch:** `feat/NX-115-channel-capability-matrix` · **Complexitate:** M · **Estimare:** 4h

## Goal
Înlocuiește scara de `hasattr()` din dispatcher (`dispatcher.py:74,80,93,101`) și consumer (`consumer.py:67`) cu o matrice **declarativă de capabilități** per `ChannelSender` (`{TEXT, RICH, CARDS, CAROUSEL, EDIT, TYPING, MEDIA, OFFER}`). Rutarea de randare devine un singur pas table-driven „Reply semantic → render conștient de capabilități", plus clamp de lungime per canal la sender. Un canal nou se adaugă declarând capabilități, nu editând scara `if/elif`.

## Business Context
Asistentul e un singur creier care vorbește pe mai multe canale cu capabilități divergente: **web** (buton, doar TEXT azi prin `WebSender`), **Telegram** (RICH + CAROUSEL + EDIT + carduri, canal de test), **WhatsApp** (TEXT + TYPING + MEDIA, canal primar de producție). Azi divergența e *emergentă* — dedusă din existența metodelor (`hasattr(sender, "send_rich")`) — nu *declarată*. Asta face imposibil de raționat „ce randează canalul X", duplică logica de degradare și ascunde buguri: `_safe_typing` pune `channel_kind` default pe `"whatsapp"` la cheie lipsă (`consumer.py:66`), deci un canal necunoscut ar trimite typing greșit pe WhatsApp. Pe măsură ce adăugăm canale (web rich-render NX-127, viitor Instagram/email), scara `if/elif` devine un punct unic de regresie. O matrice explicită = contract verificabil (test: capabilitățile declarate == metodele reale) și extensibilitate fără atingerea dispatcher-ului — exact patternul „capability negotiation" din platformele de messaging din 2026.

## Technical Description

### 1. Capabilități declarative pe `ChannelSender` (`src/channels/base.py`)
Adaugă un enum `Capability` (str-enum) și o expunere declarativă pe protocol:

```python
class Capability(str, Enum):
    TEXT = "text"        # send_text (obligatoriu pt orice sender)
    RICH = "rich"        # send_rich (recomandare structurată model iZi)
    CARDS = "cards"      # send_products (listă compactă cu butoane-link)
    CAROUSEL = "carousel"# send_carousel_card
    EDIT = "edit"        # edit_message_media (navigare carusel R2)
    TYPING = "typing"    # mark_typing (semnal inbound NX-90)
    MEDIA = "media"      # fetch_media (download inbound; informativ aici)
    OFFER = "offer"      # randare Reply.offer (NX-114) nativ vs aplatizat în text
```

Protocolul `ChannelSender` capătă un atribut/clasă `capabilities: frozenset[Capability]` (atribut de clasă pe fiecare implementare). `TEXT` e obligatoriu. Plus, opțional, `max_text_len: int | None` și `max_caption_len: int | None` pentru clamp.

### 2. Render table-driven în dispatcher (`src/worker/dispatcher.py:72-110`)
Scara `if/elif hasattr` se înlocuiește cu o funcție pură care, dată fiind perechea `(ptype, payload, sender.capabilities)`, alege ramura de randare și apelează metoda potrivită — sau **degradează grațios** la următoarea capabilitate suportată, terminând mereu pe `send_text` (P6, niciodată tăcere). Ordinea de preferință:
- `payload.rich` + `RICH` → `send_rich`; altfel aplatizare în `send_text(payload["text"])`.
- `ptype == "edit_media"`: cere `EDIT`; lipsă → `mark_failed` „edit nesuportat" (nu degradăm la text, e o navigare UI, nu conținut nou).
- `ptype == "carousel"` + produse + `CAROUSEL` → `send_carousel_card`.
- `ptype in ("carousel","products")` + produse + `CARDS` → `send_products`.
- altfel → `send_text` (textul conține deja recomandarea).

`Reply.offer` (NX-114, neutru) se randează: canal cu `OFFER` → buton/CTA nativ (web button, WhatsApp interactive/CTA); fără `OFFER` → offer-ul e deja aplatizat în `payload["text"]` (floor). Comportament **channel-aware** explicit: web = buton, WhatsApp = CTA/text.

### 3. Clamp de lungime la sender (capabilitate de transport)
La fiecare sender, înainte de apelul de transport: trunchiere la limita platformei — Telegram caption 1024 / body 4096, WhatsApp body 4096. Trunchiere cu elipsă (reutilizează patternul `_short` din `telegram/client.py:15`), aplicată în `send_text`/caption-uri. Evită respingerea întregului mesaj de către platformă (azi un text >4096 pică tăcut la retry).

### 4. Fix `_safe_typing` default (`src/worker/consumer.py:59-76`)
`event.get("channel_kind", "whatsapp")` → `event.get("channel_kind")`; `None`/necunoscut → no-op devreme (sender lipsă sau fără `TYPING` în capabilități). Lookup-ul de typing folosește `Capability.TYPING in sender.capabilities` în loc de `hasattr(sender, "mark_typing")`.

### 5. Capabilități per sender existent
- `MetaClient` (`src/meta_client.py`): `{TEXT, TYPING, MEDIA}` (+ OFFER prin CTA când implementăm; azi TEXT-floor). `max_text_len=4096`.
- `TelegramClient` (`src/channels/telegram/client.py`): `{TEXT, RICH, CARDS, CAROUSEL, EDIT, TYPING}`. `max_text_len=4096`, `max_caption_len=1024`.
- `WebSender` (`src/channels/web/sender.py`): `{TEXT}` azi (RICH vine în NX-127). `max_text_len=None`.

### 6. Test de divergență (`tests/test_dispatcher.py`)
Azi `dispatcher.py:74-110` are zero acoperire pe ramurile de degradare. Adăugăm: (a) test de consistență — pentru fiecare sender înregistrat, fiecare capabilitate declarată are metoda reală corespunzătoare și invers (nici o metodă „bogată" nedeclarată); (b) fake rich-capable vs text-only care exercită fiecare ramură de degradare (rich→text, carousel→cards→text, edit pe canal fără EDIT → dead).

**Generic across verticals:** infra pură agnostică de vertical; un vertical/canal nou se activează declarând capabilități, fără cod nou în dispatcher.

## Test Cases
**Happy:**
- Telegram + payload `rich` → `send_rich` apelat (canal cu `RICH`); provider_id legat pe mesaj, outbox `sent`.
- WhatsApp + payload `products` → degradează la `send_text` (fără `CARDS`), textul ajunge integral.
- Web + text simplu → `send_text` pe `WebSender`, `{TEXT}` suficient.

**Edge:**
- Text de 5000 caractere pe WhatsApp → trunchiat la 4096 cu elipsă; mesajul pleacă (nu respins).
- Caption carusel >1024 pe Telegram → trunchiat la 1024.
- `_safe_typing` cu `channel_kind` lipsă/necunoscut → no-op (nu trimite typing pe WhatsApp).
- `edit_media` pe sender fără `EDIT` → `mark_failed`/`dead` cu mesaj explicit (nu degradare la text).

**Failure:**
- Sender care declară `RICH` dar nu are `send_rich` → testul de consistență pică (contract încălcat).
- Sender care are `send_rich` dar nu-l declară → testul de consistență pică (capabilitate nedeclarată).
- Eroare de transport în ramura aleasă → `mark_failed` cu backoff (comportament neschimbat, acoperit de testul existent).
