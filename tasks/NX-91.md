# NX-91 — Validator: prinde cifre fără valută (numere halucinate)
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7-1 (agent) · **Branch:** `feat/NX-91-validator-bare-numbers` · **Complexitate:** S · **Estimare:** 3h

## Goal
Validatorul (stagiul 8, inline în agent) prinde azi DOAR prețurile cu valută explicită
(`_PRICE_RE` = `\d+ (lei|ron)`). Un număr halucinat FĂRĂ valută — „costă 89", „ai 12 bucăți
în stoc", „rating 4.9" — trece nepedepsit, deși poate fi un preț/stoc/rating inventat care
NU există în `ctx.retrieval`. Taskul extinde validatorul să prindă și cifrele „suspecte" fără
valută și să le verifice contra faptelor grounded (prețuri, stoc, rating din produsele
retrievate + sumele grounded de bot); la dubiu → retry de recompunere fără cifre. Întărește
invariantul „ZERO cifre inventate structural", nu doar „zero prețuri cu lei inventate".

## Business Context
O cifră inventată distruge încrederea la fel ca un preț inventat: „mai ai 3 pe stoc" când nu
e adevărat = client supărat + comandă pierdută. Gaura actuală e fix la marginea regexului —
modelul scapă halucinația scoțând valuta („costă 89, super ok"). Costul e zero (verificare în
cod determinist, fără LLM în plus pe calea fericită), iar câștigul e direct: mai puține
promisiuni false despre preț/stoc/disponibilitate.

## Technical Description

### Starea curentă (src/worker/stages/agent.py:35–114)
```python
_PRICE_RE = re.compile(r"(\d{1,6}(?:[.,]\d{1,2})?)\s*(?:lei|ron)", re.IGNORECASE)
# _prices_ok: pentru fiecare match al _PRICE_RE, valoarea trebuie să fie ∈ allowed
#   (prețuri produse retrievate + grounded_prices). Toleranță 0.5 lei.
# _valid = _prices_ok AND _links_ok.
```
`_valid` e chemat din `_finalize` (rută SALES, cu produse), `_finalize_grounded` (rută ORDER,
status comandă) și direct în `agent_stage` (SALES fără produse). Toate trei vor beneficia de
regula nouă fără modificări la apelanți — extindem `_valid` în interior.

### Ținta — al doilea control: cifre „bare" suspecte
Adăugăm un regex `_BARE_NUM_RE` care prinde numere „cu greutate" (cele care arată a preț/stoc/
cantitate), DAR cu un filtru de zgomot strict ca să NU declanșăm pe numere legitime de proză
(„top 3 produse", „pasul 2", „pe locul 1", procente — vezi mai jos). Apoi `_bare_numbers_ok`:
fiecare astfel de cifră trebuie să fie grounded într-un set de valori PERMISE derivat din
`ctx.retrieval` + sumele grounded.

```python
# numere cu ≥2 cifre SAU cu zecimale (filtrăm 0-9 simple: „top 3", „2 opțiuni")
_BARE_NUM_RE = re.compile(r"(?<![\w./-])(\d{2,6}(?:[.,]\d{1,2})?|\d[.,]\d{1,2})(?![\w%])")
# (?![\w%]) → NU prinde procente (89% = treaba NX-30) și nu prinde nr. lipite de litere

def _allowed_numbers(products: list[dict[str, Any]], grounded_prices: set[float]) -> set[float]:
    """Toate numerele pe care botul AVEA voie să le spună: prețuri (price/sale_price),
    stoc, rating, variante — din produsele retrievate — + sumele grounded (total comandă)."""
    allowed: set[float] = set(grounded_prices)
    for p in products:
        for key in ("price", "sale_price", "stock", "stock_total", "rating"):
            v = p.get(key)
            if v is not None:
                allowed.add(round(float(v), 2))
        for var in p.get("variants") or []:
            for key in ("price", "sale_price", "stock"):
                v = var.get(key)
                if v is not None:
                    allowed.add(round(float(v), 2))
    return allowed

def _bare_numbers_ok(reply: str, products, grounded_prices: set[float]) -> bool:
    """Orice cifră «grea» din reply care NU e deja un preț cu valută validat trebuie să fie
    grounded (toleranță 0.5). Anul curent + numere mici de proză sunt scoase de regex/whitelist."""
    if not get_settings().validator_bare_numbers_enabled:   # kill-switch (fail-open la fals-pozitive)
        return True
    priced = {float(t.replace(",", ".")) for t in _PRICE_RE.findall(reply)}  # deja acoperite de _prices_ok
    allowed = _allowed_numbers(products, grounded_prices)
    for token in _BARE_NUM_RE.findall(reply):
        value = float(token.replace(",", "."))
        if any(abs(value - p) <= 0.5 for p in priced):       # ex. „89 lei" → numărul 89 e ok
            continue
        if value in _SAFE_BARE:                              # whitelist proză: 24 (ore), anul curent
            continue
        if not any(abs(value - a) <= 0.5 for a in allowed):
            return False
    return True
```
`_SAFE_BARE` = set mic, documentat: `{24.0, 48.0, 100.0}` (24h window, „100% mulțumit” fără %,
procente uzuale de proză) + anul curent. Conservator: dacă apare un fals-pozitiv în live,
extinzi whitelistul sau dezactivezi prin `validator_bare_numbers_enabled` — nu rescrii regula.

`_valid` devine:
```python
def _valid(reply, products, allowed_links=None, allowed_prices=None) -> bool:
    return (
        _prices_ok(reply, products, allowed_prices)
        and _links_ok(reply, products, allowed_links)
        and _bare_numbers_ok(reply, products, allowed_prices or set())
    )
```

### Comportament la eșec (deja existent, refolosit)
Niciun cod nou de recuperare: când `_valid` întoarce `False`, `_finalize` face deja **1 retry**
de recompunere (prompt `_RECO_SYSTEM` cu „FOLOSEȘTE EXACT doar aceste prețuri”) → la eșec
**fallback determinist** (`_deterministic_reply`, doar prețuri reale). Pe ruta ORDER,
`_finalize_grounded` recompune fără numere → fallback sigur fără cifre. Deci „reformulare fără
cifre la dubiu” e exact calea existentă; NX-91 doar declanșează retry-ul și pentru cifrele bare.
Important: îmbunătățim promptul de retry să spună explicit „NU pune numere de stoc/cantitate/
rating care nu sunt în listă” ca retry-ul să aibă șanse reale să treacă a doua oară.

## Test Cases
Toate fără apeluri reale: `_bare_numbers_ok`/`_valid` sunt funcții pure; `agent_stage` rulează
cu `FakeLLM` (scriptează `final`/`retry`) + tool-uri catalog monkeypatch-uite, ca în
`tests/test_agent.py`. ZERO OpenAI/DB real în CI.

**Happy Path:**
1. `_valid("Îți recomand Crema — 82.99 lei, in stock", PRODUCTS)` → `True` (preț + valută real).
2. `_valid("Are rating 4.6 și mai sunt pe stoc", PRODUCTS)` → `True` (4.6 = rating retrievat,
   inclus în `_allowed_numbers`).

**Edge Cases:**
1. „Îți arăt top 3 produse, pasul 2 e simplu” → numere mici (1 cifră) ignorate de regex → `True`.
2. „Reducere 20% la a doua” → `%` exclus din regex (NX-30) → regula bare NU pică → `True`.
3. „Revin în 24 de ore” → 24 ∈ `_SAFE_BARE` → `True` (fără fals-pozitiv pe proză uzuală).

**Failure Cases:**
1. „Crema costă 89, super preț” (89 ∉ retrieval, fără valută) → `_bare_numbers_ok` `False` →
   în `agent_stage`: retry de recompunere → fallback determinist (preț real) + `validator_rejected`.
2. „Mai ai 47 bucăți pe stoc” când stocul retrievat e 12 → `False` (cifră de stoc halucinată).
3. Retry-ul tot inventează o cifră bare → al doilea `_valid` pică → fallback determinist
   (`_deterministic_reply`), niciodată tăcere (P6).

## Cost
**$0 adăugat pe calea fericită** — controlul e regex + set lookup în cod, fără LLM/embeddings.
Cost LLM apare DOAR când validarea pică și se declanșează retry-ul de recompunere (1 apel mini
suplimentar), exact ca azi pentru prețuri cu valută. Mărginirea costului: retry HARD-limitat la
**1** (apoi fallback determinist), deci în cel mai rău caz +1 apel mini per tur respins.
Estimare la 1000 mesaje, presupunând ~3% reply-uri respinse pe cifre bare (rată mică, calibrabilă
din `validator_rejected`): ~30 apeluri mini de recompunere extra / 1000 mesaje ≈ neglijabil
(sub câțiva cenți la 1000). Kill-switch `validator_bare_numbers_enabled` plafonează costul la 0
dacă regula s-ar dovedi prea zgomotoasă în prod.
