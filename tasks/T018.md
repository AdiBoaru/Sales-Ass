**Owner:** S · **Faza:** MVP · **Zi:** Z1 · **Branch:** — · **Complexitate:** XS · **Estimare:** 0.5h

## Goal
Igienă pe singura sursă de adevăr: conexiunea corectă pentru bot (Postgres direct, nu API HTTP), cheile separate de rol și backup-ul point-in-time confirmat funcțional.

## Business Context
Aici stau datele tuturor clienților. Un DROP greșit fără PITR = business mort.

## Technical Description
Din dashboard: **connection string Postgres** (Settings → Database) — folosește pooler-ul pe portul 6543 în mod `transaction` pentru aplicație (asyncpg) și conexiune directă 5432 pentru migrări. Verifică planul include PITR (sau măcar daily backups + activează PITR dacă bugetul permite — recomandat înainte de clienți reali). service_role key NU se folosește de bot (botul = asyncpg cu rolurile din schemă: bot_runtime etc. — create în 001/002); anon key nefolosit deloc. Confirmă în Settings → Backups că există restore points.

## Principii CLAUDE.md aplicabile
Rolurile DB din CLAUDE.md (bot_runtime/catalog_sync/dashboard_read) — conexiunea aplicației va folosi bot_runtime, nu superuser. (Creat în 002 dacă nu există — verifică la T034.)

2. Verifică PITR/backups în dashboard.
3. Test conexiune asyncpg de pe mașina ta cu rolul aplicației.
