## Goal

revendicƒГ joburi scadente din `proactive_jobs` (`scheduled_at <= now`,

### ArhitecturƒГ: control plane вЖТ per tenant (ca dispatcher-ul)
Motorul NU »Щtie dinainte ce tenan»Ыi au joburi scadente. Un ciclu:
1. **control plane** (`admin_conn`, cross-tenant, non-PII): `business_ids_with_due_jobs()`

### Claim-ul unui job (`FOR UPDATE SKIP LOCKED`, fƒГrƒГ status `claimed`)
Schema n-are status intermediar; folosim lock pe r√Ґnd √Ѓn TX, ca outbox-ul. Un job e
вАЮrevendicat" c√Ґt e »Ыinut lock-ul; devine terminal (`sent`/`skipped_*`/`failed`) √ОN
ACEEA»ШI TX cu `enqueue_outbox`, deci nu existƒГ fereastrƒГ √Ѓn care un al doilea worker
√Ѓl vede `scheduled` »Щi liber:
```python
# src/db/queries/proactive.py
async def claim_due_jobs(conn, business_id, *, limit=20) -> list[dict]:
    rows = await conn.fetch(
        """
        select id::text, kind, contact_id::text, conversation_id::text,
               payload, template_id::text, scheduled_at
        from proactive_jobs
        where business_id = $1 and status = 'scheduled' and scheduled_at <= now()
        order by scheduled_at
        for update skip locked
        limit $2
        """,
        business_id, limit,
    )
    return [_row(r) for r in rows]   # payload: json.loads dacƒГ vine text

async def mark_job(conn, business_id, job_id, status: str) -> None:
    # status вИИ {'sent','skipped_no_window','skipped_no_optin','cancelled','failed'}
    await conn.execute(
        "update proactive_jobs set status = $3, executed_at = now() "
        "where business_id = $1 and id = $2",
        business_id, job_id, status,
    )
```
**BuclƒГ per job, totul √Ѓn TX-ul de claim:**
```python
async with tenant_conn(business_id) as conn:
    async with conn.transaction():
        jobs = await claim_due_jobs(conn, business_id)
        for job in jobs:
            await _process_job(conn, business_id, job)
```
Un singur job stricat nu trebuie sƒГ rupƒГ restul lotului вЖТ fiecare `_process_job`
prinde excep»Ыiile lui »Щi marcheazƒГ `failed` (nu propagƒГ). (DacƒГ vrei izolare de TX
per job ca un rollback sƒГ nu pice tot lotul, claim per job √Ѓntr-un savepoint вАФ
`async with conn.transaction():` imbricat. v1: TX pe lot + try/except per job +
`mark_job(failed)`, simplu »Щi suficient pentru volumele de proactiv.)

### Construirea payload-ului + poarta NX-71
`_process_job` apeleazƒГ poarta din NX-71 (`decide_proactive_send`) √ОNAINTE de orice
√Ѓnqueue. Poarta returneazƒГ (form indicativ вАФ contractul exact e al NX-71):
`PROCEED_FREE` (mesaj liber, √Ѓn fereastra 24h), `PROCEED_TEMPLATE` (doar template
aprobat din `wa_templates`, √Ѓn afara ferestrei) sau motivul de skip
(`skipped_no_window`/`skipped_no_optin`):
```python
async def _process_job(conn, business_id, job):
    decision = await decide_proactive_send(conn, business_id, job)   # NX-71
    if decision.skip:                       # no_optin / no_window fƒГrƒГ template
        await mark_job(conn, business_id, job["id"], decision.skip_status)
        ctx_emit("proactive_skipped", kind=job["kind"], reason=decision.skip_status)
        return
    payload = _build_payload(business_id, conn, job, decision)       # NX-70
    idem = f"proactive:{job['id']}"          # idempoten»ЫƒГ per job (vezi mai jos)
    new_id = await enqueue_outbox(
        conn, business_id, job["conversation_id"], idem, payload, kind="proactive",
    )
    await mark_job(conn, business_id, job["id"], "sent")
    ctx_emit("proactive_enqueued", kind=job["kind"], deduped=new_id is None)
```
`_build_payload(kind, ...)` per sursƒГ (text/template gata de dispatcher):
- **awb_update**: payload jobului are deja `order_id`/`awb`/`carrier` (pus de cine
  programeazƒГ jobul вАФ vezi вАЮSurse"); format вАЮComanda ta a fost expediatƒГ, AWB
  {awb} {carrier}". Citire confirmativƒГ op»ЫionalƒГ din `shipments`
  (`where business_id=$1 and order_id=$2`).
- **back_in_stock**: din `back_in_stock_subscriptions` + `products` (nume, url) вЖТ
  вАЮ{produs} e din nou pe stoc".
- **abandoned_cart**: din `checkout_links` neconvertit (`converted_order_id is null`,
  `expires_at > now()`) вЖТ reamintire + link cu `?ref=`.
- **follow_up**: text generic din `payload.body` al jobului (sau template).
Payload-ul de outbox respectƒГ forma pe care dispatcher-ul o »Щtie deja:
`{"type":"text","to":<sender_external_id>,"text":...}` sau (template, NX-71/dispatcher
follow-up) `{"type":"template",...}`. Dispatcher-ul actual trateazƒГ `type in
(text|products|carousel|edit_media)`; `type=template` e marcat вАЮfollow-up" √Ѓn
dispatcher вЖТ outbox 'dead' azi. **De aceea v1 al NX-70 emite DOAR `type=text`**
(fereastra 24h); calea template e blocatƒГ √Ѓn motor (skip `no_window` dacƒГ poarta
cere template) p√ҐnƒГ ce dispatcher-ul capƒГtƒГ send-template вАФ vezi Out of Scope.

### Idempoten»ЫƒГ per job (P5, idempotency_key)
`idempotency_key = f"proactive:{job_id}"`. DacƒГ motorul ruleazƒГ de douƒГ ori pe
acela»Щi job √Ѓnainte de mark (crash √Ѓntre enqueue »Щi mark), `enqueue_outbox` cade pe
`on conflict do nothing` вЖТ `new_id is None`, NU al doilea mesaj. Marcarea `sent` √Ѓn
aceea»Щi TX cu enqueue garanteazƒГ: ori ambele, ori niciuna.

### `to` = id-ul de canal al destinatarului (PII doar la margine)
Mesajul proactiv pleacƒГ spre `sender_external_id` (wa_id / chat.id) din
`channel_identities` (lookup `where business_id=$1 and contact_id=$2` вЖТ `external_id`).
**Acesta e singurul loc cu PII**; intrƒГ √Ѓn `payload.to` (necesar dispatcher-ului),
NU √Ѓn loguri »Щi NU √Ѓn analytics (P12). Logurile motorului folosesc `job_id` +
`kind`, niciodatƒГ telefon.

### C√Ґmpuri TurnContext scrise aici
**NICIUNUL.** Proactivul ruleazƒГ √ОN AFARA pipeline-ului (scheduler separat, fƒГrƒГ
`TurnContext`). Nu existƒГ mesaj inbound, nu existƒГ tur. Analytics se emit direct
