# NX-171d — DDL embeddings versionabile: PK compus + query fără duplicate

**Owner:** S · **Faza:** P1 · **Track:** Catalog v3 · **Branch:** `feat/NX-171d-embeddings-versioned` · **Complexitate:** M · **Estimare:** 1 zi
**Parent plan:** [docs/CATALOG-PRODUS-V3.md](../docs/CATALOG-PRODUS-V3.md) (Pachet 5, felia 4) · **Depinde de:** NX-170 (doc embedding)

## Goal
Reparsă cheia `product_embeddings` (azi PK = `product_id`) care **blochează versiuni paralele** (produs × tip document × model). PK compus `(product_id, doc_type, model)` + **query-uri care garantează un singur rând per produs** (altfel join-ul pe `product_id` întoarce același produs de N ori → duplicate în retrieval).

## Business Context
Ca să putem ține în paralel modele diferite sau tipuri de document (product / review / usage), cheia trebuie versionată. Dar imediat ce există >1 rând per produs, orice join naiv pe `product_id` dublează rezultatele de căutare — bug de retrieval subtil și grav.

## Technical Description
- **Migrare 029** `product_embeddings`: re-cheie de la PK(`product_id`) la PK compus `(product_id, doc_type, model)`. Adaugă `doc_type text` (default `'product'`). Backfill rândurile existente cu `doc_type='product'`.
- **Query-uri (HIGH — anti-duplicate):** orice join pe embeddings TREBUIE să filtreze explicit `doc_type = <activ>` + `model = <model activ>` (din config), NU doar `product_id`. Rezultat garantat: **un singur rând per produs**. Toate query-urile rămân filtrate pe `business_id`.
- Ajustează `embed_products` (scrie cu `doc_type`/`model`) + read-path (`catalog.py`).
