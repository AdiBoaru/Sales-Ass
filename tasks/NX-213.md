# NX-213 — Optimizare latență + cost (după ce fluxul e corect)

**Owner:** Claude (build) / Codex (verify) · **Faza:** 10 (Quality Overhaul) · **Status:** DRAFT — nu se implementează până la review dependențe + DoD · **Complexitate:** M
**Parent plan:** [docs/QUALITY-OVERHAUL-2026.md](../docs/QUALITY-OVERHAUL-2026.md) · **Depinde de:** NX-201 (baseline spans), NX-210/211 (fluxul nou)
**Decizii ADR aplicabile:** D12

## Goal
De la ~14s la ținte comerciale — pe fluxul NOU, ghidat de spans-urile din NX-201
(arhitectura de latență e impusă din F0; aici e optimizarea fină).

## Scope
- Profil + context + prefetch în paralel; exact/lexical/vector în paralel.
- Evidence hydration batch (fără N+1).
- **Fast path pentru preț/stoc/link/detalii exacte (fără buclă) — CU contractul din D2:**
  fast path-ul care termină turul ocolește agentul ȘI AnswerPlan-ul, deci implementează
  propriul contract tipizat + validator determinist: identitate/autorizare verificată pentru
  status comandă · evidence + version pentru preț/stoc/FAQ (facts live, nu snapshot stale) ·
  cache NICIODATĂ cross-tenant sau cross-locale · P6 (fallback formulat, nu tăcere).
- Max 2 apeluri LLM sincrone pentru recomandarea normală.
- Reranker doar când aduce valoare (adaptiv, din NX-209); critic doar pe declanșatoare (NX-211).
- Prompt caching (prefix static byte-identic — existent, de prezervat pe promptul nou).
- Query/embedding cache; memoria + analytics complet async.
- Reasoning effort redus pe tururile simple, ridicat doar la escaladare.
- Streaming SSE pe web (primul token 1-2s; canal web = repo FE separat → update contract
  docs/FRONTEND-CONTRACT-IZI.md dacă se schimbă formatul).

## Ținte
Facts simple 1,5-3s · recomandare normală 3-6s · query complex 6-10s · P90 general <12s.

## Definition of Done
- [ ] Țintele atinse pe sim + live, măsurate cu spans-urile NX-201 (nu impresii).
- [ ] Streaming funcțional pe web fără regresii de contract FE (update
      docs/FRONTEND-CONTRACT-IZI.md dacă formatul se schimbă).
- [ ] Costul mediu/tur raportat înainte-vs-după (cu tarifele reconciliate).
- [ ] **Contractul fast path (D2) implementat + testat:** test identitate/autorizare pe status
      comandă · test evidence+version pe preț/stoc (snapshot stale respins) · test adversarial
      cache cross-tenant/cross-locale · test P6 pe eșecul fast path-ului (fallback la agent,
      nu tăcere).
- [ ] Invariante: nicio optimizare nu slăbește izolarea (`business_id` pe toate căile paralele/
      batch + test cross-tenant pe cache-urile noi de query/embedding — cheia include
      business_id + locale); zero PII în cache-uri/spans; P6 — timeout pe orice cale paralelă
      → fallback formulat, nu tăcere; kill-switch per optimizare (streaming, fast path, cache);
      fallback funcțional fără reranker/critic/cache; test e2e; `ruff` + `ruff format --check` +
      `pytest -x -q` verzi.
