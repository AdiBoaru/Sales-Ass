# NX-120 — Ingestion DoS hardening: request body-size cap + fail-CLOSED web cost/rate guard

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** Wave1 · **Branch:** `feat/NX-120-ingestion-dos-hardening` · **Complexitate:** M · **Estimare:** 4h

## Goal
Oprești epuizarea resurselor și a banilor pe ingestie înainte de orice muncă scumpă: (1) respingi POST-uri cu body peste un cap mic (Content-Length + stream-limit) ÎNAINTE de a citi corpul, pe TOATE endpointurile de ingestie; (2) pentru calea SINCRONĂ `/web/chat` (care cheltuie LLM real per request) faci rate-limit-ul și cost-cap-ul **fail-CLOSED** (429 când limiterul nu poate fi evaluat din cauza Redis); (3) prefixezi cheile SSE pubsub/backlog cu tenantul. Defense-in-depth la Traefik (`client-max-body`).

## Business Context
Asistentul rulează un singur creier pe un VPS mic (1vCPU / 4GB / 0-swap, vezi memory deploy-vps-topology). Două suprafețe sunt deschise spre internet: webhook-ul Meta (semnat) și widget-ul web (`/web/chat`, ajuns din browser, gardat DOAR de un token public livrat în browser). Azi:
- **Nicio limită de body** pe ingestie: `src/webhook/app.py:83 raw = await request.body()` bufferizează integral un POST de mulți MB ÎNAINTE de verificarea semnăturii/parsare. Pe 0-swap un singur request mare poate OOM-ui procesul. Atacatorul nici nu are nevoie de semnătură validă — bufferizarea se întâmplă înainte de verificare.
- **Rate-limit fail-OPEN pe Redis error** (`src/web/app.py:116` și `:201`): la Redis căzut, `web_rate_limited` aruncă `RedisError`, dar FastAPI o transformă în 500 sau (dacă e prinsă) lasă requestul să treacă. Pe `/web/chat` asta înseamnă pipeline complet (triaj nano + agent mini) per request → un atacator cu token-ul public poate arde cota LLM a tenantului fix când guard-ul e indisponibil. `daily_cost_cap_usd` e doar upper-bound al businessului, nu protejează per-vizitator.
- **Cheile SSE fără tenant prefix** (`web:out:{visitor_id}`, `web:backlog:{visitor_id}` — `src/channels/web/sender.py:20-25`, consumate în `src/web/app.py:233-281`): `visitor_id` e generat per tenant (`web_<uuid4>`), deci coliziunea e improbabilă, dar lipsa prefixului încalcă P7 (`business_id` pe tot) și complică izolarea/observabilitatea.

Costul per token e plătit de fiecare tenant, în orice vertical — e hardening de infra/cost generic, nu specific beauty/HVAC/auto.

## Technical Description

### 1. Body-size cap (toate endpointurile de ingestie)
Helper nou determinist `enforce_body_cap(request, max_bytes)` (cod pur, fără LLM):
- citește `Content-Length`; dacă lipsește SAU > `max_bytes` → `413 Payload Too Large` imediat, **fără** a citi corpul;
- citește corpul cu un stream-limit (acumulează din `request.stream()` și abandonează la depășire) ca să prinzi și cazul fără/cu `Content-Length` mincinos (chunked).

Capuri (config nou, valori mici, defaults în cod):
- **Webhook Meta / orders**: `webhook_max_body_bytes` default `262144` (256KB) — payload-urile Meta inbound sunt mici; 256KB e generos.
- **Web** (`/web/chat`, `/web/messages`, `/web/bootstrap`): `web_max_body_bytes` default `16384` (16KB) — `message`/`text` sunt deja capate la 2000 caractere în Pydantic (`src/web/app.py:54,65`), 16KB acoperă overhead-ul JSON.

Aplicare:
- `src/webhook/app.py:83` — `await enforce_body_cap(request, settings.webhook_max_body_bytes)` ÎNAINTE de `request.body()`/verify (returnezi `raw` din helper ca să nu citești de două ori); idem pe `/webhook/orders/{business_id}` (`:110`).
- `src/web/app.py` — pe `/web/chat` (`:189`), `/web/messages` (`:107`). FastAPI deja validează corpul prin `WebChatIn`/`WebMessageIn`, deci aici e suficient gard pe `Content-Length` la intrarea în handler (parametrul `request: Request` există deja).

**Defense in depth (Traefik):** documentezi în `DEPLOY.md` un middleware `buffering` cu `maxRequestBodyBytes` (256KB webhook / 16KB web) — gardul de aplicație rămâne sursa de adevăr (un `curl` direct la container ocolește Traefik), dar Traefik oprește pachetele mari la margine.

### 2. Fail-CLOSED pe calea care cheltuie LLM (`/web/chat`)
`web_rate_limited` (`src/web/app.py:86`) azi propagă `RedisError` (fail-open implicit). Schimbare:
- semnătură nouă cu intenție explicită: `web_rate_limited(redis, token, ip, visitor_id, *, fail_closed: bool) -> bool`. La `RedisError`/`OSError`:
  - `fail_closed=True` (folosit de `/web/chat`) → return `True` (tratat ca „limitat" → 429);
  - `fail_closed=False` (folosit de `/web/messages`, care doar pune un envelope pe stream, fără spend imediat — spend-ul se evaluează în worker) → return `False` (fail-open păstrat, comportament curent).
- `/web/chat` (`:201`): `if await web_rate_limited(..., fail_closed=True): raise HTTPException(429)`.
- `/web/messages` (`:116`): `fail_closed=False` (explicit, documentat).

**Cost-cap per token ÎNAINTE de pipeline** pe `/web/chat`: înainte de `handle_turn(..., deliver=False)` (`:222`) verifici `cost_over_budget(redis, business_id, cap)` (refolosești `src/worker/limits.py:53`) → la depășire `429` cu mesaj generic (NU rulezi pipeline-ul, NU cheltui). Plus un cost-cap **per-vizitator** zilnic (`webcost:{business_id}:{visitor_id}:{yyyymmdd}`) ca un singur token furat să nu poată epuiza tot `daily_cost_cap_usd`-ul tenantului. La Redis error pe acest check: fail-CLOSED (429), consecvent cu rate-limit-ul. (Convergența cost-guard-ului per-turn precis e NX-125; aici e gardul de admitere ieftin pe suprafața expusă.)

### 3. Prefix de tenant pe cheile SSE (P7)
`out_channel`/`backlog_key` (`src/channels/web/sender.py:20-25`) primesc `business_id` (sau token-ul, care mapează 1:1 la tenant) și emit `web:out:{business_id}:{visitor_id}` / `web:backlog:{business_id}:{visitor_id}`. Apelanții: `WebSender.send_text` (`:43-47`), `src/web/app.py:_replay_after` (`:238`) și subscribe-ul SSE (`:268,281`). Token-ul e deja disponibil în toate (sesiune verificată / outbox payload). Migrare blândă: chei vechi expiră singure (backlog TTL 300s), pubsub e efemer — fără migrare de date.

### 4. Bind bootstrap la CORS allowlist server-side
`/web/bootstrap` (`:96`) emite o sesiune fără a verifica `Origin`. CORS-ul de browser (`src/webhook/app.py:152-157`) blochează doar CITIREA răspunsului de către JS cross-origin, nu procesarea pe server (un bot ignoră CORS). Adaugi verificare server-side a `Origin` contra `web_cors_origins_list`: dacă header-ul `Origin` e prezent ȘI nu e în allowlist → `403`. (Origin absent = apel non-browser/same-origin → nu blochezi, ca să nu rupi testele/curl-ul de health; suprafața reală de abuz e browser-driven.) Verificarea fină origin↔token per canal rămâne NX-25.

### Channel-aware
Suprafața web (`/web/chat`) e singura ajunsă din browser doar cu token public → cap strâns (16KB) + fail-CLOSED + cost-cap per-vizitator. WhatsApp/Meta are corpuri mici și e gardat de semnătura Meta → cap mai larg (256KB) și fail-open pe Redis păstrat (Meta face retry, NX-51 deduplică). Diferența nu apare în pipeline (agnostic de canal) — trăiește la margine (P5/canale).

## Principii CLAUDE.md aplicabile
- **P5 (single exit / outbox):** SSE-ul doar retransmite ce a publicat dispatcher-ul; prefixul de tenant pe chei NU mută punctul de ieșire, doar îl namespace-uiește corect.
- **P6 (never silent):** la 413/429 clientul primește un răspuns explicit (nu tăcere, nu 500 opac); webhook-ul Meta rămâne pe contractul lui de retry.
- **P7 (business_id everywhere):** cheile SSE/backlog + cost-cap per-vizitator includ `business_id`; cap-ul de body protejează un VPS partajat de toți tenanții.
- **P2 (LLM doar triaj+agent):** tot ce adaugi (cap, rate-limit, cost-check, origin-check) e cod pur determinist la margine — zero LLM. Cost-cap-ul OPREȘTE LLM-ul înainte să pornească.
- **P12 (PII un singur loc):** la 429/413 NU loghezi token/IP/visitor_id în clar (visitor_id e id de canal); păstrezi convenția `webrl:*`/`webcost:*` fără PII (ca azi în `src/worker/limits.py`).

## Test Cases
**Happy:**
- POST mic valid pe `/web/chat` cu Redis OK sub praguri → 200 + `{content, products, suggestions}`.
- Webhook Meta semnat, body mic → 200 (neschimbat).
- SSE: publish prin WebSender pe `web:out:{business_id}:{visitor_id}` → subscriberul cu același token primește evenimentul; replay backlog după Last-Event-ID funcționează cu prefixul nou.

**Edge:**
- Body exact la `max_bytes` → trece; la `max_bytes + 1` → 413.
- `Content-Length` mincinos (declară mic, trimite chunked mare) → stream-limit prinde → 413.
- `/web/messages` peste rate-limit cu Redis OK → 429 (comportament curent neschimbat).
- `Origin` prezent și allowlistat la `/web/bootstrap` → 200; same-origin (fără Origin) → 200.
- Cost-cap per-vizitator atins de un visitor, dar altul sub cap → primul 429, al doilea 200.

**Failure:**
- `web_rate_limited(fail_closed=True)` cu `FakeRedis` care aruncă `RedisError` → returnează `True` → `/web/chat` 429, `handle_turn` neapelat (mock assert).
- `web_rate_limited(fail_closed=False)` cu `RedisError` → returnează `False` → `/web/messages` 202/200.
- `cost_over_budget` aruncă `RedisError` pe `/web/chat` → 429 (fail-CLOSED), pipeline neatins.
- Webhook cu body uriaș și semnătură INVALIDĂ → 413 (nu 403): respins înainte de verificarea semnăturii, fără bufferizare completă.
