# G5b-2 — Semantic cache: invalidare + caching recomandări de produs (dynamic)
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G5b-1 · **Branch:** `feat/G5b-2-cache-invalidation` · **Complexitate:** M · **Estimare:** 4h

## Goal
Deblochează caching-ul tierului `dynamic` (recomandări de produs — apeluri mini,
scumpe) **în siguranță**, prin invalidare la schimbarea datelor. Mecanismul central:
**retrieval-signature cu snapshot de preț** — un răspuns cu produse stochează prețurile
care l-au fundamentat; la fiecare hit re-validăm prețul curent, iar la orice diferență
regenerăm. „Scade un preț mâine → următorul lookup resetează automat acel cache."
Niciodată un preț învechit servit (respectă invariantul validatorului).

## Business Context
Aici se taie apelurile SCUMPE (agent mini). Recomandările de produs repetate
(„cremă pentru ten uscat sub 80 lei") în fereastra de preț stabil se servesc din cache;
când prețul/oferta se schimbă, cache-ul se invalidează singur. Asta face economia reală
pe un bot de vânzări (vezi `docs/semantic-cache-design.md` §7) FĂRĂ risc reputațional.

## Technical Description

### 1. Schema (migrare 007)
```sql
alter table semantic_cache
  add column retrieval_signature jsonb,   -- [{product_id, price}] care a fundamentat
  add column data_version integer;        -- versiunea de date a businessului la scriere
alter table businesses add column data_version integer not null default 1;
```

### 2. Write-back pentru `dynamic` (extinde G5b-1)
Când `ctx.reply.products` e setat (recomandare de produs) ȘI
`classify_volatility(body)=='dynamic'` ȘI răspunsul a trecut gate-ul:
- `retrieval_signature` = `[{product_id, price} for p in ctx.reply.products]`
  (au deja id+preț din R2).
- `data_version` = `businesses.data_version` curent.
- `volatility_class='dynamic'`, `expires_at = now() + TTL_dynamic` (config, default
  30 min — backstop scurt; invalidarea reală e price-check + version).
- upsert idempotent pe `(business_id, canonical_hash)`.

### 3. Lookup `dynamic` cu price-check (self-healing, PRIMAR)
`cache_stage` servește acum și `dynamic`. Pe un candidat de hit dynamic (L1 sau L2),
ÎNAINTE de a servi:
```python
sig = entry["retrieval_signature"]                       # [{product_id, price}]
current = await current_prices(conn, business_id, [s["product_id"] for s in sig])
if entry["data_version"] != business_data_version or \
   any(current.get(s["product_id"]) != s["price"] for s in sig):
    await delete_entry(conn, business_id, entry["id"])   # învechit → purjă lazy
    emit cache_lookup{layer:'stale_evict'}               # tratează ca MISS
    # pipeline continuă (regenerează cu preț proaspăt)
else:
    serve (ctx.reply + from_cache); emit cache_lookup{layer, similarity}
```
`current_prices(conn, business_id, product_ids)` = SELECT prețul curent (min variant,
ca în `search_products`) pe id-urile date, `where business_id = $1`. Cost: o citire mică
indexată, doar pe calea de hit dynamic.

### 4. data_version (plasă ieftină, bulk)
- `bump_data_version(conn, business_id)` → `update businesses set data_version =
  data_version + 1 where id = $1`. Apelat de jobul de sync de catalog la finalul unei
  rulări (`catalog_sync_runs`); dacă sync-ul nu e încă wired, expus și ca utilitar
  admin (`scripts/bump_cache_version.py`) pentru intervenții manuale de preț în masă.
- Lookup-urile `dynamic` compară `data_version` cu cel curent (din `businesses`, citit
  o dată per tur). Bump → toate entry-urile dynamic vechi devin instant inaccesibile.
- Entry-urile `static` IGNORĂ `data_version` (politica de retur nu se schimbă la un sync
  de prețuri).

### 5. Purjă țintită
`db/queries/semantic_cache.py`:
- `purge_business(conn, business_id)` — offboarding / reset complet.
- `purge_by_product(conn, business_id, product_id)` — șterge entry-urile al căror
  `retrieval_signature` conține product_id (`where retrieval_signature @> ...`). Pentru
  o intervenție pe un singur produs (ex. scos din stoc).

### Câmpuri TurnContext scrise aici
Niciun câmp nou (reutilizează `ctx.from_cache` din G5b-1). `data_version` business e
citit în cache_stage, nu pus pe ctx.

## Test Cases
**Happy Path:**
1. Scrie entry dynamic cu `retrieval_signature=[{p1,49.9}]`; `current_prices` întoarce
   `{p1:49.9}` → hit servit.
2. `bump_data_version` incrementează; lookup dynamic cu `data_version` vechi → miss.

**Edge Cases:**
1. Un produs din semnătură a dispărut din catalog (`current_prices` fără el) → mismatch
   → `stale_evict` (nu servește un produs inexistent).
2. Entry `static` + bump de data_version → încă servit (static ignoră data_version).

**Failure Cases:**
1. `current_prices` aruncă (DB) → tratează ca miss (fail-closed, nu servește nevalidat),
   pipeline continuă.
2. `retrieval_signature` gol/None pe un entry marcat dynamic (corupt) → miss + evict.

> Teste: `current_prices`/price-check cu DB real (scrie produs, schimbă preț, verifică
> evict); restul cu monkeypatch. ZERO apeluri de generare.

## Cost
Generare: **ZERO** adăugat; ȚINTA e să REDUCĂ apelurile mini (cele scumpe) pe
recomandări repetate. Price-check = o citire mică pe calea de hit dynamic. Economia
netă = hit_rate_dynamic × cost_agent_mini − cost_pricecheck (neglijabil). Aici se
materializează ținta de 40-60% generare-call reduction, măsurată cu instrumentarea din G5b-1.
