# NX-01 — Cloudflare în fața webhook-ului (varianta pragmatică a „WAF + API Gateway")
**Owner:** S · **Faza:** MVP · **Zi:** Z13 · **Branch:** — (infrastructură) · **Complexitate:** XS · **Estimare:** 3h

## Goal
Tot traficul public trece prin Cloudflare: TLS, WAF managed rules, rate limit pe /webhook. Accesul direct pe IP-ul VPS-ului e închis.

## Business Context
S3 din audit: cutia „WAF + API Gateway" din diagrama master, implementată azi cu 2-3 ore de configurare și 0 lei, nu cu un produs enterprise. Elimină clasa întreagă de atacuri direct-pe-IP (scanere, flood pe webhook, exploit-uri pe Caddy) înainte de primul client real.

## Technical Description
Domeniul intră în Cloudflare (free tier ajunge): DNS A record proxied (nor portocaliu), SSL/TLS mode **Full (strict)** (Caddy are deja cert valid), WAF managed rules ON, o regulă de rate limiting pe `POST /webhook` (ex. 300 req/min per IP — Meta trimite mult sub asta per IP), Bot Fight Mode OFF (poate atinge webhook-urile legitime Meta). Pe VPS: `ufw` allow 443 DOAR de pe range-urile publicate de Cloudflare (https://www.cloudflare.com/ips/), allow SSH de pe IP-ul vostru, deny rest. În Caddy: `trusted_proxies` pe aceleași range-uri ca să logezi IP-ul real al clientului (header CF-Connecting-IP), altfel rate-limit-urile aplicației văd doar IP-uri Cloudflare.

## Principii CLAUDE.md aplicabile
P7 (suprafață de atac minimă), P6 (degradare controlată — CF down e mai rar decât VPS-ul atacat).

## Implementation Steps
1. Adaugă domeniul în Cloudflare, mută nameserverele, A record proxied către VPS.
2. SSL/TLS → Full (strict); verifică handshake-ul cap-coadă.
3. WAF → managed rules ON; Security → rate limiting rule pe /webhook.
4. VPS: script `infra/ufw_cloudflare.sh` care descarcă range-urile CF și reconstruiește regulile ufw (rulabil la update-urile lor de IP-uri).
5. Caddyfile: `trusted_proxies` cu range-urile CF; loghează `CF-Connecting-IP`.
6. Re-verifică webhook-ul Meta end-to-end (semnătura X-Hub-Signature-256 NU e afectată de proxy — body-ul trece neatins).
