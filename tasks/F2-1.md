# F2-1 — Bucla de bani: tool `checkout_link` (link de cumpărare atribuibil)
**Owner:** S · **Faza:** P1 · **Zi/Ord:** pas 4 din secvențierea de producție (docs/agent-tools-architecture.md §9) — prima felie a buclei de bani · **Branch:** `feat/F2-1-checkout-link` · **Complexitate:** M · **Estimare:** ~5h

## Goal
Primul tool de **scriere** al agentului (Faza 2 din epicul tool-uri): `checkout_link(cart_items)`
construiește un link de cumpărare cu `?ref=<ref_code>`, scrie un rând în `checkout_links`
(ref_code unic per business) și întoarce URL-ul ca să-l trimită agentul. `ref_code`-ul e
ancora de **atribuire** — webhookul de comenzi (felia următoare) va face match pe el. Validatorul
(stagiul 8) acceptă linkul generat de bot (grounded prin construcție).

## Business Context
„Assisted revenue" = inima modelului de business. Bucla: conversație → recomandare → **link de
checkout (?ref=)** → comandă atribuită. Această felie livrează capătul din care pleacă
atribuirea: botul poate acum trimite un link de cumpărare real, persistat cu `ref_code`. Închiderea
buclei (webhook comenzi → match ref_code → `orders.attribution`) + `usage_daily` (revenue_attributed)
vin în feliile F2-2/F2-3.

## Technical Description

### Tool (`src/tools/commerce_tools.py`) — primul WRITE tool
- `checkout_link(cart_items: list[CartItem])`, `CartItem = {product_id, variant_id?, quantity=1}`,
  min 1 item (validare Pydantic, ca la catalog_tools).
- **Validează produsele** contra catalogului (`get_products_by_ids`, scoped pe `ctx.business.id`):
  produse inexistente sunt aruncate. Zero produse valide → `ToolResult(ok=False)` (agentul
  recomandă fără link, degradare grațioasă).
- **Cart snapshot** (jsonb, ref-uri + preț la momentul creării, principiul 8):
  `[{product_id, variant_id, name, price, quantity}]`. `total` = Σ price×qty.
- **Base URL**: `ctx.business.settings["checkout_url"]` → fallback `settings.checkout_base_url`.
  Niciunul setat → `ok=False` (checkout indisponibil; nu inventăm domeniu). `url = base + (?|&) +
  ref=<ref_code>`.
- **ref_code = `ctx.turn_id`** → idempotent per tur (re-run = ON CONFLICT). `expires_at = now +
  checkout_link_ttl_days`.
- `ToolResult(ok=True, products=<cart products>, links=[url], llm_view="Link creat: <url> | …")`.
  `links` (câmp NOU pe ToolResult) = linkurile generate de bot, pe care validatorul le acceptă.
- Emite `checkout_link_created` {items, value} (fără PII) — semnal pentru `usage_daily` (F2-3).

### Query (`src/db/queries/commerce.py`)
- `create_checkout_link(conn, business_id, conversation_id, contact_id, ref_code, cart, url,
  expires_at) -> dict` — insert idempotent: `on conflict (business_id, ref_code) do update set
  cart=excluded.cart, url=excluded.url, expires_at=excluded.expires_at returning id, ref_code, url`.
  Toate query-urile cu `business_id` (principiul 7).

### Validator (stagiul 8, `agent.py`) — acceptă linkul botului
- `ToolResult.links` se acumulează în `agent_stage` într-un set `generated_links` (ca `retrieved`).
- `_links_ok(reply, products, allowed_links)` extins: un URL din reply e valid dacă e `product_url`
  retrievat **SAU** ∈ `generated_links`. `_valid`/`_finalize` primesc `allowed_links`. Un link de
  checkout generat de bot e grounded prin construcție → nu mai e respins ca „inventat".

### Scheme + activare
- `src/agent/tool_definitions.py`: schema OpenAI `checkout_link` (strict, `cart_items` array de
  obiecte cu `product_id`/`variant_id`/`quantity`).
- `src/tools/base.py`: `_PHASE2 = ("checkout_link",)`; `enabled_tools` întoarce Faza 1 + Faza 2
  (per-business gating = ulterior).
- `agent.py` `_TOOL_SYSTEM`: o linie despre `checkout_link` — „când clientul e gata de cumpărare /
  cere linkul, cheamă checkout_link cu produsele alese".

### Config (`src/config.py` + `.env.example`)
- `checkout_base_url` (`CHECKOUT_BASE_URL`, default `""`).
- `checkout_link_ttl_days` (`CHECKOUT_LINK_TTL_DAYS`, default 7).

## Test Cases
**Happy Path:**
1. FakeLLM scriptează `checkout_link([{product_id:p1}])` → reply cu URL `?ref=` → `ctx.reply`
   conține linkul, validatorul îl acceptă, `checkout_links` scris (conn mock).
2. Tool direct: 2 items valide → cart snapshot cu preț + total, url construit din base.

**Edge Cases:**
1. Un id valid + unul inexistent → cart doar cu cel valid, `ok=True`.
2. `quantity` implicit 1; `variant_id` opțional null.
3. A doua rulare cu același `turn_id` → ON CONFLICT update (un singur rând).

**Failure Cases:**
1. Zero produse valide → `ok=False`, `llm_view` explică; agentul recomandă fără link.
2. Base URL absent (settings + business) → `ok=False`.
3. Reply cu URL inventat (non-checkout, non-product) → validator respinge → retry → fallback.

> ZERO apeluri reale: tool cu `conn`/`llm` mock (ca `test_agent`/`test_tools`); validatorul testat
> pe funcții pure. Niciun `@pytest.mark.integration`.
