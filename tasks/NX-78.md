# NX-78 — prompt_builder din categories + OpenAI prompt caching
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7 (agent tool-loop) + NX-15 (adaptor) · **Branch:** `feat/NX-78-prompt-builder` · **Complexitate:** M · **Estimare:** 4h

## Goal
Scoate verticalul hardcodat „beauty" din agent (`agent.py:43-67`) și generează system
prompt-ul agentului PER BUSINESS din `categories` (+ `intent_aliases` aprobate pentru
groundarea rutării), exact cum triajul deja se groundează pe `list_category_slugs`.
Promptul are un **prefix STATIC byte-identic per (business, locale)** ca să se prindă
prompt caching-ul OpenAI (cache automat pe prefix ≥1024 tokens → 50% reducere pe
tokenii input cache-uiți). Fără asta încălcăm principiul 9 (prompt din DB, nu hardcodat)
și plătim full price pe fiecare apel al agentului.

## Business Context
Suntem agenție multi-tenant: un magazin HVAC sau auto NU vinde beauty. Cu prompt
hardcodat „consultant într-un magazin de beauty", agentul minte verticalul oricărui
client non-beauty din ziua 1 — blocant pentru onboarding. Generarea din `categories`
face botul corect pe orice vertical fără cod nou. Prompt caching-ul taie costul agentului
(mini) cu ~50% pe partea de prefix repetat (system + tool-uri), care e cea mai mare și mai
stabilă bucată din fiecare apel — bani direcți pe retainerul lunar.

## Technical Description

### Starea actuală (de înlocuit)
`src/worker/stages/agent.py` are 3 prompturi hardcodate ca string literali la nivel de modul:
- `_TOOL_SYSTEM` (l.43-60) — „magazin de beauty online", listă de tool-uri în text liber.
- `_RECO_SYSTEM` (l.63-67) — retry de recompunere, tot „beauty".
- `_ORDER_RECO_SYSTEM` (l.210-212) — suport status comandă, neutru pe vertical (rămâne, dar mutat).

`agent_stage` le folosește direct (`run_tool_loop(_TOOL_SYSTEM, ...)`, `_finalize(... _RECO_SYSTEM)`).
Niciunul nu depinde de business → cache-friendly DAR greșit pe vertical. Triajul deja
citește `list_category_slugs(deps.conn, ctx.business.id)` — copiem pattern-ul, extins.

### Ținta: `src/agent/prompt_builder.py`
Modul pur (fără I/O LLM, fără scriere DB — doar citire catalog + formatare string).
Trei funcții, fiecare întoarce un string DETERMINIST pentru un (business, locale) dat:

```python
# src/agent/prompt_builder.py
from dataclasses import dataclass

@dataclass(frozen=True)
class PromptInputs:
    """Datele din care se compune promptul. Toate din DB, scoped pe business_id.
    Sortate determinist → același business+locale ⇒ prefix BYTE-IDENTIC (prompt caching)."""
    business_name: str
    vertical: str
    locale: str
    categories: list[str]          # nume categorii top-level, sortate alfabetic
    aliases: list[tuple[str, str]] # (phrase_norm, target) aprobate, sortate — hint de rutare

def build_agent_system(inp: PromptInputs) -> str:
    """System prompt pentru bucla de tool-calling (înlocuiește _TOOL_SYSTEM).
    STATIC pentru un (business, locale): NU conține mesajul/clientul/produsele (alea
    stau în USER, ca acum). Verticalul + categoriile vin din DB, nu hardcodate."""

def build_reco_system(inp: PromptInputs) -> str:
    """System de recompunere/retry (înlocuiește _RECO_SYSTEM), tot static per business."""

ORDER_RECO_SYSTEM: str  # neutru pe vertical → constantă, mutată din agent.py
```

`build_agent_system` produce ceva de forma (verticalul + categoriile injectate, restul fix):
```
Ești consultant de vânzări pentru {business_name}, un magazin online de {vertical} din România.
Vinzi din aceste categorii: {", ".join(categories)}.
Ai unelte ca să răspunzi GROUNDED pe catalogul real:
- search_products(...)  [... blocul de tool-uri rămâne IDENTIC, e parte din prefixul static ...]
Reguli: [... identice cu azi: max 3 apeluri, preț EXACT, nu inventa, text simplu ...]
```
Dacă `categories` e gol (client neonboardat) → omitem linia „Vinzi din aceste categorii"
(fallback la „produsele din catalog"), NU băgăm „beauty". `aliases` (max ~20, doar cele
`approved`) se pun ca hint scurt de rutare DOAR dacă există; gol → linia lipsește.

### Determinism = condiția prompt caching-ului
OpenAI cache-uiește AUTOMAT prefixul exact (≥1024 tokens) — NU există parametru
`cache_control` (asta e Anthropic). Singura pârghie e ca system+tools să fie **byte-identice**
între apeluri. Deci:
- `categories`/`aliases` se sortează DETERMINIST (alfabetic) înainte de format — altfel
  ordinea aleatoare din DB sparge cache-ul.
- Tot ce e dinamic (mesaj client, istoric, profil, produse) stă în mesajul USER, NU în system.
  `context.py:88` deja documentează exact regula asta („promptul static rămâne byte-identic").
- Tool schemas (`tool_schemas(...)`) sunt deja deterministe per rută → fac parte din prefix.
- Cache micro-optim: memoizăm `PromptInputs → str` per proces (`functools.lru_cache` pe un
  tuple hashable) ca să nu reformatăm la fiecare tur; invalidare = restart proces (acceptabil,
  categoriile se schimbă rar). NU cache cross-proces (ar fi peste scope).

### Încărcarea inputurilor (query nou, scoped)
`prompt_builder` NU atinge DB direct (modul pur). `agent_stage` încarcă inputurile printr-un
query nou în `src/db/queries/catalog.py`, pe lângă `list_category_slugs`:
```python
async def list_category_names(conn, business_id: str) -> list[str]:
    rows = await conn.fetch(
        "select name from categories "
        "where business_id = $1 and parent_id is null "  # top-level pt promptul agentului
        "order by name",
        business_id,
    )
    return [r["name"] for r in rows]
```
Aliasele de rutare (opțional, hint scurt) din `intent_aliases`:
```python
async def list_routing_aliases(conn, business_id: str, *, limit: int = 20) -> list[tuple[str, str]]:
    rows = await conn.fetch(
        "select phrase_norm, coalesce(target_value, '') as target "
        "from intent_aliases "
        "where business_id = $1 and status = 'approved' "  # DOAR aprobate (P9)
        "order by phrase_norm limit $2",
        business_id, limit,
    )
    return [(r["phrase_norm"], r["target"]) for r in rows]
```
Ambele tenant-scoped (`business_id = $1`), `conn` deja tenant_conn (RLS plasa). NB:
`categories`/`intent_aliases` NU au `locale` în schema_v2 → numele sunt language-neutral;
`locale` intră în prompt prin instrucțiunea „răspunde în limba clientului" + cheia de cache.

### Integrare în pipeline (`agent_stage`)
- Înlocuim `_TOOL_SYSTEM` literal cu `build_agent_system(inp)` și `_RECO_SYSTEM` cu
  `build_reco_system(inp)`; `_ORDER_RECO_SYSTEM` → `prompt_builder.ORDER_RECO_SYSTEM`.
- `inp` se construiește o dată la începutul `agent_stage` (după ce știm `ctx.business`,
  `ctx.language`), din cele două query-uri noi + `ctx.business.name/vertical`.
- Cheia de memoizare = `(business_id, locale)` → prefix stabil per tenant+limbă.

### Câmpuri TurnContext scrise aici
NICIUNUL nou. `agent_stage` rămâne owner-ul lui `ctx.retrieval` + `ctx.reply` (via
`ctx.set_reply`) + `ctx.emit` ca azi. `prompt_builder` e pur (nu atinge `ctx`). Promptul
e un INPUT către LLM, nu un câmp de context.

### Error handling / degradare
- Query categorii eșuat → `build_agent_system` cu `categories=[]` (fallback generic, nu crap).
  `agent_stage` prinde excepția de DB ca azi (bucla e în `try/except` → echo fallback, P6).
- `deps.llm is None` → agentul oricum face no-op înainte să atingă promptul (neschimbat).

- **P7 (business_id pe tot):** ambele query-uri noi au `where business_id = $1`; RLS plasa.

## Cost (per 1000 mesaje către agent)
Taskul NU adaugă apeluri LLM noi — doar schimbă CE prompt primește agentul (mini) existent.
Efectul e o REDUCERE de cost: prefixul system+tools (estimat ~700-1100 tokens, repetat identic
per tenant+locale) devine cache-eligibil. OpenAI aplică ~50% reducere pe tokenii input
cache-uiți (cache automat la prefix ≥1024 tokens, fără parametru — diferit de Anthropic).
La 1000 mesaje sales/agent cu prefix ~1000 tokens cache-uit: economie ≈ 1000 × 1000 × 0.5
× (preț input mini / 1M tokens) pe partea de prefix repetat — bani recuperați fără apeluri în plus.
**Ce limitează / asigură cache-ul:** prefixul byte-identic (sortare determinist + dinamicul în USER);
dacă cineva băgă context dinamic în system, cache-ul cade la 0 → testul de determinism e garda.
