# NX-180 — Evaluator conversațional + baseline (Track 0)

**Owner:** Claude (build) / Codex (verify) · **Faza:** P0 · **Track:** Response Quality · **Branch:** `feat/NX-180-eval-harness` · **Complexitate:** M · **Estimare:** 2-3 zile
**Parent plan:** [docs/RESPONSE-QUALITY-EPIC.md](../docs/RESPONSE-QUALITY-EPIC.md) · **Depinde de:** — (independent; prerechizit pt NX-181..189)

## Goal
Construiește **rigla** înainte de orice schimbare de comportament: un harness care rulează 10-12 conversații reprezentative, fiecare de **3 ori**, prin calea web REALĂ (`/web/chat`), și produce un artefact **baseline** cu scoruri deterministe + judge LLM. Fără el, orice schimbare ulterioară dovedește doar că am schimbat, nu că am îmbunătățit.

## Business Context
Golden-ul actual (`tests/golden/cases.json`, 55 must_include/forbidden peste LLM SCRIPTAT) nu poate măsura naturalețe, repetiție sau „a răspuns la ce a întrebat". Toate deciziile de mai jos (Prompt vNext, V2-light) se gate-uiesc pe acest evaluator. Extindem la ~20 conversații înainte de rollout larg.

## Technical Description
- **Set de conversații** (`qa-suite/conversations/*.json`): 10-12 scenarii × 2-5 tururi — direct-match, follow-up „dintre ele care e mai X", detail 1 produs, refine (constrângere nouă), cheaper, mixed-intent (produs + livrare), no-exact, repeat, brand absent, contraindicație.
- **Runner** (`scripts/sim/eval_run.py`): rulează pe pipeline real (`/web/chat` + fakeredis, ca `web_audit.py`), 3 rulări/caz, model + catalog fixate.
- **Gate deterministe** (per tur): rută/hint așteptat, produse permise/interzise, respectarea constrângerilor hard, prețuri/linkuri grounded, zero reply gol, card-count potrivit, zero repetare de carduri la follow-up direct, safety.
- **Judge LLM secundar** (1-5): răspunde la întrebare, folosește contextul, natural, non-repetitiv, concis, recunoaște incertitudinea. Judge-ul **nu poate anula** un fail determinist.
- **Agregare:** mediana peste cele 3 rulări + varianța (semnal de instabilitate). Output = `reports/eval-baseline-<data>.json` (fără PII: doar ID-uri normalizate, scoruri, tokens, latență).

## Principii CLAUDE.md aplicabile
- **P12 (fără PII):** raportul stochează doar `response_mode`, `query_fingerprint_hash`, scoruri, tokens, latență — nu text brut / telefon / conținut istoric.
- **Web-only (NX-179):** calea = `/web/chat` real, NU `sim/server.py`.

## Implementation Steps

## Definition of Done
- [ ] 10-12 conversații × 3 rulări **se FINALIZEAZĂ** pe pipeline real (harness-ul rulează la capăt; NU „verde" — baseline-ul ÎNREGISTREAZĂ inclusiv problemele actuale, altfel forțăm concluzia „sistemul e bun").
- [ ] Raport baseline generat: per-caz naturalness (judge 1-5), follow-up correct (det), card-repeat (det), grounding pass (det), tokens + latență **p95 PER TUR** (separat de joburile post-tur), varianță între rulări.
- [ ] Zero PII în raport (doar ID-uri + scoruri normalizate); transcript **redactat** pentru auditarea verdictului judge.
- [ ] `ruff` + `pytest -x -q` verzi.

## Reproductibilitate & măsurare (cerințe obligatorii)
- **Reproductibilitate fixată:** hash catalog + versiune model + versiune/hash prompt judge; **cache OPRIT + state RESETAT** între rulări; comparație **paired ON/OFF intercalată**.
- **Denominator explicit:** definește scor **per tur** vs per conversație (declară care).
- **Prag ≥95% follow-up:** setul trebuie să conțină **destule tururi follow-up** ca pragul să fie măsurabil — un singur scenariu × 3 nu poate valida 95%. Scalează numărul de tururi follow-up (nu doar conversații) până la un denominator util.
- Baseline = fotografia realității (incl. eșecuri), NU o poartă verde.

## Test Cases
**Happy:** rularea produce un JSON baseline cu scoruri per caz + mediană.
**Failure:** un caz cu preț inventat scriptat → gate determinist îl marchează FAIL, judge-ul nu-l poate „salva".
