# NX-42 — pip-audit + renovate + digest pin
**Owner:** S · **Faza:** P1-O3 · **Zi/Ord:** după T005 (CI test verde) · **Branch:** `chore/NX-42-supply-chain` · **Complexitate:** S · **Estimare:** 4h

## Goal
Închide suprafața de supply-chain: (1) un job nou `audit` în CI rulează `pip-audit`
pe lockfile și **pică pe orice CVE de severitate high/critical**, deci nicio
dependență vulnerabilă cunoscută nu intră în main; (2) `renovate.json` deschide PR-uri
automate de bump pe `requirements*.txt` + GitHub Actions, ca să nu rămânem pe versiuni
vechi din inerție; (3) imaginile de bază din `Dockerfile` (și `redis` din compose) sunt
referite prin **digest sha256 imutabil**, nu prin tag mutabil — build-ul de prod e
reproductibil și nu poate fi schimbat sub noi de un repush al tag-ului `python:3.12-slim`.

## Business Context
Botul rulează pe VPS în containere care vorbesc cu DB-ul clientului, cu cheia OpenAI
și cu token-urile Meta. Un pachet tranzitiv vulnerabil sau un tag de bază reotrăvit
(`python:3.12-slim` re-pushed) e exact vectorul prin care un atacator ajunge la datele
mai multor tenanți deodată. Hardening-ul de supply-chain e ieftin (rulează în CI, $0)
și e o cerință de bază pentru orice client enterprise care cere un security review
înainte de semnătură. Renovate ține fereastra de patch mică fără muncă manuală.

## Technical Description

### 1) Job CI `audit` (pip-audit pe lockfile, fail pe high)
Job nou în `.github/workflows/ci.yml`, paralel cu `lint`/`test` (nu blochează DB).
`pip-audit` cere un set de versiuni rezolvat; le avem deja pinuite exact în
`requirements.txt` + `requirements-dev.txt`, deci auditem fix ce ajunge în imagine:

```yaml
  audit:
    name: Supply-chain audit (pip-audit)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
          cache: pip
      - run: pip install pip-audit==2.9.0
      # Auditează EXACT versiunile pinuite (runtime + dev), nu mediul rezolvat live.
      # --strict: orice avertisment de rezolvare (dep ne-pinuită) = fail, nu warn.
      # Ignorăm DOAR CVE-uri triate explicit, cu motiv, în .pip-audit.toml.
      - run: pip-audit -r requirements.txt -r requirements-dev.txt --strict --desc
```

`pip-audit` întoarce **exit code != 0 când găsește orice vulnerabilitate** → jobul
pică automat. Nu există „severity high" nativ în `pip-audit` (PyPI Advisory DB nu
scorează uniform CVSS), deci poarta noastră e „**orice vulnerabilitate cunoscută →
fail**", cu o supapă de triaj explicită: CVE-urile fără patch disponibil sau
neaplicabile (ex. vulnerabilitate doar pe Windows, noi rulăm Linux) se trec în
`.pip-audit.toml` cu `ignore-vulns` + comentariu cu motiv și data revizuirii. Asta e
mai strict decât „doar high" și evită debate-ul de scoring; lista de ignore e
audit-abilă în PR.

```toml
# .pip-audit.toml — supapă de triaj. Fiecare intrare = CVE conștient acceptat.
[tool.pip-audit]
strict = true
[tool.pip-audit.ignore]
# GHSA-xxxx-... = motiv (ex. „no fix yet, not in our call path"), revizuit 2026-06.
# (gol la start)
```

Jobul `audit` se adaugă la **required checks** pe branch protection, ca `lint`/`test`.

### 2) Renovate (bump automat pe deps + actions)
`renovate.json` la rădăcină. Renovate citește `requirements.txt`,
`requirements-dev.txt` (manager `pip_requirements`), pinurile de Action din
`ci.yml` (manager `github-actions`) și pinurile de imagine din `Dockerfile` +
`docker-compose.yml` (manager `dockerfile` / `docker-compose`). Config:

```jsonc
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["config:recommended", ":dependencyDashboard", ":semanticCommits"],
  "timezone": "Europe/Bucharest",
  "schedule": ["before 6am on monday"],   // un val pe săptămână, nu zgomot zilnic
  "prHourlyLimit": 3,
  "prConcurrentLimit": 6,
  "labels": ["dependencies"],
  "rangeStrategy": "pin",                  // păstrăm pin exact, nu range-uri
  "vulnerabilityAlerts": { "labels": ["security"], "schedule": ["at any time"] },
  "lockFileMaintenance": { "enabled": false },
  "packageRules": [
    {
      "description": "Patch+minor pe deps de runtime → auto-merge dacă CI verde",
      "matchManagers": ["pip_requirements"],
      "matchUpdateTypes": ["patch", "minor"],
      "automerge": true,
      "automergeType": "pr"
    },
    {
      "description": "Major → niciodată auto; necesită review uman",
      "matchUpdateTypes": ["major"],
      "automerge": false,
      "addLabels": ["needs-review"]
    },
    {
      "description": "GitHub Actions pinuite pe digest, grupate",
      "matchManagers": ["github-actions"],
      "groupName": "github-actions",
      "pinDigests": true
    },
    {
      "description": "Imagini Docker — pin pe digest, grupate, fără auto-merge",
      "matchManagers": ["dockerfile", "docker-compose"],
      "groupName": "docker base images",
      "pinDigests": true,
      "automerge": false
    }
  ]
}
```

`vulnerabilityAlerts` „at any time" = când GitHub semnalează un advisory pe o
dependență, Renovate deschide PR-ul de fix imediat, în afara ferestrei de luni.
Auto-merge pe patch/minor **se bazează pe CI verde** (jobul `audit` + `test`),
deci un bump care reintroduce o vulnerabilitate sau sparge testele NU se
auto-merge-uiește. Major rămâne manual (poate sparge API). `rangeStrategy: pin`
păstrează pinurile exacte din `requirements.txt` (sursa pentru `audit` și Docker).

> NB operațional: Renovate cere ca **GitHub App „Renovate"** să fie instalat pe
> repo (pas manual, intră în `TODO-MANUAL.md`). Fără app, `renovate.json` e inert —
> nu sparge nimic, dar nici nu lucrează. Self-hosted via Action e alternativă (out of scope).

### 3) Digest pin pe imaginile de bază
`Dockerfile` referă acum `python:3.12-slim` (tag mutabil). Trecem la digest imutabil,
cu tag-ul păstrat în comentariu pentru lizibilitate (Renovate îl re-rezolvă la bump):

```dockerfile
# syntax=docker/dockerfile:1
# python:3.12-slim — pin pe digest (Renovate actualizează la bump). Tag = doar comentariu.
FROM python:3.12-slim@sha256:<DIGEST_REZOLVAT> AS builder
...
FROM python:3.12-slim@sha256:<DIGEST_REZOLVAT> AS runtime
```

La fel pentru `redis:7-alpine` în `docker-compose.yml` →
`redis:7-alpine@sha256:<DIGEST>`. Digestul curent se rezolvă cu
`docker buildx imagetools inspect python:3.12-slim` (sau `docker pull` + `inspect`)
și se pune o singură dată; după aceea Renovate (manager dockerfile/docker-compose,
`pinDigests: true`) îl ține la zi prin PR. Cele două stagii buildER/runtime trebuie
să folosească **același** digest (consistență builder→runtime).

### Câmpuri TurnContext scrise aici
**Niciunul.** Task pur de infra CI/Docker — nu atinge pipeline-ul, worker-ul,
DB-ul aplicației sau `TurnContext`. Zero cod Python de runtime.

### Securitate & PII
`pip-audit` și Renovate citesc DOAR manifeste de pachete — niciun secret, niciun PII.
Output-ul jobului `audit` conține nume de pachete + ID-uri CVE, nimic sensibil →
poate rămâne în logurile publice de CI. GitHub App-ul Renovate primește scope minim
(content + PR), fără acces la secretele de Actions (cheile OpenAI/Meta/DB rămân în
secrets, neexpuse PR-urilor de la fork). Digest pin = protecție contra re-otrăvirii
unui tag de bază (supply-chain integrity).

## Implementation Steps
1. Adaugă jobul `audit` în `.github/workflows/ci.yml` (pip-audit pinuit, `--strict`,
   `-r requirements.txt -r requirements-dev.txt`).
2. Creează `.pip-audit.toml` (supapă de triaj, gol la start, cu comentariu de format).
3. Rezolvă digestul curent pentru `python:3.12-slim` și `redis:7-alpine`
   (`docker buildx imagetools inspect ...`).
4. Editează `Dockerfile`: ambele `FROM` → `python:3.12-slim@sha256:<DIGEST>`
   (același digest pe builder + runtime), tag în comentariu.
5. Editează `docker-compose.yml`: `image: redis:7-alpine@sha256:<DIGEST>`.
6. Creează `renovate.json` (config de mai sus).
7. **Test poarta CVE:** pe un branch throwaway, adaugă un pachet cu CVE high cunoscut
   în `requirements-dev.txt` (ex. o versiune veche notoriu vulnerabilă) → confirmă că
   jobul `audit` pică; apoi scoate-l (NU intră în PR-ul final).
8. Local: `docker compose build` cu digesturile noi trebuie să reușească
   (validează că digesturile sunt corecte și imaginile se trag).
9. `TODO-MANUAL.md`: adaugă „Instalează GitHub App Renovate pe repo" + „Adaugă `audit`
   la required status checks pe branch protection main".
10. PR; verifică în Actions că `lint` + `test` + `audit` rulează toate verzi.

## Test Cases
> Acest task nu rulează cod de aplicație și nu cheamă niciun LLM → „testele" sunt
> verificări de CI/build, nu pytest. ZERO apeluri reale (pip-audit lovește PyPI
> Advisory DB public; e parte din scopul jobului, nu un apel de model/serviciu plătit).

**Happy Path:**
1. PR cu `requirements*.txt` curat → jobul `audit` verde, `renovate-config-validator`
   pasează, `docker compose build` reușește cu digesturile pinuite.
2. Renovate (după instalarea App) deschide un PR de patch pe o dependență → CI verde
   (inclusiv `audit`) → se auto-merge-uiește conform `packageRules`.

**Edge Cases:**
1. CVE fără patch disponibil sau neaplicabil (ex. doar Windows) → se adaugă în
   `.pip-audit.toml` `ignore` cu motiv → jobul `audit` redevine verde, ignore-ul e
   vizibil în diff și audit-abil.
2. Bump major propus de Renovate → NU se auto-merge-uiește, primește label
   `needs-review` (verificat în config + comportamentul PR-ului).
3. Tag-ul `python:3.12-slim` e re-pushed upstream → build-ul nostru rămâne pe digestul
   pinuit (neschimbat), Renovate deschide un PR separat cu noul digest pentru review.

**Failure Cases:**
1. Se introduce deliberat un pachet vulnerabil în `requirements-dev.txt` (branch
   throwaway) → jobul `audit` pică cu exit code != 0, PR-ul e blocat la merge.
2. Digest greșit/inexistent în `Dockerfile` → `docker compose build` eșuează la pull
   („manifest unknown") — prins local înainte de PR.
3. `renovate.json` invalid (typo în `packageRules`) → `renovate-config-validator`
   raportează eroarea înainte de commit (nu așteptăm ca Renovate să tacă în prod).
