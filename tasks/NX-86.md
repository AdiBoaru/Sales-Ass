# NX-86 — XAUTOCLAIM reaper + dead-letter pentru inbound blocat
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după NX-51 (dedupe L2) · **Branch:** `feat/NX-86-inbound-reaper` · **Complexitate:** M · **Estimare:** 4h

## Goal
Închide două găuri de pierdere tăcută pe calea inbound (stagiul 2). (1) Dacă un
consumer moare după `XREADGROUP` dar înainte de `XACK`, entry-ul rămâne pe veci în
PEL (Pending Entries List) — `consume_once` citește doar entries NOI (`>`), deci
mesajul nu mai e procesat niciodată. (2) `claim_inbound` commit-ează up-front (NX-51):
un crash în mijlocul turului marchează `provider_msg_id` ca văzut FĂRĂ a fi produs
vreun reply, iar la retry-ul Meta dedupe-ul L2 îl respinge → clientul nu primește
nimic. Dispatcher-ul de outbox are deja reaper implicit (visibility timeout); acest
task aduce echivalentul pe inbound: XAUTOCLAIM periodic + strategie de recuperare a
claim-urilor orfane (dead-letter).

## Business Context
Un mesaj inbound pierdut = un client ignorat = vânzare ratată + percepția că „botul
e mort". Crash-urile de worker (OOM, deploy, kill) sunt normale în producție pe VPS
cu docker compose; fără reaper, fiecare crash în fereastra greșită îngroapă tăcut
mesaje. Principiul 6 („niciodată tăcere") cere ca un mesaj de client să fie ori
procesat, ori vizibil într-o coadă moartă — niciodată dispărut fără urmă.

## Technical Description

### Problema 1 — entries blocate în PEL (consumer mort între READ și ACK)
`consume_once` (consumer.py:107) face `xreadgroup(..., {STREAM_INBOUND: ">"})` →
doar entries nelivrate niciunui consumer. Un entry deja livrat unui consumer care a
murit înainte de `xack` (consumer.py:130) rămâne „pending" pe acel consumer mort și
nu mai e re-citit. Soluția standard Redis Streams: `XAUTOCLAIM` periodic re-asignează
entries mai vechi decât un `min-idle-time` către consumer-ul viu, care le reprocesează.

```python
# src/worker/consumer.py — reaper de PEL
REAPER_MIN_IDLE_MS = 60_000   # un entry „idle” >60s la un consumer = orfan
REAPER_BATCH = 50

async def reap_pending(pool, redis, consumer_name, debouncer) -> int:
    """Reclamă entries orfane din PEL și le reprocesează. Întoarce câte a tratat.

    XAUTOCLAIM scanează PEL-ul grupului, transferă entries cu idle > min-idle
    către `consumer_name`, le întoarce ca (id, fields) și le procesăm exact ca
    în consume_once (același cod de rutare + ACK în finally)."""
    cursor = "0-0"
    handled = 0
    while True:
        cursor, entries, _deleted = await redis.xautoclaim(
            STREAM_INBOUND, CONSUMER_GROUP, consumer_name,
            min_idle_time=REAPER_MIN_IDLE_MS, start_id=cursor, count=REAPER_BATCH,
        )
        for msg_id, fields in entries:
            try:
                if fields is None:           # entry a fost deja șters din stream → curăță PEL
                    await redis.xack(STREAM_INBOUND, CONSUMER_GROUP, msg_id)
                    continue
                event = json.loads(fields["data"])
                if event.get("kind", "message") == "message":
                    await debouncer.add(event)
                else:
                    await process_event(pool, redis, event)
            except Exception:  # noqa: BLE001 — un entry stricat nu oprește reaper-ul
                log.exception("reaper: eroare la entry %s", msg_id)
            finally:
                await redis.xack(STREAM_INBOUND, CONSUMER_GROUP, msg_id)
            handled += 1
        if cursor == "0-0":   # XAUTOCLAIM a terminat ciclul de scanare
            break
    return handled
```

Reaper-ul rulează pe un interval (NU pe fiecare iterație de buclă — ar bate degeaba
PEL-ul gol). În `run_consumer`: după N cicluri de `consume_once` fără mesaje noi,
sau pe un timer (`last_reap_at`, fiecare ~30s) → `reap_pending`. Idle-ul de 60s e
mai mare decât orice tur normal (LLM + DB ~secunde), deci nu fură entries pe care un
consumer viu le procesează încă.

### Problema 2 — claim orfan (crash DUPĂ claim_inbound, ÎNAINTE de reply)
`claim_inbound` (inbound_dedupe.py:14) face `INSERT ... ON CONFLICT DO NOTHING` care
se commit-ează imediat (processor.py:282). Dacă turul crapă după acest punct (pipeline,
DB, OOM), rândul din `inbound_dedupe` rămâne, dar nu există nici message outbound, nici
outbox. Când reaper-ul (Problema 1) sau retry-ul Meta readuce același `provider_msg_id`,
`claim_inbound` întoarce `False` → mesajul e respins ca „duplicat" deși NU a fost servit.

**Strategie: claim cu watermark de finalizare + dead-letter pentru orfani.**

Adăugăm două coloane pe `inbound_dedupe` (migrare 009): `claimed_at` (când a fost
revendicat) și `completed_at` (NULL până turul ajunge la final). `claim_inbound`
devine „claim-or-resume":

```sql
-- claim NOU sau RECLAMĂ un orfan (claimed dar nefinalizat și expirat)
insert into inbound_dedupe (business_id, provider_msg_id, claimed_at)
values ($1, $2, now())
on conflict (business_id, provider_msg_id) do update
   set claimed_at = now()
 where inbound_dedupe.completed_at is null                       -- nefinalizat
   and inbound_dedupe.claimed_at < now() - make_interval(secs => $3)  -- expirat (orfan)
returning (xmax = 0) as is_new, claimed_at
```
- Rând nou (`is_new=true`) → procesează (cazul curent).
- Conflict + `completed_at IS NULL` + claim vechi (> `CLAIM_TTL_S`, ex. 300s) → `DO UPDATE`
  reușește → e un orfan reclamabil → procesează din nou (turul precedent a murit).
- Conflict + `completed_at IS NOT NULL` → `WHERE` blochează update-ul → 0 rânduri
  întoarse → duplicat REAL (deja servit) → skip, ca azi.
- Conflict + claim recent (alt consumer procesează ACUM) → 0 rânduri → skip (evită
  dubla procesare concurentă; ferestrele de claim nu se suprapun cu idle-ul reaper-ului).

`mark_inbound_completed(conn, business_id, provider_msg_id)` setează `completed_at =
now()` și se apelează la finalul lui `handle_turn`, în ACEEAȘI tranzacție cu scrierea
Sender-ului în outbox (atomicitate: ori reply-ul e în coadă ȘI mesajul e marcat
finalizat, ori niciuna). Astfel un crash între claim și outbox lasă `completed_at`
NULL → recuperabil; un tur reușit e marcat finalizat → nu se reprocesează.

Pentru turele cu early-exit fără outbox (ex. mesaj deduplicat în amonte, gate care
tace deliberat), `completed_at` se setează tot la final — „finalizat" = „turul a
rulat până la capăt", nu „a trimis ceva".

### Câmpuri TurnContext scrise aici
**Niciunul.** Task de infrastructură stagiul 2 (consumer + dedupe L2), sub
pipeline. Nu atinge `TurnContext`. Singura schimbare în `handle_turn` e apelul de
`mark_inbound_completed` în TX-ul de finalizare.

### Logging & PII
Reaper-ul și recuperarea logă DOAR `provider_msg_id`, `business_id` și `msg_id`-ul de
stream — niciun `body`, niciun telefon (P12; PII trăiește doar în `channel_identities`).
`provider_msg_id` e id de provider, nu PII.

## Principii CLAUDE.md aplicabile
- **P6 (niciodată tăcere):** scopul întreg. Un entry orfan în PEL sau un claim orfan
  devine recuperabil, nu pierdut. Echivalentul inbound al „dead, nu pierdut tăcut" de
  pe outbox.
- **P7 (business_id pe tot):** `claim_inbound` și `mark_inbound_completed` filtrează pe
  `business_id = $1`; RLS pe `inbound_dedupe` (004) rămâne plasa de izolare. Reaper-ul
  rulează pe stream-ul Redis unic (pre-tenant), dar reprocesarea trece prin
  `process_event` → `tenant_conn(business_id)` ca orice mesaj.
- **P10 (observabilitate din runner):** reaper-ul e în consumer, nu în stagii; stagiile
  nu știu că un mesaj e o reluare. Emite eveniment de la nivelul worker-ului.
- **P12 (PII):** logurile reaper-ului nu conțin body/telefon.

## Database Changes
Migrare `009_inbound_dedupe_completion.sql`:
```sql
alter table public.inbound_dedupe
  add column if not exists claimed_at   timestamptz not null default now(),
  add column if not exists completed_at timestamptz;

-- backfill: rândurile existente = deja procesate (înainte de schema cu watermark)
update public.inbound_dedupe set completed_at = first_seen where completed_at is null;

-- index pentru reaper-ul de orfani și pentru cleanup
create index if not exists idx_inbound_dedupe_orphan
  on public.inbound_dedupe (claimed_at) where completed_at is null;
```

filtrează pe `business_id = $1` (`claim_inbound`, `mark_inbound_completed`). `locale`
nu se aplică (tabel de control, nu faq/cache/template).

lui standard — aici marcăm doar faptul recuperării.

## Test Cases
**Happy Path:**
1. Mesaj nou → `claim_inbound` întoarce „procesează", tur normal, `mark_inbound_completed`
   setează `completed_at` în TX-ul outbox (stub DB verifică ambele apeluri în ordine).
2. PEL gol → `reap_pending` întoarce 0, fără efecte secundare; `run_consumer` nu-l
   apelează mai des decât intervalul (test pe `last_reap_at`).

**Edge Cases:**
1. Orfan recuperabil: `inbound_dedupe` are rând cu `completed_at NULL` + `claimed_at`
   vechi → al doilea `claim_inbound` reușește `DO UPDATE` → reprocesare; `inbound_reclaimed`
   emis cu `source=dedupe_orphan`.
2. Claim concurent: `completed_at NULL` + `claimed_at` RECENT (alt consumer activ) →
   `claim_inbound` întoarce skip (0 rânduri) → fără dublă procesare.
3. XAUTOCLAIM întoarce un entry cu `fields=None` (entry șters din stream între timp) →
   reaper-ul doar ACK-uiește, fără să încerce să-l proceseze, fără excepție.
4. Cleanup: rând finalizat vechi → șters; orfan recent (`completed_at NULL`, sub pragul
   lung) → păstrat.

**Failure Cases:**
1. Crash în mijlocul turului: claim făcut, dar `mark_inbound_completed` NU rulează
   (simulat: TX rollback înainte de commit) → `completed_at` rămâne NULL → la
   reprocesare orfanul e reclamat și servit (regresia NX-51 e închisă — test explicit).
2. Un entry din batch-ul de reaper aruncă la `json.loads` (payload corupt) → e logat,
   ACK-uit (nu blochează PEL-ul), reaper-ul continuă cu restul batch-ului.
3. `XAUTOCLAIM` aruncă `ResponseError` (grup inexistent după FLUSHALL) → prins în
   `run_consumer`, logat, bucla continuă (nu omoară worker-ul); `ensure_group` îl
   recreează la următorul ciclu.

> Teste fără infra reală: fake Redis cu `xautoclaim` scriptat (listă de
> `(cursor, entries, deleted)`), stub `asyncpg.Connection` pentru claim-or-resume și
> `mark_inbound_completed` (verifică SQL-ul + parametrii `business_id`). ZERO Redis,
> Postgres sau OpenAI real în CI.

## Cost
Niciun apel LLM/embeddings — task pur de infrastructură (Redis Streams + Postgres).
$0 adăugat per 1000 mesaje. Cost marginal: un `XAUTOCLAIM` la ~30s (neglijabil, scanează
doar PEL-ul, de regulă gol) + 2 coloane pe `inbound_dedupe` + un `UPDATE` în plus per tur
(`mark_inbound_completed`, în TX-ul deja existent al Sender-ului). Economia reală:
mesajele care altfel s-ar pierde tăcut sunt servite → vânzări recuperate, nu cost.
