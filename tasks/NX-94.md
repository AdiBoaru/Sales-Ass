# NX-94 — Orders webhook: HMAC peste corpul brut (nu secret în header)
**Owner:** S · **Faza:** MVP · **Zi/Ord:** după F2-2 (endpoint comenzi) · **Branch:** `feat/NX-94-orders-hmac` · **Complexitate:** S · **Estimare:** 2h

## Goal
Webhook-ul de comenzi (`POST /webhook/orders/{business_id}`) trebuie să verifice o
**semnătură HMAC-SHA256 peste corpul BRUT** cu `ORDERS_WEBHOOK_SECRET`, NU să compare un
header de secret static. Azi endpointul citește `X-Orders-Secret` și-l compară constant-time
cu secretul din settings — secretul circulă în clar la fiecare cerere și payload-ul n-are
nicio legare de integritate. Reutilizăm exact pattern-ul Meta din `signature.py`: orice
header scurs din loguri/proxy NU mai autentifică nimic, fiindcă atacatorul tot nu poate
semna un corp pe care nu-l cunoaște. Verificarea se face ÎNAINTE de orice parsare JSON.

## Business Context
`/webhook/orders` alimentează `orders.attribution` → cifra „botul a generat X RON", exact
numărul care vinde retainerul. Un secret care călătorește în clar în fiecare request (loguri
de access, proxy, APM) e o cheie de injecție de comenzi false → revenue umflat artificial
(descoperit de client = încredere pierdută). HMAC peste corp leagă autentificarea de
conținut: nu poți rejuca un secret scurs fără să cunoști și să semnezi un payload valid.

## Technical Description

### Starea curentă (de înlocuit) — `app.py:121-124`
```python
expected = secret
provided = request.headers.get("X-Orders-Secret") or ""
if not expected or not hmac.compare_digest(provided, expected):
    return PlainTextResponse("forbidden", status_code=403)
raw = await request.body()           # corpul citit DUPĂ verificarea de secret
```
Problema: `provided` e secretul în clar; corpul nu intră deloc în autentificare.

### Ținta — HMAC peste corpul brut (pattern-ul `signature.py` al Meta)
Adăugăm o funcție pură în `src/webhook/signature.py`, simetrică cu `verify_meta_signature`
(prefix `sha256=`, `hmac.compare_digest`, fail-closed). Refolosim aceeași formă, doar alt
secret și alt header:
```python
def verify_orders_signature(secret: str, raw_body: bytes, signature_header: str | None) -> bool:
    """True dacă `signature_header` e HMAC-SHA256 valid peste `raw_body` cu `secret`.

    Identic ca semantică cu verify_meta_signature: timp constant, fail-closed
    (fără secret SAU fără antet → False). `raw_body` = EXACT octeții primiți."""
    if not secret or not signature_header or not signature_header.startswith(_PREFIX):
        return False
    expected = hmac.new(secret.encode(), raw_body, hashlib.sha256).hexdigest()
    received = signature_header[len(_PREFIX):]
    return hmac.compare_digest(expected, received)
```
(Dacă apare duplicare reală cu `verify_meta_signature`, se poate extrage un helper privat
`_verify_hmac_sha256(secret, raw, header)` apelat de ambele — opțional, nu obligatoriu la
complexitatea asta.)

### Endpoint rescris — `app.py` `receive_order`
Citim corpul ÎNTÂI (octeții bruți), verificăm semnătura, abia apoi parsăm — exact ca
`receive_webhook` (Meta), principiul 7: nu deserializăm input neîncrezut:
```python
@app.post("/webhook/orders/{business_id}")
async def receive_order(
    business_id: str,
    request: Request,
    secret: str = Depends(get_orders_secret),
    redis: Redis = Depends(redis_dep),
) -> Response:
    raw = await request.body()
    signature = request.headers.get("X-Orders-Signature")
    if not verify_orders_signature(secret, raw, signature):
        return PlainTextResponse("invalid signature", status_code=403)
    try:
        order = json.loads(raw)
    except json.JSONDecodeError:
        return PlainTextResponse("bad request", status_code=400)
    try:
        await enqueue_inbound(redis, {"kind": "order", "business_id": business_id, "order": order})
    except RedisError:
        return PlainTextResponse("service unavailable", status_code=503)
    return PlainTextResponse("ok", status_code=200)
```
Header nou: `X-Orders-Signature: sha256=<hex>`. Vechiul `X-Orders-Secret` dispare complet.
`import hmac` din `app.py` rămâne folosit doar dacă nu mai e nevoie nicăieri → de scos dacă
devine neutilizat (ruff F401).

### Margine subțire, fără DB (neschimbat)
Endpointul rămâne fără DB: doar verifică + parsează + pune envelope `kind='order'` pe stream;
atribuirea reală (`process_order`) e în worker, tenant-scoped. `business_id` vine din path,
autentificat acum de HMAC (un corp semnat cu secretul businessului). Nu atingem `TurnContext`
(comenzile nu-s evenimente de canal, nu trec prin pipeline) — **niciun câmp TurnContext scris**.

### Error handling
- Fără secret configurat (`""`) → `verify_orders_signature` întoarce False → 403 (fail-closed,
  ca azi „gol → 403"). Endpointul nu acceptă orbește.
- Header lipsă / prefix greșit / hex greșit → 403, nimic pe stream.
- Corp semnat corect dar JSON invalid → 400 (semnătura e validă, dar payload-ul e rupt).
- Redis jos → 503 (emitentul reîncearcă), ca path-ul Meta.

## API Changes
`POST /webhook/orders/{business_id}` — schimbare de **contract de auth** (nu de path/body):
- **Înainte:** header `X-Orders-Secret: <secret în clar>`.
- **Acum:** header `X-Orders-Signature: sha256=<hmac_sha256(ORDERS_WEBHOOK_SECRET, raw_body)>`.
- Erori: 403 (semnătură lipsă/invalidă SAU secret neconfigurat), 400 (JSON invalid pe corp
  semnat corect), 503 (Redis jos), 200 (acceptat).
Breaking change pentru orice emitent existent — endpointul nu e încă live pe demo (vezi
TODO-MANUAL, secretul nu e setat), deci fără migrare de compat.

## Test Cases
Toate testele endpointului folosesc `fakeredis` + override de secret (ca azi) + un helper
`_sign(body: bytes) -> str` care calculează `"sha256=" + hmac_sha256(SECRET, body).hexdigest()`.
ZERO apeluri reale (fără DB, fără OpenAI — endpointul nu cheamă niciun LLM).

**Happy Path:**
1. Corp `_order(ref="r")` semnat cu `_sign` → 200, `xlen(STREAM_INBOUND) == 1`, envelope cu
   `kind='order'`, `business_id='biz-1'`, `order.external_id == 'ORD-1'`.
2. Corp cu diacritice/UTF-8 (`name="Cremă hidratantă"`) semnat pe bytes raw → 200 (HMAC pe
   octeții bruți validează identic, nu pe re-serializare).

**Edge Cases:**
1. Header prezent dar fără prefix `sha256=` (doar hex-ul) → 403, nimic pe stream.
2. Secret neconfigurat în settings (`get_orders_secret` override → `""`) + corp „semnat" →
   403 (fail-closed), nimic pe stream.

**Failure Cases:**
1. Semnătură calculată peste un ALT corp (atacator nu cunoaște corpul real) → 403, stream gol.
2. Corp semnat corect (`_sign`) dar JSON invalid (`b"{not json"` semnat) → 400 (semnătura
   trece, parse-ul cade), nimic pe stream.
3. Vechiul `X-Orders-Secret: <SECRET>` trimis fără `X-Orders-Signature` → 403 (contractul vechi
   nu mai e acceptat).
