# NX-23 — Web Widget W4: identitate vizitator + merge
**Owner:** S · **Faza:** P2 (V1.5, Epic E26 · W4) · **Zi/Ord:** după NX-20/NX-22 (canal web LIVE) · **Branch:** `feat/NX-23-visitor-identity-merge` · **Complexitate:** M · **Estimare:** 4h (umbrelă → vezi [PROPUNERE] breakdown)

## Goal
Un vizitator anonim pe widgetul web are deja o identitate stabilă (`visitor_id`
semnat în cookie first-party → `channel_identities` cu `channel_kind='webchat'`),
deci poate purta o conversație și un istoric ÎNAINTE să-și dea telefonul. Când
ulterior își lasă telefonul (sau scrie de pe WhatsApp cu același număr), cele două
identități se contopesc într-un singur `contact`: istoricul web + WhatsApp devine
unul singur, fără contacte duplicate. Merge-ul e logat în `audit_log` și
**REVERSIBIL** (păstrăm legătura veche în `details`), pentru că un merge greșit pe
un număr introdus de mâna greșită nu trebuie să fie ireparabil. Identity resolution
rămâne o regulă de pipeline — trăiește în Gates (stagiul 3), nu în vreun loc nou.

## Business Context
Pe web, 90% din vizitatori nu se prezintă niciodată. Dacă fiecare reîncărcare ar
crea un contact nou, dashboard-ul s-ar umple de duplicate și atribuirea conversiilor
(checkout_links → orders) s-ar rupe. `visitor_id` în cookie ține firul. Merge-ul
web↔WhatsApp e exact momentul în care un lead anonim devine un client cunoscut: tot
ce a citit pe web (produse comparate, FAQ, coș) trebuie să apară în continuarea pe
WhatsApp, unde se închide vânzarea. Zero duplicate pe demo = condiția ca raportul de
lead-uri să fie credibil pentru client.

## Technical Description

### 1. Identitatea de vizitator (semnată, first-party)
NX-20 (Gateway SSE) emite la prima vizită un `visitor_id` (uuid) într-un cookie
first-party `nx_vid`, **semnat** cu un secret per-tenant (HMAC-SHA256, payload =
`business_id.visitor_id`). Semnătura previne ca un vizitator să se dea drept altul
(furt de istoric). Verificarea trăiește în gateway (NX-20), NU în pipeline —
pipeline-ul primește deja `sender_external_id = visitor_id` validat în envelope-ul
neutru. La primul mesaj, `get_or_create_contact(conn, business_id, 'webchat',
visitor_id)` creează contactul + `channel_identities('webchat', visitor_id)` exact
ca pentru orice alt canal (cod existent, zero modificări pe drumul de creare).

> Cookie-ul `nx_vid` e PII de canal → trăiește pe browserul clientului + reflectat
> DOAR în `channel_identities.external_id`. NICIODATĂ în loguri (logger-ul redactează
> `visitor_id` la fel ca telefonul / wa_id, principiul 12).

### 2. Declanșatorul de merge: capturarea telefonului
Două căi produc o cerere de merge, ambele convergând în aceeași funcție:
- **Web → telefon:** vizitatorul își lasă telefonul în widget (formular „vrei pe
  WhatsApp?"). Gateway-ul normalizează E.164 și pune în envelope un câmp
  `link_identity = {channel_kind:'whatsapp', external_id:'+407...'}`. Gate-ul vede
  câmpul și încearcă merge-ul.
- **WhatsApp → web (același număr):** userul scria pe web, apoi deschide WhatsApp.
  Aici NU avem dovada că e aceeași persoană decât dacă widgetul a trimis deja
  telefonul → cazul colaps în primul (avem deja identitatea whatsapp existentă).

Merge-ul rulează în Gates pentru că „identity resolution e regulă de pipeline"
(CLAUDE.md, stagiul 3). Adăugăm o poartă NOUĂ, DUPĂ rezolvarea de bază a contactului
(care s-a întâmplat deja în processor) și ÎNAINTE de moderare/risc — un câmp
`link_identity` din envelope ⇒ încearcă merge, apoi continuă turul normal pe
contactul rezultat (winner).

```python
# src/worker/stages/gates.py — poartă nouă (NU early-exit; doar reatașează ctx.contact)
async def _maybe_merge_identity(ctx: TurnContext, deps: PipelineDeps) -> None:
    link = ctx.message.link_identity  # {channel_kind, external_id} sau None
    if not link:
        return
    winner = await merge_contacts_by_identity(
        deps.conn,
        ctx.business.id,
        current_contact_id=ctx.contact.id,           # identitatea web curentă
        link_channel_kind=link["channel_kind"],      # 'whatsapp'
        link_external_id=link["external_id"],         # E.164 normalizat
        actor="web_widget",
    )
    if winner is not None and winner.id != ctx.contact.id:
        ctx.contact = winner                          # restul turului curge pe winner
        ctx.emit("identity_merged", merged_into=winner.id)  # FĂRĂ telefon/visitor_id (P12)
```

### 3. Funcția de merge (cod determinist, tranzacțional, reversibil)
`merge_contacts_by_identity` (în `src/db/queries/contacts.py`) trăiește integral
într-o tranzacție tenant-scoped pe `business_id`. Regula de winner: dacă identitatea
țintă (`whatsapp`/`external_id`) NU există → o atașăm pe contactul curent (web) și
gata, nu e merge (un singur contact). Dacă există un alt contact cu identitatea
whatsapp → acela e **winner** (canalul „greu", cu istoric de vânzare), iar contactul
web devine **loser** și se reatașează tot la winner. Politica „cine câștigă" e
deliberat simplă și documentată: whatsapp > webchat (canalul de bani câștigă).

```sql
-- în tranzacție; conn tenant-scoped; toate query-urile au business_id = $1
-- 0. caut contactul deja legat de identitatea țintă (winner candidat)
select c.id from contacts c
join channel_identities ci on ci.contact_id = c.id
where ci.business_id = $1 and ci.channel_kind = $2 and ci.external_id = $3;

-- CAZ A: nu există → doar leg identitatea de contactul curent (NU e merge)
insert into channel_identities (business_id, contact_id, channel_kind, external_id)
values ($1, $current, $2, $3)
on conflict (business_id, channel_kind, external_id) do nothing;

-- CAZ B: există și e alt contact → reatașez TOATE rândurile loser → winner
-- (ordine: copii înainte de a goli loser-ul; fiecare cu business_id = $1)
update channel_identities       set contact_id = $winner where business_id=$1 and contact_id=$loser;
update conversations            set contact_id = $winner where business_id=$1 and contact_id=$loser;
update messages                 set contact_id = $winner where business_id=$1 and contact_id=$loser;
update conversation_summaries   set contact_id = $winner where business_id=$1 and contact_id=$loser;
update checkout_links           set contact_id = $winner where business_id=$1 and contact_id=$loser;
update orders                   set contact_id = $winner where business_id=$1 and contact_id=$loser;
update back_in_stock_subscriptions set contact_id = $winner where business_id=$1 and contact_id=$loser;
update appointments             set contact_id = $winner where business_id=$1 and contact_id=$loser;
update proactive_jobs           set contact_id = $winner where business_id=$1 and contact_id=$loser;
-- loser rămâne ca tombstone (NU îl ștergem fizic → reversibilitate):
update contacts set lifecycle = 'churn_risk', profile = profile || '{"merged_into":"<winner>"}'::jsonb
  where business_id=$1 and id=$loser;
-- merge lead_score: winner ia max-ul (un lead cald nu se răcește prin merge)
update contacts set lead_score = greatest(lead_score, $loser_score)
  where business_id=$1 and id=$winner;
```

`back_in_stock_subscriptions` are `unique(business_id, contact_id, product_id,
variant_id)` → un UPDATE care ar duplica abonamentul poate da conflict. Tratăm cu
`on conflict do nothing` printr-un `insert ... select` + `delete` pe loser, SAU mai
simplu: `update ... where not exists(...)` apoi `delete` rămășițele loser-ului.
La fel pentru orice constraint unic pe `(business_id, contact_id, ...)`.

### 4. Audit + reversibilitate
În aceeași tranzacție, scriem în `audit_log` snapshot-ul necesar pentru un undo:
```sql
insert into audit_log (business_id, actor, action, entity, entity_id, details)
values ($1, $actor, 'identity_merge', 'contact', $winner::text,
  jsonb_build_object(
    'loser_contact_id', $loser::text,
    'winner_contact_id', $winner::text,
    'link_channel_kind', $2, 'link_external_id_hash', encode(digest($3,'sha256'),'hex'),
    'reattached', jsonb_build_object('conversations', $n_conv, 'messages', $n_msg, ...)
  ));
```
`details` ține `loser_contact_id` + numărul de rânduri reatașate per tabel → un job
de undo (out of scope aici, dar deblocat) poate recrea contactul loser și muta
rândurile înapoi. **NU stocăm telefonul brut** în `audit_log` — doar hash-ul (P12);
identificarea unei intrări de audit se face pe hash, ca în `channel_identities`.

### Câmpuri TurnContext scrise aici
- `ctx.contact` — reatașat la winner-ul de merge. **Owner: Gates** (la fel ca
  `is_blocked` / handoff state azi). Niciun alt stagiu nu rescrie `contact` după Gates.
- `ctx.events` — `identity_merged` (acumulat, owner = orice stagiu prin `ctx.emit`).
- Câmp NOU pe `InboundMessage`: `link_identity: dict | None` (umplut de marginea web
  în envelope, citit de Gates). Owner = ingestia (parser-ul web), read-only în pipeline.

`ctx.reply` NU se atinge aici: merge-ul e silent, turul continuă la moderare/triaj/agent.

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** merge-ul e o poartă în Gates, nu un orchestrator separat;
  nu sare înapoi, doar reatașează `ctx.contact` și lasă turul să curgă mai departe.
- **P2 (LLM doar triaj+agent):** ZERO LLM aici — winner/loser, normalizare E.164,
  reatașare = cod 100% determinist. Nicio excepție de adaptor necesară.
- **P3 (un owner per câmp):** `ctx.contact` are un singur scriitor post-Gates (Gates).
- **P7 (business_id pe tot):** fiecare UPDATE/SELECT din merge are `business_id = $1`;
  RLS (bot_runtime + app.business_id) e plasa care transformă un merge cross-tenant
  într-un „zero rânduri", nu într-o scurgere.
- **P12 (PII într-un loc):** telefonul/visitor_id stau DOAR în `channel_identities`;
  `audit_log` și `analytics_events` primesc DOAR hash-uri / id-uri de contact, niciodată
  numărul brut. Logger-ul redactează `link_external_id`.

## Events de emis (analytics)
- `identity_merged` {merged_into: winner_contact_id, n_reattached?} — la merge reușit
  (caz B). FĂRĂ telefon/visitor_id (P12). Permite raportul „lead-uri web convertite
  pe WhatsApp" în dashboard.
- `identity_linked` {channel_kind} — la caz A (doar atașare, fără merge de contacte).
  Optional; util pentru a distinge „prima identitate de canal nouă" de un merge real.

## Dependencies
- **NX-20** (Gateway SSE / W1) — în main: produce `visitor_id` semnat + envelope web.
- **NX-22** (Canal web în gates/sender / W3) — în main: `channel_kind='webchat'` rulează
  prin pipeline (altfel nu există identitate web de contopit).
- **G5a** (gates_stage) — în main (#54): poarta de merge se adaugă în `gates_stage`.
- **NX-60** (contract envelope neutru) — `link_identity` se adaugă în contract.
- Recomandat: **NX-25** (CORS/CSP allowlist / W3) în main, ca telefonul capturat să nu
  poată fi injectat de pe un domeniu neautorizat.

## Out of Scope
- **Job de undo / consolă de unmerge** — `audit_log` ține datele pentru reversibilitate,
  dar UI-ul/scriptul de undo e task de margine separat (V1.5+).
- **Verificarea telefonului prin OTP** înainte de merge — V1 are încredere în telefonul
  capturat de widget (rate-limitat + CORS allowlist din NX-25); OTP = hardening ulterior.
- **Merge automat fuzzy** (nume/email apropiate) — DOAR pe identitate exactă de canal.
- **Politici de winner per-business** (ex. un vertical preferă webchat) — V1 = regulă
  fixă whatsapp > webchat.
- **Merge cross-business** — interzis structural (business_id pe tot); nu e un caz.

## Test Cases
**Happy Path:**
1. Web → telefon nou: contact web are 2 conversații; merge caz A; un singur contact,
   acum cu identitate `webchat` + `whatsapp`; `identity_linked` emis.
2. Web (istoric) → telefon care există deja pe un contact WhatsApp cu o comandă:
   merge caz B; după merge winner are conversațiile web + comanda; `ctx.contact`=winner;
   `identity_merged{merged_into}` emis; loser e tombstone (`profile.merged_into`).

**Edge Cases:**
1. `link_identity` = telefonul DEJA legat de contactul curent → no-op (winner==current),
   niciun eveniment de merge, turul continuă.
2. loser ȘI winner au abonament back-in-stock identic (același product/variant) →
   reatașarea respectă unique-ul, fără excepție, abonamentul duplicat e curățat.
3. `link_identity` cu E.164 ne-normalizat / gol → marginea normalizează; gate-ul pe gol
   = no-op (nu încearcă merge).
4. lead_score: loser=80, winner=10 → după merge winner=80 (greatest).

**Failure Cases:**
1. Eroare DB în mijlocul reatașării → tranzacția face rollback complet: niciun rând
   mutat parțial, niciun audit_log scris, `ctx.contact` rămâne neschimbat; turul continuă
   pe contactul vechi (P6: niciodată tăcere — botul răspunde, merge-ul se reîncearcă data viitoare).
2. `link_identity` cu telefonul altui business (`business_id` diferit) → SELECT-ul de
   winner întoarce zero (RLS + filtru) → caz A pe propriul tenant, NU atinge contactul străin.
3. bot_runtime fără INSERT pe `audit_log` (grant lipsă) → merge-ul eșuează la audit →
   rollback; test care prinde lipsa grantului și forțează follow-up-ul de DDL.

> Teste fără apeluri reale: merge = cod determinist, ZERO LLM/embeddings → niciun mock de
> OpenAI necesar. Cazurile A/B rulează pe DB de integrare cu tranzacție rollback-uită
> (pattern existent `test_tenant_isolation.py`); cazurile de gate folosesc stub-uri DB.
> ZERO apeluri reale în CI.

## Cost LLM
**$0.** Taskul nu cheamă niciun LLM/embedding — winner/loser, normalizare, reatașare și
audit sunt cod determinist. Costul marginal e câteva UPDATE-uri tranzacționale per merge
(eveniment rar: o dată per vizitator care își lasă telefonul), mărginit natural de
frecvența capturării de telefon. Beneficiu: elimină contacte duplicate (cost de raportare
+ atribuire greșită) și păstrează istoricul de vânzare unificat.
