# NX-53 — Test de izolare CONCURENT (50 tururi paralele, 2 tenanți)
**Owner:** S · **Faza:** MVP · **Zi:** Z3 · **Branch:** test/NX-53-isolation · **Complexitate:** S · **Estimare:** 4h

## Goal
Dovada executabilă că izolarea multi-tenant ține sub concurență reală — exact scenariul pe care testul existent (secvențial) nu îl atinge.

## Business Context
Validează NX-50. „RLS testat live" devine „RLS testat live SUB LOAD" — propoziția pe care o poți spune unui client enterprise cu dovadă în CI.

## Technical Description
`pytest` async: 50 de task-uri lansate cu `asyncio.gather`, alternând tenant A / tenant B (25/25). Fiecare task: ia conexiune prin `tenant_connection(bid)`, INSERT un marker unic (`isolation_probe` în `analytics_events` sau tabel temporar dedicat), SELECT toate markerele vizibile, assert: (1) `current_user == 'bot_runtime'`; (2) `current_setting('app.business_id') == bid`; (3) markerele vizibile aparțin DOAR tenantului propriu. La final: numărătoare globală ca admin → 25+25, zero încrucișări.

## Implementation Steps
1. Fixture cu 2 business_id-uri de test (create/rollback per sesiune de test).
2. `tests/test_isolation_concurrent.py` cu scenariul de mai sus; marker `@pytest.mark.slow`.
3. Rulează de 20× local (`pytest --count=20` via pytest-repeat sau buclă) — 0 eșecuri.
4. Job CI separat `isolation-concurrent` (nu blochează PR-urile obișnuite; obligatoriu pe main + nightly).
