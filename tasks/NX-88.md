# NX-88 — Post-tur: extractor profil (nano) + update lead_score
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G6-2 (context builder) · **Branch:** `feat/NX-88-post-turn-profile` · **Complexitate:** M · **Estimare:** 4h

## Goal
Botul „învață" clientul: după fiecare tur, un pas asincron (post-livrare) extrage
semnale de profil din conversație (nano) și actualizează `contacts.profile` +
recalculează `contacts.lead_score`. Azi ambele câmpuri sunt **read-only la runtime**
— se citesc în context (G6-2) dar nimeni nu le scrie, deci profilul rămâne gol și
lead-score-ul înghețat. Fără pasul ăsta, personalizarea și calificarea de lead nu
au sursă de date.

## Business Context
Lead-score-ul e ce transformă conversațiile în pipeline comercial: clientul vrea să
știe cine e gata de cumpărat (handoff prioritar, follow-up, export CRM — NX-31).
Profilul (skin_type, buget tipic, branduri preferate) face următoarea recomandare
mai bună fără să reîntrebe. Ambele cresc rata de conversie, deci direct banii
clientului. Pasul e **post-tur** (nu blochează răspunsul) → zero impact pe latența percepută.

## Technical Description

### Plasare: hook post-tur în `handle_turn` (după outbox, nu blochează livrarea)
Hook-urile post-tur trăiesc deja la [processor.py:401-406](src/worker/processor.py#L401-L406)
(`_cache_writeback`, `_summarize_if_needed`), apelate DUPĂ scrierea în outbox. Adăugăm
al treilea hook, în aceeași zonă, cu `llm`-ul **guardat de cost guard** (peste buget → `None`
→ hook sărit, ca celelalte):
```python
await _extract_profile_and_score(conn, llm, business, conv["id"], contact, ctx)
```
Rulează best-effort: orice excepție e prinsă și logată, NU rupe `TurnResult` (reply-ul
e deja în outbox). Nu se rulează în shadow mode (NX-93).

### Extractorul de profil (nano, prin adaptorul unic)
Un singur apel `nano` pe istoricul scurt + ultimul tur, cu output JSON validat Pydantic:
```python
class ProfileDelta(BaseModel):
    profile_patch: dict[str, str | int | float | bool]   # chei candidate
    lead_signals: LeadSignals                              # ex. {intent_strength, has_budget, asked_price, ready_to_buy}
```
**Whitelist de chei (P-cheie):** `profile_patch` trece printr-un filtru de chei permise
**per vertical** (skin_type, budget_band, fav_brands… pentru beauty) — cheile necunoscute
sunt **aruncate** + se emite `profile_key_dropped`. v1 = whitelist minimă din config;
lista bogată per vertical e **NX-43** (dependență de date, nu blochează). Modelul nu
poate scrie chei arbitrare în DB.

### lead_score = formulă deterministă (NU numărul inventat de LLM)
LLM-ul dă doar **semnale** (`lead_signals`); scorul final e calculat în **cod** dintr-o
formulă transparentă (ex. ponderi pe intent + buget declarat + apropiere de checkout),
clampat 0..100. Motiv: scorul trebuie să fie explicabil și stabil între versiuni de model
(intră în contracte / export CRM). LLM-ul nu scrie niciodată direct `lead_score`.

### Scriere în DB (un singur owner per câmp)
Query nou în `contacts.py` (azi singurul UPDATE e `block_contact` la [contacts.py:90](src/db/queries/contacts.py#L90)):
```sql
update contacts
   set profile = profile || $3::jsonb,        -- merge, nu overwrite
       lead_score = $4,
       updated_at = now()
 where business_id = $1 and id = $2;
```
`bot_runtime` are UPDATE pe `contacts` (003). Merge `||` ca să nu pierdem chei vechi.

### Câmpuri TurnContext / owner
NU scrie câmpuri pe `TurnContext` (rulează după ce turul s-a închis). Owner exclusiv pe
`contacts.profile` + `contacts.lead_score` la runtime = **acest hook** (P3: un singur
proprietar). Citește `ctx.history`, `ctx.message`, `ctx.route`, `ctx.retrieval`.

## Principii CLAUDE.md aplicabile
- **P2 (LLM doar triaj+agent):** extractorul e al treilea apel acceptat doar prin
  adaptorul unic `src.agent.llm` (nano, post-tur), nu generare în pipeline — în spiritul
  summarizer-ului (G6-2) deja stabilit ca hook post-tur. Scorul e cod determinist.

- **P12 (PII):** evenimentele logă DOAR chei/contoare (`profile_key_dropped {key}`,
  `lead_score_updated {old,new}`), NICIODATĂ corpul mesajului sau valori PII.

5. `src/worker/processor.py`: `_extract_profile_and_score(...)` + apel în `handle_turn`
   după `_summarize_if_needed` (best-effort, guardat de `llm`, sărit în shadow).
6. Teste (mock `extract_profile` + fake conn): vezi Test Cases; `ruff check . && ruff
   format . && pytest -x -q` verde.

## Test Cases
**Happy Path:**
1. `extract_profile` mock → `{profile_patch:{skin_type:"uscat"}, lead_signals:{...}}` →
   patch aplicat + scor calculat + `profile_updated`/`lead_score_updated` emise.
2. Două tururi succesive → al doilea face merge peste primul (chei cumulate, nu pierdute).

**Edge Cases:**
1. `profile_patch` cu cheie necunoscută (`fav_color`) → aruncată, `profile_key_dropped`,
   restul patch-ului aplicat.
2. `lead_signals` slabe (doar browsing) → scor mic dar nenul; ready_to_buy → scor mare (formulă).
3. Shadow mode (NX-93) → hook sărit complet.

**Failure Cases:**
1. `extract_profile` aruncă / JSON invalid → deltă goală, fără UPDATE, fără excepție propagată.
2. `update_contact_profile_and_score` aruncă (DB) → prins, logat, `TurnResult` valid.
3. `llm=None` → hook sărit, zero apeluri nano.

> Teste fără apeluri reale: `LLMClient.extract_profile` cu `AsyncOpenAI` mock (JSON fabricat);
> processor cu fake conn/redis. ZERO apeluri OpenAI/DB reale în CI.

## Cost

JSON mic out). La ~$X/1M tokens nano → fracțiune de cent per tur. **Ce limitează costul:**

guard (G2c) — peste plafonul zilnic → `llm=None` → sărit; (3) input bugetat (același istoric
de max 8 ca în context builder, nu conversația întreagă).
