# NX-82 — Tool request_human + activare tool-uri per business
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7-3 (tool-uri în main) · **Branch:** `feat/NX-82-request-human-tool` · **Complexitate:** S · **Estimare:** 3.5h

## Goal
Două găuri mici, înrudite, închise împreună: (1) agentul (stagiul 7) capătă un TOOL
`request_human` ca să escaladeze EXPLICIT la om când recunoaște că nu mai poate ajuta
(frustrare, cerere complexă, ceva în afara catalogului) — azi handoff-ul vine DOAR din
poarta de risc determinist în Gates (stagiul 3), deci agentul nu are pârghie; (2)
`enabled_tools(business, route)` ignoră complet `business` (`noqa: ARG001`) — activarea de
tool-uri e doar pe rută, identică pentru toți clienții. Le facem reale: toolset-ul se taie și
per-business din `businesses.settings`, ca un client să poată dezactiva `request_human`
(n-are operator) sau `checkout_link` (n-are coș) fără cod nou.

## Business Context
Escaladarea explicită din agent prinde cazurile pe care pattern-urile de risc nu le văd
(„nu înțelegi ce vreau", „vreau altceva, nu produse") — alternativa e botul care se învârte
și clientul care pleacă. Activarea per-business e bani direct: clienții noștri sunt verticale
diferite (beauty cu coș vs salon cu programări); să nu oferim un tool pe care clientul nu-l
poate onora (checkout fără coș, human fără operator) înseamnă mai puține promisiuni neacoperite
și un setup per-client fără modificări de cod.

### (1) `request_human` ca TOOL al agentului
Tool nou, contract uniform (`async def tool(ctx, deps, args) -> ToolResult`), înregistrat cu
`@register("request_human")`. Reutilizează funcția existentă din Gates — NU duplicăm logica de
handoff (un singur loc setează `handoff_until`):
```python
# src/tools/handoff_tools.py
class RequestHumanArgs(BaseModel):
    reason: str = Field(min_length=1, max_length=200)  # de ce escaladează (scurt, fără PII)

@register("request_human")
async def request_human_tool(ctx, deps, args) -> ToolResult:
    a = RequestHumanArgs(**args)
    from src.worker.stages.gates import request_human  # funcția existentă, sursă unică de handoff
    await request_human(deps.conn, ctx, a.reason, source="agent")  # business_id din ctx (P7)
    await notify_operator(deps, ctx, a.reason)                     # best-effort, vezi mai jos
    # Mesaj de tranziție GROUNDED prin construcție (nu produs/preț) → trece validatorul.
    return ToolResult(ok=True, llm_view="Am anunțat un coleg; spune-i clientului că revine cineva.")
```
Diferența față de poarta de risc: `source="agent"` (urmărim în analytics cine a escaladat).
Agentul, după ce primește `llm_view`, formulează el mesajul de tranziție în limba clientului
(text neutru, fără cifre → validatorul de preț/link îl acceptă fără sume permise). Dacă agentul
cheamă `request_human`, turul tot iese prin Sender ca orice reply de agent; turul URMĂTOR cade
pe poarta 3 din Gates (`handoff_until` în viitor → tăcere) — exact ca handoff-ul de risc.

Schema OpenAI în `tool_definitions.py` (prefix STATIC, `strict: True`, `business_id` NU apare):

### Notificare operator (best-effort, fără LLM)
`notify_operator(deps, ctx, reason)` postează pe `operator_alert_webhook` (Slack/generic) dacă e
setat: business slug, `conversation_id`, `reason`. **FĂRĂ PII** — niciun telefon/nume/corp de

### (2) Activare tool-uri per business
`enabled_tools` taie pe DOUĂ axe: rută (ca acum) ∩ allow/deny per business din `settings`.
Convenție în `businesses.settings`:
```json
{ "tools": { "disabled": ["checkout_link"], "request_human_enabled": true } }
```
```python
_SALES_TOOLS = ("search_products", "get_product_details", "compare_products", "checkout_link")
_ORDER_TOOLS = ("check_order",)
# request_human e disponibil pe AMBELE rute (escaladarea nu ține de sales/order),
# dar default OFF: îl oferim doar dacă businessul a confirmat că are operator.
def enabled_tools(business: Any, route: str | None = None) -> list[str]:
    base = _ORDER_TOOLS if route == "order" else _SALES_TOOLS
    names = list(base)
    cfg = (getattr(business, "settings", None) or {}).get("tools", {}) if business else {}
    if cfg.get("request_human_enabled"):     # opt-IN: doar businessuri cu operator
        names.append("request_human")
    disabled = set(cfg.get("disabled", []))  # deny-list per business
    return [n for n in names if n in TOOL_REGISTRY and n not in disabled]
```
`business=None` (apel din teste / fără config) → comportamentul vechi exact (doar rută),
deci nu spargem `test_enabled_tools_phase1_and_2`. Apelantul (`agent.py:267`) deja pasează
`ctx.business` — zero schimbări acolo.

### Câmpuri TurnContext scrise aici
Niciun câmp nou. Tool-ul `request_human` NU scrie direct în `ctx` — scrie în DB
(`conversations.handoff_until`/`risk_flags` via `set_handoff`) și emite `handoff_requested`

**Edge Cases:**

2. `set_handoff` aruncă (DB jos) → `run_tool` prinde excepția tool-ului → `ToolResult(ok=False,
   llm_view="Unealta a eșuat.")`, turul continuă (agentul răspunde fără handoff, nu tace).

## Cost
**Zero cost LLM adăugat.** Tool-ul e cod determinist; agentul DECIDE să-l cheme în cadrul aceleiași
bucle de tool-calling deja bugetate (max 3 tool calls/tur, cap dur în adaptor) — niciun apel LLM
suplimentar. `notify_operator` e un POST HTTP (≈0 cost). Ce mărginește costul: limita de 3 tool
calls/tur (un `request_human` consumă un slot, nu deschide apeluri noi) + faptul că escaladarea
TERMINĂ conversația botului (turul următor tace pe `handoff_until`), deci scade tokenii viitori.
