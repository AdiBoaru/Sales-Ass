# NX-17 — schema_version în payload-urile din stream
**Owner:** S · **Faza:** P1 · **Zi/Ord:** O2 · **Branch:** `feat/NX-17-schema-version` · **Complexitate:** S · **Estimare:** 3h

## Goal
Pune un câmp de versiune (`v: 1`) în FIECARE payload care trece prin marginile sistemului:
envelope-urile neutre de pe stream-ul `inbound` (message/status/callback/order) și payload-ul
din `outbox`. Consumerul citește `v` și acceptă atât versiunea curentă `N` cât și `N-1`. Asta
face rolling deploy-ul sigur: un worker `v2` proaspăt pornit poate procesa mesajele `v1` rămase
în coada Redis (sau în `outbox`) de la worker-ul vechi, fără să crape și fără să le piardă tăcut
(principiul 6). Regula de evoluție: chei NOI doar opționale, NICIODATĂ redenumiri sau ștergeri —
compatibilitate înainte ȘI înapoi.

## Business Context
Deploy-urile pe VPS (docker compose) sunt rolling: webhook/poller (producători) și worker +
dispatcher (consumeri) repornesc la momente diferite, iar Redis Streams + `outbox` păstrează
mesaje în zbor între versiuni. Fără un câmp de versiune, prima schimbare de formă a envelope-ului
(ex. când NX-60 adaugă o cheie nouă) transformă orice mesaj „de tranziție" într-un `KeyError`
într-un `except` care îl ACK-uiește tăcut — adică un mesaj de client pierdut la fiecare deploy.
`v` e cheltuiala de 1 octet care face ca evoluția schemei de mesaje să fie o non-problemă.

## Technical Description

### Starea actuală (ce există)
Patru producători pun envelope pe stream-ul unic `inbound`:
1. `InboundEvent.to_dict()` → `{"kind": "message", ...}` (din `webhook/meta.py` + `telegram/poller.py`)
2. `StatusEvent.to_dict()` → `{"kind": "status", ...}`
3. `CallbackEvent.to_dict()` → `{"kind": "callback", ...}` (din `telegram/poller.py`)
4. Envelope inline de comandă în `webhook/app.py`: `{"kind": "order", "business_id": ..., "order": ...}`

Un singur consumer: `consume_once` → `process_event` (`worker/consumer.py`) rutează după `event.get("kind")`.
Payload-ul de `outbox` e construit inline în `processor.py` (`payload = {"type": "text"|"carousel", ...}`)
și consumat de `dispatch_row` (`dispatcher.py`). **Niciun câmp de versiune nu există azi** (grep `"v"`/`schema_version` → 0).

### Țintă — constanta + producătorii
Adaug o constantă unică (un singur loc de adevăr) și o pun în toate envelope-urile:

```python
# src/channels/base.py
ENVELOPE_VERSION = 1  # versiunea curentă a envelope-ului de stream (NX-17).
                      # Bump DOAR la schimbare incompatibilă. Chei noi opționale = NU bump.

@dataclass
class InboundEvent:
    ...
    def to_dict(self) -> dict[str, Any]:
        return {"v": ENVELOPE_VERSION, "kind": "message", **asdict(self)}
# idem StatusEvent ("status") și CallbackEvent ("callback")
```

Envelope-ul de comandă (inline în `webhook/app.py`) primește și el `v`:
```python
await enqueue_inbound(
    redis,
    {"v": ENVELOPE_VERSION, "kind": "order", "business_id": business_id, "order": order},
)
```

Payload-ul de `outbox` (construit în `processor.py`) capătă `v` în payload-ul jsonb:
```python
payload = {"v": ENVELOPE_VERSION, "type": "text", "to": sender_external_id, ...}
```
(`enqueue_outbox` îl serializează ca atare; nicio coloană nouă — `v` trăiește în `payload jsonb`.)

### Țintă — consumerii (acceptă N și N-1)
Un singur punct de verificare în `process_event`, înainte de rutarea pe `kind`:
```python
SUPPORTED_ENVELOPE_VERSIONS = frozenset({0, 1})  # N și N-1; 0 = envelope vechi FĂRĂ câmpul `v`

def _envelope_version(event: dict) -> int:
    # Lipsa cheii = pre-NX-17 (v0). Default tolerant = compatibilitate înapoi gratis.
    return int(event.get("v", 0))

async def process_event(pool, redis, event: dict) -> None:
    v = _envelope_version(event)
    if v not in SUPPORTED_ENVELOPE_VERSIONS:
        # versiune din VIITOR (worker vechi vede mesaj de la worker nou cu v necunoscut):
        # NU ACK tăcut, NU crash — log + re-raise ca să NU se piardă (rămâne în PEL).
        log.warning("envelope versiune nesuportată v=%s kind=%s — skip (rămâne neACK-uit)",
                    v, event.get("kind"))
        raise UnsupportedEnvelopeVersion(v)
    ...  # rutarea existentă pe kind
```
Simetric, dispatcher-ul (`dispatch_row`) verifică `payload.get("v", 0) in SUPPORTED_ENVELOPE_VERSIONS`
înainte de a trimite (un `v` necunoscut în outbox → `mark_failed` cu backoff, NU `dead` — un worker
mai nou îl va prelua; rândul rămâne vizibil, principiul 6).

**Asimetria deliberată:** `v` mai mic sau egal (N-1, v0) = procesăm normal (chei lipsă sunt
opționale → default-uri). `v` mai mare (din viitor) = NU ACK-uim/NU marcăm dead → mesajul așteaptă
worker-ul nou. Aceasta e exact garanția de rolling deploy: nici producătorul nou nu rupe consumerul
vechi (mesajul așteaptă), nici producătorul vechi nu rupe consumerul nou (v0 acceptat).

### Regula de evoluție (documentată în cod + CONTRIBUTING)
- Cheie NOUĂ → opțională, cu default sigur la consumer. NU bump `v`.
- Redenumire / ștergere / schimbare de tip a unei chei → INTERZIS într-o singură versiune.
  Procedura corectă: adaugi cheia nouă (opțională) → deploy → backfill → consumerul preferă cheia
  nouă cu fallback la cea veche → după ce coada s-a golit de v vechi, bump `v` și scoți cheia veche.
- `SUPPORTED_ENVELOPE_VERSIONS` ține mereu cel puțin `{N-1, N}`.

### Câmpuri TurnContext scrise aici
**Niciunul.** NX-17 atinge marginile (envelope de stream + payload outbox) și rutarea din consumer,
ÎNAINTE ca `TurnContext` să fie construit (`handle_turn`). Nu scrie niciun câmp din `TurnContext`.

### Error handling & securitate
- `v` necunoscut din viitor → re-raise (rămâne în Pending Entries List Redis / scadent în outbox),
  nu ACK tăcut. Asta inversează intenționat comportamentul „ACK chiar și la eșec" din `consume_once`
  DOAR pentru cazul versiunii incompatibile — un payload malformat (alt `Exception`) rămâne ACK-uit
  ca acum (nu blochează coada). Distincția: `UnsupportedEnvelopeVersion` e singura excepție pe care
  bucla o lasă să propage înainte de ACK.
- `v` e un int mic, fără PII — sigur de logat (P12). Nimic din corpul mesajului nu intră în logul de versiune.
- Izolare multi-tenant: NX-17 nu adaugă query-uri noi cu date; envelope-ul de comandă păstrează
  `business_id` (autentificat de secret la webhook), neschimbat.

## Principii CLAUDE.md aplicabile
- **P5 (un singur punct de ieșire):** `outbox` rămâne singura ieșire; `v` se adaugă în payload-ul lui,
  nu se creează un al doilea canal. `ENVELOPE_VERSION` e un singur loc de adevăr.
- **P6 (niciodată tăcere):** un envelope cu `v` din viitor NU e ACK-uit tăcut / NU devine `dead` —
  rămâne în coadă pentru worker-ul nou. Pierderea tăcută la deploy e exact ce previne taskul.
- **P7 (business_id pe tot):** envelope-ul de comandă păstrează `business_id`; `v` nu schimbă izolarea.
- **P12 (PII):** `v` e non-PII; logurile de versiune nu conțin corpul mesajului / telefoane.

## Implementation Steps
1. `src/channels/base.py`: adaugă `ENVELOPE_VERSION = 1`; pune `"v": ENVELOPE_VERSION` în `to_dict()`
   pentru `InboundEvent`, `StatusEvent`, `CallbackEvent`.
2. `src/webhook/app.py`: importă `ENVELOPE_VERSION`; adaugă `"v"` în envelope-ul inline `kind="order"`.
3. `src/worker/processor.py`: adaugă `"v": ENVELOPE_VERSION` în dict-ul `payload` (text + carusel).
4. `src/worker/consumer.py`: `SUPPORTED_ENVELOPE_VERSIONS`, `UnsupportedEnvelopeVersion`,
   `_envelope_version`; verificarea la începutul `process_event`; în `consume_once`, lasă
   `UnsupportedEnvelopeVersion` să NU fie ACK-uită (re-raise înainte de `xack`), restul rămâne ca acum.
5. `src/worker/dispatcher.py`: în `dispatch_row`, verifică `payload.get("v", 0)` în set; necunoscut →
   `mark_failed` (backoff, nu dead).
6. `CONTRIBUTING.md`: secțiune scurtă „Versionarea payload-urilor de stream/outbox" cu regula de evoluție.
7. Teste (`tests/test_envelope_version.py`): v lipsă (v0) procesat; v1 procesat; v=99 (viitor) →
   `UnsupportedEnvelopeVersion` + nu ACK-uit; payload outbox v necunoscut → mark_failed nu dead;
   `to_dict()` conține `v`. Verifică round-trip „producător v1 → consumer v1".
8. `ruff check . && ruff format . && pytest -x -q` verde.

## Out of Scope
- **Migrarea efectivă la v2** (o schimbare reală de formă) — NX-17 livrează DOAR mecanismul `v=1` +
  toleranța N/N-1. Prima schimbare incompatibilă reală e un task viitor care va folosi acest mecanism.
- **Versionare pe alte structuri** — `conversations.state` (are deja `state_version` optimistic lock),
  `messages.payload`, `analytics_events.properties`: nu sunt în zbor între workeri la deploy, nu intră aici.
- **Negociere de versiune / handshake** între producător și consumer — overkill; toleranța statică N/N-1 e suficientă.
- **Dead-letter queue dedicat** pentru envelope din viitor — v1 le lasă neACK-uite în PEL (Redis) /
  scadente în outbox; un DLQ separat e follow-up dacă apare nevoia.

## Test Cases
**Happy Path:**
1. `InboundEvent(...).to_dict()` → conține `{"v": 1, "kind": "message", ...}`; `process_event` îl
   rutează la `handle_turn` (mock) fără excepție.
2. Round-trip rolling deploy: payload de outbox scris cu `v=1` de `processor` → `dispatch_row` îl
   acceptă și apelează sender-ul (mock) — confirmă producător-nou → consumer-nou.

**Edge Cases:**
1. Envelope vechi FĂRĂ cheia `v` (dict construit manual, simulează mesaj v0 rămas în coadă de la
   worker-ul anterior) → `_envelope_version` întoarce 0 → procesat normal (compatibilitate înapoi).
2. Envelope `kind="status"` cu `v=1` → procesat de ramura de status, fără să atingă rutarea de mesaj.
3. Payload outbox FĂRĂ `v` (rând vechi din DB înainte de deploy) → `dispatch_row` îl tratează ca v0 →
   trimite normal.

**Failure Cases:**
1. Envelope cu `v=99` (din viitor — worker vechi citește mesaj de la producător nou) →
   `process_event` ridică `UnsupportedEnvelopeVersion`; în `consume_once`, `redis.xack` NU se apelează
   pentru acel msg_id (verificat pe fake redis) → mesajul rămâne în PEL pentru worker-ul nou.
2. Payload outbox cu `v=99` → `dispatch_row` → `mark_failed` (backoff), statusul NU e `dead`
   (verificat pe stub de outbox) → un dispatcher mai nou îl va prelua.

> Teste 100% deterministe (fără LLM, fără Redis/DB real): `to_dict()` sunt funcții pure;
> `process_event`/`dispatch_row` rulate cu fake redis + stub-uri de conn/sender. ZERO apeluri reale în CI.
> Acest task NU cheamă LLM/embeddings — deci fără cost per mesaj (cost adăugat: $0).
