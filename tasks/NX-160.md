# NX-160 — Generic Customer Memory v2 (capture broad → classify safety → canonicalize → inject safe)

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** Memory · **Branch:** `feat/NX-160-generic-memory-v2` · **Complexitate:** L · **Estimare:** 2-3 zile → **[PROPUNERE] livrat feliat (6 felii, vezi Implementation Steps)**

## Goal
Ridică memoria de la „facts pentru beauty" la **memorie generică pe orice business** (magazin,
service auto, salon, restaurant/brutărie, clinică, sală, florărie, agenție). NX-148 a livrat
mecanismul, dar o simulare pe cod real a scos două defecte: (1) în live lipsesc migrările `022`+`023`
→ `conversation_facts` nu există → `upsert_facts` crapă `UndefinedTableError` înghițit tăcut de
savepoint → **memoria e practic moartă**; (2) `select_whitelisted_facts` e **fail-closed** →
modelul nu nimerește cheile exacte din whitelist (a emis `budget_max_lei`/`preferred_brand`/
`fragrance_free_preference` vs `budget_band`/`fav_brands`/`restriction`) → **4/5 facts aruncate**.
Direcția corectă (agreată Claude + Codex): **capture broad → classify safety → canonicalize if
possible → inject only safe memory**. Design complet: [`docs/MEMORY-ARCHITECTURE-2026.md`](../docs/MEMORY-ARCHITECTURE-2026.md).

## Business Context
O agenție SaaS onboardează orice magazin. Whitelist-ul per-vertical scris de mână = cold-start pe
fiecare client nou (zero config → zero memorie) + reține doar ce a prezis un om. Memoria trebuie să
fie o **capacitate generică a platformei**: un service auto reține „Golf 7 diesel, plăcuțe, buget
300 lei", o brutărie reține „fără gluten, 6 persoane, 150 lei" — fără ca nimeni să scrie o listă de
chei. Memoria bună = botul „te știe" între tururi → recomandare mai bună → conversie. Memoria moartă
(cazul live de azi) = re-întreabă, sună robotic, pierde vânzarea.

## Technical Description

### Trei niveluri, clar separate (§2-3 din doc)
Memoria are DOI consumatori cu nevoi OPUSE: **modelul** (recall în prompt) vrea vocabular DESCHIS
(generalitate); **codul** (search/analytics/CRM/profile) vrea vocabular CANONIC (chei stabile).
Whitelist-ul unic îi confundă. Soluția — 3 niveluri:

1. **Raw Candidate Memory** — ce extrage modelul liber (`raw_key`: `preferred_brand`, `vehicle_model`,
   `diet_preference`, `preferred_time`). Fără poartă fail-closed.
2. **Safe Recall Memory** — doar facts SIGURE care pot ajunge în prompt (`visibility='inject'`):
   preferințe, buget, restricții comerciale, stil, mărime, brand, program, scop. NU telefon/email/
   adresă/card/CNP/diagnostic medical.
3. **Canonical Business Memory** — `canonical_key` pt cod (search/profile/analytics). Surse: nucleu
   universal (`budget_band`, `fav_brands`, `restriction`, `size`, `use_case`, `recipient`,
   `style_pref`, `preferred_time`) + `DomainPack.profile_whitelist`/`fact_type_whitelist`/
   `searchable_facets` + (viitor) facets derivate din catalog.

### Fluxul post-tur
`extract (raw_key + canonical_key + source_ref) → safety_filter (clasează + setează visibility) →
canonicalizer (determinist, alias map) → upsert → [tur următor] fetch visibility='inject' → facts_block safe`

### Safety Gate (diferența critică)
`Sunt diabetic, ce desert?` → NU se memorează `health_condition=diabetic` injectabil. Se marchează
`safety_class='health', visibility='candidate'` (semnal intern, NU folosit automat). DAR `fără zahăr`
formulat ca preferință de produs → `restriction`, `inject`. Asta e linia: **condiție medicală ≠
preferință comercială**. Leagă P0-safety (`safety_medical_guardrail`).

### Canonicalizer (determinist, fără LLM nou în v1)
Alias map universal: `preferred_brand|brand_preference→fav_brands`, `budget_max_lei|max_budget|
price_limit→budget_band`, `fragrance_free_preference|avoid_fragrance→restriction`,
`vehicle_make_model→vehicle_model`, `appointment_time_preference→preferred_time` + mapări din
DomainPack. Fără match sigur → rămâne raw candidate (nu-l folosește codul, dar dacă e safe se poate
injecta cu `raw_key` formatat prezentabil).

### contacts.profile — un singur adevăr
Pe termen scurt păstrăm `contacts.profile` ca azi (nu rupem codul), dar îl **actualizăm din canonical
facts high-confidence**. Țintă medie: `profile` = materializare/cache din canonical facts safe.

## Implementation Steps (feliat)
**Felia 1 — deploy fix + no silent dead layer**
1. Rulează `scripts/migrate.py` pe live → aplică `022`+`023`; `scripts/migrate.py --check` verde.
2. `upsert_facts`: `UndefinedTableError`/orice fail de persistare → emite `facts_persist_failed {error_type}` (nu doar `pass`). Ideal: poarta de migrări și pe web/webhook (pipeline in-process), nu doar worker.

**Felia 2 — schema v2**
3. Migrare `docs/024_conversation_facts_memory_v2.sql`: adaugă `raw_key`, `canonical_key NULL`,
   `memory_key NOT NULL`, `safety_class NOT NULL default 'unknown'`, `visibility NOT NULL default
   'candidate'`, `source_message_id uuid NULL`. Unique NOU `(business_id, contact_id, memory_key)`;
   DROP vechiul unique `(…, fact_type)`. `fact_type` rămâne alias backcompat.

**Felia 3 — extractor**
4. Prompt nano cu chei canonice disponibile (nucleu + DomainPack) → output `raw_key`+`canonical_key`+
   `source_ref`. `Message` primește `id`; `get_messages_for_extraction` selectează `id`; transcript cu
   refs `m1/m2/m3` → model întoarce `source_ref` → cod mapează la `message.id` (fallback: ultimul inbound al turului).

**Felia 4 — safety_filter + canonicalizer**
5. `src/worker/memory_safety.py`: clasează `safety_class` + setează `visibility` (PII/financial→drop,
   medical→candidate, comercial→inject). `src/worker/canonicalize.py`: alias map universal + DomainPack.

**Felia 5 — read path**
6. `fetch_relevant_facts`: DOAR `visibility='inject'` + ne-expirate + tenant. `facts_block` formatat
   safe (labels prezentabile, nu snake_case brut, nu candidates, nu PII).

**Felia 6 — teste**
7. Pure + integration + un smoke cu nano real (beauty/auto/restaurant/service/PII/medical/source/RLS).

## Test Cases
**Happy Path:**
- Beauty: „ten sensibil, buget 100 lei, prefer CeraVe, fără parfum" → `skin_type`+`budget_band`+
  `fav_brands`+`restriction`, toate `inject`.
- Auto: „Golf 7 diesel, plăcuțe frână până în 300 lei" → `vehicle_model`+`fuel_type`+`part_category`+`budget_band`.
- Restaurant/brutărie: „fără gluten, 6 persoane, buget 150 lei" → `diet_preference/restriction`+`quantity_context`+`budget_band`.
- Service: „programare după ora 18" → `preferred_time`.

**Edge Cases:**
- Medical: „sunt diabetic/însărcinată/alergie severă" → `safety_class` sensibil, `visibility!='inject'`;
  DAR „fără zahăr" ca preferință → `restriction`, `inject`.
- `source_message_id` nenul pe fact persistat; fact re-menționat → `last_seen` bump, nu duplicat; canonicalizare stabilă (`preferred_brand`→`fav_brands`).

**Failure Cases:**
- PII (telefon/email/adresă) → `drop`, nimic injectat, ZERO valori în analytics.
- Tabel lipsă/`UndefinedTableError` → `facts_persist_failed` emis (nu tăcere).
- RLS/tenant: facts ale altui `business_id` invizibile; upsert/fetch respectă `bot_runtime`.
- LLM replay/mock în CI — ZERO apeluri reale (smoke-ul cu nano real e separat, out-of-CI).
