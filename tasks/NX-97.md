# NX-97 — Provisioning bot_runtime LOGIN în prod (apply_005 + DATABASE_URL_BOT)
**Owner:** S · **Faza:** P1 · **Zi/Ord:** înainte de trafic real cu >1 client · **Branch:** `feat/NX-97-bot-runtime-login-cutover` · **Complexitate:** S · **Estimare:** 1.5h

## Goal
Activează în PROD calea de izolare sigură pentru tenant path: rolul de LOGIN
`bot_runtime` (conexiune directă, fără bypassrls) în loc de modul compat (admin +
`SET ROLE` în init). Codul e deja livrat (NX-50), dar pentru că migrarea `005` nu e
aplicată și `DATABASE_URL_BOT` e nesetat, workerul rulează pe compat — exact
scurgerea **P0-A** (identitate de sesiune ce se poate scurge sub multiplexarea
poolerului) pe care NX-50 trebuia s-o închidă. RLS ca defense-in-depth e practic
inactiv. Task de **deploy/ops**, zero cod nou: aplici `005`, setezi env-ul de prod,
verifici că tenant path se loghează direct ca `bot_runtime` fără `SET ROLE`.

## Business Context
Multi-tenant pe o singură schemă: un query scăpat fără filtru de tenant TREBUIE să
întoarcă 0 rânduri, nu datele altui client. Atâta vreme cât workerul rulează pe
compat (login ca `postgres` + `SET ROLE bot_runtime`), un `SET ROLE` care se scurge
sub pooler-ul în transaction-mode poate servi date cross-tenant sau un drum
superuser care ocolește RLS. Pentru o agenție care vinde „datele clientului tău sunt
izolate", asta e riscul de încredere numărul unu. Cutover-ul îl elimină structural,
fără să schimbe nimic în comportamentul botului.

## Technical Description
**Codul e complet — NU se modifică în acest task.** Verificat în
`src/db/connection.py`:

- `get_bot_pool()` (linia 144) decide modul după settings:
  `_bot_login_mode = bool(s.database_url_bot)`; `dsn = s.database_url_bot or
  s.supabase_db_url`. Dacă `DATABASE_URL_BOT` e setat → login DIRECT ca
  `bot_runtime`, init = `_assert_bot_role` (verifică `current_user == 'bot_runtime'`),
  `statement_cache_size=100`. Dacă e gol → compat: login ca admin, init =
  `_init_bot_conn_compat` care face `set role bot_runtime` o singură dată.
- `tenant_conn(business_id)` (linia 197) NU mai face `SET ROLE` în niciun caz —
  setează doar `app.business_id` per checkout și îl resetează la release. Asta e
  esența: identitatea vine din credențiale, nu din comandă de sesiune.
- `_check_isolation` (linia 124) e plasa NX-04: la fiecare checkout, dacă
  `current_user != 'bot_runtime'` SAU `app.business_id` lipsește/diferă → `IsolationError`
  ÎNAINTE de primul query (default `DB_ISOLATION_ASSERT=strict`).
- Boot-ul workerului (`src/worker/consumer.py:156`) cheamă `get_bot_pool()` EAGER →
  o parolă greșită în `DATABASE_URL_BOT` crapă la pornire, nu la primul mesaj.

Diferența observabilă post-cutover, în acelaşi `tenant_conn`:
- compat: `current_user` raportat de pool e `postgres`/`postgres.<ref>` (logarea
  reală), coborât la `bot_runtime` prin `set role` în init;
- login: `current_user == 'bot_runtime'` din credențiale, fără niciun `set role`.

Migrarea `docs/005_bot_runtime_login.sql` face DOAR
`alter role bot_runtime login password :'bot_password'` — niciun obiect nou, niciun
GRANT nou (toate există din `003`/`004`). `scripts/apply_005.py` rulează ca admin,
citează parola server-side cu `format(%L)` (zero injection) și verifică
`rolcanlogin=t, rolbypassrls=f, rolsuper=f`.

**Nuanță de conectivitate (Supabase):** login cu rol custom (`bot_runtime`) merge fie
pe conexiunea DIRECTĂ (`db.<ref>.supabase.co:5432`), fie pe Session pooler cu user
`bot_runtime.<ref>`. Confirmi care funcționează din rețeaua VPS-ului (aceeași clasă
de problemă ca nuanța IPv4 din T018). Dacă login-ul direct nu merge din rețea, rămâi
pe compat și documentezi — NU forța.

## Implementation Steps
1. **Pre-check (ca admin):** confirmă că `bot_runtime` există ca `nologin` (din 003):
   `select rolname, rolcanlogin from pg_roles where rolname='bot_runtime'`.
   Dacă lipsește → rulează întâi `scripts/apply_003.py` (dependență).
2. **Generează parola** `bot_runtime` în vault (lungă, aleatoare). Notează că trebuie
   percent-encodată în DSN dacă are caractere speciale (`@`→`%40` etc. — vezi memoria
   DB URL password encoding).
3. **Aplică 005:** `BOT_RUNTIME_PASSWORD='<din vault>' python scripts/apply_005.py`.
   Confirmă output-ul: `LOGIN activat=True`, `bypassrls=False`, `superuser=False`.
4. **Probă de izolare conectat DIRECT ca bot_runtime** (din SQL editor cu noile
   credențiale sau script ad-hoc — vezi blocul de verificare din `005.sql`):
   - `select current_user` → `bot_runtime`
   - `select count(*) from products` → `0` (fără `app.business_id` → RLS fail-closed)
   - `select set_config('app.business_id', '6098812a-50fc-44bd-a1ba-bc77e6399158', false); select count(*) from products` → 500 (doar tenantul demo).
5. **Setează env-ul de prod (VPS):** `DATABASE_URL_BOT=postgresql://bot_runtime:<parola>@<host>:5432/postgres`
   în `.env`-ul de pe VPS (separat de `SUPABASE_DB_URL`, care rămâne admin/migrări).
   Folosește host-ul confirmat funcțional la pasul 4 (direct vs Session pooler).
6. **Restart workeri** (`docker compose up -d worker dispatcher`) și verifică în
   loguri că `get_bot_pool()` a pornit fără `RuntimeError` din `_assert_bot_role`
   (parolă/rol corect).
7. **Verificare runtime (DoD):** rulează un script de probă care deschide
   `tenant_conn(demo)` și citește `current_user` → trebuie `bot_runtime`, fără ca
   modul compat să fi rulat (`_bot_login_mode is True`). Vezi Test Cases.
8. **Actualizează TODO-MANUAL.md** (bifează pașii NX-50) + `docs/PROJECT_STATUS.md`
   (cutover P0-A făcut în prod). Niciun cod de aplicație modificat.

**Happy Path:**
1. apply_005 cu `BOT_RUNTIME_PASSWORD` corect → toate 3 verificările OK; rolul devine
   LOGIN fără bypassrls/superuser.
2. Cu `DATABASE_URL_BOT` setat și worker repornit, `tenant_conn(demo)` deschide
   conexiune, `current_user == 'bot_runtime'`, `_bot_login_mode True`, citește 500
   produse pentru demo (RLS filtrează corect la tenant).

**Edge Cases:**
1. `DATABASE_URL_BOT` gol (status quo) → cod cade pe compat (login admin + `SET ROLE`),
   `_assert_bot_role` tot trece (rol efectiv `bot_runtime`) dar `_bot_login_mode` e
   False → probă runtime PICĂ DoD-ul (semnalează că cutover-ul nu s-a făcut).
2. Login direct nu merge din rețeaua VPS (Supabase blochează rolul custom pe portul
   ales) → rămâi pe Session pooler cu user `bot_runtime.<ref>`; dacă nici acela →
   rămâi pe compat și documentezi în PROJECT_STATUS (nu forța).

**Failure Cases:**
1. `DATABASE_URL_BOT` cu parolă greșită → `get_bot_pool()` (eager la boot,
   consumer.py:156) crapă la pornirea workerului, nu la primul mesaj — fail-fast,
   nu un drum tăcut.
2. `DATABASE_URL_BOT` setat din greșeală spre un rol privilegiat (ex. `postgres`) →
   `_assert_bot_role` ridică `RuntimeError` la init („rol efectiv X, aștept
   'bot_runtime'") → pool-ul NU pornește, nu rulează un drum care ocolește RLS în tăcere.
3. La un checkout, `app.business_id` ajunge nesetat/greșit (reuse murdar) →
   `_check_isolation` ridică `IsolationError` ÎNAINTE de query + `log.critical`
   (fără PII), GUC resetat — regresia de izolare pică zgomotos.
