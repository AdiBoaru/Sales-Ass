# NX-87 — Debounce durabil + creare conversație fără race
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după NX-51 (dedupe L2) · **Branch:** `feat/NX-87-debounce-conv-race` · **Complexitate:** M · **Estimare:** 4h

## Goal
Închide două găuri de fiabilitate pe calea inbound, ambele apărute când treci debounce-ul
de pe canalul de TEST (Telegram) spre producție: (1) bufferul de debounce e in-process, iar
mesajele se XACK-uiesc ÎNAINTE de flush → un crash în fereastra de 3s pierde mesajele
tampon (deja confirmate pe stream, deci ne-recuperabile); (2) `get_or_create_conversation`
face SELECT-apoi-INSERT fără unique parțial / advisory lock → două prime-mesaje strict
simultane pentru un contact pot crea două conversații deschise paralele (state split,
istoric rupt). Creăm o conversație race-safe (exact pattern-ul dovedit de `get_or_create_contact`)
și garantăm că niciun mesaj confirmat nu se pierde înainte de a fi procesat.

## Business Context
Pe producție (WhatsApp), Meta face retry agresiv și userii trimit rafale de mesaje scurte.
Dacă un crash de worker pierde tampon-ul de debounce, clientul a scris și botul tace —
exact eșecul pe care principiul 6 îl interzice, și pe un canal unde clientul plătește.
Două conversații deschise pe același contact înseamnă state împărțit (agentul „uită" ce-a
afișat în cealaltă), atribuire de conversie ruptă și un dashboard care numără dublu. Ambele
sunt invizibile la trafic mic și apar fix când volumul crește — adică exact când contează.

## Technical Description

### Starea actuală (de ce e gaură)
- `src/worker/consumer.py:consume_once` face `debouncer.add(event)` apoi `redis.xack(...)`
  în `finally` — deci mesajul e ACK-uit pe stream IMEDIAT, dar procesarea reală se întâmplă
  `delay` secunde mai târziu, în `Debouncer._flush_later`, din alt task. Între ACK și flush,
  un crash pierde tot bufferul (nu mai e re-livrat de XREADGROUP, fiind deja ACK-uit).
- `src/db/queries/conversations.py:get_or_create_conversation` (74-95): `SELECT open` →
  dacă None, `INSERT`. Fără unique parțial pe open-conv și fără advisory lock, două tururi
  concurente trec amândouă de SELECT (zero rânduri) și inserează două conversații `status='open'`.
  Comentariul din cod recunoaște deja gaura („Hardening (advisory lock) → follow-up").
- Contrast: `get_or_create_contact` (contacts.py:100-154) e DEJA race-safe prin
  `unique(business_id, channel_kind, external_id)` + `ON CONFLICT DO NOTHING` + re-SELECT
  câștigătorul. Replicăm exact acest model pentru conversație. (Crearea contactului NU e
  atinsă de acest task — e deja corectă.)

### Fix 1 — ACK după flush (debounce durabil pe calea de TEST)
Mutăm responsabilitatea de ACK din `consume_once` în `Debouncer`: bufferul ține și `msg_id`-ul
de stream alături de eveniment; ACK-ul se face DOAR după ce `_process(combined)` a reușit,
pentru TOATE mesajele din lot. Astfel un crash înainte de flush lasă mesajele ne-ACK-uite →
XREADGROUP le re-livrează (sau XAUTOCLAIM, follow-up). Idempotența la re-livrare e deja
acoperită de dedupe L2 (`claim_inbound`, NX-51), deci re-procesarea unui mesaj parțial nu
dublează efecte.

```python
# debounce.py — bufferul ține (event, msg_id); ACK delegat
class Debouncer:
    def __init__(self, process, ack, *, delay=DEBOUNCE_SECONDS):
        self._process = process            # async (combined_event) -> None
        self._ack = ack                    # async (list[str msg_id]) -> None
        ...
        self._buffers: dict[tuple, list[tuple[dict, str]]] = {}

    async def add(self, event, msg_id: str) -> None:
        key = self._key(event)
        self._buffers.setdefault(key, []).append((event, msg_id))
        ...

    async def _flush_later(self, key):
        ...
        items = self._buffers.pop(key, [])
        if not items:
            return
        events = [e for e, _ in items]
        msg_ids = [m for _, m in items]
        try:
            await self._process(self._combine(events))
            await self._ack(msg_ids)         # ACK DOAR la succes → crash = re-livrare
        except Exception:
            log.exception("debounce: lot eșuat — mesaje NEACK-uite (re-livrare)")
```

În `consumer.py`, mesajele de tip `message` NU se mai ACK-uiesc în `finally`; doar statusurile
(procesate imediat) și mesajele cu eroare de PARSE (json invalid → ACK + log, n-are sens re-livrate)
se ACK-uiesc inline. `run_consumer` injectează `ack = lambda ids: _ack_batch(redis, ids)`.

> **Producție (notat, nu în acest task):** varianta complet durabilă mută bufferul în Redis
> (cheie per expeditor + ZADD cu timestamp), ca un restart de worker să nu piardă nici lotul
> in-flight. Aici facem ACK-after-flush (recuperare la crash via re-livrare stream), suficient
> pentru a închide gaura fără a rescrie mecanismul. Buffer-Redis = NX separat dacă mutăm pe prod.

### Fix 2 — conversație deschisă race-safe (unique parțial + ON CONFLICT)
Migrare `009`: unique parțial pe conversația deschisă per (business, contact, canal):

```sql
create unique index if not exists uq_conversations_one_open
  on public.conversations (business_id, contact_id, channel_id)
  where status = 'open';
```

`get_or_create_conversation` urmează exact pattern-ul `get_or_create_contact`: fast-path SELECT,
iar la creare un INSERT cu `ON CONFLICT DO NOTHING` pe indexul parțial; dacă pierde cursa
(`returning` gol) → re-SELECT câștigătorul.

```python
existing = await get_open_conversation(conn, business_id, contact_id, channel_id)
if existing is not None:
    return existing
row = await conn.fetchrow(
    f"""
    insert into conversations (business_id, contact_id, channel_id, locale)
    values ($1, $2, $3, $4)
    on conflict (business_id, contact_id, channel_id) where (status = 'open')
        do nothing
    returning {_CONV_COLS}
    """,
    business_id, contact_id, channel_id, locale,
)
if row is not None:
    return _row_to_dict(row)
# am pierdut cursa → câștigătorul există acum
won = await get_open_conversation(conn, business_id, contact_id, channel_id)
if won is None:  # pragma: no cover — imposibil după conflict pe indexul parțial
    raise RuntimeError("conversation open conflict fără rând existent")
return won
```

PostgreSQL acceptă `ON CONFLICT ... WHERE` ca arbiter pentru indexul parțial — predicatul
`(status = 'open')` trebuie să-l reproducă exact pe cel al indexului. Avantaj față de advisory
lock: zero lock global de ținut, izolare strictă pe `business_id` (predicatul indexului e
tenant-scoped prin coloane), și INSERT-ul orfan nici nu apare (DO NOTHING, nu rollback ca la
contact). Păstrăm `ON CONFLICT` în loc de advisory lock tocmai pentru că nu introduce o cheie
de lock globală peste businessuri (un advisory lock ar trebui derivat din hash-ul cheii →
risc de coliziune cross-tenant; constraint-ul DB e curat).

### Câmpuri TurnContext scrise
**Niciunul nou.** `get_or_create_conversation` rămâne stratul de date (alimentează
`conv["id"]`/`state`/`state_version` din care `handle_turn` construiește `TurnContext`).
Debounce-ul operează pe envelope-ul brut, ÎNAINTE de pipeline — nu atinge `TurnContext`.

### Error handling
- Debounce: lot eșuat → mesaje ne-ACK-uite + log (FĂRĂ corpul mesajului, P12) → re-livrare.
- Conversație: conflict pe indexul parțial = drum normal (re-SELECT), NU eroare propagată.
- Re-procesarea unui mesaj re-livrat e idempotentă prin `claim_inbound` (dedupe L2) — al doilea
  pas pe același `provider_msg_id` iese devreme cu `deduped=True`.

## Principii CLAUDE.md aplicabile
- **P6 (niciodată tăcere):** ACK-after-flush garantează că un mesaj confirmat ajunge procesat
  chiar și după crash — nu mai există fereastra de 3s în care un mesaj dispare tăcut.
- **P7 (business_id pe tot):** indexul parțial e tenant-scoped prin coloane (`business_id,
  contact_id, channel_id`); `get_or_create_conversation` păstrează `where business_id = $1`.
  Conversația deschisă unică e enforced în DB ca plasă, izolarea primară rămâne în cod.
- **P3 (un singur proprietar per câmp):** o singură conversație deschisă = un singur owner
  pentru `conversations.state` per (contact, canal). Două conversații deschise ar însemna doi
  scriitori pe „același" state logic — exact ce P3 interzice.
- **P12 (PII):** logurile de debounce/conflict conțin DOAR id-uri + count, niciodată body sau
  `sender_external_id` (PII de canal stă în channel_identities).

## Test Cases
**Happy Path:**
1. `tests/test_debounce.py` — două mesaje rapide ale aceluiași user → un flush, iar
   fake-ack-ul primește AMÂNDOUĂ `msg_id`-urile DUPĂ `_process` (verifică ordinea: process
   înainte de ack, cu un `process` care înregistrează un timestamp/ordinal).
2. `tests/test_conversation_race.py` — un singur apel `get_or_create_conversation` pe un
   contact fără conversație → creează una `status='open'`; al doilea apel întoarce aceeași.

**Edge Cases:**
1. Expeditori diferiți în debounce → două loturi, ACK separat pe `msg_id`-urile fiecăruia
   (testul existent `test_separate_senders_not_merged` adaptat la noua semnătură).
2. Conversație deschisă DEJA existentă → fast-path SELECT, fără INSERT (verifică prin
   `state_version`/`id` neschimbat și că nu apare un al doilea rând).
3. Lot cu un singur mesaj → flush, un singur `msg_id` ACK-uit (testul `test_single_message_passes_through`
   extins cu asserția de ack).

**Failure Cases:**
1. `_process` aruncă în `_flush_later` → fake-ack NU e apelat (mesajele rămân ne-ACK-uite),
   excepția e prinsă și logată, worker-ul nu moare.
2. Cursă reală de creare: `asyncio.gather` pe două `get_or_create_conversation` concurente
   (două conexiuni tenant-scoped pe același business) → exact un `status='open'` în DB,
   amândouă apelurile întorc același `id`, niciun INSERT orfan rămas.
3. Re-livrare după „crash": un mesaj re-procesat (același `provider_msg_id`) după ce a fost
   ne-ACK-uit → `claim_inbound` (dedupe L2, stub în test) îl marchează `deduped=True`, fără
   dublare de outbox.
