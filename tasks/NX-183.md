# NX-183 — ResponseEnvelope V2-light: contract + compose + renderer text-only

**Owner:** Claude (build) / Codex (verify) · **Faza:** P1 · **Track:** Response Quality · **Branch:** `feat/NX-183-envelope-v2` · **Complexitate:** M · **Estimare:** 3-4 zile
**Parent plan:** [docs/RESPONSE-QUALITY-EPIC.md](../docs/RESPONSE-QUALITY-EPIC.md) · **Depinde de:** NX-180 (baseline), NX-181 (măsurat), **gate de decizie** (se construiește DOAR dacă evaluatorul confirmă reziduul structural). **NU depinde de NX-184** — NX-183 e self-contained (`response_shape` opțional cu fallback); NX-184 depinde unidirecțional de NX-183.

## Goal
Rupe forma rigidă intro→carduri→pick→education păstrând siguranța factuală. Modelul scrie `lead` liber, natural, legat de întrebarea curentă; selectează produse și dovezi din meniuri OPACE prezentate de cod. **Codul compune orice afirmație factuală.** Renderer-ul acceptă explicit răspunsuri **text-only, zero carduri**, inclusiv un răspuns factual inline („X e mai lejer") compus de cod.

## Decizie de produs (închisă)
Alegem **siguranța factuală** fără să sacrificăm naturalețea:
- `lead` = conversațional liber, dar FĂRĂ afirmații tehnice noi (scrub ca `intro` azi: cifre/claims/medical).
- Motivele factuale = **microcopy compact compus de cod** din evidence, NU propoziții rigide, NU text liber pe atribute.
- BINE: `Pentru ce ai descris, varianta asta pare cea mai potrivită.` + `Textură lejeră · Potrivită pentru ten gras · În buget`
- NU: `Am selectat acest produs deoarece are textura lejeră și este potrivit pentru tenul gras.`
- Parafrazarea controlată a microcopy-ului = DOAR ulterior, dacă evaluatorul arată că sună prea mecanic. Nu în v1.

## Contract (UNIC — schema V2)
```json
{
  "lead": "Dintre cele două, aș merge pe aceasta.",
  "answer": {"product_id": "p1", "evidence_ids": ["e1"], "presentation": "inline"},
  "products": [
    {"product_id": "p1", "evidence_ids": ["e1", "e3"], "reason_style": "best_if"}
  ],
  "follow_up": null
}
```
- `lead` — mesajul conversațional (scrubuit; fără afirmații tehnice noi).
- `answer` (nullable) — răspunsul FACTUAL text-only: codul compune numele produsului + afirmația din evidence (`presentation: "inline"` = fără card). Pentru „dintre ele care e mai lejeră?" răspundem factual FĂRĂ relistare.
- `products` — recomandări-card; **poate fi gol** (text-only e valid). `reason_style`: `best_if | good_for | note` → microcopy per-locale compus determinist.
- `follow_up` (nullable) — cel mult o întrebare, doar când ajută.
- **`evidence_ids` sunt OPACE** (`e1`, `e2`…): generate de COD per produs în bundle (mapate la fapte reale: fațete/reason_codes/pros). Modelul **selectează dintr-un meniu**, nu poate inventa dovezi (nu există căi semantice ghicibile ca `texture.light`). Id necunoscut → eliminat determinist.
- `product_id` validat prin membership. Prețuri, nume, linkuri, stoc = injectate de cod ca azi.

## Technical Description
- **Renderer text-only:** `render` cere azi `rich.items` ne-gol (`finalize.py:372`) → adaugă **ramură explicită** text-only (`lead` + `answer` compus, zero carduri) ȘI adaptor V2→`RichReply` pt recommendation/detail (lead→intro, microcopy→`reason`), ca să NU rescrii call-site-urile (`flatten`, `flatten_framing`, `card_products`, cross-sell).
- **Meniu de evidence:** bundle-ul per produs listează evidence-urile disponibile ca `[e1] <fapt>`; modelul referă doar id-urile.
- **`response_shape` opțional** în input (de la NX-181/planner); lipsă/invalid → fallback `recommendation`. NX-184 îl îmbogățește ulterior.
- **Fallback tehnic:** doar pe eroare de apel structurat → V1/proză; **NU** reintroduce produse respinse (setul rămâne cel gated).

## Implementation Steps
1. Schema V2 (contractul unic de mai sus) + validare structured outputs.
2. Generarea meniului de evidence opace per produs (cod) + membership la întoarcere.
3. Compunere microcopy determinist (`reason_style` × locale) + compunere `answer` inline.
4. Ramură text-only în `render` + adaptor V2→RichReply.
5. Fallback tehnic care păstrează setul gated.
6. Cache: namespace `envelope_version` (pre-triaj); direct/detail/repeat rămân `cacheable=False`.
7. Evaluator ON vs Prompt-vNext-only.
