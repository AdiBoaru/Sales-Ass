# NX-60 — Abstracție de canal: envelope neutru + ChannelSender registry
**Owner:** S · **Faza:** MVP (test multi-canal) · **Ord:** înainte de NX-61/62 · **Branch:** feat/NX-60-channel-abstraction · **Complexitate:** M · **Estimare:** 6h

## Goal
Izolează cuplajul de canal la cele DOUĂ margini ale sistemului — ingestia și
trimiterea — printr-un envelope neutru și un registru de sender-e, astfel încât
adăugarea unui canal nou (Telegram, mai târziu Instagram/web) să NU atingă
worker-ul, pipeline-ul sau stratul de date. Pipeline-ul e deja agnostic; acest
task face explicit contractul.

## Business Context
Vrem să testăm rapid produsul vorbind direct pe Telegram (pe VPS), fără să
atingem calea WhatsApp existentă. Fără abstracție, codul Telegram ar refolosi
nume Meta („phone_number_id", „wa_id") cu sens schimbat — confuz și fragil.
Un contract de canal curat e și fundația pentru web widget (E26).

## Technical Description
Azi cuplajul de canal e în: `webhook/meta.py` (parser), `webhook/signature.py`
(verify Meta), `redis_bus.py` (cheia de dedupe), `worker/consumer.py`
(`resolve_channel_by_phone`), `worker/processor.py` (`_CHANNEL_KIND="whatsapp"`
hardcodat), `dispatcher.py` + `meta_client.py` (trimitere) și `outbox.claim_due`
(întoarce `phone_number_id`). Pipeline-ul/runner-ul sunt deja neutre.

**1. Envelope neutru (inbound).** `InboundEvent` capătă câmpuri agnostice:
`channel_kind` (whatsapp|telegram|...), `channel_account_id` (provider_account_id
al canalului RECEPTOR — phone_number_id la WA, bot id la TG), `sender_external_id`
(id-ul userului pe canal — wa_id / chat_id), `sender_name`, `provider_msg_id`,
`content_type`, `body`, `media_id`, `timestamp`, `payload`. Parserul Meta
mapează la aceste nume (wa_id→sender_external_id, phone_number_id→channel_account_id).
`to_dict` păstrează `kind` (message|status). Dedupe layer 1 se cheamă pe
`(channel_account_id, provider_msg_id)`.

**2. Rezolvare generică.** `resolve_channel(conn, channel_kind, provider_account_id)`
în `db/queries/channels.py` (înlocuiește/încadrează `resolve_channel_by_phone`,
care rămâne wrapper subțire cu `kind='whatsapp'`). Consumer-ul folosește
`event["channel_kind"]` + `event["channel_account_id"]`.

**3. Identity resolution agnostică.** `processor.handle_turn` ia `channel_kind`
din event (NU hardcodat); `get_or_create_contact` primește `channel_kind`.

**4. ChannelSender registry (outbound).** `src/channels/base.py`:
```python
class ChannelSender(Protocol):
    async def send_text(self, account_id: str, to: str, text: str) -> str: ...  # → provider_msg_id
```
Un registru `{channel_kind: ChannelSender}`. `MetaClient` implementează protocolul
(`account_id`=phone_number_id). `claim_due` întoarce `channel_kind` +
`channel_account_id` (din join-ul outbox→conversations→channels), iar
`dispatch_row` alege sender-ul din registru după `channel_kind`.

## Principii CLAUDE.md aplicabile
P1 (pipeline neutru — cuplajul stă la margini, nu curge înăuntru) · P5 (un singur
punct de ieșire, acum cu mai multe transporturi în spate) · P12 (PII de canal tot
în `channel_identities`, indiferent de canal).

## Implementation Steps
1. `webhook/meta.py`: redenumește câmpurile `InboundEvent` la cele neutre + adaugă
   `channel_kind="whatsapp"` în parser; idem `StatusEvent` (channel_account_id).
2. `redis_bus.seen_before`: semnătură pe `(channel_account_id, provider_msg_id)`.
3. `webhook/app.py`: folosește noile nume.
4. `db/queries/channels.py`: `resolve_channel(kind, provider_account_id)`.
5. `worker/consumer.py`: rezolvă pe `channel_kind` + `channel_account_id` din event.
6. `worker/processor.py`: `channel_kind` din event în identity resolution.
7. `src/channels/base.py`: `ChannelSender` Protocol + `SENDER_REGISTRY`.
8. `outbox.claim_due`: întoarce `channel_kind` + `channel_account_id` (join există deja).
9. `meta_client.MetaClient`: conformă cu protocolul; înregistrată pt `whatsapp`.
10. `dispatcher.dispatch_row`: alege sender din registru pe `channel_kind`.
11. Actualizează testele existente la numele neutre (zero regresie funcțională).
