# NX-113 — Hybrid retrieval: FTS/pg_trgm + vector fused via RRF + deterministic rerank

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** Wave1 · **Branch:** `feat/NX-113-hybrid-retrieval-rrf` · **Complexitate:** L · **Estimare:** ~10h (XL — vezi [PROPUNERE] split la final)

## Goal
Înlocuiește retrieval-ul „ori-ori" (semantic SAU `name ILIKE '%query%'`) cu retrieval HIBRID adevărat: lexical real (Postgres FTS `tsvector` pe `name+brand+ai_summary` via `websearch_to_tsquery`, plus `pg_trgm` pentru SKU/cod-piesă) ȘI vector, rulate AMÂNDOUĂ pe câte un pool de ~30-50 candidați, fuzionate prin Reciprocal Rank Fusion (RRF), apoi un rerank DETERMINIST (boost in-stock / sale / concern-overlap) înainte de trunchierea la 6 — FĂRĂ niciun apel LLM suplimentar.

## Business Context
Retrieval-ul de produse e levierul nr. 1 de calitate, generic peste verticale. Azi:
- Calea lexicală e un singur `p.name ILIKE '%întreaga-frază%'` (`catalog.py:137`) care nu prinde aproape nicio propoziție naturală („caut o cremă bună pentru ten gras sub 80 lei").
- Pe demo-ul live `product_embeddings=0`, deci calea lexicală spartă E calea efectiv folosită.
- Semantic și lexical sunt mutual exclusive (ori-ori în `catalog_tools.py:160-198`), fără fuziune și fără reranker — „reranker"-ul din CLAUDE.md e neimplementat.

Standardul 2026 de product-RAG e BM25/FTS + vector fuzionate prin RRF, apoi rerank. E dublu spart pentru verticale cu SKU/cod-piesă (HVAC/auto) unde lexicalul contează cel mai mult — un cod „DAE-15X" nu se regăsește semantic, dar se regăsește lexical. Singura inteligență (creier) multi-canal, multi-vertical trebuie să găsească produsul corect indiferent de cum scrie clientul.

## Technical Description

### Calea curentă (de înlocuit)
- `search_products()` (`catalog.py:91`): filtre SQL dure + `p.name ILIKE` (linia 137).
- `search_products_semantic()` (`catalog.py:300`): JOIN pe `product_embeddings`, `ORDER BY pe.embedding <=> qvec`, vectorul inline-uit ca text literal (`catalog.py:319`, `[x,...]`).
- `search_products_tool()` (`catalog_tools.py:130`): alege semantic XOR sql-only pe `has_embeddings()` + relax-ladder.

### Modelul hibrid nou (în `catalog.py`)
1. **Lexical real** — funcție nouă `search_products_lexical(conn, business_id, query_text, *, filters..., pool=50)`:
   - `tsvector` generat pe `name + coalesce(brand,'') + coalesce(ai_summary,'')` (vezi Database Changes), interogat cu `websearch_to_tsquery('simple', $q)` → rank `ts_rank_cd`.
   - Tolerant la typo / SKU: `OR` pe `pg_trgm` similarity (`name % $q` / `similarity(name,$q)`) — esențial pentru cod-piesă (generic, nu beauty-specific).
   - Aplică ACELEAȘI filtre dure (category/brand/concerns/price/in_stock) ca semantic — paritate.
   - Întoarce ~`pool` rânduri cu rank, nu 6.
2. **Vector pool** — `search_products_semantic()` extins cu param `pool` → întoarce ~`pool` candidați (nu doar 6) când e chemat pentru fuziune.
3. **RRF fusion** — funcție pură Python `rrf_fuse(lexical_ranked, vector_ranked, *, k=60) -> list[product_id-ordered]`. Scor `sum(1/(k+rank_i))` pe fiecare listă; produsele apar în AMÂNDOUĂ urcă natural. Pură, deterministă, testabilă fără DB.
4. **Dedup vs displayed_products** — exclude id-urile deja afișate (`ctx.state.displayed_products`) ÎNAINTE de trunchiere (paritate cu „arată-mi altele"; complementar cu NX-119).
5. **Rerank determinist** — funcție pură `deterministic_rerank(fused, *, boosts)`: boost mic, stabil pentru `in_stock`, `sale_price < price`, `concern_overlap` (nr. concerns cerute care apar în `attributes->'concerns'`). NICIUN LLM (P2). Concern-overlap citește cheile din taxonomie; pe demo via `map_concerns` (`src/tools/taxonomy.py`), iar generic-across-verticals via taxonomy-from-DB (dependență moale pe **NX-124** — nu hardcodăm beauty).
6. **Trunchiere la 6** abia la final (hard cap din arhitectură).

### Orchestrare (în `catalog_tools.py:search_products_tool`)
- Rulează AMÂNDOUĂ retrieverele MEREU (nu XOR), pe pool, prin relax-ladder existent (`_relax_ladder`).
- Dacă LLM/embeddings lipsesc → doar lexical (degradare grațioasă, P6), dar lexicalul e ACUM real (FTS+trgm), nu `ILIKE`.
- Dacă FTS-ul nu prinde nimic dar vectorul da → fuziunea e doar vector (și invers).

### Channel-aware
Doare ambele canale, dar e mai vizibil pe **widget-ul web** (NX-127): acolo userii scriu fraze libere lungi („vreau ceva pentru tenul meu sensibil care se înroșește, buget 100 lei"), unde FTS+RRF câștigă cel mai mult față de WhatsApp (mesaje scurte). Output-ul rămâne neutru de canal: tool-ul întoarce `products` (max 6×8 câmpuri) → web le randează ca carduri/butoane, WhatsApp ca CTA/text. Retrieval-ul NU știe de canal (cuplaj doar la margini).

### Observabilitate (din runner / emit, P10)
Extinde event-ul `product_search` (`catalog_tools.py:201`) cu: `top_cosine_distance` (cel mai bun vector, semnal de calitate), `relax_depth` (treapta la care a ieșit ceva), `zero_result` (bool), `fused` (bool — au contribuit ambele), `lexical_pool`/`vector_pool` (counts). FĂRĂ text de query/concerns în properties (P12).

### Codec pgvector nativ
Înregistrează codecul nativ asyncpg pentru `vector` (la crearea pool-ului, `src/db/connection.py`) → eliminăm inline-ul vectorului ca text de ~15KB pe fiecare query (`catalog.py:319`, `embed_products.py:81`). Trimitem `list[float]` direct. Reduce alocările și latența pe hot path.

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** retrieval rămâne un stagiu/tool determinist, fără loop de orchestrare.
- **P2 (LLM doar la triaj+agent):** RRF + rerank sunt cod pur; singurul apel extern rămâne `embed([query])`. ZERO LLM nou.
- **P4 (buget context):** pool intern 30-50, dar tool result rămâne max 6×8 câmpuri spre model.
- **P6 (niciodată tăcere):** degradare lexical-only când lipsesc embeddings/LLM; relax-ladder înainte de gol.
- **P7 (business_id everywhere):** fiecare query nou are `WHERE business_id = $1`; RLS plasă.
- **P8 (state=refs):** dedup-ul citește `displayed_products` (id+name+price), nu obiecte.
- **P9 (prompt/taxonomie din DB):** concern-overlap pe taxonomy-from-DB (NX-124), nu beauty-hardcodat.
- **P10 (observabilitate din runner):** retrieverele nu știu că sunt măsurate; `ctx.emit` scrie metricile.
- **P11 (locale în chei):** FTS folosește `'simple'` config (limbă-agnostic) ca să nu rateze RO/HU/EN; nu introducem stemming legat de o singură limbă.

## Implementation Steps
1. **Migrare FTS+trgm** (`docs/0NN_fts_index.sql`, alege următorul număr liber la impl): `create extension if not exists pg_trgm`; coloană generată `search_tsv tsvector` (sau index funcțional) pe `name+brand+ai_summary` cu config `'simple'`; index GIN pe `search_tsv`; index GIN trgm pe `lower(name)` (+ `external_id`/SKU dacă există). `grant select` pe coloane noi e implicit (deja pe `products`).
2. **Codec pgvector nativ** în `src/db/connection.py` la init pool (`register_vector` / codec custom asyncpg). Înlocuiește literal-ul text din `search_products_semantic` (`catalog.py:319`) și `embed_products.py:81` cu `list[float]`.
3. **`search_products_lexical()`** în `catalog.py`: FTS `websearch_to_tsquery('simple', $q)` + `ts_rank_cd` OR `pg_trgm similarity`, aceleași filtre dure, întoarce pool cu rank.
4. **Param `pool`** pe `search_products_semantic()` (default 6 = compat; fuziunea cere ~50).
5. **`rrf_fuse()`** (pură) + **`deterministic_rerank()`** (pură) în `catalog.py` (sau modul nou `src/db/queries/fusion.py` dacă e mai curat — pură, fără DB).
6. **Refactor `search_products_tool()`** (`catalog_tools.py`): rulează ambele retrievere pe pool prin ladder → fuse → dedup vs `displayed_products` → rerank → top 6. Degradare lexical-only fără LLM.
7. **Extinde `ctx.emit("product_search", ...)`** cu `top_cosine_distance`, `relax_depth`, `zero_result`, `fused`, pool counts.
8. **Backfill embeddings**: rulează `python -m src.jobs.embed_products` pe demo (deblochează calea vector); nu e cod, e date — notează în PR.
9. **Teste** în `tests/test_retrieval_ranking.py` (fișier existent): adaugă cazuri RRF/rerank pure + assert pe SQL generat (fără DB).
10. `ruff check . && ruff format . && pytest -x -q` verde.

## Test Cases
**Happy:**
- `rrf_fuse`: produs prezent în ambele liste (lexical rank 3, vector rank 1) iese peste un produs prezent doar într-una — assert ordine deterministă.
- `deterministic_rerank`: la scor RRF egal, in-stock + sale + concern-overlap urcă; tie-break stabil pe `id`.
- `search_products_lexical` (assert SQL, fără DB): conține `websearch_to_tsquery`, `ts_rank_cd`, `pg_trgm`/`similarity`, `business_id = $1`.

**Edge:**
- Frază naturală lungă pe care `name ILIKE` o rata → FTS o prinde (validare pe SQL generat / test integration opțional).
- SKU/cod-piesă cu typo („DAE15X" vs „DAE-15X") → `pg_trgm` îl recuperează.
- Doar un retriever produce rezultate (embeddings=0 → vector gol) → fuziunea = lista lexicală, fără crash; `fused=False` în emit.
- Toate candidații deja în `displayed_products` → după dedup pool gol → relax/`zero_result=True`, fără tăcere.

**Failure:**
- `embed([query])` aruncă → fallback lexical-only, `mode` reflectă degradarea, niciun mesaj tăcut (P6).
- `query` gol / doar stopwords → `websearch_to_tsquery` întoarce query gol → cad pe `pg_trgm`/relax, niciun SQL invalid.
- Tenant fără embeddings ȘI fără LLM → lexical-only real (nu vechiul `ILIKE`).
