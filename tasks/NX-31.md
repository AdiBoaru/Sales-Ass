# NX-31 — Export CRM: webhook `lead.qualified` + CSV zilnic
**Owner:** S · **Faza:** P2 · **Zi/Ord:** după NX-88 (lead_score) + NX-52 (secret per business) · **Branch:** `feat/NX-31-crm-export` · **Complexitate:** L · **Estimare:** 12h

> independente (fiecare ≤4h, fiecare cu PR + DoD proprii):
> - **NX-31a — Webhook `lead.qualified` semnat HMAC OUTBOUND** (~4h): la trecerea unui
>   contact peste pragul de calificare (din NX-88), POST best-effort către URL-ul CRM al

>   ref la secretul outbound în secret manager) + validare la onboarding (NX-41) + feature

## Goal
Clientul vrea leadurile calificate în propriul CRM, nu doar în dashboardul nostru. Două
canale de export, ambele **outbound**, ambele tenant-scoped: (1) **real-time** — un eveniment
`lead.qualified` POST-at către URL-ul HTTPS al clientului în momentul în care `contacts.lead_score`
(scris de NX-88) trece pragul de calificare, semnat HMAC peste corpul brut ca să nu poată fi
falsificat; (2) **batch** — un CSV zilnic per tenant cu leadurile/conversațiile zilei, urcat
în storage privat, descărcabil printr-un signed URL cu expirare. Niciun apel LLM aici — totul
e cod determinist (HMAC + HTTP + agregare SQL + CSV).

## Business Context
Calificarea de lead (NX-88) e inutilă dacă rămâne în baza noastră: clientul lucrează în CRM-ul
lui (HubSpot, Pipedrive, foaie internă). Push-ul real-time pune leadul fierbinte în fața
echipei de vânzări în secunde (exact când e „ready_to_buy"), iar CSV-ul zilnic e plasa de
reconciliere + sursa pentru rapoartele clientului. E o funcție de **retenție de retainer**:
clientul vede botul alimentând direct pipeline-ul lui comercial. Outbound semnat = clientul
are încredere că datele vin de la noi și n-au fost injectate.

## Technical Description

### A. Declanșatorul `lead.qualified` (NX-31a) — refolosește hook-ul post-tur din NX-88
NX-88 are deja hook-ul post-tur `_extract_profile_and_score` care scrie `contacts.lead_score`
și emite `lead_score_updated {old, new}`. NU adăugăm un al doilea apel LLM și NU schimbăm
proprietarul câmpului (P3). Atașăm declanșatorul **lângă** acel hook, ca cod pur:
```python
# în processor.py, în interiorul _extract_profile_and_score, DUPĂ update_contact_profile_and_score:
threshold = float(business.settings.get("crm", {}).get("qualify_threshold", 70))
crossed_up = old_score < threshold <= new_score   # doar la TRECEREA în sus a pragului
if crossed_up and business.settings.get("crm", {}).get("export_enabled"):
    await _enqueue_lead_qualified(conn, business, contact, new_score)
```
„Trecere în sus" (edge, nu nivel) → un singur eveniment per calificare, nu unul la fiecare tur
cât scorul rămâne peste prag. Best-effort: orice excepție prinsă, nu rupe turul (reply-ul e
deja în outbox, P6).

### Payload-ul + semnătura outbound (simetric cu NX-52/NX-94, dar OUTBOUND)
Corpul e construit în cod, serializat o singură dată (bytes canonici), apoi semnat pe acei
bytes — exact ce semnăm e exact ce trimitem:
```python
payload = {
    "event": "lead.qualified",
    "occurred_at": now_utc_iso(),
    "business_id": str(business.id),
    "lead": {
        "contact_id": str(contact.id),          # ID intern, NU telefon (P12)
        "display_name": contact.display_name,
        "lead_score": float(new_score),
        "lifecycle": contact.lifecycle,
        "locale": contact.locale,
    },
}
raw = json.dumps(payload, separators=(",", ":"), ensure_ascii=False).encode("utf-8")
signature = "sha256=" + hmac.new(crm_secret.encode(), raw, hashlib.sha256).hexdigest()
```
Header: `X-Nativx-Signature: sha256=<hex>` + `Content-Type: application/json`. Secretul
(`crm_secret`) e **separat** de `orders_webhook_secret` (NX-52): acela e INBOUND (clientul
semnează spre noi), ăsta e OUTBOUND (noi semnăm spre client). Refolosim doar *forma* HMAC din
`signature.py`, adăugând o funcție pură `sign_outbound(secret, raw) -> str` (nu duplicăm
`verify_*`). PII: în payload mergem cu `contact_id` intern + `display_name`; **telefonul NU
intră** — rămâne doar în `channel_identities` (P12). Dacă clientul vrea telefonul, e o
discuție de DPA (Out of Scope).

### Livrare prin `outbox` + dispatcher (NU POST direct din hook) — P5
Hook-ul NU face `httpx.post` direct (ar fi un al doilea punct de ieșire + fără retry). Scrie
un rând în `outbox` cu `kind='crm_webhook'`, exact ca un mesaj de chat, și dispatcher-ul îl
livrează cu retry/backoff idempotent:
```python
async def _enqueue_lead_qualified(conn, business, contact, score):
    await insert_outbox(
        conn, business_id=business.id, conversation_id=None,
        idempotency_key=f"leadqual:{business.id}:{contact.id}:{score_bucket(score)}",
        kind="crm_webhook",
        payload={"url": business.settings["crm"]["webhook_url"], "body": payload, "sig": signature},
    )
```
`idempotency_key` include un bucket de scor → re-calificarea după churn nu se dublează în
aceeași zi, dar o re-trecere peste prag mai târziu da. Dispatcher-ul capătă o ramură nouă pe
`kind`: pentru `crm_webhook` face `httpx.post(url, content=body, headers={...sig...})`,
`raise_for_status`; 2xx → `sent`, 4xx → `dead` (URL/secret greșit la client, nu retry-uim
la infinit), 5xx/timeout → retry cu backoff (`next_attempt_at`). `httpx.AsyncClient` injectat
(ca `MetaClient`) → testele folosesc `MockTransport`, zero apeluri reale (P-teste).

### B. CSV zilnic per tenant (NX-31b) — job nocturn, conn ADMIN
Pattern identic cu `rollup_usage.py` (cross-tenant, citește append-only → `admin_conn`):
```
python -m src.jobs.export_crm_csv            # ziua de ieri (UTC)
python -m src.jobs.export_crm_csv 2026-06-15 # o zi anume
```
Pentru fiecare business cu `crm.export_enabled`: un SELECT agregat (toate cu `business_id = $1`)
peste leadurile zilei (contacte cu activitate + scor), serializat în CSV cu `csv.writer` pe un
buffer, urcat în **storage privat** (Supabase Storage, bucket privat per environment) la calea
`crm-exports/{business_id}/{day}.csv`, apoi un **signed URL** cu expirare (ex. 7 zile). Calea
include `business_id` → izolare fizică pe path în plus față de izolarea logică. Un business
care crapă e logat și sărit (nu oprește restul, ca `rollup_usage`).
```python
# coloane CSV (fără PII brută): contact_id, display_name, lead_score, lifecycle,
#   locale, first_seen_day, msgs_in, msgs_out, last_intent
# telefonul NU e în CSV (P12). display_name e nume comercial, nu identificator de canal.
```
Urcarea în storage o face un mic client injectabil (`StorageClient` Protocol cu `put`/
`signed_url`) → testul pasează un fake in-memory, fără rețea.

### C. Config per tenant (NX-31c)
`businesses.settings.crm` (jsonb, fără migrare):
```json
{"crm": {"export_enabled": true, "webhook_url": "https://client.example/hooks/nativx",
         "secret_ref": "crm_secret/nativex-demo", "qualify_threshold": 70}}
```
`secret_ref` = referință în secret manager (NU secretul în DB — la fel ca `channels.credentials_ref`).
Resolverul citește secretul din env/secret manager după `secret_ref`. Validare la onboarding
(NX-41): `webhook_url` e HTTPS, `secret_ref` rezolvă la un secret nenul. Feature flag global
`crm_export_enabled` în `config.py` (kill-switch).

### Câmpuri TurnContext scrise aici
**Niciunul.** NX-31a rulează în hook-ul post-tur (după închiderea turului, ca NX-88) —
nu atinge `TurnContext`. NX-31b/c sunt joburi/config offline. Singurele scrieri DB:
`outbox` (insert, owner = Sender/hook post-tur ca azi) și storage (extern).

## Database Changes
**None (DDL).** `businesses.settings` (jsonb) ține config-ul CRM — fără coloană nouă.
`outbox` are deja `kind`, `idempotency_key UNIQUE`, `payload` — refolosit cu `kind='crm_webhook'`.
- Query CSV (NX-31b), agregat, mereu cu `business_id = $1`:
```sql
select c.id as contact_id, c.display_name, c.lead_score, c.lifecycle, c.locale,
       count(m.*) filter (where m.direction = 'inbound')  as msgs_in,
       count(m.*) filter (where m.direction = 'outbound') as msgs_out
from contacts c
left join messages m
       on m.contact_id = c.id and m.business_id = c.business_id
      and m.created_at >= $2 and m.created_at < $3      -- fereastra zilei
where c.business_id = $1 and c.lead_score > 0 and c.erased_at is null
group by c.id;
```
(`erased_at is null` → nu exportăm contacte șterse GDPR.) Niciun lookup pe faq/cache/templates
aici, deci `locale` nu e cheie de cache — e doar coloană exportată.

## Events de emis (analytics)
- `lead_qualified_export_enqueued` {contact_id, score} — la enqueue în outbox (FĂRĂ telefon, P12).
- `crm_webhook_delivered` {contact_id, http_status} — la 2xx din dispatcher.
- `crm_webhook_failed` {contact_id, http_status, attempt} — la 4xx (dead) sau retry epuizat.
- `crm_csv_exported` {day, rows} — la urcarea CSV-ului zilnic (job).
- Evenimentele se emit din worker/job (au conn tenant / admin). `lead_score_updated` rămâne
  emis de NX-88, neschimbat — NX-31 nu-l dublează.

## Out of Scope
- **Telefonul / PII brută în payload sau CSV** — rămâne în `channel_identities`; export de
  telefon = discuție DPA separată, nu acest task.
- **Alte evenimente outbound** (`order.placed`, `cart.abandoned` spre CRM) — aici doar
  `lead.qualified`; restul = carduri viitoare pe același mecanism.
- **Conectori CRM specifici** (HubSpot/Pipedrive SDK, mapare de câmpuri) — noi livrăm un
  webhook generic + CSV; maparea o face clientul/integratorul.
- **Rotirea self-service a secretului outbound** (dashboard) — P2, ca la NX-52.
- **Replay protection cu timestamp/nonce** pe webhook-ul outbound — HMAC + `occurred_at` în
  payload; nonce dedicat = separat.
- **Backfill istoric** (export retroactiv al leadurilor vechi peste prag) — doar de-acum înainte.
- **Format CSV configurabil per client** (coloane la alegere) — v1 = set fix de coloane.

## Test Cases
Toate testele fără rețea reală: webhook prin `httpx.MockTransport`, storage prin fake
in-memory, DB prin fake conn. ZERO apeluri OpenAI/HTTP/DB reale în CI. (Niciun LLM implicat.)

**Happy Path:**
1. **(31a)** `old=60, new=72, threshold=70, export_enabled=true` → `_enqueue_lead_qualified`
   scrie outbox `kind='crm_webhook'`, body conține `event="lead.qualified"` + `contact_id`,
   semnătura `sha256=...` validează cu `verify` peste raw + secret. `lead_qualified_export_enqueued` emis.
2. **(31a)** Dispatcher procesează rândul → `MockTransport` întoarce 200 → `outbox.status='sent'`,
   `crm_webhook_delivered {http_status:200}` emis.
3. **(31b)** Job pe o zi cu 3 contacte cu scor > 0 → CSV cu 3 rânduri + header, urcat la
   `crm-exports/<biz>/<day>.csv` în fake storage, `signed_url` returnat, `crm_csv_exported {rows:3}`.

**Edge Cases:**
1. **(31a)** `old=75, new=80` (deja peste prag, rămâne peste) → trecere în sus = False → NU
   enqueue (nu re-emitem la fiecare tur).
2. **(31a)** Două tururi care trec pragul în aceeași zi (același bucket de scor) → al doilea
   `insert_outbox` lovește `idempotency_key UNIQUE` → un singur rând.
3. **(31a)** Corp cu diacritice (`display_name="Ștefan Cremă"`) → HMAC pe bytes UTF-8 raw
   validează identic la destinatar (re-serializarea NU schimbă semnătura — semnăm exact ce trimitem).
4. **(31b)** Contact cu `erased_at` setat → exclus din CSV; contact cu `lead_score=0` → exclus.

**Failure Cases:**
1. **(31a)** Endpoint client întoarce 403 (secret greșit la el) → `outbox.status='dead'`, FĂRĂ
   retry, `crm_webhook_failed {http_status:403}` emis.
2. **(31a)** Endpoint client întoarce 503 / timeout → retry programat (`next_attempt_at` în
   viitor, `attempts++`), NU `dead` din prima.
3. **(31a)** `insert_outbox` aruncă (DB) în hook → prins, logat, turul rămâne valid (reply deja
   în outbox), fără excepție propagată (best-effort, P6).
4. **(31b)** `StorageClient.put` aruncă pentru un business → logat, sărit, restul businessurilor
   se exportă (job nu se oprește).
5. **(31c)** Business fără `settings.crm.webhook_url` dar cu `export_enabled=true` → niciun
   enqueue + warning logat (config incomplet), nu excepție.

## Cost
**Niciun apel LLM/embeddings** — scorul vine gata din NX-88, exportul e HMAC + HTTP + SQL + CSV.
Cost OpenAI adăugat = **$0 / 1000 mesaje**. Costuri marginale: (1) un POST HTTP per lead
calificat (negligibil; mărginit de faptul că se emite DOAR pe *trecerea* pragului, nu per tur —
edge, nu nivel); (2) un job nocturn O(contacte active/zi) per tenant + un upload storage/zi
(mărginit de fereastra de o zi și de filtrul `lead_score > 0`). Ce limitează costul de
infrastructură: feature flag `crm_export_enabled` (off implicit), `idempotency_key` (zero
dubluri), și retry-ul cu plafon (`dead` la 4xx, backoff la 5xx — nu hammering la un endpoint mort).
