# NX-112 — Persist clarify-filled constraints + processor = singurul scriitor explicit de state

**Owner:** Claude · **Faza:** P0 · **Track/Wave:** Wave0 · **Branch:** `feat/NX-112-persist-state-fields` · **Complexitate:** S · **Estimare:** 3h

## Goal
Slotul umplut de `clarify_resume_stage` (`ctx.state.constraints[field] = answer`, `clarify.py:46`) trebuie PERSISTAT la write-back, nu doar mutat in-memory pentru turul curent. Facem `processor.py` (Sender) singurul scriitor EXPLICIT al fiecărui câmp din `ConversationState` — merge canonic din `ctx.state`, fără să ne bazăm pe mutația prin referință de dict partajat. În același timp, conectăm sau ștergem cheile moarte `active_search` / `asked_intents`.

## Business Context
Slot-filling pe mai multe ture e nucleul oricărui asistent de vânzări single-brain, indiferent de vertical (buget beauty, BTU la HVAC, serviciu la salon, fitment auto, mărime la fashion). Azi, dacă userul răspunde „buget 200 lei" la o întrebare de clarificare pusă de bot, slotul ajunge la agent în turul curent dar se PIERDE silențios la write-back: `from_jsonb` reconstruiește `constraints` dintr-un dict detașat, iar `processor.py:492-504` construiește `new_state` DOAR din `conv['state']` + `displayed_products` + `pending_question` + `state_patch` — nu citește niciodată `ctx.state.constraints` modificat. Turul următor re-citește state stale → botul „uită" bugetul și fie reîntreabă, fie ignoră constrângerea. Confirmat independent de auditurile code-2026, n8n și industry; netestat (`test_clarify.py:215` verifică doar că `constraints` PRE-existente supraviețuiesc, nu și un slot NOU umplut de stagiu). Pe lângă asta, `active_search` și `asked_intents` sunt declarate + hidratate (`models.py:131,134,169,172`) dar NICIO etapă nu le scrie vreodată — chei moarte care induc în eroare.

## Technical Description

### Bug-ul de write-back (P3 încălcat)
`ConversationState.from_jsonb` (`models.py:142-176`) hidratează `constraints = raw.get('constraints') or {}` — un dict NOU, detașat de `conv['state']` (dict-ul brut din DB). `clarify_resume_stage` mută `ctx.state.constraints[field] = answer` (`clarify.py:46`) pe acel dict detașat. Dar Sender-ul (`processor.py:492-504`) compune:
```python
new_state = conv["state"]                                   # dict brut DB, NU ctx.state
new_state = {**conv["state"], "displayed_products": ...}     # doar pe rich
new_state = {**new_state, "pending_question": ...}           # NX-130
if ctx.state_patch: new_state = {**new_state, **ctx.state_patch}  # NX-79 cart
```
`ctx.state.constraints` modificat nu intră NICIODATĂ în `new_state`. Mutația in-place a unui dict detașat nu se propagă la write-back. Singurul motiv pentru care funcționează pe `conv['state']` deja existent e că `conv['state']` (brut) e copiat în `new_state` — dar slotul NOU trăiește pe `ctx.state.constraints`, nu pe brut.

### Fix (processor = single explicit writer, P3)
În TX-ul de write-back (`processor.py`, blocul `async with conn.transaction()`), după liniile existente de `new_state`, merge canonic câmpurile owned de `ConversationState` din `ctx.state`:
```python
new_state = {
    **new_state,
    "constraints": ctx.state.constraints,
    "active_search": ctx.state.active_search,
    "asked_intents": ctx.state.asked_intents,
    "cart": ctx.state.cart,
}
```
Notă ordine: `state_patch` (NX-79, owner = Agent, ex. `cart_add → {"cart":[...]}`) trebuie să rămână ULTIMUL merge ca să nu fie suprascris de `ctx.state.cart`; alternativ, lăsăm `cart` afară din merge-ul canonic dacă `cart_add` scrie deja prin `state_patch` și `ctx.state.cart` nu e ținut sincron — vezi pasul 3. `displayed_products` și `pending_question` rămân gestionate de blocurile existente (owner Sender, derivate din `ctx.reply`), NU din `ctx.state`, ca să nu re-introducem produse vechi pe un reply non-rich.

### `active_search` / `asked_intents` — conectează sau șterge
Decizie luată la implementare, în acest task:
- **`asked_intents`**: conectat ca semnal ieftin anti-loop — în `clarify_resume_stage` (și/sau la setarea unei noi întrebări via `ctx.set_clarify`), append-uim `field`-ul cerut în `ctx.state.asked_intents` (dedup, cap 8 intrări). Devine vizibil în context_blocks ca „am întrebat deja X" → triaj/agent nu reîntreabă la nesfârșit același slot. NX-116 (triage anti-loop) îl va citi.
- **`active_search`**: dacă în main NU există încă un scriitor de retrieval care să-l populeze, îl ȘTERGEM din `ConversationState` (model + `from_jsonb`) ca să nu rămână cheie moartă. NX-119 (search sessions) și NX-113 (hybrid retrieval) îl vor re-introduce cu owner real (retrieval scrie setul de filtre ca să-l reamintească pe follow-up). Preferința: ștergem acum (YAGNI), îl reintroducem când are scriitor real. Nu lăsăm chei hidratate-dar-nescrise.

### Channel-aware
Calea de state e agnostică de canal: același `new_state` se scrie pentru WhatsApp/Telegram (async, prin outbox) și pentru web sync (`deliver=False`, `processor.py:487`). Bug-ul e cel mai vizibil pe WhatsApp/Telegram multi-tur (userul răspunde la întrebări în ture separate); calea web sync partajează EXACT același write-back, deci fix-ul acoperă toate canalele fără cod specific de canal. Nu există buton web vs CTA text aici — state pur.

### Generic-across-verticals
`constraints` / `asked_intents` rămân data-shape-agnostic: chei dinamice (`field` vine din `pending_question`, nu dintr-un vocabular hardcodat de sloturi). HVAC `btu`, salon `service`, auto `fitment`, fashion `size` curg prin EXACT același cod, fără ramuri per vertical. Nicio cheie de slot hardcodată în processor sau model.

## Principii CLAUDE.md aplicabile
- **P3 (single-owner-per-field)** — miezul taskului: `processor` devine singurul scriitor explicit pentru fiecare câmp din `ConversationState`; eliminăm mutația prin referință de dict partajat care violează „un câmp = un scriitor".
- **P8 (state = ref-uri)** — `constraints`/`asked_intents` rămân valori scalare/string mici; `cart`/`displayed_products` rămân ref-uri, nu obiecte de produs.
- **P4 (context budget)** — `asked_intents` capat la 8 intrări, `constraints` rămân în bugetul de 8KB (CHECK DB 003 ca plasă).
- **P7 (business_id everywhere)** — `patch_conversation_state` deja primește `business.id`; nu introducem query fără business_id.
- **P12 (PII one place)** — `clarify_resume_stage` deja emite `clarify_resumed` FĂRĂ `answer` (poate fi PII); păstrăm asta, nu logăm valori de constraint.

## Test Cases
**Happy:** Stagiu umple `constraints['budget_max']='200'` → `new_state['constraints']['budget_max']=='200'`; al doilea `handle_turn` cu acel state → constraint încă prezent (nu stale). `clarify_resume_stage` pe `pending_question={field:'budget_max',resume_route:'sales'}` cu body „200 lei" → `constraints['budget_max']=='200 lei'` în `new_state` ȘI `'budget_max' in new_state['asked_intents']`.
**Edge:** Constraint pre-existent (`{'x':1}`) + slot nou umplut → AMBELE în `new_state` (nu se pierde niciunul). `asked_intents` cu 8 intrări + a 9-a → capat la 8, cea mai veche scoasă, fără duplicate. Reply non-clarify (set_reply normal) → `constraints` din `ctx.state` tot persistat (nu doar pe ramura clarify). Rich reply cu `displayed_products` + constraint nou → ambele persistate (merge-ul de produse nu suprascrie constraints). `cart_add` via `state_patch` + constraint nou → `cart` ȘI `constraints` corecte (ordine merge OK).
**Failure:** State brut corupt (`constraints` lipsă / non-dict) → `from_jsonb` default `{}`, write-back nu crapă. `pending_question` fără `field` → `clarify` folosește default `'intent'`, `asked_intents` primește `'intent'`, fără excepție. Body gol la clarify → slotul NU se consumă, `constraints` neschimbate, `asked_intents` neschimbate.
