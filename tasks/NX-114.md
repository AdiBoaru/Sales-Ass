# NX-114 — DomainPack skeleton: politică+taxonomie per-tenant/per-vertical ca DB config + `Reply.offer` neutru

**Owner:** Claude · **Faza:** P1 · **Track/Wave:** Wave0 · **Branch:** `feat/NX-114-domainpack-skeleton` · **Complexitate:** L · **Estimare:** ~7h (vezi [PROPUNERE] split la final)

## Goal
Definește un contract de configurare **DomainPack** încărcat din DB per `(business, vertical)` (taxonomie concern, termeni de risc/legal pe locale, saluturi adiționale, whitelist de profil, statusuri de comandă „finalizată", monedă) și un **loader** care îl atașează pe `BusinessConfig`. Separat, adaugă în contractul `Reply` un câmp **`offer` tipizat și neutru de canal** (kind, label, url/payload), astfel încât un singur obiect semantic să fie randat ca buton (web), CTA interactiv (WhatsApp) sau buton inline (Telegram) — exact la marginea de canal. Acesta este SKELETON-ul + wiring-ul; NU migrăm încă fiecare consumator hardcodat (acela e follow-up per-feature).

## Business Context
Asistentul e un singur „creier" care servește mai multe brand-uri pe mai multe verticale (beauty, HVAC, auto, salon) și mai multe canale (WhatsApp primar, web widget, Telegram test). Astăzi cunoștința per-vertical e **hardcodată în cod**: maparea concern (`src/tools/taxonomy.py:58`, doar `beauty` → alte verticale primesc `{}` și pierd tăcut filtrele), termenii de risc/legal (`src/worker/stages/gates.py:60` RISK_PATTERNS, doar RO), saluturile (`src/worker/stages/greeting.py:35` `_GREETINGS` global), whitelist-ul de profil (`src/worker/profile.py:38` `_WHITELIST`). Onboardarea unui client HVAC/auto/salon cere astăzi **modificări de cod + redeploy**, ceea ce încalcă direct principiul 9 (config/prompt din DB). DomainPack mută toată politica per-vertical într-un **rând de config**, deci un vertical nou = config, nu deploy.

Separat, nu există un obiect de ofertă tipizat în `Reply`: link-urile de checkout și CTA-urile sunt astăzi „scuipate" de LLM ca text simplu în corpul mesajului, în loc să fie randate ca buton (web) / CTA (WhatsApp). `Reply.offer` introduce **seam-ul channel-aware** pe care îl vor consuma ChannelProfile (NX-115) și checkout-ul (NX-104), păstrând cuplajul de canal DOAR la cele două margini (NX-60).

## Technical Description

### 1. Contractul DomainPack (`src/domain/pack.py` — NOU)
Un dataclass `DomainPack` (lightweight, intern, ca restul din `src/models.py`) cu câmpuri pur de config, toate cu default-uri sensibile și agnostice de vertical:

```python
@dataclass(frozen=True)
class DomainPack:
    vertical: str                                  # ecommerce | beauty_salon | auto_service | other
    concern_map: dict[str, str]                    # termen liber norm → cheie canonică attributes->'concerns'
    risk_terms: dict[str, dict[str, list[str]]]    # locale → {reason: [phrase_norm,...]} (human_request/legal_complaint)
    greetings: dict[str, list[str]]                # locale → saluturi adiționale (peste baza din greeting.py)
    profile_whitelist: frozenset[str]              # chei permise în contacts.profile (peste minimul agnostic)
    settled_order_statuses: tuple[str, ...]        # statusuri „finalizat" pt check_order (delivered/closed/...)
    currency: str = "RON"                           # moneda afișată (din businesses.currency family)
```

- `DomainPack` e **complet derivat din date**: nicio mapare hardcodată în acest fișier. Sursa = `businesses.settings["domain_pack"]` (jsonb) + un set de **default-uri per-vertical** încărcate dintr-un fișier de seed JSON (`src/domain/defaults/`), NU din cod inline. Default-urile per-vertical = punctul de plecare; `settings["domain_pack"]` face merge deasupra (override per-tenant). Așa un tenant beauty cu jargon propriu nu cere alt vertical.
- Normalizarea cheilor (lower + strip diacritice, exact `_norm` din `taxonomy.py`/`gates.py`) se face **o singură dată la încărcare**, ca lookup-urile downstream să fie O(1) și deja normalizate.

### 2. Loader-ul (`src/domain/loader.py` — NOU)
`load_domain_pack(business: BusinessConfig) -> DomainPack`:
- citește `business.vertical` → încarcă default-ul JSON corespunzător din `src/domain/defaults/<vertical>.json` (fallback pe `other.json` dacă verticalul nu are fișier);
- face deep-merge cu `business.settings.get("domain_pack", {})` (override per-tenant câștigă);
- normalizează toate cheile/frazele și întoarce un `DomainPack` `frozen`.
- **Pur, fără I/O DB**: primește `BusinessConfig` deja încărcat (care conține `settings` jsonb). Default-urile JSON se citesc o dată la boot și se cache-uiesc (`@lru_cache` pe vertical), deci zero cost per-tur.

### 3. Wiring pe `BusinessConfig` (`src/db/queries/businesses.py:43`)
`load_business` rămâne owner-ul lui `BusinessConfig`. Adăugăm câmpul `domain_pack: DomainPack | None = None` pe `BusinessConfig` (`src/models.py:54`) și îl populăm în `load_business` apelând `load_domain_pack`. `currency` se citește din coloana existentă `businesses` (schema are `currency` family la nivel de tabele comerț; pt businesses folosim `settings["currency"]` cu fallback `"RON"` — vezi Database Changes). Owner unic al câmpului = loader-ul de business (P3). Stagiile citesc `ctx.business.domain_pack`, nu re-încarcă.

### 4. `Reply.offer` — seam neutru de canal (`src/models.py:248`)
Un dataclass nou `Offer` + câmp pe `Reply`:

```python
@dataclass
class Offer:
    kind: str            # "checkout" | "open_url" | "quick_reply" | "book"
    label: str           # textul afișat pe buton/CTA (per-locale, vine de la emitent)
    url: str | None = None       # pt checkout/open_url (din catalog/checkout_links, NU inventat)
    payload: str | None = None   # pt quick_reply: token rutat (ex. "offer:reorder") — neutru
```

`Reply.offer: Offer | None = None`. Semantica:
- **Web**: marginea (WebSender, NX-127) randează un buton (`url` → link, `payload` → mesaj nou).
- **WhatsApp**: marginea randează un CTA interactiv (interactive button/url message). Dacă canalul nu suportă tipul, fallback la text — DAR fallback-ul e decis de ChannelProfile (NX-115), NU aici.
- **Telegram**: buton inline (url) sau reply-keyboard (payload), ca `Chip`-urile existente (`src/models.py:227`).

`Offer` e **payload channel-neutral**: emitentul (agent/checkout) setează intenția semantică; CUM se randează e exclusiv la margine (NX-60). În acest card adăugăm DOAR câmpul + helper-ul `set_offer` pe `TurnContext` + aplatizarea în text (floor) ca un Sender vechi să nu crape. **NU** implementăm randarea bogată pe niciun canal (acela e NX-115 + NX-127 + NX-104). Floor-ul: dacă `offer.url` există, se append-uiește la `reply.text` ca link simplu (comportamentul de azi, dar acum derivat din câmp tipizat, nu „scuipat" de LLM).

### Channel-aware & generic-across-verticals
- **Channel-aware**: `Reply.offer` ESTE seam-ul. Un singur obiect → buton web / CTA WhatsApp / inline Telegram, fără cod de canal în pipeline.
- **Generic-across-verticals**: DomainPack ESTE fundația generică. Toată politica per-vertical iese din cod în DB/JSON-seed; un vertical nou = `src/domain/defaults/<vertical>.json` + (opțional) override în `businesses.settings`.

## Principii CLAUDE.md aplicabile
- **P3 (single-owner-per-field)**: `BusinessConfig.domain_pack` are owner unic = `load_business`; `Reply.offer` are owner = stagiul care emite oferta (agent/checkout). Nimeni altcineva nu le scrie.
- **P7 (business_id everywhere)**: DomainPack e per-`(business, vertical)`; sursa de config (`businesses.settings`) e deja tenant-scoped + RLS (`id = current_business_id()`). Loader-ul nu atinge DB direct (primește `BusinessConfig` deja izolat).
- **P9 (prompt/config din DB)**: inima cardului — politica/taxonomia per-vertical din DB+seed, nu hardcodată. Default-urile JSON sunt seed editabil, nu cod de logică.
- **P11 (locale in keys)**: `risk_terms` și `greetings` sunt keyed pe locale; un termen de risc în limba greșită NU prinde — exact ca regula de cache/FAQ.
- **P5 (single exit)** & **NX-60 (cuplaj doar la margini)**: `Reply.offer` curge prin outbox; randarea de canal stă la margine.
- **P12 (PII într-un loc)**: whitelist-ul de profil din DomainPack rămâne barieră — nicio cheie de contact (telefon/email/nume) nu intră vreodată în whitelist (la fel ca azi în `profile.py`).

**Edge:**
- `vertical="other"` → fallback pe `other.json`, fără excepție.
- `vertical="hvac"` (nu are fișier) → fallback `other.json`.
- `risk_terms` lookup pe locale absent (ex. „hu" inexistent) → listă goală, nu KeyError.
- Termen concern cu diacritice/uppercase („Ten Grăs") → normalizat la aceeași cheie ca „ten gras".
- `settings["domain_pack"]` cu tip greșit (string în loc de dict) → ignorat defensiv, se folosește default-ul (P6 — niciun crash).

**Failure:**
- Fișier default JSON corupt/lipsă → loader loghează warning + cade pe `other` sau pe pack gol; pipeline-ul NU crapă (fail-safe, `domain_pack=None` acceptabil).
- `DOMAIN_PACK_ENABLED=false` → `load_business` întoarce `BusinessConfig` cu `domain_pack=None`; testele de regresie (taxonomy/gates/greeting) rămân verzi pe constantele de cod.
