> face în două bucăți ≤4h, fiecare livrabilă și testabilă separat:
> - **NX-41a (~3h) — business + channel + identity:** `upsert_business` (plafoane + privacy_url
>   validat + secret coalesce), refolosire `upsert_channel`, `upsert_channel_identity` + contact

> - **NX-41b (~3h) — plafoane fine + secret one-time + FAQ seed + validări:** mini-migrarea

## Goal
Un singur script CLI care provisionează un tenant complet, de la zero, într-o rulare:
`businesses` (cu plafoane + `orders_webhook_secret` + `privacy_url` validat) → `channels`
→ `channel_identities` (operatorul / numărul de test) → un seed minim de `faqs`. Rerulabil
fără efecte secundare: fiecare resursă face `ON CONFLICT`, fiecare pas printează
`created`/`exists`. Azi onboarding-ul e un amestec de SQL manual + `seed_telegram_channel.py`
+ insert-uri ad-hoc — un client nou cere o jumătate de zi de copy-paste fragil. Scriptul ăsta
devine **API-ul de onboarding** la faza B (UI-ul dashboard îl va chema în loc de SQL manual),
deci forma lui (idempotent, output per resursă, validări la intrare) trebuie corectă din prima.

## Business Context
Modelul de business e agenție SaaS: fiecare client nou = un setup fee + un retainer. Cu cât
onboarding-ul e mai rapid și mai puțin predispus la greșeli (un `business_id` lipsă, un secret
de comenzi negenerat, un `privacy_url` uitat = launch-blocking pe AI Act), cu atât marja de
setup crește și riscul de „am uitat un pas" scade. Un onboarding manual care omite
`orders_webhook_secret` rupe atribuirea de revenue (cifra care vinde retainerul, NX-52); unul
care omite `privacy_url` blochează go-live-ul (T180, art. 50). Scriptul codifică checklist-ul
de onboarding ca **cod verificabil**, nu ca disciplină umană.

## Technical Description

### Rolul de execuție — ADMIN, nu bot_runtime
Onboarding-ul scrie în tabele care sunt **read-only pentru bot** (`businesses`, `channels`) și
creează rândul `businesses` ÎNAINTE ca tenantul să existe — deci RLS-ul tenant (`app.business_id`)
nici n-ar avea ce filtra. Scriptul rulează pe `admin_conn` (pool privilegiat, `SUPABASE_DB_URL`),
exact ca `seed_telegram_channel.py` și migrările. **NU** folosește `tenant_conn`/`bot_runtime`.
Asta e o excepție documentată de la „business_id pe tot": e operația de **control plane** care
NAȘTE tenantul (la fel ca lookup-ul canal→business). Suprafața rămâne strict provisioning.

### Fluxul — un pas per resursă, fiecare cu raport `created`/`exists`
Toate într-o singură tranzacție (`async with conn.transaction()`): dacă orice pas crapă
(ex. `privacy_url` invalid descoperit târziu, secret manager indisponibil), nu rămâne un tenant
pe jumătate creat. Ordinea respectă FK-urile: business → channel → identity → faqs.

```
[1] business   → upsert pe slug (UNIQUE) → returnează business_id
                 + daily_cost_cap_usd, orders_webhook_secret, settings.privacy_url
[2] channel    → upsert pe (kind, provider_account_id) → channel_id   (refolosim upsert_channel)
[3] identity   → upsert operator/număr test pe (business_id, channel_kind, external_id)
[4] faqs       → upsert seed minim (din --faq-file JSON sau set default) pe cheie logică
```

### [1] Business — upsert idempotent + plafoane + secret + privacy_url
`businesses` are `UNIQUE (slug)` dar nu o constrângere pe care un `ON CONFLICT` să-l prindă
natural pentru câmpurile de config; folosim `slug` ca ancoră de idempotență. Funcție nouă
`upsert_business(conn, *, slug, name, vertical, default_locale, supported_locales, timezone,
daily_cost_cap_usd, privacy_url, orders_webhook_secret) -> dict` în `src/db/queries/businesses.py`:

```sql
insert into businesses
    (slug, name, vertical, status, default_locale, supported_locales, timezone,
     daily_cost_cap_usd, orders_webhook_secret, settings)
values ($1, $2, $3, 'onboarding', $4, $5, $6, $7, $8,
        jsonb_build_object('privacy_url', $9::text))
on conflict (slug) do update set
    name              = excluded.name,
    vertical          = excluded.vertical,
    default_locale    = excluded.default_locale,
    supported_locales = excluded.supported_locales,
    timezone          = excluded.timezone,
    daily_cost_cap_usd = excluded.daily_cost_cap_usd,
    -- secretul se generează O SINGURĂ DATĂ: la re-rulare păstrăm cel existent (nu-l rotim
    -- accidental, ar invalida integrarea de comenzi a clientului — vezi Out of Scope)
    orders_webhook_secret = coalesce(businesses.orders_webhook_secret, excluded.orders_webhook_secret),
    -- merge pe settings, nu overwrite (nu pierdem feature flags setate manual)
    settings          = businesses.settings || jsonb_build_object('privacy_url', $9::text)
returning id::text as id, orders_webhook_secret, (xmax = 0) as created
```

Observații de proiectare:
- `status='onboarding'` la creare (nu `active`) — activarea e un pas uman ulterior, după ce
  catalogul e seedat + canalele verificate. `--activate` e explicit Out of Scope.
- `orders_webhook_secret` se generează în Python cu `secrets.token_hex(32)` DOAR dacă lipsește;
  `coalesce(businesses.orders_webhook_secret, excluded....)` în SQL garantează că re-rularea NU
  rotește un secret deja folosit. Schema-add: coloana `businesses.orders_webhook_secret` vine din
  migrarea NX-52; NX-41 o populează. La output secretul se afișează **o singură dată** (la
  `created=True`), cu avertisment „salvează-l acum"; la `exists` se afișează doar `***set***`.
- `privacy_url` e **obligatoriu** și **validat la intrare** (vezi mai jos) — fără el scriptul
  refuză să creeze businessul (regula T180: fără privacy_url, businessul nu poate fi activat).

### Validare `privacy_url` (T180)
Funcție pură `validate_privacy_url(url: str) -> str` în `scripts/create_tenant.py` (sau helper
`src/onboarding/validation.py`): URL parsabil, schemă `https` (nu `http` — politica de
confidențialitate pe HTTP e respinsă), host nevid. Eșec → `sys.exit(2)` cu mesaj clar ÎNAINTE de
orice scriere în DB. NU facem fetch HTTP la URL (verificăm forma, nu disponibilitatea — un 404
temporar nu trebuie să blocheze onboarding-ul; verificarea de conținut e a omului).

### [2] Channel — refolosim `upsert_channel`
Deja există și e idempotent pe `(kind, provider_account_id)` (NX-63). Pentru WhatsApp,
`provider_account_id = phone_number_id`; pentru Telegram, `bot id`. Scriptul îl cheamă direct;
pentru Telegram poate (opțional, `--telegram-token`) valida tokenul cu `getMe` și deriva bot id-ul,
refolosind logica din `seed_telegram_channel.py` (extragem `_get_bot` într-un helper reutilizabil,
fără să dublăm). Output: `created`/`exists` din flag-ul `created` deja întors.

### [3] channel_identity — operatorul / numărul de test
Funcție nouă `upsert_channel_identity(conn, *, business_id, contact_id, channel_kind, external_id)
-> dict` în `src/db/queries/channel_identities.py` (fișier nou sau în `contacts.py`):

```sql
insert into channel_identities (business_id, contact_id, channel_kind, external_id, verified)
values ($1, $2, $3, $4, true)
on conflict (business_id, channel_kind, external_id) do nothing
returning id::text as id, (xmax = 0) as created
```

Pentru a avea un `contact_id`, scriptul creează întâi un contact „seed" (operatorul de test) cu
`upsert_contact` (dacă nu există deja pentru perechea identitate). Identitatea operatorului e
PII (telefon E.164 / id Telegram) → trăiește DOAR aici (P12). Acest pas e **opțional**
(`--operator-id`): un tenant poate fi creat fără o identitate seed și să-și primească contactele
organic din primul mesaj inbound.

### [4] faqs — seed minim, idempotent, cu locale
`faqs` NU are un unique natural pe conținut. Adăugăm idempotența la nivel logic: scriptul citește
un `--faq-file` JSON (`[{question, answer, locale}]`) sau un set default mic (program, livrare,
retur — în `default_locale`); pentru fiecare, upsert pe `(business_id, locale, lower(question))`.
Cum nu există un index unic pe asta, folosim pattern „check-then-insert" în aceeași tranzacție SAU
adăugăm un unique parțial. **Decizie:** adăugăm `unique (business_id, locale, md5(question))`
într-o mini-migrare (`docs/009_faq_seed_unique.sql`) ca `ON CONFLICT` să funcționeze curat:

```sql
insert into faqs (business_id, question, answer, locale, is_active)
values ($1, $2, $3, $4, true)
on conflict (business_id, locale, md5(question)) do update set
    answer = excluded.answer, is_active = true
returning id::text as id, (xmax = 0) as created
```

**Embedding-ul rămâne NULL** la seed — `faqs.embedding` se populează de jobul de embed (separat,
deblocat după NX-41), exact ca produsele. NX-41 NU cheamă embeddings (ar fi LLM/embed din afara
celor 2 puncte; jobul de embed e locul corect). Lookup-ul de FAQ în runtime e mereu
`business_id + locale + cosine` (P11) — seed-ul respectă cheia (locale obligatoriu per rând).

### Câmpuri TurnContext scrise
**Niciunul.** NX-41 e un script de provisioning offline — nu intră în pipeline, nu primește și
nu produce `TurnContext`. (Notat explicit ca să fie clar că nu atinge contractul central.)

### Output & exit codes
- Per resursă: o linie `[business] created: <id>` / `[channel] exists: <id>` / ...
- La final: rezumat + (la `created`) secretul de comenzi afișat o dată cu avertisment.
- Exit `0` = succes; `2` = validare la intrare (privacy_url, argumente lipsă); `1` = eroare DB.
- **Loguri fără PII**: `external_id`-ul operatorului (telefon/id) NU se printează în clar — doar
  `channel_kind` + `***`. Secretul de comenzi se afișează DOAR la creare (one-time), niciodată
  în re-rulare (P12).

## Principii CLAUDE.md aplicabile
- **P2 (LLM doar triaj+agent):** NX-41 NU cheamă niciun LLM/embedding. `faqs.embedding` rămâne
  NULL → populat de jobul de embed dedicat. `orders_webhook_secret` e generat criptografic

- **P9 (promptul din DB) / idempotență:** scriptul codifică checklist-ul de onboarding ca date

- **P11 (limba e parte din cheie):** seed-ul de `faqs` are `locale` obligatoriu per rând; cheia de
  idempotență include `locale`.
- **P12 (PII într-un loc):** identitatea operatorului (PII) intră DOAR în `channel_identities`;
  logurile nu conțin telefoane/id-uri în clar; secretul de comenzi afișat o singură dată.

## Test Cases
Toate testele rulează pe o **tranzacție rollback-uită** pe `admin_conn` (ca `test_tenant_isolation.py`)
— zero efecte persistente, ZERO apeluri externe (fără OpenAI/embeddings — NX-41 nu cheamă LLM; fără
Telegram getMe în CI — `--telegram-token` testat cu `_get_bot` mock).

**Happy Path:**
1. `create_tenant(slug='acme', name='Acme', vertical='ecommerce', privacy_url='https://acme.ro/privacy',
   daily_cost_cap=5.0, channel_kind='whatsapp', provider_account_id='PNID-1')` → `businesses` creat
   cu `status='onboarding'`, `daily_cost_cap_usd=5.0`, `settings.privacy_url` setat,
   `orders_webhook_secret` nevid; `channels` creat; output `[business] created`, `[channel] created`.
2. Rulare cu `--faq-file` (2 întrebări RO) → 2 rânduri `faqs` cu `locale='ro'`, `embedding IS NULL`,
   `is_active=true`.

**Edge Cases:**
1. **A doua rulare identică** → toate `exists`, `business_id`/`channel_id` identice cu prima,
   `orders_webhook_secret` byte-identic cu prima (coalesce a păstrat-o).
2. Re-rulare cu un FAQ schimbat (același `question`, alt `answer`) → UPDATE pe rândul existent
   (`answer` nou), count `faqs` neschimbat (idempotență pe cheia logică).
3. Tenant fără `--operator-id` → niciun `channel_identities` creat; restul resurselor OK (identitatea
   e opțională).
4. `settings` cu un feature flag pre-existent (rulare 2 după ce cineva a setat manual `settings.foo`)
   → `privacy_url` actualizat dar `settings.foo` PĂSTRAT (merge `||`, nu overwrite).

**Failure Cases:**
1. `--privacy-url http://acme.ro/privacy` (HTTP) → `SystemExit(2)`, mesaj clar, ZERO rânduri în DB
   (validarea precede tranzacția).
2. `--privacy-url` lipsă → `SystemExit(2)` din argparse/validare, nimic scris.
3. Eroare DB la pasul [3] (ex. FK violation simulat) → tranzacția întreagă rollback: `businesses` și
   `channels` NU rămân persistate (test verifică zero rânduri după excepție).
4. `--telegram-token` invalid (getMe mock → `ok=False`) → exit clar înainte de orice scriere DB.

> Niciun apel real: `_get_bot` (Telegram) și conexiunea DB sunt pe tranzacție rollback-uită / mock.
> NX-41 nu cheamă OpenAI/embeddings → fără cost LLM (vezi mai jos).

## Cost
**$0 LLM/embeddings.** NX-41 nu cheamă niciun model: `orders_webhook_secret` e generat cu `secrets`
(criptografic, gratuit), `faqs.embedding` rămâne NULL (populat ulterior de jobul de embed dedicat,
cost contabilizat acolo). Onboarding-ul e o operație rară (per client nou, nu per mesaj) → impact de
cost recurent nul.
