# NX-14 — Supabase regiune UE + decizie OpenAI EU
**Owner:** S · **Faza:** P1 · **Ord:** O1 · **Branch:** `feat/NX-14-eu-data-residency` · **Complexitate:** S · **Estimare:** 4h

## Goal
Confirmă (sau planifică migrarea) regiunii proiectului Supabase ca să fie în UE
ÎNAINTE de primul contact real în DB, și scrie un ADR datat despre data residency
la OpenAI. E un task de **guvernanță + verificare**, nu de feature: după ce primul
număr de telefon/nume real intră în DB, mutarea regiunii devine un *transfer de date
personale* (GDPR Art. 44-49) — adică o operațiune cu cost legal, nu un click. Deci
fereastra de a alege regiunea „gratis" se închide la primul client real. ADR-ul
prinde decizia (regiune Supabase + dacă rutăm OpenAI prin endpoint EU) în scris,
datată și reversibilă-explicit, ca să nu fie o presupunere tribală.

## Business Context
Vindem unui retailer din România date despre clienții LUI (telefoane E.164, nume,
istoricul conversațiilor). Pentru un client B2B serios (și pentru DPA-ul pe care
agenția îl semnează cu el), „unde stau datele" e o întrebare de contract, nu de
infrastructură. Un proiect Supabase în `us-east-1` care procesează PII de cetățeni
UE = expunere de compliance pe care nu vrei s-o descoperi la primul audit al
clientului. Costul deciziei acum: ~4h de verificare + scris. Costul ne-deciziei:
migrare de date personale sub presiune + o conversație neplăcută cu clientul.

## Technical Description

Taskul are **trei livrabile** și **zero cod de runtime în calea fierbinte** (nu
atingem pipeline-ul, nu scriem în `TurnContext`). Nu se cheamă niciun LLM aici.

### 1. Verificarea regiunii Supabase (manual + dovadă în repo)
Regiunea unui proiect Supabase se vede în **Dashboard → Project Settings → General →
Region** (ex. `eu-central-1` Frankfurt, `eu-west-2` Londra) și e **imutabilă** după
creare — Supabase NU oferă „change region" in-place; o mutare reală înseamnă proiect
nou + dump/restore + re-pointare a `SUPABASE_DB_URL`. De aici regula „înainte de date
reale".

Pașii de verificare (Adi, manual — vezi TODO-MANUAL.md) + capturarea dovezii ca text
verificabil, nu screenshot:
```bash
# regiunea proiectului via Management API (token din dashboard, NU în repo)
curl -s -H "Authorization: Bearer $SUPABASE_MGMT_TOKEN" \
  https://api.supabase.com/v1/projects/$PROJECT_REF | python -m json.tool
# câmpul "region" → trebuie să înceapă cu "eu-"
```
Verificare programatică de la nivelul DB (rulează cu `admin_conn`, control plane —
NU pe `bot_runtime`, e mentenanță non-tenant):
```sql
-- indiciu de timezone server (eu = ok); confirmarea finală e câmpul region din API
select now() at time zone 'UTC' as utc_now, current_setting('TimeZone') as srv_tz;
```
Rezultatul (regiunea confirmată) se scrie în ADR (vezi 3) și se bifează în TODO-MANUAL.

### 2. Decizia „migrăm sau nu" (arbore, în ADR)
```
Regiune Supabase ∈ UE?
├── DA  → ADR: status "Accepted", regiune notată, fără migrare. Gata.
└── NU  → DB conține deja PII real (channel_identities non-goale, contacts cu nume)?
         ├── NU (doar date demo seedate) → migrare IEFTINĂ acum:
         │     proiect nou în UE → re-rulează schema_v2 + 003/004 + re-seed →
         │     repointează SUPABASE_DB_URL/DATABASE_URL_BOT. Cost: ~1h, ZERO transfer PII.
         └── DA → migrare = transfer de date personale (GDPR): necesită temei legal +
               notificare. ESCALADARE: nu se face în acest task; ADR-ul documentează
               riscul + planul, deciziei i se atașează owner + termen.
```
Check „avem deja PII real?" (rulat cu `admin_conn`, read-only, fără a loga valori):
```sql
-- DOAR numărătoare, NU selectăm telefoane/nume (P12 — PII nu iese în output/loguri)
select
  (select count(*) from channel_identities where external_id is not null) as identities,
  (select count(*) from contacts where display_name is not null and erased_at is null) as named_contacts;
-- 0 + 0 → încă pe date demo → fereastra de migrare gratis e deschisă
```
Pentru clientul demo curent (`6098812a-…`, `nativex-demo`) avem 500 produse seedate
și (conform CLAUDE.md) `channel_identities` doar de la testele Telegram throwaway —
deci foarte probabil suntem în ramura „date demo, migrare ieftină dacă regiunea nu e UE".

### 3. ADR OpenAI EU data residency (`docs/decisions/`)
Creăm directorul `docs/decisions/` (nu există încă) cu un ADR în formatul standard
(MADR-lite), **datat**, despre rutarea apelurilor OpenAI prin endpoint-ul EU.

Context tehnic real: clientul `AsyncOpenAI` se construiește azi **fără `base_url`**
(`src/agent/llm.py`, `get_llm()` → `AsyncOpenAI(api_key=s.openai_api_key)`), deci
toate apelurile (triaj nano, agent mini, embeddings, moderation) merg pe endpoint-ul
default global. OpenAI oferă **EU data residency** (procesare în regiunea UE pentru
proiecte eligibile) — la nivel de cod se traduce într-un `base_url`/proiect EU +
posibil cost mai mare. Estimarea din seed: **~+10% pe nano**.

ADR-ul NU implementează rutarea EU acum — o **documentează ca decizie** (poate fi
`Accepted` = „rutăm de la început" sau `Deferred` = „amânat până la primul client cu
cerință de DPA, dar costul și pașii sunt scriși"). Dacă viitorul task de implementare
o cere, hook-ul de cod e trivial și **deja anticipat**: un `openai_base_url` opțional
în settings → pasat în `AsyncOpenAI(base_url=...)`. NU îl activăm în acest task; doar
notăm în ADR că ăsta e punctul de schimbare (un singur loc, principiul 2 — adaptorul unic).

## Principii CLAUDE.md aplicabile

  dacă se implementează vreodată, ca `base_url` în `get_llm()` — UN singur loc.
  Acest task nu cheamă niciun model.
- **Roluri DB (CLAUDE.md → securitate):** verificările sunt admin/control-plane, nu
  bypass RLS de tenant; `service_role` rămâne doar pentru migrări.

## Implementation Steps

## Test Cases
**Happy Path:**
1. `tests/test_adr_eu_residency.py`: ADR-ul există la calea așteptată și conține
   secțiunile obligatorii (`## Context`, `## Decision`, `## Status`, dată ISO) +
   mențiunea explicită a unei regiuni Supabase și a poziției OpenAI EU. Asertă pe text,
   ZERO apeluri externe.
2. Regiune dashboard = `eu-central-1` → ADR status „Accepted", fără task de migrare;
   `pytest -x -q` verde.

**Edge Cases:**
1. Regiune NU-UE + inventar PII = `0/0` (doar date demo) → ADR documentează planul de
   re-creare ieftin în UE + task în TODO-MANUAL; testul ADR tot trece (verifică
   prezența secțiunii „Migration Plan").
2. `docs/decisions/` nu există la pornire → pasul 3 îl creează; testul ADR e roșu
   ÎNAINTE și verde DUPĂ crearea fișierului (dovadă că testul chiar verifică ceva).
3. Câmpul `region` lipsește din răspunsul Management API (token cu scope insuficient)
   → scriptul de verificare iese cu mesaj clar („region indisponibil, verifică
   manual în dashboard"), nu crapă cu KeyError.

**Failure Cases:**
1. Regiune NU-UE + inventar PII > 0 (PII real în DB) → scriptul/ADR marchează
   ESCALADARE (transfer de date personale): testul ADR cere prezența secțiunii
   „Risk / Escalation" în acest caz; nu se execută nicio migrare automată.
2. `SUPABASE_MGMT_TOKEN` lipsă/invalid la rularea `scripts/check_region.py` → exit
   non-zero cu mesaj clar, fără a expune tokenul în output (P12 — secretele nu intră
   în loguri); fallback documentat = verificare manuală în dashboard.

> Testele nu ating OpenAI, Supabase live sau rețeaua: `test_adr_eu_residency.py`
> citește doar fișierul ADR din repo. Verificările de regiune/PII sunt pași MANUALI
> (Adi, cu credențiale reale), nu rulează în CI.

## Cost
**Zero cost de runtime în acest task** — niciun apel LLM/embeddings. Singurul cost
*viitor* pe care îl DECIDE ADR-ul: dacă alegem rutarea OpenAI EU (`Accepted`),
estimarea seed este **~+10% pe nano** (triajul). Limitarea costului: nano e modelul
ieftin, iar volumul lui e deja redus de straturile gratuite (alias/cache, stagiul 4,
țintă 40-60% opriri) și de moderation gate (NX-15, mesaje flagged nu ajung la nano).
Pe agent (mini) impactul +10% e mai semnificativ în absolut, dar mini e oricum
bugetat dur (max 3 tool calls, prompt caching 75-90% discount din prefix static).
Decizia finală „merită +10% pentru EU residency" se ia când există un client cu
cerință contractuală — ADR-ul o ține deschisă și ieftin de comutat (un `base_url`).
