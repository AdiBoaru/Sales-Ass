# NX-80 — Tool-uri agent: subscribe_back_in_stock + delivery_eta
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7-3 (check_order) · **Branch:** `feat/NX-80-tools-backinstock-eta` · **Complexitate:** S · **Estimare:** 3.5h

## Goal
Completează două din cele 12 tool-uri documentate, ambele lipsă din `TOOL_REGISTRY`:
`subscribe_back_in_stock(product_id, variant_id)` — clientul vrea notificare când un produs
fără stoc revine → INSERT idempotent în `back_in_stock_subscriptions`; și
`delivery_eta(product_id, address)` — clientul întreabă „când ajunge la mine?”. Prima e un
WRITE tool real (ancora pe care proactivul NX-70 o citește la restock). A doua, neavând încă
integrare curier/ERP, e un **stub determinist documentat** (ETA estimativ din zile de
expediere, marcat [PROPUNERE]) — onest, fără să inventeze date de transport.

## Business Context
Un „nu mai e pe stoc” fără captură de intenție = lead pierdut. `subscribe_back_in_stock`
transformă un out-of-stock într-o promisiune de revenire pe care proactivul o monetizează la
restock (CLAUDE.md: „back-in-stock” în proactive_jobs). `delivery_eta` ridică conversia la
`ready_to_buy`: incertitudinea „cât durează livrarea?” e o frână clasică de coș. Până la
integrarea reală de curier, un ETA estimativ onest e mai bun decât tăcere sau o cifră inventată
(care ar fi tăiată oricum de validator dacă ar fi numerică-grounded — vezi mai jos).

## Technical Description

### Stare curentă (citită din cod)
`TOOL_REGISTRY` (src/tools/base.py) are 5 tool-uri: `search_products`, `get_product_details`,
`compare_products`, `checkout_link`, `check_order`. `_SALES_TOOLS` =
`("search_products","get_product_details","compare_products","checkout_link")`. Contractul
uniform: `async def tool(ctx, deps, args: dict) -> ToolResult`, scoped pe `ctx.business.id`
(modelul NU primește business_id), args validate Pydantic ÎNAINTE de execuție, eșec prins în
`run_tool` → `ToolResult(ok=False)` (nu rupe turul). `back_in_stock_subscriptions` are deja
GRANT `select,insert,update,delete` + RLS pe `bot_runtime` (docs/003, liniile 59-62). DDL:
`unique (business_id, contact_id, product_id, variant_id)`, `variant_id` NULLABLE.

### `subscribe_back_in_stock` — WRITE tool (back_in_stock_subscriptions)

**Query nou** `src/db/queries/commerce.py::subscribe_back_in_stock`:
```python
async def subscribe_back_in_stock(
    conn, business_id, contact_id, product_id, variant_id=None
) -> dict:
    """INSERT idempotent în back_in_stock_subscriptions. Re-subscribe pe aceeași
    (product, variant) = ON CONFLICT no-op (resetează notified_at→NULL: vrea iar
    notificare). `business_id = $1` (P7). Întoarce {id, created}."""
    row = await conn.fetchrow(
        """
        insert into back_in_stock_subscriptions
            (business_id, contact_id, product_id, variant_id)
        values ($1, $2, $3, $4::uuid)
        on conflict (business_id, contact_id, product_id, variant_id)
            do update set notified_at = null
        returning id::text as id, (xmax = 0) as created
        """,
        business_id, contact_id, product_id, variant_id,
    )
    return dict(row)
```
**Capcană UNIQUE + NULL** (de documentat în cod): în Postgres `NULL` e DISTINCT în unique
constraint, deci `variant_id IS NULL` NU prinde `ON CONFLICT` → re-subscribe pe produs (fără
variantă) ar duplica. Mitigare în tool: dacă `variant_id is None`, facem un SELECT-guard
prealabil (există deja un rând activ pe `business_id+contact_id+product_id+variant_id is null`?)
→ dacă da, no-op cu `created=False`. Query helper `has_back_in_stock_sub` (SELECT cu
`variant_id is not distinct from $4`, scoped pe business). Alternativa „migrare la
`NULLS NOT DISTINCT`” e Out of Scope (atinge schema_v2, separat).

**Tool** `src/tools/commerce_tools.py::subscribe_back_in_stock_tool`:
```python
class BackInStockArgs(BaseModel):
    product_id: str = Field(min_length=1)
    variant_id: str | None = None

@register("subscribe_back_in_stock")
async def subscribe_back_in_stock_tool(ctx, deps, args) -> ToolResult:
    a = BackInStockArgs(**args)
    # 1. produsul există în catalog (scoped pe business) — nu abonăm la fantome
    products = await get_products_by_ids(deps.conn, ctx.business.id, [a.product_id], limit=1)
    if not products:
        return ToolResult(ok=False, error="not_found", llm_view="Produsul nu există în catalog.")
    p = products[0]
    # 2. dacă deja e pe stoc, nu are sens să abonezi — spune-i să-l ia acum
    if p.get("availability") == "in_stock":
        return ToolResult(ok=True, products=[p],
            llm_view=f"{p['name']} este pe stoc acum — nu e nevoie de notificare.")
    # 3. insert idempotent (cu guard pe variant NULL)
    if a.variant_id is None and await has_back_in_stock_sub(
            deps.conn, ctx.business.id, ctx.contact.id, a.product_id, None):
        ctx.emit("back_in_stock_subscribed", product_id=a.product_id, created=False)
        return ToolResult(ok=True, products=[p],
            llm_view=f"Ești deja pe lista de notificare pentru {p['name']}.")
    res = await subscribe_back_in_stock(
        deps.conn, ctx.business.id, ctx.contact.id, a.product_id, a.variant_id)
    ctx.emit("back_in_stock_subscribed", product_id=a.product_id, created=res["created"])
    return ToolResult(ok=True, products=[p],
        llm_view=f"Te anunț când {p['name']} revine pe stoc.")
```
`contact_id` vine din `ctx.contact.id` (rezolvat din `channel_identities`), NU din args (P12 —
PII-ul de contact nu trece prin model). `product_id` validat contra catalogului scoped pe
business — nu abonezi la un id ghicit dintr-un alt tenant. `products=[p]` în `ToolResult` (preț
grounded pentru validator, oglindind pattern-ul existent).

### `delivery_eta` — stub determinist documentat [PROPUNERE]

**Nu există integrare curier/ERP** în cod (src/proactive/ = doar `__init__.py`; nicio config de
curier). Deci tool-ul returnează un ETA estimativ DETERMINIST, marcat onest, NU o dată inventată:
```python
class DeliveryEtaArgs(BaseModel):
    product_id: str = Field(min_length=1)
    # adresa: doar text liber, folosită DOAR pt a distinge București/regiune (heuristică),
    # NU se persistă, NU intră în loguri (PII de livrare — P12). Niciun query pe ea.
    address: str | None = Field(default=None, max_length=200)

@register("delivery_eta")
async def delivery_eta_tool(ctx, deps, args) -> ToolResult:
    a = DeliveryEtaArgs(**args)
    products = await get_products_by_ids(deps.conn, ctx.business.id, [a.product_id], limit=1)
    if not products:
        return ToolResult(ok=False, error="not_found", llm_view="Produsul nu există în catalog.")
    p = products[0]
    if p.get("availability") in ("out_of_stock", "discontinued"):
        return ToolResult(ok=True, products=[p],
            llm_view=f"{p['name']} nu e pe stoc acum, deci nu pot estima livrarea.")
    # ETA estimativ din settings business (zile lucrătoare), fallback config global.
    lo, hi = _eta_business_days(ctx.business)   # ex. (1, 3) din business.settings sau default
    ctx.emit("delivery_eta_estimated", product_id=a.product_id, days_lo=lo, days_hi=hi)
    return ToolResult(ok=True, products=[p],
        llm_view=f"Estimativ {p['name']} ajunge în {lo}-{hi} zile lucrătoare (estimare, nu garanție).")
```
`_eta_business_days(business)` citește `business.settings["delivery_eta_days"]` (ex.
`{"lo":1,"hi":3}`), fallback `config.delivery_eta_days_default` (default `[1,3]`). Stub-ul NU
inventează AWB/tracking (alea vin la expediere reală via `shipments`, citite de `check_order`).
ETA-ul e exprimat ca INTERVAL de zile, NU ca dată calendaristică concretă → nu există „cifră de
preț” de validat; e mesaj text, marcat „estimare, nu garanție”. Când apare integrarea reală de
curier (task viitor), corpul lui `_eta_business_days` se înlocuiește cu apelul la API — semnătura
tool-ului nu se schimbă. **Marcat [PROPUNERE]**: dacă produsul echipă vrea să NU expunem ETA fără
integrare reală, tool-ul poate fi pus în spatele unui flag `delivery_eta_enabled` (default false)
— atunci nu intră în `_SALES_TOOLS` și agentul nu-l vede.

## Test Cases
Pattern: query-urile de DB monkeypatch-uite (ca în `tests/test_tools.py`), `run_tool` real,
`PipelineDeps` cu `conn`/`llm` fake. ZERO DB/LLM real în CI.

**Happy Path:**
1. `subscribe_back_in_stock` pe produs `out_of_stock` existent → `subscribe_back_in_stock`
   apelat cu `(business_id, contact.id, product_id, None)`, `ok=True`, event
   `back_in_stock_subscribed{created=True}`, llm_view „te anunț când revine”.
2. `delivery_eta` pe produs `in_stock`, `business.settings={"delivery_eta_days":{"lo":2,"hi":4}}`
   → llm_view conține „2-4 zile lucrătoare”, event `delivery_eta_estimated{days_lo=2,days_hi=4}`.

**Edge Cases:**
1. Re-abonare cu `variant_id=None` când există deja rând → `has_back_in_stock_sub` True →
   `created=False`, fără al doilea INSERT.
2. Abonare la produs `in_stock` → niciun INSERT, llm_view „este pe stoc acum”.
3. `delivery_eta` fără `business.settings` → fallback `config.delivery_eta_days_default=[1,3]`.
4. `delivery_eta.address` lung/cu diacritice → trunchiat la max_length, NU apare în event.

**Failure Cases:**
1. `product_id` din alt tenant (lipsă în catalogul scoped) → `get_products_by_ids` întoarce []
   → `ok=False, error=not_found`, niciun INSERT (izolare).
2. `subscribe_back_in_stock` (DB) aruncă → `run_tool` prinde → `ToolResult(ok=False)`, turul NU
   se rupe (P6), eroarea logată fără args/PII.
3. `args` invalide (`product_id` gol) → `BackInStockArgs` ridică ValidationError → prins în
   `run_tool` → degradare grațioasă, fără scriere.
