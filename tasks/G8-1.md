# G8-1 — Evals: golden harness + gate CI (regresii de pipeline)
**Owner:** S · **Faza:** P1 · **Zi/Ord:** pas 3 din secvențierea de producție (docs/agent-tools-architecture.md §9) · **Branch:** `feat/G8-1-golden-ci` · **Complexitate:** M · **Estimare:** ~4h

## Goal
Un **harness golden** care rulează un set curat de inputuri prin pipeline-ul REAL
(gates → limbă → cache → triaj → agent → fallback) cu un LLM **scriptat** (determinist,
ZERO OpenAI) și verifică rezultatul față de așteptări: **rută**, **fapte obligatorii**,
**interdicții** (preț/link inventat) și **niciodată tăcere** (P6). Plus un **gate CI**
(pytest) care pică build-ul la orice regresie. Prima felie din epicul „evals" — acum că
agentul ia acțiuni cu tool-calling (G7-1), regresiile de *selecție de tool* / validator
trebuie prinse automat.

## Business Context
Tool-calling-ul (G7-1) a mărit suprafața de regresie: agentul alege ce unealtă cheamă, iar
validatorul e singura plasă împotriva halucinațiilor de preț/link. Un golden CI verde =
garanție executabilă că modificările viitoare (prompturi, tool-uri, validator) nu strică
rutarea, grounding-ul sau invariantul „zero prețuri inventate". Schema are deja tabelele
(`golden_tests`, `conversation_evals`); aici livrăm motorul + gate-ul, sursa CI = fixture JSON.

## Technical Description

### Harness (`src/evals/golden.py`) — reutilizabil, agnostic de sursă/LLM
- `GoldenExpect(route?, must_include, forbidden, expect_reply=True)` — contractul de
  așteptări (oglindește `golden_tests.expected`: rută / fapte / interdicții).
- `GoldenResult(case_id, passed, failures)` — rezultatul + motivele.
- `evaluate_reply(ctx, expect, *, case_id) -> GoldenResult` — **checker PUR** peste un
  `TurnContext` deja rulat: rută (când e dată), `must_include` (toate prezente, case-insensitive),
  `forbidden` (niciunul prezent), `expect_reply` (un reply produs ⇒ P6; `False` ⇒ tăcere/halt
  așteptat). Reutilizabil în CI ȘI în producție (rulat pe pipeline real).
- `GoldenCase(id, input, language, expect, fixtures)` + `load_cases(path)` — încarcă fixture-urile
  JSON. `fixtures` = artefacte CI (script LLM + catalog + categorii), opace pentru checker.
- `run_case(ctx, deps, stages, expect, *, case_id) -> GoldenResult` = `run_pipeline` + `evaluate_reply`.

### Gate CI (`tests/test_golden.py`) — determinist, fără OpenAI/DB
- `ScriptedLLM`: implementează exact metodele pe care le cheamă pipeline-ul —
  `moderate` (fixtures.moderation), `embed` (zerouri), `classify_json` (fixtures.triage),
  `complete` (fixtures.retry), `run_tool_loop` (rulează fixtures.tool_calls prin `execute`,
  întoarce fixtures.final). ZERO apeluri reale.
- Stub-uri DB (monkeypatch, ca în `test_agent`): `list_category_slugs` (triaj),
  `exact_lookup`/`semantic_lookup` (cache → miss curat), `search_products_semantic` /
  `get_products_by_ids` (tool-uri → catalogul cazului), `set_handoff` (risc, no-op).
- Parametrizat pe toate cazurile din `cases.json` → fiecare caz e un test; `failures` în mesaj.

### Cazuri seed (`tests/golden/cases.json`) — plasa de regresie
1. `greeting-simple` — „salut" → rută `simple`, reply produs.
2. `sales-grounded` — cerere de cremă → rută `sales`, reply conține prețul REAL (82.99), fără 999.
3. `invented-price-blocked` — modelul inventează 999/888 → validator → fallback determinist cu
   prețul real; **interzis** 999/888 (invariantul anti-halucinație).
4. `moderation-neutral` — mesaj flagged → răspuns neutru (fără rută, gated înainte de triaj).
5. `risk-handoff` — „vreau să vorbesc cu un om" → mesaj de tranziție către coleg (handoff).
6. `order-never-silent` — „unde e comanda mea" → rută `order`, niciun stagiu nu o acoperă încă →
   fallback produce reply (P6: niciodată tăcere).

## Test Cases
**Happy Path:**
1. `evaluate_reply` pe un ctx cu reply corect, rută corectă, fapt prezent → `passed=True`.
2. Cele 6 cazuri din `cases.json` rulate prin `run_case` → toate `passed`.

**Edge Cases:**
1. `expect_reply=False` + ctx cu halt (tăcere) → pass; cu reply → fail.
2. Rută dată dar `ctx.route is None` (gated înainte de triaj) → fail dacă `route` cerut; cazurile
   gated (moderare/risc) au `route=None` în expect.

**Failure Cases:**
1. `forbidden` prezent în reply (preț inventat) → `passed=False`, motivul listat.
2. `must_include` absent → `passed=False` cu „missing fact: …".
3. Reply lipsă când `expect_reply=True` → `passed=False` cu „no reply (tăcere)".

> ZERO apeluri reale: `ScriptedLLM` (moderate/embed/classify_json/complete/run_tool_loop) +
> stub-uri DB prin monkeypatch. Gate-ul rulează în CI exact ca testele unit.
