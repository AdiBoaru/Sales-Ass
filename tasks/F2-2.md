# F2-2 — Bucla de bani: webhook comenzi → match `ref_code` → atribuire
**Owner:** S · **Faza:** P1 · **Zi/Ord:** pas 4 (bucla de bani), felia 2 · **Branch:** `feat/F2-2-orders-attribution` (stacked pe `feat/F2-1-checkout-link`) · **Complexitate:** M · **Estimare:** ~5h

## Goal
Închide bucla de atribuire: un webhook de comenzi de la platforma magazinului → `orders`
(idempotent), iar dacă comanda poartă `?ref=<ref_code>` (din linkul botului, F2-1) → match pe
`checkout_links` → `orders.attribution='assisted'` + `attributed_checkout_link_id` +
`checkout_links.converted_order_id`. Aici se naște „assisted revenue".

## Business Context
Fără această felie, linkurile generate de F2-1 nu se transformă în revenue MĂSURAT. Atribuirea
e metrica de bani a agenției (funnel: conversație → recomandare → checkout link → comandă
atribuită). `usage_daily` (F2-3) agregă apoi `order_attributed` în `revenue_attributed`.

## Technical Description

### Webhook subțire (margine, FĂRĂ DB — ca path-ul Meta)
- `POST /webhook/orders/{business_id}` ([app.py](src/webhook/app.py)): verifică un secret partajat
  (`X-Orders-Secret` == `ORDERS_WEBHOOK_SECRET`, compare_digest), parsează JSON, pune un envelope
  `{kind:'order', business_id, order}` pe stream-ul `inbound` (reutilizat) → ACK 200. Zero DB.
  Comenzile NU sunt evenimente de canal (n-au `channel_account_id`) → `business_id` vine din path
  (autentificat de secret); rezolvarea pe canal nu se aplică.

### Worker — rutare `kind='order'` ([consumer.py](src/worker/consumer.py))
- `process_event`: pentru `kind='order'` → `tenant_conn(business_id)` → `process_order(conn,
  business_id, order)`, ÎNAINTE de `resolve_channel` (care e doar pt evenimente de canal).

### Engine de atribuire ([src/webhook/orders.py](src/webhook/orders.py))
- `OrderIn`/`OrderItemIn` (Pydantic, neutru de platformă): `external_id, status, total, currency,
  ref?, placed_at, items[]`. Adaptorul platformă→neutru (Shopify/Woo) = out of scope.
- `process_order(conn, business_id, order) -> dict`:
  1. `ref` prezent → `get_checkout_link_by_ref` → dacă există: `attribution='assisted'`,
     `checkout_link_id`, `contact_id`/`conversation_id` din link.
  2. `upsert_order` (idempotent pe `business_id+external_id`; pe conflict update DOAR status/total
     — re-livrarea nu downgradează atribuirea). Întoarce `inserted` (xmax=0).
  3. `inserted` → `insert_order_items` (evită dublarea la re-livrare).
  4. atribuit → `mark_checkout_converted` (set `converted_order_id` dacă e null — idempotent).
  5. `insert_events`: `order_received` + (dacă atribuit) `order_attributed {attribution, total}`,
     fără PII (orders n-are telefon; contactul vine via channel_identities).

### Query-uri ([src/db/queries/commerce.py](src/db/queries/commerce.py))
- `get_checkout_link_by_ref(conn, business_id, ref_code)` (id, contact_id, conversation_id, converted_order_id).
- `upsert_order(...)` → `on conflict (business_id, external_id) do update set status, total, updated_at`,
  `returning id, (xmax=0) as inserted`.
- `insert_order_items(conn, order_id, items)` (executemany).
- `mark_checkout_converted(conn, business_id, checkout_link_id, order_id)` (set dacă null).

### Config
- `orders_webhook_secret` (`ORDERS_WEBHOOK_SECRET`, default `""` → endpoint 403 dacă negol cerut).

## Principii CLAUDE.md aplicabile
- **Webhook subțire, fără DB:** endpointul doar enqueue (ca path-ul Meta); DB-ul e în worker.
- **P7:** fiecare query cu `business_id`; match-ul `ref_code` e per business.
- **P12:** zero PII în loguri/analytics; telefonul nu intră în `orders`.
- **Idempotență:** `(business_id, external_id)` + items doar la insert + `converted_order_id` set-if-null.
- **P6:** payload invalid → 400/403 explicit, nu 500/tăcere; order necunoscut fără ref → `attribution='none'`.

### Câmpuri scrise
`orders`, `order_items`, `checkout_links.converted_order_id`, `analytics_events`. Niciun câmp pe TurnContext.

## Implementation Steps

## Test Cases
**Happy Path:**
1. `process_order` cu ref valid → atribuit (query-uri monkeypatch); events corecte.
2. Endpoint cu secret valid → fakeredis are 1 envelope `kind='order'`, 200.

**Edge Cases:**
1. ref inexistent → `attribution='none'`, fără `mark_checkout_converted`.
2. Re-livrare (`inserted=False`) → `insert_order_items` NU se cheamă a doua oară.
3. Comandă fără items → upsert ok, fără order_items.

**Failure Cases:**
1. Secret lipsă/greșit → 403, fără enqueue.
2. JSON invalid → 400.
3. `OrderIn` invalid (lipsă external_id/total) în worker → ValidationError prins, logat (turul de
   order nu blochează consumerul).

> ZERO apeluri reale: engine cu query-uri monkeypatch-uite; endpoint cu fakeredis + override secret.

## Cost
