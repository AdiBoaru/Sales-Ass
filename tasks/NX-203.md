# NX-203 — Retrieval benchmark set (200-500 query-uri) + harness ir-measures

**Owner:** Claude (build) / Codex (verify) · **Faza:** 1 (Quality Overhaul) · **Status:** DRAFT rev.2 (post-review Codex) · **Complexitate:** M
**Parent plan:** [docs/QUALITY-OVERHAUL-2026.md](../docs/QUALITY-OVERHAUL-2026.md) · **Depinde de:** — (paralel cu NX-202)
**Decizii ADR aplicabile:** D3, D15

## Goal
Metrul de măsură pentru retrieval: setul pe care se aleg documentele, embeddings-urile și
rerankerul — SEPARAT de golden set-ul conversațional (alt volum, alte metrici).

## Scope
1. **200-500 query-uri ro-RO** cu qrels. Acoperire: exact match (nume/brand/SKU) · colocvial
   (concern_map) · compuse (multi-constraint) · typo/diacritice lipsă · negative („fără parfum",
   „nu brandul X").
2. **Metodologia qrels (obligatorie):**
   - **relevanță GRADUALĂ** (ex. 0-3), nu binar relevant/irelevant;
   - **hard constraints așteptate** per query;
   - **produse INTERZISE explicit** (nu doar „nerelevante");
   - **proveniența per query:** `real_sanitized` | `synthetic` | `paraphrase`, cu **minim de
     query-uri reale sanitizate per categorie**;
   - **`catalog_version`** pe care s-au făcut etichetele (etichetele expiră la schimbarea catalogului);
   - **verificare umană pentru TOATE variațiile generate** (nimic sintetic neverificat);
   - **test de integritate qrels** (produse inexistente, contradicții relevant/interzis, duplicate).
3. **Split OBLIGATORIU tuning vs holdout, cu felii SINGLE-USE per gate (versionat):** setul se
   împarte de la început în **dev/tuning** (pe el se aleg documentele, embeddings-urile,
   rerankerul, ponderile) și **holdout de retrieval** partiționat în FELII dedicate, stratificate
   pe categoriile de query: **H1** (gate-ul de switch NX-207), **H2** (gate-ul NX-209),
   **H3** (evaluarea NX-210). **O felie se deschide O SINGURĂ DATĂ, la gate-ul ei; după
   deschidere e „arsă"** — se retrogradează în pool-ul de tuning și NU mai validează nimic
   (reutilizarea aceluiași holdout la gate-uri succesive = contaminare: deciziile dintre
   gate-uri ar fi informate de el). Feliile se realimentează din query-urile noi etichetate de
   ritualul săptămânal (NX-215). Split-ul + istoricul deschiderilor sunt versionate în repo;
   mutarea unui query între seturi = schimbare cu review.
4. **Harness `ir-measures`** (dev-only): Recall@20, nDCG@6, Top-6 hit rate, încălcări de
   constrângeri, latență, cost per query. **Pin în `requirements-dev.txt`** (nu există azi).
   Output **machine-readable** și comparabil între rulări, cu **configurația completă a rulării**
   înregistrată (model embeddings, `document_version`, ponderi/weights, reranker, data).
   **Intervale de încredere** raportate (nu doar medii punctuale).
5. **Baseline-ul configurației ACTUALE** (lexical + semantic + fusion existente din
   `src/tools/catalog_tools.py`) publicat — pe AMBELE splits (o singură dată; ulterior
   holdout-ul doar la gate-uri).

## Out of Scope
Schimbări de retrieval (NX-207/NX-209). Aici doar setul + harness-ul + baseline.

## Definition of Done
- [ ] ≥200 query-uri cu qrels graduale, versionate în repo, cu proveniență + `catalog_version`.
- [ ] Split tuning/holdout versionat, stratificat, cu felii H1/H2/H3 dedicate per gate;
      holdout exclus din orice buclă de alegere; istoricul deschiderilor înregistrat
      (o felie deschisă = arsă, retrogradată în tuning; verificabil în istoric).
- [ ] Test de integritate qrels verde; toate variațiile generate verificate uman.
- [ ] `ir-measures` pinned în `requirements-dev.txt`; harness rulabil cu o comandă.
- [ ] Output machine-readable cu config-ul rulării + intervale de încredere.
- [ ] Baseline publicat: Recall@20 / nDCG@6 / Top-6 / latență / cost pe configurația actuală.
- [ ] Zero PII în query-urile sanitizate (test dedicat); toate rulările scoped pe `business_id`.
