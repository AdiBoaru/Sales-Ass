# NX-181 — Prompt vNext (relaxare _RICH_RULES, standalone)

**Owner:** Claude (build) / Codex (verify) · **Faza:** P1 · **Track:** Response Quality · **Branch:** `feat/NX-181-prompt-vnext` · **Complexitate:** S · **Estimare:** 1 zi
**Parent plan:** [docs/RESPONSE-QUALITY-EPIC.md](../docs/RESPONSE-QUALITY-EPIC.md) · **Depinde de:** NX-180 (baseline)

## Goal
Schimbare izolată, cu risc mic: relaxează `_RICH_RULES` și injectează un `response_shape` + semnal anti-repetiție în mesajul USER. Măsurată SEPARAT vs baseline. **Așteptare temperată:** promptul de pe origin/main e DEJA de-templatizat (intro „1-2 fraze naturale", education „opțională, mai bine gol", anti-repetiție explicită) → probabil plafonează; evaluatorul decide dacă mai e nevoie de V2.

## Business Context
Template-ul nu e impus de schema JSON (permite pick/education=null, suggestions=[]) ci de prompt + calea rich mereu-carduri. Curățăm ce mai împinge probabilistic modelul spre aceeași formă, fără să atingem schema.

## Technical Description
- `src/agent/prompt_builder.py` (`_RICH_RULES`): scoate „ideal 4"; recomandă **1-3 produse după nevoie**; intro maxim o frază, poate fi gol; education implicit gol (deja opțional — întărește regula de aur); chips doar dacă oferă acțiune utilă; șterge exemplele care ancorează formulări.
- **`response_shape`** minimal în mesajul USER (nu în system → prompt caching intact): `recommendation | direct_followup | detail`, derivat DETERMINIST din semnalele existente ale planner-ului (attr_query/detail/rehidratare). **Vocabular unic:** `response_shape` e SINGURUL nume (nu există `response_hint` nicăieri în cod); owner unic = planner. NX-181 îl introduce minimal, NX-184 îl îmbogățește cu `obligations`.
- **Anti-repetiție** în mesajul USER, definit concret: **prima propoziție a răspunsului anterior, normalizată + trunchiată** (NU un hash opac — nu ajută modelul). Fără analytics, fără state nou dacă istoricul o conține deja.

## Principii CLAUDE.md aplicabile
- **Prompt caching:** semnalele per-tur stau în USER, prefixul system rămâne byte-identic.
- **P9:** promptul se generează din DB; schimbăm regulile invariante, nu hardcodăm per-caz.

## Câmpuri TurnContext scrise
Niciunul nou (doar conținutul promptului + input per-tur).

## Implementation Steps
1. Editează `_RICH_RULES` (relaxări de mai sus), în spatele kill-switch-ului.
2. Derivă `response_shape` din planner + pasează-l în USER (`_finalize_rich`).
3. Adaugă semnalul anti-repetiție (prima propoziție anterioară normalizată+trunchiată) în USER.
4. Rulează evaluatorul (NX-180) ON vs OFF (paired, intercalat); compară.
