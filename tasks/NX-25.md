# NX-25 — Web Widget W3: CORS/CSP + allowlist domenii per tenant
**Owner:** S · **Faza:** P2 · **Zi/Ord:** Epic E26 (V1.5), după W1 (NX-20) · **Branch:** `feat/NX-25-web-cors-allowlist` · **Complexitate:** S · **Estimare:** 3h

## Goal
Widget-ul web (gateway SSE din NX-20) e public prin design: oricine vede `widget.js`
și `data-token`-ul în HTML-ul clientului. Acest task pune frâna de origine: fiecare
request către `/web/*` se acceptă DOAR dacă vine de pe un domeniu din allowlist-ul
tenantului (`businesses.settings.web.allowed_origins`). Token-ul public identifică
tenantul, NU autentifică — securitatea reală e perechea (token → allowlist de origini)
+ rate limit agresiv. Embed de pe un domeniu neautorizat → `403`, fără sesiune, fără
niciun apel LLM. Fără asta, oricine ar putea cloda widget-ul altui client pe propriul
site și i-ar arde bugetul de tokeni.

## Business Context
Modelul de business e SaaS per tenant cu cost guard zilnic. Token-ul public e vizibil în
sursa paginii oricărui client → fără allowlist, un actor rău (sau un concurent) embeddează
widget-ul pe `evil.example` și consumă cota LLM a clientului nostru până la `daily_cost_cap`,
lăsându-l fără bot pe site-ul lui real. Allowlist-ul de origini transformă token-ul public
într-un identificator inofensiv: util doar de pe domeniile pe care le-am autorizat la onboarding.

## Technical Description

Toată logica e **cod determinist, zero LLM** (P2). Trăiește la marginea web (NX-60):
`src/web/` (gateway-ul SSE). NU atinge pipeline-ul (stagiile 3-9) și NU scrie niciun câmp
din `TurnContext` — respinge requestul ÎNAINTE ca un envelope să ajungă pe stream-ul `inbound`.

### Sursa allowlist-ului: `businesses.settings` (jsonb, fără migrare)
Per tenant, în `businesses.settings`:
```jsonc
{
  "web": {
    "allowed_origins": ["https://shop.client.ro", "https://www.client.ro"],
    "theme": { /* NX-21 */ }
  }
}
```
Origini = scheme + host (+ port dacă != 443), match EXACT, case-insensitive pe host,
fără trailing slash, fără path. `http://` permis DOAR pe `localhost`/`127.0.0.1` (dev);
în prod allowlist-ul e doar `https://`. Wildcard de subdomeniu (`https://*.client.ro`)
explicit **out of scope** v1 (vezi mai jos).

Token-ul public → tenant se rezolvă în NX-20 (`resolve_web_token(token) → {business_id,
allowed_origins}`, pe `admin_conn` — control plane, ca `resolve_channel`, fiindcă la
momentul requestului încă nu avem tenant scope). Acest task adaugă `allowed_origins`
la ce întoarce acel lookup și consumă valoarea.

### Verificarea originii (cod, nu doar middleware CORS)
CORS-ul browserului blochează DOAR citirea răspunsului cross-origin de către JS — NU
oprește serverul să proceseze requestul (un `curl`/bot ignoră complet CORS). Deci verificarea
e în DOUĂ straturi:

1. **Guard în handler (dur, sursa de adevăr)** — pe fiecare `/web/*`, înainte de orice
   lucru, compară `Origin` (sau `Referer` pentru SSE GET, unde Origin poate lipsi) cu
   allowlist-ul tenantului rezolvat din token:
```python
# src/web/security.py
def normalize_origin(value: str | None) -> str | None:
    if not value:
        return None
    p = urlsplit(value.strip())
    if not p.scheme or not p.hostname:
        return None
    host = p.hostname.lower()
    port = f":{p.port}" if p.port and p.port not in (80, 443) else ""
    return f"{p.scheme.lower()}://{host}{port}"

def origin_allowed(origin: str | None, allowed: list[str]) -> bool:
    norm = normalize_origin(origin)
    if norm is None:
        return False  # fără Origin valid → respins (nu fail-open pe securitate)
    allow = {normalize_origin(a) for a in allowed} - {None}
    return norm in allow

async def require_allowed_origin(request: Request, tenant) -> str:
    origin = request.headers.get("origin") or request.headers.get("referer")
    if not origin_allowed(origin, tenant.allowed_origins):
        # P12: logăm originea respinsă (nu e PII) + business_id, NU token-ul întreg
        log.warning("web: origin respins biz=%s origin=%s", tenant.business_id, origin)
        raise HTTPException(status_code=403, detail="origin not allowed")
    return normalize_origin(origin)
```
> Notă SSE: `EventSource` (GET `/web/stream`) NU trimite `Origin` în toate browserele;
> cădem pe `Referer`. Dacă lipsesc ambele → `403`. Sesiunea SSE moștenește originea
> validată la `POST /web/messages` (din NX-20), deci stream-ul nu poate fi deschis de
> pe altă origine decât cea care a creat sesiunea.

2. **Middleware CORS reflectat per-tenant (UX corectă în browser)** — răspundem cu
   `Access-Control-Allow-Origin: <origin>` DOAR dacă originea e în allowlist (reflectare,
   NU `*`, fiindcă trimitem cookie first-party — NX-23). `Access-Control-Allow-Credentials:
   true`, metode `GET, POST, OPTIONS`, headere `Content-Type, Last-Event-ID`,
   `Access-Control-Max-Age: 600`. Preflight `OPTIONS` se rezolvă în middleware (200),
   fără să atingă handlerul. `Vary: Origin` mereu (corectitudine cache/CDN).
   Nu folosim `CORSMiddleware` global (e single allowlist, noi avem per-tenant) → middleware
   custom care rezolvă tenantul din token și citește `allowed_origins`.

### CSP + headere de securitate pe răspunsuri
Pe răspunsurile gateway-ului web (și pe `widget.js` servit, dacă îl servim noi):
```
Content-Security-Policy: default-src 'none'; connect-src 'self'; frame-ancestors 'none'
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Cross-Origin-Resource-Policy: same-site
```
`frame-ancestors 'none'` pe endpoint-urile gateway (nu pe widget.js, care e injectat în
pagina clientului). CSP-ul widget-ului în sine (shadow DOM) e NX-21; aici doar headerele
API. `Cache-Control: no-store` pe `/web/stream` și `/web/messages`.

### Rate limit agresiv (token public = ușor de abuzat)
Token-ul nefiind secret, rate limit-ul e mai dur decât pe WhatsApp. Refolosim contorul
Redis existent (`src/web/` îl apelează; pattern „contor Redis" din pipeline):
- per `(business_id, visitor_id)`: ex. 10 mesaje / 30s;
- per `(business_id, ip)`: ex. 60 req / min (acoperă vizitatori fără cookie + boți);
- depășire → `429` cu `Retry-After`, ZERO apel LLM.
NX-20 definește pragurile exacte; acest task se asigură că verificarea de origine rulează
ÎNAINTE de rate limit (origine greșită = `403` instant, nici nu atinge contorul).

### Ordinea la marginea web (fiecare cu early-exit)
```
1. OPTIONS (preflight)     → middleware CORS răspunde 200, fără handler
2. rezolvă token → tenant  → token necunoscut/inactiv → 403 (NX-20)
3. require_allowed_origin   → origine ne-allowlistată/absentă → 403   [ACEST TASK]
4. rate limit (Redis)       → depășit → 429
5. handler real             → envelope pe stream `inbound` (NX-20/NX-22)
```

### Câmpuri TurnContext scrise aici
**Niciunul.** Respingerea se face la marginea HTTP, înainte de pipeline. Dacă requestul
trece, envelope-ul neutru (canal `web`) e produs de NX-20/NX-22, nu de acest task.

## Principii CLAUDE.md aplicabile
- **P2 (LLM doar triaj+agent):** verificarea de origine + CORS/CSP = cod pur determinist.
  Un request respins NU ajunge la niciun apel LLM — exact rostul lui (protejează costul).
- **P5 (un singur punct de ieșire):** nu introducem o cale nouă de trimitere; doar gardăm
  intrarea web înainte de `inbound`.
- **P6 (niciodată tăcere) — NUANȚAT:** P6 e despre răspunsul către client în pipeline.
  Aici e securitate de margine: o origine neautorizată merită `403` explicit, NU fail-open
  (contrar lui NX-15 unde moderation e best-effort). Un actor rău nu primește degradare grațioasă.
- **P7 (business_id pe tot):** allowlist-ul e citit per `business_id`; cheile Redis de rate
  limit includ `business_id`; lookup-ul token→tenant e singura excepție control-plane (ca
  `resolve_channel`), unde tocmai derivăm `business_id`.
- **P11 (locale parte din cheie):** nu se aplică direct (nu atingem faqs/cache/templates aici).
- **P12 (PII într-un loc):** logurile de respins conțin DOAR `business_id` + originea (host
  public, nu PII) — NICIODATĂ token-ul întreg (prefix scurt max), niciun `visitor_id` brut,
  niciun telefon. Token-ul se redactează în logger ca și secretele.

## Implementation Steps

   `security_headers(response)` (CSP + nosniff + referrer + CORP).
2. `src/web/middleware.py`: middleware CORS per-tenant (rezolvă token din query/header →
   `allowed_origins`, reflectă originea allowlistată, gestionează `OPTIONS` preflight, `Vary: Origin`).

## Database Changes
**None (DDL).** Allowlist-ul trăiește în `businesses.settings` (jsonb existent în schema_v2).
Lookup-ul (read-only, control plane, NU bot_runtime — `businesses` e read-only pentru bot):
```sql
-- rulat pe admin_conn (token→tenant, înainte de a avea tenant scope)
select id::text as business_id,
       coalesce(settings->'web'->'allowed_origins', '[]'::jsonb) as allowed_origins
from businesses
where id = $1;        -- business_id derivat din token în NX-20
```
> Nu e un query de tenant runtime (n-are `where business_id = $1` ca filtru de izolare,
> ci `where id = $1` ca lookup de control plane — îl DERIVĂM, ca `resolve_channel`).
Scrierea allowlist-ului (onboarding, rol admin):
```sql
update businesses
set settings = jsonb_set(settings, '{web,allowed_origins}', $2::jsonb, true)
where id = $1;
```

## API Changes
Nu adaugă endpointuri noi (acelea sunt în NX-20). Modifică comportamentul `/web/*`:
- **`OPTIONS /web/messages`, `OPTIONS /web/stream`** → `200` cu headere CORS reflectate dacă
  originea e allowlistată; altfel `403`.
- **`POST /web/messages`** → `403 {"detail":"origin not allowed"}` dacă `Origin`/`Referer` nu
  e în allowlist; `429` cu `Retry-After` la depășire rate limit; altfel comportamentul NX-20.
- **`GET /web/stream`** → `403` dacă originea (din Referer/sesiune) nu e allowlistată.
- Toate răspunsurile `/web/*` capătă headerele de securitate (CSP, nosniff, CORP, Vary: Origin).

## Events de emis (analytics)
- `web_origin_rejected` `{reason: "not_in_allowlist" | "missing_origin", origin_host}` — pe `403`
  de origine. FĂRĂ token, FĂRĂ visitor_id (P12). Util pentru a detecta clonarea widget-ului.

> Notă: emiterea în `analytics_events` cere `business_id`; pe origine respinsă ÎL avem (token-ul
> s-a rezolvat la tenant — refuzăm originea, nu tenantul). Dacă token-ul însuși e necunoscut
> (respins în NX-20 înainte de acest guard), NU emitem aici (n-avem business_id) — e cazul NX-20.
> Rate limit (`429`) emite în NX-20, nu aici (ca să nu dublăm).

- **Token-ul în sine + rotația** — generarea/rezolvarea token public per tenant e NX-20.
- **Rate limit (pragurile + contorul Redis)** — definit în NX-20; aici doar ordonăm: origine
  înainte de rate limit.
- **CSP-ul widget-ului (shadow DOM, surse permise în pagina clientului)** — NX-21; aici doar
  headerele endpointurilor API.
- **WAF / anti-DDoS la nivel de CDN** — infra (VPS/Cloudflare), nu cod aplicație.
- **UI de management al allowlist-ului în dashboard** — onboarding manual prin script v1.

## Test Cases
Toate cu `fastapi.testclient.TestClient` + `resolve_web_token`/redis mock-uite — **zero rețea,
zero LLM, zero DB real** în CI.

**Happy Path:**
1. Token tenant A + `Origin: https://shop.client.ro` (în allowlist) → `POST /web/messages` 200,
   envelope pus pe stream (fake redis `xadd` apelat o dată), header `Access-Control-Allow-Origin`
   = originea reflectată.
2. Preflight `OPTIONS` cu origine allowlistată → 200, headere CORS corecte, handlerul real
   NU e atins (mock pe handler aruncă dacă e apelat).

**Edge Cases:**
1. `Origin: https://SHOP.Client.RO/` (uppercase + trailing slash) vs allowlist
   `https://shop.client.ro` → match (normalizare), 200.
2. `EventSource` pe `GET /web/stream` fără `Origin` dar cu `Referer: https://shop.client.ro/p/5`
   → originea derivată din Referer e allowlistată → stream deschis.
3. `http://localhost:5173` cu `web_require_https_origins=false` (dev) → permis; același cu
   `=true` (prod) → `403`.
4. Token tenant A folosit cu `Origin` din allowlist-ul tenantului B → `403` (allowlist-uri izolate).

**Failure Cases:**
1. `Origin: https://evil.example` (necunoscut) → `403`, `web_origin_rejected{reason:"not_in_allowlist"}`
   emis, `xadd` neapelat (banul LLM neatins).
2. Request fără `Origin` și fără `Referer` (curl/bot) → `403 missing_origin`, fără sesiune.
3. `businesses.settings.web.allowed_origins` lipsă/`[]` (tenant neconfigurat) → orice origine → `403`
   (fail-closed: tenant fără allowlist = widget web nedeschis, nu deschis pentru toți).
4. `allowed_origins` cu o intrare malformată (`"client.ro"` fără scheme) → `normalize_origin`
   o ignoră (None), nu crapă; restul allowlist-ului funcționează.

## Cost
**Zero cost LLM/embeddings** — task de margine pur determinist (CORS/CSP/allowlist + rate limit).
Dimpotrivă, scopul lui economic e să PREVINĂ cost: fiecare `403`/`429` oprește requestul înainte
de triaj(nano)+agent(mini), deci protejează `daily_cost_cap_usd` al tenantului împotriva
clonării widget-ului pe domenii neautorizate. Latență adăugată per request legitim: o normalizare
de string + un set lookup (μs) — neglijabil.
