# NX-07 — Pacing proactiv + quiet hours
**Owner:** S · **Faza:** P1 · **Zi/Ord:** O2, după NX-70 (motorul) + NX-05 (pauză quality) · **Branch:** `feat/NX-07-proactive-pacing` · **Complexitate:** M · **Estimare:** 6h

## Goal
Pune o poartă de **pacing** în motorul proactiv (NX-70), ÎNTRE poarta de gating
(NX-71: consent/24h/template) și `enqueue_outbox`: limitează ritmul mesajelor
proactive ca să nu pară spam și să nu ardă quality rating-ul. Trei reguli:
(1) **cap per contact** — max 2 proactive/săptămână; (2) **cap per business/oră** —
spread, nu burst; (3) **quiet hours** — nimic între 21:00–09:00 în
`businesses.timezone`. Mesajele care lovesc o limită NU se pierd: jobul se
**reprogramează** (împinge `scheduled_at` în viitor), nu se aruncă. Frâna corectă
pe a doua sursă de bani (proactivul), complementară pauzei pe quality (NX-05).

## Business Context
Quality rating-ul WhatsApp scade aproape întotdeauna din cauza mesajelor proactive
percepute ca spam (vezi NX-05). NX-71 ține proactivul *legal* (consent + fereastră
+ template aprobat), dar legal ≠ ne-spammy: 5 template-uri într-o oră către același
om sau un burst de 200 de mesaje la 3 dimineața tot îți strică reputația și te duce
spre restricție. Pacing-ul e disciplina de ritm care păstrează livrabilitatea pe
care se bazează toți banii din proactiv (AWB, recuperare coș, restock). Quiet hours
e și o chestiune de bun-simț: un client nu vrea ping la 3 noaptea.

from dataclasses import dataclass

@dataclass(frozen=True)
class PacingVerdict:
    allow: bool
    defer_until: datetime | None = None   # dacă allow=False: când să re-încerce
    reason: str = "ok"                    # 'ok'|'quiet_hours'|'contact_cap'|'business_rate'

### Regula 1 — Quiet hours (21:00–09:00 în timezone-ul businessului)
Cod pur, fără DB. Convertim `now` (UTC) în ora locală a businessului cu
`zoneinfo.ZoneInfo(business.timezone)` și verificăm fereastra de tăcere. Pragurile
vin din config (`proactive_quiet_start_hour=21`, `proactive_quiet_end_hour=9`):
```python
from zoneinfo import ZoneInfo

def _quiet_hours_defer(tz_name: str, now_utc: datetime) -> datetime | None:
    s = get_settings()
    try:
        tz = ZoneInfo(tz_name)
    except Exception:                       # timezone invalid în DB → fallback Bucharest, nu crăpa
        tz = ZoneInfo("Europe/Bucharest")
    local = now_utc.astimezone(tz)
    start, end = s.proactive_quiet_start_hour, s.proactive_quiet_end_hour   # 21, 9
    in_quiet = local.hour >= start or local.hour < end                     # fereastră peste miezul nopții
    if not in_quiet:
        return None
    # următoarea oră de "end" locală (azi dacă suntem înainte de zori, mâine dacă seara)
    resume_day = local.date() if local.hour < end else (local.date() + timedelta(days=1))
    resume_local = datetime.combine(resume_day, time(hour=end), tzinfo=tz)
    return resume_local.astimezone(timezone.utc)
```
Fereastra `21:00–09:00` trece peste miezul nopții → condiția e `hour >= 21 OR hour < 9`.
Reprogramarea se face EXACT la 09:00 local convertit în UTC (mesajul nu se pierde,
pleacă dimineața). Quiet hours se evaluează PRIMA (cea mai ieftină, fără round-trip).

### Regula 2 — Cap per contact (max 2 proactive / 7 zile)
Sursa de adevăr = **DB** (durabilă peste restart, nu Redis): câte joburi proactive
au fost efectiv trimise către contactul ăsta în ultimele 7 zile. „Trimis" =
`status='sent'` cu `executed_at` recent (jobul curent încă e `scheduled`, deci nu se
numără pe sine):
```python
# src/db/queries/proactive.py (extinde modulul NX-70)
async def count_recent_proactive(conn, business_id, contact_id, *, since) -> int:
    return await conn.fetchval(
        """
        select count(*)
        from proactive_jobs
        where business_id = $1 and contact_id = $2
          and status = 'sent' and executed_at >= $3
        """,
        business_id, contact_id, since,   # since = now - interval '7 days'
    )
```
```python
async def _contact_cap_hit(conn, business_id, contact_id, now) -> bool:
    s = get_settings()
    since = now - timedelta(days=s.proactive_contact_cap_days)        # 7
    n = await count_recent_proactive(conn, business_id, contact_id, since=since)
    return n >= s.proactive_contact_cap_max                            # 2
```
Deferim la începutul ferestrei următoare (`now + 1 zi` la `09:00` local, prin
`_next_window_start`) — nu re-evaluăm la fiecare ciclu de 5s degeaba.

### Regula 3 — Cap per business/oră (spread, nu burst)
Contor Redis per (business, oră-curentă-UTC), exact ca flag-urile din NX-15. Bucket
orar fix (cheia conține ora) cu TTL 1h: numărăm câte au plecat în ora curentă; peste
plafon → deferim cu un mic spread (`proactive_spread_minutes`, default 3 min) ca să
distribuim restul, nu să le aruncăm:
```python
async def _business_rate_ok(redis, business_id, now) -> bool:
    if redis is None:                       # fără Redis → fail-OPEN (nu opri tot proactivul)
        return True
    s = get_settings()
    bucket = now.strftime("%Y%m%d%H")       # bucket orar UTC
    key = f"prorate:{business_id}:{bucket}"
    try:
        n = await redis.incr(key)
        if n == 1:
            await redis.expire(key, 3600)
    except Exception:
        log.warning("pacing: contor Redis eșuat → fail-open")
        return True
    return n <= s.proactive_business_hourly_cap     # default 30/oră
```
**Atenție la corectitudinea contorului:** `incr` se face în `_business_rate_ok`, dar
mesajul chiar pleacă DOAR dacă quiet+contact-cap au trecut deja (ele rulează ÎNAINTE,
deci ajungem aici doar pe drumul „va fi trimis"). Dacă `enqueue_outbox` dă dedup
(`new_id is None`, re-run idempotent), NU re-incrementăm (contorul reflectă intenția
de trimitere o singură dată per job — vezi „idempotent" mai jos). Contorul e o
estimare de spread, nu facturare (sursa de adevăr la volum = `usage_daily`), deci
mici scurgeri la restart sunt acceptabile (TTL 1h le curăță).

### Reprogramare — `proactive_jobs` NU are `next_attempt_at`
Seed-ul zice „mesajele amânate primesc `next_attempt_at`". În schema reală
`next_attempt_at` e pe `outbox`, NU pe `proactive_jobs`. Mecanismul echivalent pe
proactive_jobs = **împinge `scheduled_at`** (claim-ul motorului e `scheduled_at <=
now() and status='scheduled'`, deci un `scheduled_at` în viitor scoate jobul din
lot fără să-l pierzi). Jobul rămâne `scheduled` (NU `skipped_*` — skip e verdict
NX-71, defer e pacing):
```python
async def reschedule_job(conn, business_id, job_id, defer_until) -> None:
    await conn.execute(
        """
        update proactive_jobs
           set scheduled_at = greatest($3, scheduled_at)   -- nu trage niciodată mai devreme
         where business_id = $1 and id = $2 and status = 'scheduled'
        """,
        business_id, job_id, defer_until,
    )
```
`greatest(...)` garantează că un defer nu poate accidental grăbi un job programat
mai târziu. Idempotent: rerularea aceluiași defer e no-op (același `scheduled_at`).

## Test Cases
> Toate cu `conn`/`redis` fake (stub pe `count_recent_proactive` + fake `incr`) și
> `now` INJECTAT (param `now=` în `check_pacing`) — zero dependență de ceasul real,
> ZERO DB/Redis real, ZERO apel Meta/Telegram în CI. NX-07 nu cheamă niciun LLM.

**Happy Path:**
1. `now` = 14:00 local, contact cu 0 proactive recente, rate sub plafon →
   `check_pacing.allow=True`, niciun defer, niciun event de pacing.
2. `now` = 10:00 local (după quiet), contact cu 1 proactiv în 7 zile, rate ok →
   `allow=True` → jobul ajunge la `enqueue_outbox` (verificat că `_process_job`
   continuă).

**Edge Cases:**
1. `now` = 21:00 local fix (marginea ferestrei) → în quiet → defer la 09:00 local
   următor; `now` = 09:00 local fix → NU în quiet → trece (margini inclusiv/exclusiv
   testate explicit: `>= 21` și `< 9`).
2. Quiet peste miezul nopții: `now` = 02:00 local → defer la 09:00 ACEEAȘI zi (nu +1).
3. `businesses.timezone = "Nonsense/Zone"` → fallback `Europe/Bucharest`, decizia
   se ia pe Bucharest, fără excepție.
4. Re-run idempotent: defer aplicat de două ori pe același job → `scheduled_at`
   identic (`greatest`), no-op al doilea (fără dublu-incrementare Redis: incr-ul nu
   se atinge pe calea de defer quiet/contact, doar pe `_business_rate_ok`).

**Failure Cases:**
1. `redis=None` (fără Redis) → `_business_rate_ok` fail-open `True`; pacing decide
   doar pe quiet+contact-cap; mesajul nu e blocat de absența contorului.
2. `redis.incr` aruncă (rețea) → fail-open `True`, warning logat, fără excepție
   propagată; restul lotului continuă.
3. `count_recent_proactive` aruncă (DB) → excepția e prinsă în `_process_job` (NX-70)
   → jobul → `failed` + `proactive_failed`, restul lotului neafectat (moștenit din
   contractul de izolare per job al NX-70).
