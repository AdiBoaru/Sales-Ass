# NX-85 — Lock per conversație pentru ordonare multi-consumer
**Owner:** S · **Faza:** P1 · **Zi/Ord:** stagiul 2 hardening (după debounce R1) · **Branch:** `feat/NX-85-conversation-lock` · **Complexitate:** S · **Estimare:** 3.5h

## Goal
Garantează ordinea per-conversație când rulează **>1 replică de worker**. Acum ordinea
ține DOAR fiindcă un singur consumer procesează totul (vezi TODO `consumer.py:11`).
`XREADGROUP` distribuie entry-urile pe consumeri → 2+ replici pot procesa concurent /
dezordonat aceeași conversație, ceea ce dă `state_version` conflict (optimistic lock din
`patch_conversation_state`), istoric incoerent și răspunsuri întrețesute. Soluția: un
**lock Redis pe (business_id, identitatea expeditorului)** ținut pe durata procesării unui
tur, cu **skip-and-retry** (re-coadă) când e ocupat — fără lock cross-conversație, deci
paralelismul ÎNTRE conversații diferite rămâne intact (exact ce vrem la scale).

## Business Context
Producția rulează `docker compose` cu posibil >1 replică de worker (scale orizontal pe VPS).
Fără lock, două mesaje rapide ale aceluiași client pot ateriza pe consumeri diferiți și se
calcă reciproc: clientul vede două răspunsuri în ordine greșită, sau unul se pierde pe
`state_version` conflict. Asta arată rupt și costă bani (turul pierdut e re-procesat sau,
mai rău, taie vânzarea). Lock-ul e plasa care face scale-out-ul sigur, păstrând debit-ul
(conversații diferite în paralel).

## Technical Description

### De ce lock pe identitatea expeditorului, nu pe conversation_id direct
`conversation_id` se rezolvă abia în `handle_turn` (după `get_or_create_conversation`).
Dar pe calea inbound a unui tur, identitatea **`(channel_kind, channel_account_id,
sender_external_id)`** mapează 1:1 la conversația activă a contactului pe acel canal —
exact cheia pe care `Debouncer._key` o folosește deja (`debounce.py:41`). Deci lock-ul îl
luăm **în `process_event`, în jurul apelului care rezolvă tenant+conversație+pipeline**,
keyed pe această triplă (proxy stabil pentru conversație, disponibil ÎNAINTE de DB).
`business_id` real intră în cheie după `resolve_channel` (vezi mai jos), ca izolarea
multi-tenant să fie în cheie (principiul 7) — dar `channel_account_id` deja mapează 1:1 la
business, deci nu există coliziune cross-tenant nici înainte de rezolvare.

### Lock primitive (Redis SET NX EX + release prin Lua compare-del)
Adăugăm helperele în `src/redis_bus.py` (lângă `seen_before`, același pattern `SET NX EX`):
```python
# TTL-ul lock-ului: peste durata maximă realistă a unui tur (LLM + tool calls + DB),
# dar mic, ca un worker mort să nu țină conversația blocată mult. Sub block-ul XREADGROUP.
_LOCK_TTL_SECONDS = 30

# Release sigur: ștergem DOAR dacă tokenul e al nostru (alt worker putea prelua după expirare).
_RELEASE_LUA = """
if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end
"""

def _lock_key(business_id: str, sender_key: str) -> str:
    # business_id în cheie (P7); sender_key = channel_kind:account:sender (1:1 conversație).
    return f"convlock:{business_id}:{sender_key}"

async def acquire_conv_lock(redis: Redis, business_id: str, sender_key: str, token: str) -> bool:
    """SET NX EX atomic — True dacă am prins lock-ul (nimeni nu procesează conversația)."""
    return bool(await redis.set(_lock_key(business_id, sender_key), token, nx=True, ex=_LOCK_TTL_SECONDS))

async def release_conv_lock(redis: Redis, business_id: str, sender_key: str, token: str) -> None:
    """Release best-effort: del DOAR dacă tokenul e al nostru (compare-and-del, Lua atomic)."""
    try:
        await redis.eval(_RELEASE_LUA, 1, _lock_key(business_id, sender_key), token)
    except Exception:  # noqa: BLE001 — release eșuat → lock-ul expiră singur la TTL
        log.warning("convlock: release eșuat (expiră la TTL)")
```
`token` = `uuid4().hex` per încercare → un worker nu eliberează lock-ul reluat de altul
după expirare (cazul clasic de lock distribuit).

### Integrare în consumer (skip-and-requeue, NU blocare)
Lock-ul se ia în `process_event`, DUPĂ `resolve_channel` (avem `business_id`), în jurul
ramurii `message`/`callback`/`status`→pipeline. Cheia de sender o reconstruim din envelope
(aceleași câmpuri ca `Debouncer._key`). Dacă lock-ul e **ocupat**, NU blocăm consumer-ul
(ar serializa tot worker-ul) — facem **re-XADD pe `inbound` cu un mic delay logic** și
ACK-uim entry-ul curent (re-coadă), ca worker-ul să treacă la altă conversație:
```python
async def process_event(pool, redis, event):
    ...  # order/status routing rămân neschimbate (status NU se serializează — e idempotent)
    business_id = channel["business_id"]
    if event.get("kind", "message") in ("message", "callback"):
        sender_key = f"{channel_kind}:{channel_account_id}:{event.get('sender_external_id','')}"
        token = uuid4().hex
        if not await acquire_conv_lock(redis, business_id, sender_key, token):
            # ocupat → un alt worker procesează ACELAȘI sender; re-coadă (nu pierdem, nu blocăm).
            await _requeue_busy(redis, event)
            return
        try:
            await _route_locked(pool, redis, business_id, channel, event)  # load_business + pipeline
        finally:
            await release_conv_lock(redis, business_id, sender_key, token)
        return
    await _route_unlocked(...)  # status: fără lock (idempotent pe provider_msg_id)
```
`_requeue_busy` = `enqueue_inbound(redis, {**event, "_requeues": n+1})` cu cap
(`settings.conv_lock_max_requeues`, default 10) → după cap, log + drop (mesaj logat, nu
tăcut: principiul 6 e onorat de turul care DEȚINE lock-ul, nu de acesta). Anti-busy-spin:
înainte de re-XADD, un `await asyncio.sleep(settings.conv_lock_requeue_delay_ms/1000)`
mic (default 150ms) ca să nu ardem CPU re-cozând la infinit aceeași conversație ocupată.

### Interacțiunea cu Debouncer (R1) — lock-ul e DUPĂ debounce
Fluxul real: `consume_once` → `debouncer.add(event)` (coalesce 3s) → la flush, `_handle`
→ `process_event`. Deci lock-ul protejează **lotul debounce-uit**, exact granularitatea
corectă (un tur = un lot). Debouncer-ul e per-proces, deci NU dă serializare cross-worker
— de aceea lock-ul Redis e necesar pe lângă el, nu în loc.

### Câmpuri TurnContext scrise
**Niciunul.** Lock-ul trăiește integral în `consumer.py` + `redis_bus.py`, ÎNAINTE de
construirea `TurnContext` (în `handle_turn`). Contractul pipeline-ului rămâne neatins
(principiul 3: niciun câmp nou, niciun owner nou). Singura modificare de comportament:
`process_event` poate ieși devreme (re-coadă) fără a atinge DB.

### Error handling / degradare
- `redis is None` (config fără Redis în test) → lock sărit, comportament de azi (un consumer).
- `acquire` aruncă (Redis jos) → **fail-open**: procesăm fără lock (indisponibilitatea
  guard-ului nu blochează traficul — același principiu ca rate-limit/cost-guard din
  `limits.py`). Logăm `convlock_acquire_failed` și mergem mai departe.
- worker mort cu lock ținut → lock-ul expiră la `_LOCK_TTL_SECONDS` (30s) → conversația se
  deblochează singură. `XAUTOCLAIM` (entry-ul stuck) e task separat (vezi Out of Scope).

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** lock-ul e infrastructură de coadă (stagiul 2), NU un stagiu și
  NU un loop de orchestrare — pipeline-ul rămâne liniar, agnostic de lock.
- **P3 (un owner per câmp):** zero câmpuri TurnContext noi, zero owner nou; lock-ul e în
  afara contractului `TurnContext`.
- **P6 (niciodată tăcere):** re-coadă în loc de drop; fail-open la Redis jos; cap pe
  re-cozări cu log (nu drop tăcut). Răspunsul vine de la turul care deține lock-ul.
- **P7 (business_id pe tot):** cheia de lock include `business_id` (`convlock:{business_id}:
  {sender_key}`) → izolare multi-tenant și în spațiul de lock-uri.
- **P12 (PII):** `sender_external_id` (wa_id/chat.id) intră în cheia Redis (efemeră, TTL
  30s, nu e log), dar evenimentele de analytics logă DOAR `business_id` + `requeues`,
  NICIODATĂ identitatea expeditorului. Logurile worker-ului nu conțin `sender_external_id`.

## Test Cases
> Toate testele folosesc un **fake Redis** (dict + suport `set(nx=,ex=)` + `eval` pe
> Lua compare-del) și envelope-uri fabricate. ZERO Redis/DB/LLM real în CI. Pipeline-ul
> (`handle_turn`) e stub-uit cu un dublu care doar înregistrează ordinea apelurilor.

**Happy Path:**
1. Un singur worker, un sender, 2 mesaje în ordine → ambele procesate în ordine, lock
   luat+eliberat de două ori (curat).
2. Doi senderi diferiți, un worker, mesaje întrețesute → ambele conversații avansează,
   zero re-coadă (chei de lock distincte).

**Edge Cases:**
1. Doi workeri concurenți pe același sender: al doilea primește `acquire=False` → re-coadă
   cu `_requeues=1`; după ce primul eliberează, re-coada prinde lock-ul → ordine păstrată.
2. `redis=None` → `acquire`/`release` sărite, `process_event` rulează ca în mono-consumer.
3. Status/order events: NU iau lock (idempotente) → procesate imediat chiar cu lock de
   message activ pe același sender (test că ramura status ocolește lock-ul).

**Failure Cases:**
1. `acquire_conv_lock` aruncă (Redis jos) → fail-open: turul se procesează fără lock,
   `convlock_acquire_failed` logat, nicio excepție în sus.
2. `release_conv_lock` aruncă → prins, lock-ul expiră la TTL; turul s-a finalizat oricum.
3. Re-coadă atinge `conv_lock_max_requeues` → `conversation_lock_dropped` emis, fără buclă
   infinită (assert pe numărul de XADD-uri).

## Cost
**Zero cost LLM/embeddings** — lock-ul e pur Redis (cod determinist, stagiul 2). Nu atinge
niciun apel OpenAI. Cost operațional Redis: 2 comenzi extra per tur (`SET NX EX` + `EVAL`
release) ≈ neglijabil față de apelurile triaj/agent. La contention reală, re-cozările adaugă
XADD-uri (mărginite de `conv_lock_max_requeues`), de aceea capul + delay-ul de 150ms limitează
amplificarea. Per 1000 mesaje fără contention: ~2000 comenzi Redis O(1), sub-milisecundă
fiecare → impact de latență/cost imperceptibil.
