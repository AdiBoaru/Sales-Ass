# NX-22 — Web Widget W3: canal web în gates/sender
**Owner:** S · **Faza:** P2 (E26, V1.5) · **Zi/Ord:** W3 (după NX-20 gateway + NX-21 widget.js) · **Branch:** `feat/NX-22-web-channel-gates-sender` · **Complexitate:** M · **Estimare:** 4h

## Goal
Cablează `channel_kind='web'` ca al treilea canal de prim rang în pipeline, atingând
DOAR cele două margini (ingestie + trimitere) — exact ca Telegram (NX-60). Concret:
(1) sender-ul web (`ChannelSender`) scrie reply-ul în stream-ul SSE al sesiunii (prin
registry, nu printr-o cale paralelă); (2) typing-ul pe web e un eveniment SSE, nu un
call REST extern; (3) debounce-ul devine 800ms pentru web (chat live într-un tab, nu
WhatsApp pe telefon); (4) web NU are fereastră 24h (e o sesiune deschisă, nu o
conversație async). Pipeline-ul (stagiile 3-9) rămâne byte-identic — niciun `if
channel == "web"` în worker/runner. DoD: golden echo pe web verde.

## Business Context
Web widget-ul (E26, zona 1b din diagrama v4) e canalul care prinde vizitatorul ANONIM
direct pe site, înainte să aibă WhatsApp — exact unde se pierd cei mai mulți lead-uri.
Ca să-l adăugăm fără să riscăm calea WhatsApp deja stabilă, canalul web trebuie să intre
prin aceleași contracte ca Telegram: un `ChannelSender` + o intrare în registry, plus
2-3 reglaje de parametri (debounce, typing, 24h). Dacă apare cod specific de web în
pipeline, fundația multi-canal (NX-60) e compromisă și fiecare canal viitor (Instagram,
web) devine o ramură de `if`-uri.

## Technical Description

### Ce e deja agnostic (NU se atinge)
Verificat în cod: `gates.py` (stagiul 3) e deja channel-agnostic — NU conține niciun
check de fereastră 24h. Fereastra 24h trăiește exclusiv în PROACTIV (`in_24h_window(conv)`
= funcție SQL, schema_v2:154), care nu intră pe calea inbound. Deci „web fără 24h" NU e o
modificare în gates — e o regulă în sender-ul/marginea web (web nu poate fi destinatar de
proactiv-template) + în motorul proactiv (follow-up, vezi Out of Scope). Runner-ul,
context builder-ul, triajul și agentul operează pe `TurnContext` și nu știu de canal.

### 1. Sender web — scrie în stream-ul sesiunii (marginea de IEȘIRE)
`channel_kind='web'` capătă un `ChannelSender` (`src/channels/web/sender.py`) înregistrat
în `build_registry` (dispatcher.py:146) DOAR dacă web e activat. Spre deosebire de
Meta/Telegram (REST extern), sender-ul web publică reply-ul pe canalul Redis pub/sub al
sesiunii pe care îl ascultă gateway-ul SSE (NX-20). Contractul `ChannelSender` e respectat
exact (`send_text(account_id, to, text) -> provider_msg_id`):
```python
# src/channels/web/sender.py — implementează ChannelSender (NX-60)
class WebSender:
    """account_id = id-ul canalului web al tenantului; to = session_id (vizitatorul).
    `provider_msg_id` = uuid generat aici (web n-are id de la platformă).
    Publică pe Redis pub/sub al sesiunii; gateway-ul SSE relayează clientului."""
    def __init__(self, redis: Redis) -> None:
        self._redis = redis

    async def send_text(self, account_id: str, to: str, text: str) -> str:
        msg_id = str(uuid4())
        await self._publish(to, {"type": "message", "id": msg_id, "text": text})
        return msg_id

    async def send_products(self, account_id: str, to: str, text: str, products: list) -> str:
        # fallback W1 (ca pe canalele fără carusel nativ): listă compactă cu link-uri.
        msg_id = str(uuid4())
        await self._publish(to, {"type": "products", "id": msg_id, "text": text, "products": products})
        return msg_id

    async def mark_typing(self, account_id: str, to: str, provider_msg_id: str | None) -> None:
        # typing pe web = un eveniment SSE, NU un call REST extern.
        await self._publish(to, {"type": "typing"})

    async def _publish(self, session_id: str, event: dict) -> None:
        # canalul pub/sub e namespaced pe sesiune; payload-ul NU conține PII (doar text + id).
        await self._redis.publish(f"web:session:{session_id}", json.dumps(event))
```
Dispatcher-ul rămâne PROST: `dispatch_row` alege `registry.get("web")` exact ca pentru
WhatsApp/Telegram (dispatcher.py:63-67); `send_products`/`mark_typing` sunt detectate prin
`hasattr` (pattern existent). Reply-ul web trece PRIN outbox (principiul 5) — NU scurtăm
calea, doar transportul final e pub/sub în loc de HTTP.

### 2. Typing web = eveniment SSE (marginea de transport, NX-90)
NX-90 a stabilit `mark_typing` ca metodă OPȚIONALĂ pe `ChannelSender`, declanșată
fire-and-forget din `consumer.process_event` la `kind == "message"`. Pentru web,
`WebSender.mark_typing` publică `{"type":"typing"}` pe stream-ul sesiunii — gateway-ul SSE
îl trimite ca event `typing` la widget. Best-effort: un publish eșuat NU rupe turul
(prins de `_safe_typing` din NX-90). Zero call extern, zero cost.

### 3. Debounce 800ms pentru web (marginea de ingestie, stagiul 2)
`Debouncer` (debounce.py) are azi un singur `delay` global (3.0s). Web e chat live într-un
tab — 3s ar fi enervant. Facem `delay`-ul dependent de `channel_kind`, fără să atingem
mecanismul de coalescing:
```python
# src/worker/debounce.py — delay per canal, cheia include deja channel_kind
DEBOUNCE_SECONDS_DEFAULT = 3.0
DEBOUNCE_SECONDS: dict[str, float] = {"web": 0.8}  # din settings (vezi mai jos)

def _delay_for(self, channel_kind: str | None) -> float:
    return self._delays.get(channel_kind or "", self._default_delay)
```
`Debouncer.__init__` primește `delays: dict[str, float]` + `default_delay`; `_flush_later`
folosește `_delay_for(channel_kind)` în loc de `self._delay`. Cheia de buffer include deja
`channel_kind` (debounce.py:42-46), deci loturile web și WhatsApp nu se amestecă — singura
schimbare e ce `delay` aplicăm pe fiecare. Valorile vin din `settings.debounce_seconds`
(default 3.0) + `settings.debounce_seconds_web` (default 0.8), citite în
`consumer.run_consumer` la construcția `Debouncer`.

### 4. Web fără fereastră 24h
Două locuri, ambele la margine (NU în pipeline):
- **Sender web:** web nu trimite NICIODATĂ proactiv-template (n-are conceptul de „în afara
  ferestrei"). Sesiunea e deschisă SAU închisă — dacă e închisă, mesajul se pierde (sau se
  buffer-uiește pentru reconectare Last-Event-ID, treaba gateway-ului NX-20, nu a noastră).
- **Motor proactiv (follow-up):** la selecția destinatarilor, conversațiile cu
  `channels.kind='web'` se exclud din `awb_update`/`back_in_stock`/`abandoned_cart` —
  proactivul țintește canale async (WhatsApp). Vezi Out of Scope (motorul proactiv nu e
  încă scris; aici doar documentăm regula, fără cod).

### Identitate & PII
Sender-ul web operează pe `session_id` (id efemer de sesiune semnat, NX-20), NU pe telefon.
PII-ul de canal (visitor_id) trăiește în `channel_identities` (`channel_kind='web'`,
`external_id=visitor_id`) — identitatea + merge-ul cu WhatsApp e NX-23, în afara acestui
task. Payload-ul publicat pe Redis pub/sub conține DOAR `{type, id, text/products}` —
fără telefon, fără visitor_id (P12). Logurile sender-ului web logă `channel_kind` +
lungimea textului, NICIODATĂ corpul sau `session_id`.

### Câmpuri TurnContext scrise aici
**NICIUNUL.** Acest task atinge exclusiv marginile (sender web + debounce), NU pipeline-ul.
`TurnContext` curge neschimbat: stagiile scriu aceleași câmpuri ca pe WhatsApp/Telegram
(`ctx.reply` de către stagii, `ctx.route` de triaj etc.). Sender-ul web citește
`ctx.reply.text`/`ctx.reply.products` (deja setate) și produce rânduri de outbox prin
calea existentă — owner-ul rămâne Sender-ul (processor), exact ca azi.

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar, cuplaj DOAR la margini):** tot ce face acest task stă la cele
  două margini (ingestie/debounce + ieșire/sender). Zero `if channel == "web"` în
  worker/runner/stagii. Dacă un check de web ajunge în pipeline, designul e greșit.
- **P5 (un singur punct de ieșire):** reply-ul web tot prin Sender → outbox → dispatcher →
  `WebSender`. Pub/sub e transportul FINAL, nu o cale paralelă. Singura excepție justificată
  (moștenită din NX-90) = typing-ul, semnal efemer trimis direct, nu prin outbox.
- **P6 (niciodată tăcere / degradare grațioasă):** publish eșuat la typing = prins,
  ignorat (turul răspunde oricum). Sesiune închisă la send → mesajul se buffer-uiește
  (gateway) sau se marchează `failed` în outbox (vizibil), nu pierdut tăcut.
- **P7 (business_id pe tot):** `WebSender` operează pe `account_id` (canalul web al
  tenantului) rezolvat deja pe `business_id` în `claim_due`; canalul pub/sub e namespaced
  pe `session_id`, izolat per sesiune. Nicio scriere DB nouă în acest task.
- **P12 (PII într-un loc):** payload-ul pub/sub și logurile NU conțin telefon/visitor_id/
  session_id sau corpul mesajului — doar `channel_kind` + metrici de lungime.

## Test Cases
Teste fără rețea: `WebSender` primește un fake Redis (capturează `publish`); golden-ul
rulează pe pipeline-ul real cu LLM scriptat (`ScriptedLLM`) + stub-uri DB, ca `test_golden.py`.
ZERO OpenAI / Redis real / SSE real în CI.

**Happy Path:**
1. `WebSender.send_text("web-ch-1", "sess-abc", "Salut!")` → un singur `publish` pe
   `web:session:sess-abc` cu `{"type":"message","id":<uuid>,"text":"Salut!"}`; return = `id`.
2. Golden echo, `channel_kind="web"`: mesaj inbound web → pipeline → `ctx.reply` → outbox →
   `dispatch_row` alege `WebSender` → payload de tip `message` publicat o singură dată.

**Edge Cases:**
1. Reply cu `products` (carusel) pe web → `dispatch_row` cheamă `send_products` (hasattr
   True) → payload `type=products` cu lista de produse, nu `send_text`.
2. Debounce: două mesaje web la 0.5s distanță → coalescite într-un singur tur (sub 0.8s);
   aceleași două pe WhatsApp la 0.5s → tot coalescite (sub 3.0s) — cheia separă canalele,
   delay-ul diferă, lotul e corect per canal.
3. `web_enabled=False` → `WebSender` neînregistrat; un rând outbox web → `dead` cu log,
   restul canalelor neafectate.

**Failure Cases:**
1. `redis.publish` aruncă în `send_text` → excepția propagă la `dispatch_row`, care o prinde
   → `mark_failed` (retry/backoff sau `dead`), turul NU crapă, nimic pierdut tăcut (P6).
2. `WebSender.mark_typing` aruncă (Redis jos) → prins de `_safe_typing` (NX-90),
   turul răspunde normal, warning logat fără PII.
3. Event web fără `channel_account_id` la `resolve_channel` → canal necunoscut, ignorat cu
   log (cale existentă consumer.py:71-77), fără crash.

## Cost
**$0 LLM.** Taskul e pur transport + reglaj de parametri — niciun apel de model/embedding
introdus. Per 1000 mesaje web: 0 tokeni adăugați față de WhatsApp/Telegram (același pipeline,
aceleași 2 puncte LLM — triaj nano + agent mini). Transportul web (Redis pub/sub local) e mai
ieftin decât REST extern (zero call HTTP la platformă). Debounce-ul de 800ms reduce ușor
numărul de tururi LLM pe burst-uri de tastare rapidă (mai multe mesaje → un singur tur),
deci dacă ceva, scade costul marginal față de a procesa fiecare keystroke separat.
