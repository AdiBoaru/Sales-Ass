# NX-211 — AnswerPlan + validator determinist extins + critic semantic selectiv

**Owner:** Claude (build) / Codex (verify) · **Faza:** 8 (Quality Overhaul) · **Status:** DRAFT — nu se implementează până la review dependențe + DoD · **Complexitate:** L
**Parent plan:** [docs/QUALITY-OVERHAUL-2026.md](../docs/QUALITY-OVERHAUL-2026.md) · **Depinde de:** NX-210 (GO la gate)
**Decizii ADR aplicabile:** D7, D8, D9
**Delimitare față de NX-210:** acolo = `AnswerPlanV0` experimental/offline; AICI = **contractul
de PRODUCȚIE** (schema stabilă, validator complet, critic, property tests, kill-switch).

## Goal
Răspunsul grounded: planul cu dovezi ÎNAINTEA textului, validare pe două niveluri
(determinist pentru fapte structurale, semantic pentru afirmații) — D8: evidence ID e necesar,
nu suficient.

## Scope
1. **AnswerPlan** (output structurat al agentului, înaintea randării): `selected_products`,
   `claims[{type: fact|recommendation, text, evidence_ids, need_ids?}]`, `facts{price,stock}`,
   `uncertainties`, `unmet_constraints`.
2. **Validator determinist** (extinde validatorul existent): preț · stoc · URL · produs/variantă ·
   business_id · claims factuale vs evidence · evidence existent și actual · hard constraints ·
   acțiuni confirmate doar după success · PII. Fluxul: AnswerPlan → validare → draft → critic →
   revizie MAX o dată → validare finală.
3. **Critic semantic SELECTIV** (declanșat de condiții de COD: recomandare/comparație · claims
   medicale-juridice · evidence coverage sub prag · validator picat la prima trecere): evidence
   chiar susține claim-ul? · recomandarea răspunde nevoii? · ton · nu exagerează · nu lipsește
   o limitare importantă. Pe tier max-quality poate rula pe fiecare răspuns de vânzare.

## Gate (ținte)
Preț/link/stoc inventate = 0 · acțiuni fals-raportate = 0 · claims factuale grounded ≥99% ·
hard constraint violations = 0.

## Out of Scope
Needs profile / ancorare pe nevoi (NX-212 — AnswerPlan are `need_ids` pregătit).

## Definition of Done
- [ ] AnswerPlan (contract de PRODUCȚIE, versionat) emis pe fiecare tur de vânzare al agentului nou.
- [ ] Validator extins cu teste property-based (hypothesis, pinned în requirements-dev) pe
      garanțiile dure: niciun preț/URL în afara evidence; niciun URL/produs din alt tenant;
      fuzzy ambiguu nu se rezolvă automat; UNKNOWN nu devine MISMATCH.
- [ ] Critic selectiv cu declanșatoare de COD documentate + kill-switch (OFF → comportament vechi).
- [ ] Țintele gate atinse pe golden set + halu-suite.
- [ ] Invariante: `business_id` server-owned pe toate query-urile + test adversarial cross-tenant;
      zero PII în AnswerPlan persistat/telemetrie; **P6 — nicio cale nu produce tăcere** (validator
      picat de 2 ori → fallback formulat, nu tăcere); fallback funcțional cu criticul indisponibil;
      test e2e (nu doar unit); `ruff` + `ruff format --check` + `pytest -x -q` verzi.
