## Goal
Activează enforcement-ul de selecție — UN singur moment, cu owner unic acest card:
1. **Match Gate enforce:** `exact/alternatives/rejected` schimbă efectiv ce se afișează, cu disclosure onest.
2. **QuerySpec enforce:** QuerySpec e promovat din shadow pentru fațetele eligibile — `SearchArgs` devine proiecția OBLIGATORIE; argumente de tool care slăbesc un hard constraint NU se execută.

Doar pe fațete cu coverage suficient (NX-186) + SQL tri-state/recall verde (NX-189). Pilot 1 business, canary gradual.

## Politica de afișare (închisă — aliniată la MatchSet disjunct din NX-187)
| Situație | Comportament |
|---|---|
| toate hard MATCH | `exact` — poate fi recomandare |
| hard MISMATCH | `rejected` — nu se afișează pe NICIO cale |
| hard UNKNOWN non-safety | `alternatives` — NU se numește „exact"; etichetat explicit („nu pot confirma X") |
| **soft mismatch** | **rămâne în mulțimea determinată de hard constraints** (ex. all-hard-MATCH → `exact`); **scor penalizat + tradeoff explicat în microcopy** — NU mută apartenența |
| safety UNKNOWN/MISMATCH | **fail-closed** (SafetyPolicy decide) |
| zero exact | mesaj onest + un singur pas următor, niciodată tăcere |

Relaxarea nu mai e tăcită; doar soft se relaxează automat.

## QuerySpec enforcement (promovare din shadow — owner: acest card)
- QuerySpec (NX-185) devine sursa OBLIGATORIE pentru fațetele eligibile: `SearchArgs` = proiecția lui, nu a doua sursă de adevăr.
- Tool-loop-ul poate formula query-ul semantic, dar **args care slăbesc un hard constraint nu se execută** — proiecția corectează la valoarea din QuerySpec + emite `hard_constraint_blocked`.
- **Test obligatoriu:** tool-loop cere `price_max=150`, QuerySpec are `price_max=80` → SQL primește **80**.
- **Kill-switch OFF → comportament VECHI complet** (inclusiv proiecția: revine la args-urile modelului, ca azi).
- Nu există NX-185b — enforcement-ul QuerySpec trăiește AICI, aliniat cu enforce-ul per fațetă.

## Technical Description
- `render`/planner: consumă `MatchSet` → `exact` ca recomandări, `alternatives` etichetate, `rejected` exclus dur pe orice cale (inclusiv fallback tehnic V2).
- Mod `no_exact` (V2/NX-183): explică ce combinație lipsește + cere O relaxare (buget/atribut/brand/categorie) → alegerea se scrie în QuerySpec → re-search.
- Enforcement gate-uit **per fațetă**: (a) coverage peste prag (NX-186), (b) fațeta participă în retrieval cu paritate tri-state verde (NX-189), (c) recall acceptabil (NX-187). Fațetă sub oricare → informativă/soft (disclosure), nu blochează.
- Safety fail-closed pe toate căile.

## Câmpuri TurnContext scrise
`ctx.retrieval`/`ctx.reply` reflectă MatchSet-ul aplicat (owner: planner).

## Implementation Steps
1. Aplică MatchSet în planner ÎNAINTE de a seta `ctx.retrieval` (toate căile downstream consumă setul gated).
2. Promovare QuerySpec: proiecție obligatorie SearchArgs + blocarea slăbirii hard + telemetrie.
3. Alternatives UX + disclosure per-locale; `no_exact` prin V2.
4. Gate per-fațetă (coverage ∧ SQL-paritate ∧ recall); safety fail-closed.
5. Teste dure: `rejected` pe nicio cale; price 150→80; soft mismatch nu mută apartenența.
6. Canary rollout + evaluator.
