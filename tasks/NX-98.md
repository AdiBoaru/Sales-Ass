# NX-98 — search_products funcțional pe tenant live — fallback SQL-only + backfill product_url
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7 (catalog tools) · **Branch:** `feat/NX-98-search-fallback-url` · **Complexitate:** M · **Estimare:** 4h

## Goal
`search_products` trebuie să întoarcă produse REALE pe tenantul live, chiar și când
`product_embeddings` lipsește. Azi tool-ul cheamă DOAR calea semantică
(`search_products_semantic`), care face `JOIN product_embeddings` — cu 0 embeddings
pe demo, întoarce zero rânduri, iar calea SQL-only (`search_products`, deja scrisă,
catalog.py:53-99) nu e chemată niciodată. În plus `products.product_url` e NULL pe
demo → validatorul de link (`_links_ok`, agent.py:95-105) acceptă doar `url`-uri din
retrieval, deci agentul nu poate emite niciun link. Taskul: fallback determinist la
search SQL-only când embeddings lipsesc + asigură rularea `embed_products` + backfill
`product_url`. Parte cod, parte date/ops.

## Business Context
Fără asta, agentul (mini) e orb pe demo: orice `search_products` → „Niciun produs
găsit", deci nici recomandare, nici link de cumpărare → bucla de bani (checkout_link →
conversie) e ruptă din capăt. E gaura care face botul să pară gol deși sunt 500 de
produse seedate. Reparat, demo-ul devine prezentabil pentru primii clienți.

## Technical Description

### Starea curentă (problema, exact)
1. **`search_products_tool` (catalog_tools.py:81-98)**: cheamă NUMAI
   `search_products_semantic(...)`. Dacă întoarce gol din lipsă de buget (`price_max`),
   reia fără filtru — dar tot semantic. Nu există nicio cale care să cadă pe SQL-only.
2. **`search_products_semantic` (catalog.py:210-217)**: SQL-ul concatenează
   `" join product_embeddings pe on pe.product_id = p.id"` — INNER JOIN. Cu
   `product_embeddings` gol → 0 rânduri, indiferent de filtre.
3. **`_links_ok` (agent.py:95-105)**: `allowed = {p.get("url") for p in products if p.get("url")}`.
   Cu `products.product_url` NULL, `url` e None → mulțimea de linkuri permise e goală →
   orice link în reply e respins → retry → fallback determinist fără link.

### Țintă — fallback determinist (cod)
Tool-ul detectează lipsa de embeddings ȘI/SAU rezultatul gol și cade pe `search_products`
(SQL-only). NU mai adăugăm LLM nicăieri — embedding-ul de query rămâne unicul apel
(prin adaptorul `src.agent.llm`, deja stabilit ca excepție de tip „clasificator", P2).

`src/db/queries/catalog.py` — helper ieftin de detecție (un singur SELECT, scoped):
```python
async def has_embeddings(conn, business_id: str) -> bool:
    """True dacă tenantul are măcar un product_embedding (decide semantic vs SQL-only)."""
    row = await conn.fetchrow(
        "select 1 from product_embeddings where business_id = $1 limit 1",
        business_id,
    )
    return row is not None
```

`src/tools/catalog_tools.py` — rescrie corpul lui `search_products_tool`:
```python
a = SearchArgs(**args)
# 1. decizie de cale: semantic doar dacă avem embeddings + LLM pt query vector
use_semantic = deps.llm is not None and await has_embeddings(deps.conn, ctx.business.id)
products: list[dict[str, Any]] = []
if use_semantic:
    query_vec = (await deps.llm.embed([a.query]))[0]
    products = await search_products_semantic(
        deps.conn, ctx.business.id, query_vec, price_max=a.price_max, limit=a.limit
    )
    if not products and a.price_max is not None:
        products = await search_products_semantic(
            deps.conn, ctx.business.id, query_vec, limit=a.limit
        )
# 2. fallback SQL-only: fie n-avem embeddings/LLM, fie semantic n-a întors nimic
if not products:
    products = await search_products(
        deps.conn, ctx.business.id, query_text=a.query, price_max=a.price_max, limit=a.limit
    )
    if not products and a.price_max is not None:
        products = await search_products(
            deps.conn, ctx.business.id, query_text=a.query, limit=a.limit
        )
ctx.emit("product_search", mode=("semantic" if use_semantic and products else "sql_only"),
         count=len(products), had_price_filter=a.price_max is not None)  # FĂRĂ query (P12 — poate conține PII)
return ToolResult(ok=True, products=products, llm_view=_brief(products))
```
Note:
- `search_products` (SQL-only) folosește `query_text ilike '%...%'` pe `p.name` — matching
  slab, dar ZERO halucinație și acoperă cazul fără embeddings. Calea „bună" rămâne
  semantică; SQL-only e plasa, nu înlocuitorul.
- Detecția `has_embeddings` se cache-uiește implicit per tur (un singur search/tur tipic);
  nu merită memoizat global (embeddings apar după job, nu în timpul turului).
- `ctx.emit("product_search", mode=...)` distinge clar în analytics când rulăm degradat
  (`sql_only`) → semnal că jobul de embed trebuie rulat pe acel tenant.

### Țintă — `product_url` (date/ops, NU cod de pipeline)
Validatorul e CORECT: linkurile trebuie să fie din catalog (P8 din arhitectură — zero
linkuri inventate). Gaura e de DATE: `products.product_url` e NULL pe demo. Două căi,
ambele în acest task:
1. **Backfill demo** prin scriptul de seed/sync (admin, NU worker — catalog read-only
   pentru bot). Sursa reală de URL = câmpul din feed-ul magazinului; pe demo nu există,
   deci derivăm determinist din `slug`:
   ```sql
   -- script admin (admin_conn), per tenant, scoped pe business_id:
   update products
      set product_url = $2 || '/p/' || slug          -- $2 = base_url al tenantului
    where business_id = $1
      and product_url is null
      and slug is not null;
   ```
   Base-url-ul tenantului vine din `businesses.settings->>'store_base_url'` (cheie jsonb
   nouă, fără migrare). Dacă lipsește → scriptul NU inventează (lasă NULL, raportează N).
2. **Fără backdoor în validator**: NU relaxăm `_links_ok` să accepte linkuri non-catalog.
   Dacă `url` e NULL, agentul pur și simplu nu pune link (deja se întâmplă) — corect.
   După backfill, linkurile devin valide natural.

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** fallback-ul e o ramură deterministă ÎN tool, fără loop de
  orchestrare; o singură trecere semantic→SQL-only.
- **P2 (LLM doar triaj+agent):** singurul apel extern e `embed([query])` prin adaptorul
  unic — embedding e clasificator/retrieval, nu generare; calea SQL-only n-are LLM deloc.
- **P6 (niciodată tăcere):** degradare embeddings-absent → SQL-only → tot iese ceva
  (produse reale), nu „Niciun produs găsit" structural.
- **P7 (business_id pe tot):** `has_embeddings`, `search_products`, backfill — toate cu
  `where business_id = $1`. RLS (`bot_runtime` + `app.business_id`) e plasa.
- **P8 (state = ref-uri):** neschimbat — produsele rămân câmpuri compacte (≤6×8).
- **P12 (PII în loguri):** `product_search` NU logă `query` (poate conține nume/adrese);
  doar `mode`/`count`/`had_price_filter`.

## Test Cases
**Happy Path:**
1. `has_embeddings`→True, `search_products_semantic` mock → 3 produse → tool întoarce
   cele 3, `mode=semantic`, `search_products` (SQL) neatins.
2. `has_embeddings`→False → tool cheamă direct `search_products` (SQL) → produse,
   `mode=sql_only`, `embed` neatins.

**Edge Cases:**
1. `has_embeddings`→True dar semantic gol cu `price_max` → reia semantic fără preț; tot
   gol → cade pe SQL-only (lanțul complet de degradare).
2. SQL-only întoarce și el gol (query fără match) → `ToolResult(ok=True, products=[])`,
   `_brief` = „Niciun produs găsit." (P6: tot un răspuns, nu excepție).
3. Backfill cu `store_base_url` absent în `businesses.settings` → 0 rânduri actualizate,
   scriptul raportează N NULL rămase, NU inventează URL.

**Failure Cases:**
1. `deps.llm.embed` aruncă (API/rețea) DEȘI `has_embeddings`→True → prinsă, cade pe
   SQL-only (nu propagă excepția, nu tace — P6).
2. `deps.conn` aruncă în `has_embeddings` → eroarea se propagă ca eroare de tool normală
   (runner-ul o prinde), NU listă goală mascată.
3. Backfill rulat de două ori → idempotent (`where product_url is null`) → a doua rulare
   actualizează 0 rânduri.

> Teste fără apeluri reale: `search_products_semantic`/`search_products` ca stub-uri pe
> `deps.conn`; `deps.llm.embed` mock (vector fabricat) sau `deps.llm=None`. ZERO OpenAI/DB
> real în CI (pattern G8-1 / T140).

## Cost
Cod: **niciun apel LLM nou**. Singurul apel rămâne `embed([query])` pe calea semantică —
1 embedding (`text-embedding-3-small`, ~$0.00002/1K tokens, query ~20 tokens) per
`search_products`. La ~1 search/conversație → ~$0.0004 / 1000 mesaje. Pe calea SQL-only
(embeddings lipsă) costul e **$0** (nu se cheamă embed). Ce limitează costul: un singur
embedding per apel de tool (nu per produs), max 3 tool calls/tur (limită dură), și
fallback-ul SQL-only care NU adaugă apeluri. Jobul `embed_products` e one-shot/incremental
(re-embed doar la `content_hash` schimbat): 500 produse × ~50 tokens ≈ 25K tokens ≈
$0.0005 total pe demo, plătit o dată, nu per mesaj.
