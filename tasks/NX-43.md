# NX-43 — Whitelist chei contacts.profile (per vertical, data-driven)

**Owner:** S · **Faza:** P1-O2 · **Zi/Ord:** după NX-88 (extractor profil) · **Branch:** `feat/NX-43-profile-key-whitelist` · **Complexitate:** S · **Estimare:** 4h

## Goal
Extractorul de profil (NX-88) primește de la nano un `profile_patch` cu chei
candidate. NX-88 are doar o **whitelist minimă din config** — suficientă ca
mecanism, săracă în acoperire. Acest task înlocuiește lista plată cu o
**whitelist per vertical, citită din date** (`businesses.settings.profile_whitelist`,
cu fallback pe taxonomia din config per vertical), astfel încât modelul să NU
poată scrie chei inventate în `contacts.profile`, iar setul de chei valide
(`skin_type`, `budget_band` la beauty; `vehicle_model`, `engine_type` la auto)
să fie bogat și extensibil fără deploy de cod. Cheile necunoscute sunt aruncate
+ se emite `profile_key_dropped` (semnal pentru extinderea listei).

## Business Context
Profilul auto-învățat e ce face următoarea recomandare mai bună fără să
reîntrebe clientul (skin_type → recomandare cremă potrivită; vehicle_model →
piesa corectă). Dacă modelul scrie chei aleatorii (`fav_color`, `mood`),
profilul se umflă cu zgomot pe care niciun consumator nu-l citește, iar
exportul CRM (NX-31) devine necurat. O whitelist **bogată dar închisă**, per
vertical, ține profilul curat și acționabil — direct calitatea datelor pe care
clientul plătește. Bogăția per vertical e ce diferențiază produsul livrat unui
salon de cel livrat unui service auto, fără branch de cod per client.

## Technical Description

### Unde se aplică (în extractorul NX-88, nu un stagiu nou)
Filtrul trăiește în `src/worker/profile.py` (creat de NX-88), funcția
`filter_profile_patch(...)`, apelată de hook-ul post-tur `_extract_profile_and_score`
ÎNAINTE de `update_contact_profile_and_score`. NX-43 nu adaugă un stagiu de
pipeline și **nu scrie niciun câmp pe `TurnContext`** — rulează post-tur, după
ce reply-ul e deja în outbox (ca tot NX-88). Schimbarea e doar în **sursa
whitelist-ului** (config plat minim → set per vertical, data-driven) + bogăția lui.

### Sursa whitelist-ului — data-driven, NU hardcodat în prompt/cod de business
Schema_v2 **nu are** tabel `taxonomy` (vezi `schema_reference.md`: se adaugă
aditiv DOAR când verticalul cere filtre pe concerns — nu e cazul aici). Deci
„taxonomia" de chei de profil trăiește în două straturi, în ordinea priorității:

1. **Per business** — `businesses.settings.profile_whitelist` (jsonb: listă de
   chei) — citit deja în `BusinessConfig.settings` (zero query nou). Override
   per client (un client beauty care vrea și `hair_type`).
2. **Per vertical** — `PROFILE_KEY_TAXONOMY` în `src/agent/profile_taxonomy.py`
   (nou): un dict `vertical -> {key: spec}` versionat, importat ca dată, NU
   hardcodat în prompt și NU citit din `os.environ`. E „taxonomie în cod-dată",
   nu „chei hardcodate în prompt" — principiul 9 vizează promptul agentului
   generat din `categories`; whitelist-ul de profil e o constrângere de
   validare, separată de prompt.

### Funcția de filtrare (înlocuiește whitelist-ul plat din NX-88)
```python
# src/worker/profile.py (modificat — NX-88 a pus aici whitelist-ul minim)
def resolve_whitelist(business: BusinessConfig) -> dict[str, ProfileKeySpec]:
    """Per business (settings) UNION per vertical (taxonomie). Override pe cheie."""
    base = PROFILE_KEY_TAXONOMY.get(business.vertical, PROFILE_KEY_TAXONOMY["other"])
    extra = business.settings.get("profile_whitelist") or []
    # cheile din settings care nu au spec → tratate ca tip "str" liber (override permisiv intenționat)
    merged = dict(base)
    for k in extra:
        merged.setdefault(k, ProfileKeySpec("str"))
    return merged

def filter_profile_patch(
    patch: dict[str, Any], business: BusinessConfig
) -> tuple[dict[str, Any], list[str]]:
    """Întoarce (patch_curat, chei_aruncate). Cheie necunoscută SAU valoare ce nu
    respectă spec-ul (enum/tip) → aruncată. Modelul nu poate scrie chei arbitrare."""
    wl = resolve_whitelist(business)
    clean: dict[str, Any] = {}
    dropped: list[str] = []
    for key, value in (patch or {}).items():
        spec = wl.get(key)
        if spec is None or not _value_ok(spec, value):
            dropped.append(key)
            continue
        clean[key] = value
    return clean, dropped
```
`_value_ok(spec, value)`: enum → `value in spec.enum`; tip → izoElement
(`isinstance` cu coerciție prudentă pentru int/float din JSON). O valoare în
afara enum-ului e tot un „drop" (cheie corectă, valoare inventată = tot zgomot).

### Integrare cu emiterea de eveniment (NX-88 deja emite)
NX-88 emite `profile_key_dropped {key}` și `profile_updated {keys_set, dropped}`.
NX-43 nu schimbă contractul de evenimente — doar îmbogățește setul valid, deci
**mai puține drop-uri legitime** și drop-uri mai semnificative (chei chiar
inventate). Evenimentele rămân FĂRĂ valori (P12).

### Câmpuri TurnContext scrise aici
**Niciunul.** Rulează post-tur (NX-88), nu atinge `TurnContext`. Owner pe
`contacts.profile` la runtime rămâne hook-ul NX-88; NX-43 doar restrânge ce
ajunge în el.

## Test Cases
**Happy Path:**
1. `business.vertical="beauty_salon"`, patch `{skin_type:"gras", budget_band:"mid"}`
   → ambele trec, `dropped=[]`.
2. `business.vertical="auto_service"`, patch `{vehicle_model:"Logan", vehicle_year:2018}`
   → ambele trec (str + int coerciat din JSON).

**Edge Cases:**
1. Cheie comună (`budget_band`) validă la ambele verticale, dar `skin_type` (beauty)
   trimisă la `business.vertical="auto_service"` → `skin_type` aruncată,
   `budget_band` păstrată.
2. Override: `settings.profile_whitelist=["hair_type"]` la un beauty → `hair_type`
   acceptată chiar dacă nu e în taxonomia de bază (test).
3. `business.vertical` necunoscut/gol → fallback pe `PROFILE_KEY_TAXONOMY["other"]`
   (doar `budget_band`), restul aruncat (fail-closed pe chei, nu pe vertical).
4. Patch gol `{}` → `({}, [])`, niciun drop, niciun eveniment de drop.

**Failure Cases:**
1. `skin_type:"verde"` (în afara enum-ului) → aruncată ca drop (cheie bună,
   valoare inventată = zgomot).
2. `vehicle_year:"acum doi ani"` (text unde spec cere int) → aruncată
   (coerciția eșuează, nu scriem gunoi).
3. `profile_whitelist` în settings cu tip greșit (string în loc de listă) →
   tratat ca listă goală, fallback pe taxonomia de vertical, fără excepție.

> Teste fără apeluri reale: `filter_profile_patch` / `resolve_whitelist` sunt
> **cod pur** — primesc `BusinessConfig` fabricat + dict de patch. ZERO OpenAI,
> ZERO DB în CI (filtrul nu atinge nici LLM, nici conexiunea).
