# NX-77 — Clarificare deterministă din state.pending_question
**Owner:** S · **Faza:** P1-O1 · **Zi/Ord:** după G5b (cache) + G5c (limbă), în stratul 4 · **Branch:** `feat/NX-77-clarify-pending` · **Complexitate:** S · **Estimare:** 3h

## Goal
Stratul gratuit de clarificare (stagiul 4, fără LLM) lipsește: `state.pending_question`
e declarat (`models.py:127`) și hidratat (`:147`) dar NICIODATĂ citit de vreun stagiu.
Când botul a pus la turul precedent o întrebare de clarificare (ex. „pentru ce tip de
ten?"), iar clientul răspunde acum scurt („ten gras"), turul curent trebuie să CONSUME
deterministc răspunsul — pliază-l peste întrebarea în așteptare, curăță `pending_question`,
emite contextul îmbogățit spre triaj/agent — fără un apel LLM separat. Azi acel răspuns
re-intră orb în triaj/clarify (nano), deci plătim un tur LLM pentru ceva ce e pur de stare.

## Business Context
Conversațiile de vânzare sunt o serie de „te întreb / îmi răspunzi" (buget? tip de ten?
păr/față?). Fiecare răspuns scurt al clientului care, fără context, ar fi clasificat
„clarify" de nano = un apel nano irosit + un tur în plus pentru client. Pliind răspunsul
peste întrebarea știută determinist scurtăm bucla (mai puține tururi → conversie mai bună)
și mutăm trafic din stratul plătit (triaj nano) în stratul gratuit (țintă 40-60%, CLAUDE.md
stagiul 4). Zero cost LLM adăugat.

## Technical Description

### Forma lui `pending_question` (contract de stare)
`ConversationState.pending_question: dict | None`. Un singur owner la SCRIERE și un singur
owner la CITIRE/CURĂȚARE (vezi mai jos). Forma minimă, ref-uri nu obiecte (principiul 8):
```python
pending_question = {
    "field": "skin_type",          # ce constrângere completăm (cheie în state.constraints)
    "asked_turn": "<turn_id>",      # turul care a pus întrebarea (anti-stale)
    "kind": "constraint",           # constraint | choice (extensibil; v1 doar constraint)
    "options": ["uscat", "gras", "mixt", "sensibil"],  # opțional: valori așteptate (choice)
}
```
NU stocăm textul întrebării (ref, nu obiect) — textul a fost deja trimis clientului.

### Cine SCRIE `pending_question` (owner unic: agentul, stagiul 7)
Singurul loc care pune o întrebare „țintită" e agentul când îi lipsește o constrângere
înainte să caute (ex. nu poate filtra fără tip de ten). NX-77 adaugă DOAR mecanismul de
scriere via un câmp nou pe `ctx` (`ctx.set_pending_question(...)`), pe care procesorul îl
persistă în state în aceeași TX (la fel ca `displayed_products`). Popularea efectivă din
promptul agentului (când să întrebe) e un follow-up minimal marcat în Out of Scope — NX-77
livrează contractul + scrierea + CITIREA. Triajul NU scrie `pending_question` (clarify-ul
lui rămâne LLM, fără stare persistată).

### Cine CITEȘTE și CURĂȚĂ (owner unic: `clarify_stage`, stratul 4)
Stagiu NOU determinist `clarify_stage`, plasat în pipeline DUPĂ `cache_stage` și ÎNAINTE de
`triage_stage` (e tot strat gratuit, dar are nevoie de `ctx.language` din `language_stage`
și de un eventual cache hit înainte). Logica:
```python
async def clarify_stage(ctx, deps) -> None:
    pq = ctx.state.pending_question
    if not pq or not pq.get("field"):
        return                                   # nimic în așteptare → no-op
    body = (ctx.message.body or "").strip()
    if not body:
        return
    # 1. anti-stale: dacă răspunsul vine prea târziu (alt subiect între timp), abandonează
    #    întrebarea ca să nu pliem greșit. Heuristic determinist, fără LLM:
    if _looks_like_new_topic(body):              # ex. începe o cerere nouă / detect_risk
        ctx.consume_pending_question()           # doar curăță, lasă pipeline-ul mai departe
        ctx.emit("clarify_abandoned", field=pq["field"])
        return
    # 2. mapează răspunsul scurt la constrângere (cod pur, normalizat fără diacritice)
    value = _extract_answer(body, pq)            # normalizează + match pe options dacă kind=choice
    if value is None:
        return                                   # nu pare răspunsul → lasă triaj/agent (clarify LLM)
    # 3. consumă determinist: pliază în constraints, curăță pending_question
    ctx.merge_constraint(pq["field"], value)     # state.constraints[field] = value
    ctx.consume_pending_question()
    ctx.emit("clarify_resolved", field=pq["field"])
    # 4. FĂRĂ reply: am îmbogățit doar starea/contextul. Lăsăm pipeline-ul să continue spre
    #    agent cu constrângerea completă (search_products va filtra corect). Niciun apel LLM aici.
```
`_extract_answer` e cod determinist:
- `kind="choice"`: normalizează `body` (lowercase, fără diacritice, ca în gates `_norm`) și
  caută o `option` conținută în text → întoarce opțiunea canonică, altfel None.
- `kind="constraint"` generic: buget („sub 100", „100 lei") → refolosește `_budget`-ul din
  agent (extras într-un util comun); altfel răspuns scurt liber (≤4 cuvinte) → valoare brută.
`_looks_like_new_topic` = euristic conservator (mesaj lung > N cuvinte care nu conține nicio
opțiune, SAU `detect_risk` non-None) → preferăm să abandonăm întrebarea decât să pliem greșit.

**Precision-first:** dacă nu suntem siguri că `body` e răspunsul (None din `_extract_answer`),
NU consumăm și NU setăm reply — lăsăm triajul nano să decidă (clarify plătit ca azi). NX-77
e o scurtătură, nu o poartă dură; un eșec degradează la comportamentul actual (principiul 6).

### Persistarea (un singur scriitor de state: procesorul/Sender)
`clarify_stage` NU scrie în DB — mutează doar `ctx.state` (in-memory) + setează un flag
`ctx.state_dirty=True`. Procesorul (singurul writer de `conversations.state`, processor.py:369)
construiește `new_state` din `conv["state"]` plus mutațiile semnalate pe `ctx`:
```python
new_state = dict(conv["state"] or {})
if ctx.reply and ctx.reply.products:
    new_state["displayed_products"] = ctx.reply.products
if ctx.state_dirty:                              # NX-77: clarify a modificat constraints/pending_question
    new_state["constraints"] = ctx.state.constraints
    new_state["pending_question"] = ctx.state.pending_question   # None după consum → cheie ștearsă logic
```
Scris cu `patch_conversation_state` (optimistic lock pe `state_version`, deja există). Azi
procesorul scrie state DOAR când există reply cu produse; NX-77 extinde: dacă `ctx.state_dirty`
și NU există reply (clarify a îmbogățit starea dar pipeline-ul continuă fără early-exit), tot
trebuie persistat → mutăm patch-ul de state pe ramura „există reply SAU state_dirty".

### Câmpuri TurnContext pe care le scrie NX-77
- `ctx.state.pending_question` și `ctx.state.constraints` — owner CITIRE/CURĂȚARE: `clarify_stage`.
- `ctx.state_dirty: bool` (CÂMP NOU pe TurnContext, default False) — owner: `clarify_stage`
  (și `set_pending_question` când agentul scrie); citit de procesor pentru persistare.
- Helpers noi pe `TurnContext`: `set_pending_question(field, kind, options=None)`,
  `consume_pending_question()`, `merge_constraint(key, value)` — toate setează `state_dirty`.
- `clarify_stage` NU setează `ctx.reply` (nu e un răspuns; e îmbogățire de stare → pipeline continuă).

## Principii CLAUDE.md aplicabile
- **P1 (pipeline liniar):** `clarify_stage` e un stagiu nou în ordine fixă (după cache,
  înainte de triaj); fără loop, fără săritură înapoi. Nu produce reply, doar mută starea.
- **P2 (LLM doar triaj+agent):** consumul e 100% cod determinist (`_extract_answer`, `_norm`).
  ZERO apel LLM/embed în acest stagiu — e exact stratul gratuit din CLAUDE.md stagiul 4.
- **P3 (un singur proprietar per câmp):** `pending_question` are owner unic de scriere
  (agent via `set_pending_question`) și owner unic de citire/curățare (`clarify_stage`).
  Persistarea în DB rămâne la unicul writer de state (procesorul).
- **P4 (buget de context în cod):** `pending_question` e mic (ref, fără textul întrebării);
  `constraints` rămâne sub bugetul de 8KB tăiat de context builder.
- **P6 (niciodată tăcere):** incertitudinea (nu pare răspunsul) NU tace și NU setează reply —
  lasă triajul nano (clarify plătit). Stale → abandonează întrebarea, continuă pipeline-ul.
- **P8 (state = ref-uri):** `pending_question` stochează `field`/`options`, nu textul/obiectul.
- **P11 (limba e parte din cheie):** `clarify_stage` rulează DUPĂ `language_stage`; mesajele
  de opțiuni (dacă vreodată generate) ar fi locale-aware — v1 nu generează text, deci N/A direct.

## Events de emis (analytics)
- `clarify_resolved` {field} — răspuns consumat determinist, constrângere completată (fără valoarea
  brută dacă e PII-sensibilă; pentru constrângeri de produs — buget/ten — valoarea e ne-PII, dar
  v1 logă DOAR `field` ca să nu riscăm; principiul 12).
- `clarify_abandoned` {field} — întrebarea în așteptare abandonată (răspuns stale / subiect nou).
- (Opțional, când agentul scrie:) `clarify_asked` {field} — emis de agent la `set_pending_question`.

Justificare pentru ce NU emitem: nu emitem un event pe „pending_question prezent dar body nu pare
răspunsul" (no-op tăcut) — ar fi zgomot; turul curge normal spre triaj și se va vedea în
`intent_detected`. Toate event-urile sunt fără corpul mesajului / fără PII (principiul 12).

## Out of Scope
- **Popularea „când întreabă" din promptul agentului** — logica prin care agentul DECIDE că-i
  lipsește o constrângere și pune întrebarea (vs să caute direct). NX-77 livrează contractul +
  scrierea + citirea; rafinarea promptului agentului = follow-up [PROPUNERE].
- **Clarify LLM / multi-pas** — întrebări înlănțuite, dezambiguizare semantică. Rămâne la
  triajul nano (clarify plătit). NX-77 e DOAR scurtătura deterministă pe o întrebare știută.
- **`kind="choice"` cu fuzzy/sinonime largi** — v1 face match literal pe `options` normalizate;
  sinonime bogate (ex. „piele grasă" → „gras") = follow-up sau intent_aliases.
- **Mapare `field` → coloană de filtrare în `search_products`** — clarify scrie în
  `state.constraints`; cum le folosește exact agentul la filtrare e treaba lui (deja citește
  `constraints` în `context.py`).
- **TTL/expirare în timp a `pending_question`** — v1 folosește euristica „subiect nou";
  expirarea pe vârstă (ex. > 30 min) = follow-up dacă apare nevoia.

## Test Cases
**Happy Path:**
1. `pending_question={field:"skin_type", kind:"choice", options:["uscat","gras","mixt","sensibil"]}`,
   body „ten gras” → `constraints["skin_type"]="gras"`, `pending_question` curățat, `state_dirty`,
   `clarify_resolved{field:"skin_type"}` emis, fără reply; pipeline continuă spre agent.
2. `pending_question={field:"budget", kind:"constraint"}`, body „sub 100 lei” →
   `constraints["budget"]=100.0` (via `_budget`), curățat, `clarify_resolved` emis.

**Edge Cases:**
1. Body cu diacritice/majuscule („GRAS” / „grás”) → normalizat → match pe opțiune (test pe `_norm`).
2. `pending_question` prezent dar body gol/whitespace → no-op (nu consumă, nu curăță, fără event).
3. Body lung care nu conține nicio opțiune și nu pare răspuns scurt → `extract_answer→None` →
   state neschimbat, triajul preia (clarify nano).
4. State vechi cu `pending_question` parțial (fără `field`) → `clarify_stage` no-op defensiv.

**Failure Cases:**
1. Subiect nou clar („caut altceva, un parfum”) cu `pending_question` setat →
   `looks_like_new_topic→True` → `clarify_abandoned` emis, `pending_question` curățat, fără reply,
   pipeline continuă (triajul clasifică noua cerere).
2. `detect_risk(body)` non-None (ex. „vreau un om”) cu `pending_question` setat → abandonează
   întrebarea, lasă gates/risc din turul anterior să fi tratat deja; clarify nu pliază peste un risc.
3. Procesorul: `patch_conversation_state` aruncă `StateConflict` (scriere concurentă) pe un tur
   doar-state (fără reply) → eroarea se propagă/loghează ca azi (optimistic lock), starea nu se
   corupe; retry-ul turului re-evaluează (idempotent prin `inbound_dedupe`).

> Teste fără apeluri reale: `clarify_stage` e cod pur — `deps.llm` poate fi un mock care
> aruncă la orice atingere (dovedește zero LLM). Persistarea se testează cu stub DB sau
> tranzacție rollback-uită pe `business_id` demo. ZERO apeluri OpenAI/DB real în CI.

## Cost
**$0 LLM adăugat.** `clarify_stage` e 100% cod determinist (string-match normalizat + regex de
buget) — niciun apel OpenAI/embed. Efectul pe cost e NEGATIV (economisește): fiecare răspuns
scurt consumat aici NU mai ajunge la triaj (nano) → ~−1 apel nano/tur clarificat. Pe 1000 mesaje
cu ~15% follow-up-uri de clarificare consumate determinist → ~150 apeluri nano evitate
(triaj nano ~300 tok input → economie neglijabilă în $ dar reduce latența cu un round-trip LLM
și scurtează bucla de vânzare). Mărginirea costului: stagiul nu poate escalada — fără LLM, fără
DB write propriu (persistarea e în TX-ul existent al Sender-ului, fără query suplimentar).
