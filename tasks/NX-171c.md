# NX-171c — DDL content quality: content_status + schema_version + verified_at (cu backfill sigur)

**Owner:** S · **Faza:** P1 · **Track:** Catalog v3 · **Branch:** `feat/NX-171c-content-status` · **Complexitate:** S-M · **Estimare:** 1 zi
**Parent plan:** [docs/CATALOG-PRODUS-V3.md](../docs/CATALOG-PRODUS-V3.md) (Pachet 5, felia 3) · **Depinde de:** NX-168e (date validate)

## Goal
Adaugă quality-gate la nivel de DB: `content_status ∈ {draft, reviewed, published, rejected}` + `schema_version` + `verified_at`. Doar `published` ajunge la bot. **Cu strategie de backfill sigură** ca filtrul să NU golească accidental catalogul.

## Business Context
Auditul static (168d) prinde incoerența pe JSON; `content_status` e plasa la nivel de DB: un produs incomplet/nerevizuit NU e servit clientului. Dar un filtru `published` fără backfill = catalog gol în producție → outage de recomandare.

## Technical Description — secvența de backfill OBLIGATORIE (HIGH — nu doar demo)
Filtrul `published` afectează **TOȚI tenanții**, nu doar demo. SQL nu poate ști dacă un produs trece auditul Python → backfill-ul e un **job explicit**. Auditul cere **snapshot COMPLET** `{products, categories}` (verifică duplicate cross-produs: SKU/GTIN/nume) → **NU se poate rula per produs**. Flag per-tenant, implicit **off**.
1. **Migrare 028a:** adaugă `content_status` **nullable** (fără filtru, fără default încă), + `schema_version int`, `verified_at timestamptz`.
2. **Job de backfill** (`src/jobs/backfill_content_status.py`, per tenant): rulează `audit(contract="v3")` **O DATĂ pe catalogul complet** al tenantului → citește `violations` **machine-readable** (`entry.product_slugs`, NU parsează text CLI) și marchează fiecare produs implicat. Produs care apare în ≥1 violation → `draft`; restul → `published`. Idempotent, re-rulabil.
3. **Default `draft`** pentru produse noi.
4. **`NOT NULL`** pe `content_status` DUPĂ backfill (pe toți tenanții backfill-uiți).
5. **Filtru `published`** în read-path (`catalog.py`) în spatele unui **feature flag per-tenant, default off**. Se activează per tenant DOAR după ce backfill-ul a rulat pt el.
6. **Test-plasă per tenant:** `visible_count > 0` pt FIECARE tenant înainte de a-i activa flagul (nu doar demo). Alt tenant fără backfill → flag off → zero risc de catalog gol.
