# NX-96 — State 8KB: tăiere în cod (nu doar CHECK în DB)
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G6 (context builder) · **Branch:** `feat/NX-96-state-truncate` · **Complexitate:** S · **Estimare:** 2h

## Goal
Principiul 4 cere bugetul de state impus în COD + CHECK în DB ca plasă. Acum
există DOAR CHECK-ul (`chk_state_size < 8192`, 003): calea de scriere
(`patch_conversation_state`) nu trunchiază niciodată, deci un state supradimensionat
e RESPINS de DB cu eroare, nu tăiat în cod. Adăugăm o trunchiere defensivă a
state-ului la sub 8KB ÎNAINTE de patch (taie cele mai vechi ref-uri din
`displayed_products`), ca CHECK-ul DB să rămână ce a fost gândit: ultima plasă,
nu mecanismul primar.

## Business Context
State-ul e memoria scurtă a botului (produse arătate, constrângeri, întrebarea
în așteptare). Dacă el crește peste 8KB (carusel mare + state vechi acumulat),
acum patch-ul aruncă din DB → turul eșuează → riscă tăcere sau retry inutil pe
calea de bani. Tăierea în cod ține botul funcțional indiferent cât a vorbit
clientul, fără să cheltuim nimic în plus (cod determinist, zero LLM).

## Technical Description

### Starea curentă (reală, ce verificăm)
- `src/worker/context.py:59-95` impune bugete **per-bloc de PROMPT** (`state_block`
  taie la `max_chars=400`, `max_products=3`) — adică ce intră în promptul LLM. NU
  atinge ce se PERSISTĂ în `conversations.state`.
- `src/db/queries/conversations.py:98-136` — `patch_conversation_state` serializează
  `new_state` cu `json.dumps` și face UPDATE cu optimistic lock. Nu măsoară, nu taie.
- `src/worker/processor.py:369-391` (Sender) construiește `new_state` ca
  `{**conv["state"], "displayed_products": ctx.reply.products}` — `ctx.reply.products`
  poate avea câmpuri pline (name, price, url, image, ai_summary scurt). Pe conversații
  lungi, `displayed_products` se acumulează → state-ul crește. Aceeași cale e folosită
  și de navigarea caruselului (NX-85/NX-77 `handle_callback`).
- `docs/003_bot_runtime_role.sql:137` — `chk_state_size check (pg_column_size(state) < 8192)`
  e singura apărare azi. Un patch > 8KB → `CheckViolationError` din asyncpg, neprins.

### Target — un singur loc de trunchiere, pe calea de scriere
Trunchierea trăiește în `patch_conversation_state` (unicul drum spre
`conversations.state`), ca ORICE caller (Sender + callback carusel + viitoare) să
fie acoperit automat. Funcție pură, testabilă izolat:

```python
# src/db/queries/conversations.py
STATE_MAX_BYTES = 7800  # marjă sub CHECK-ul DB de 8192 (header jsonb + creștere la edit)

def _serialized_size(state: dict[str, Any]) -> int:
    # aproximează pg_column_size: bytes ai textului json (compact, ca în UPDATE)
    return len(json.dumps(state, separators=(",", ":")).encode("utf-8"))

def truncate_state(state: dict[str, Any], *, max_bytes: int = STATE_MAX_BYTES) -> dict[str, Any]:
    """Aduce state-ul sub `max_bytes` tăind cele mai VECHI ref-uri întâi.
    Determinist, fără LLM. Ordinea de sacrificiu (de la cel mai puțin valoros):
      1. `displayed_products`: păstrează cele mai RECENTE; aruncă din coadă
         (lista e cronologică, ultimul afișat = ultimul element).
      2. dacă tot e mare cu 0 produse → golește `active_search` (re-derivabil din triaj).
    NU atinge `pending_question`/`constraints`/`asked_intents` (memorie critică
    de conversație) decât în ultimă instanță (vezi pasul 4)."""
    if _serialized_size(state) <= max_bytes:
        return state
    out = dict(state)
    products = list(out.get("displayed_products") or [])
    # taie cele mai vechi (din față) câte un produs până încape ori rămâne 1
    while products and _serialized_size({**out, "displayed_products": products}) > max_bytes:
        if len(products) <= 1:
            break
        products = products[1:]
    out["displayed_products"] = products
    if _serialized_size(out) > max_bytes and out.get("active_search"):
        out["active_search"] = {}
    return out
```

`patch_conversation_state` aplică trunchierea înainte de `json.dumps`:

```python
new_state = truncate_state(new_state)         # P4: tăiere în COD, înainte de DB
... update conversations set state = $4::jsonb ... where business_id = $1 and id = $2 ...
```

### Observabilitate (cine știe că s-a tăiat)
`patch_conversation_state` e stratul de date — NU emite analytics (nu are
`TurnContext`, principiul 10: doar runner-ul/stagiile emit). Semnalează prin
**return**: schimbăm semnătura să întoarcă și un flag (sau logăm `WARNING` fără
PII). Practic: dacă `truncate_state` a tăiat, scriem un `log.warning` cu
`conversation_id` + `business_id` + `bytes_before/after` (zero corp de mesaj,
zero PII — P12) și lăsăm Sender-ul (processor) să emită eventul `state_truncated`
pe `ctx` (el are TurnContext). Varianta minimă: `truncate_state` e pură și
expusă, iar processor-ul o apelează explicit și emite eventul; `patch_*` rămâne
plasa idempotentă care re-aplică tăierea pentru caller-ele fără ctx (callback).

**Decizie:** trunchierea defensivă stă în `patch_conversation_state` (acoperă
TOATE caller-ele); processor-ul (Sender) o apelează ÎNAINTE și, dacă a redus,
emite `state_truncated` pe ctx. Dublul apel e idempotent (a doua oară nu mai taie).

### Câmpuri TurnContext scrise aici
Niciun câmp nou pe `TurnContext`. Tăierea operează pe `new_state` (dict),
construit din `ctx.state` + `ctx.reply.products` în Sender (owner: Sender, ca
acum). Eventual `ctx.emit("state_truncated", ...)` (events e append-only,
owner stagii — neschimbat). Nu atingem owner-ship-ul niciunui câmp.

## Principii CLAUDE.md aplicabile
- **P4 (buget impus în COD, nu în prompturi/DB):** exact gaura. Tăierea în
  `patch_conversation_state`/Sender face din CHECK-ul DB o plasă, nu mecanismul
  primar. Marjă (7800 < 8192) ca un edit ulterior să nu treacă pragul.
- **P8 (state = ref-uri, nu obiecte):** tăiem din `displayed_products` (ref-uri
  {product_id, name, price}); nu inventăm compresie pe obiecte mari — nu trebuie
  să existe obiecte mari în state.
- **P6 (niciodată tăcere):** un state mare NU mai trebuie să rupă turul cu
  CheckViolation → degradare grațioasă (taie memoria veche, răspunde).
- **P7 (business_id pe tot):** UPDATE-ul din `patch_conversation_state` are deja
  `where business_id = $1 and id = $2` — neatins.
- **P10 (observabilitate din runner):** stratul de date NU emite; eventul
  `state_truncated` e emis de Sender pe `ctx`, scris de runner.
- **P12 (PII):** logul de trunchiere conține DOAR id-uri + dimensiuni, niciodată
  corp de mesaj sau nume de produs.

## Implementation Steps
1. `src/db/queries/conversations.py`: adaugă `STATE_MAX_BYTES`, `_serialized_size`,
   `truncate_state` (pure). Aplică `truncate_state(new_state)` în
   `patch_conversation_state` înainte de `json.dumps`.
2. `src/worker/processor.py` (Sender, ~369-391): apelează `truncate_state` pe
   `new_state` înainte de `patch_conversation_state`; dacă a redus
   (`_serialized_size` mai mic), `ctx.emit("state_truncated", products_dropped=…,
   bytes_after=…)`. (Apelul rămâne idempotent cu cel din `patch_*`.)
3. `src/config.py` + `.env.example` (opțional): `state_max_bytes` (default 7800)
   dacă vrem reglaj fără redeploy; altfel constantă în modul.
4. Edge: dacă state-ul depășește 7800 cu 0 produse ȘI `active_search` gol
   (doar `pending_question`/`constraints` uriașe) — caz patologic improbabil.
   Lăsăm patch-ul să meargă spre DB; dacă CHECK-ul aruncă, e bug real de upstream
   (cineva a băgat obiect mare în state, P8) — îl prindem în test, nu îl mascăm.
5. Teste (pur Python, fără DB/LLM): `truncate_state` cu state mic = identic;
   state mare cu N produse → păstrează cele mai recente, scade sub prag;
   idempotență (a doua tăiere = no-op); `active_search` golit doar la nevoie.
   Plus un test pe `patch_conversation_state` cu conn mock care verifică că
   `json.dumps` primit e < `STATE_MAX_BYTES`.
6. `ruff check . && ruff format . && pytest -x -q` verde.

## Test Cases
**Happy Path:**
1. State tipic (2-3 produse, constrângeri mici) → `truncate_state` no-op, identic.
2. `handle_turn` pe o conversație cu `displayed_products` mare seedat → reply
   trimis, state persistat < 8KB, `state_truncated` emis cu `products_dropped > 0`.

**Edge Cases:**
1. Exact pe prag (`_serialized_size == STATE_MAX_BYTES`) → no-op (condiția e `<=`).
2. Un singur produs uriaș (P8 încălcat upstream) → rămâne 1 produs (nu tăiem sub 1);
   dacă încă > prag, `active_search` golit; dacă tot > prag, lasă DB-ul să decidă.
3. `displayed_products` gol + `active_search` mare → golește `active_search`, scade sub prag.

**Failure Cases:**
1. State patologic > 7800 chiar și cu 1 produs + `active_search` gol (doar
   `constraints`/`pending_question` uriașe) → `patch_*` lasă să meargă spre DB;
   dacă `chk_state_size` aruncă, e bug de upstream — testul ASERTĂ că trunchierea
   a încercat tot ce putea (produse=1, active_search={}) înainte. (Semnal real, nu mascat.)
2. `truncate_state` cu `state=None`/dict gol → întoarce `{}` fără excepție.
3. `StateConflict` (optimistic lock pierdut) rămâne neschimbat — trunchierea nu
   interferează cu logica de versiune (test: conflict încă ridicat după trunchiere).

> Teste 100% pur Python pe funcția determinist `truncate_state` + conn mock pentru
> `patch_conversation_state`. ZERO apeluri reale (fără DB, fără OpenAI) în CI.

## Cost
**$0 LLM** — task pur determinist (cod), niciun apel OpenAI/embeddings. Nici nu
trebuie: P2 interzice LLM în afara triaj/agent, iar memoria veche se ARUNCĂ, nu
se rezumă cu model. Cost de runtime: o serializare `json.dumps` în plus per patch
(neglijabil; aceeași operație se făcea oricum la scriere). Ce limitează: tăierea
e O(n) pe `displayed_products` (n ≤ câteva zeci), bounded de plafonul de 7800 bytes.
