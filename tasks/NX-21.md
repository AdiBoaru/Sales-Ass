# NX-21 — Web Widget W2: widget.js
**Owner:** J · **Faza:** P2 (V1.5 — Epic E26, după clientul 1 stabil pe WhatsApp) · **Zi/Ord:** W2 (după NX-20) · **Branch:** `feat/NX-21-web-widget-js` · **Complexitate:** M · **Estimare:** 4h

## Goal
Widget de chat embeddabil cu o singură linie de `<script data-token>` pe site-ul clientului,
care consumă gateway-ul SSE din NX-20 (POST `/web/messages` + GET `/web/stream`). Front-end
static pur (vanilla JS, fără build, fără framework) izolat în **shadow DOM** ca să nu existe
NICIUN conflict CSS/JS cu site-ul gazdă. Temă din `businesses.settings`, i18n RO/HU/EN,
disclosure „AI" vizibil PERMANENT (art. 50 AI Act). NU conține logică de pipeline — pipeline-ul
și worker-ul rămân agnostice de canal (canalul `web` se adaugă în gates/sender la NX-22).

## Business Context
Web widget = al doilea canal după WhatsApp, deschide vânzarea direct pe site-ul magazinului
(vizitatorul nu trebuie să aibă WhatsApp). Embed de o linie = friction zero la onboarding-ul
unui client nou (agenție SaaS: cu cât setup-ul e mai scurt, cu atât marja e mai bună).
Shadow DOM e non-negociabil: dacă widgetul strică layout-ul magazinului, clientul îl scoate
în 5 minute. Disclosure-ul AI permanent ne ține pe partea corectă a AI Act (sancțiuni reale
pentru UE) și e și onest față de vizitator.

## Technical Description

### Ce e și ce NU e
- **E:** un fișier static `web/widget.js` (+ `web/embed.js` mic, loaderul) servit de pe un CDN
  / static host. Vanilla JS, zero dependențe, zero build step. Tot UI-ul trăiește în shadow DOM.
- **NU e:** cod Python, nu atinge pipeline-ul, nu vorbește cu DB, nu cunoaște `business_id`
  (cunoaște DOAR `data-token` public — secretul/maparea token→tenant e în gateway, NX-20/NX-25).

### Embed de o linie (pe site-ul terț)
```html
<script src="https://cdn.nativx.tech/widget/embed.js" data-token="pub_8f3a...e21" async></script>
```
`embed.js` (loader subțire, ~1KB): citește `data-token` din propriul `<script>`
(`document.currentScript`), creează host-elementul, atașează `shadowRoot` (mode `open`),
apoi încarcă `widget.js` (bundle-ul real) în interior. Un singur tag, nimic de configurat în
pagină. Token-ul e PUBLIC (identifică tenantul, NU autentifică — rate-limitat agresiv la gateway,
vezi NX-25); nu punem niciodată secrete în front-end.

### Shadow DOM — izolare totală
```js
const host = document.createElement('div');
host.id = 'nativx-assistant-root';
document.body.appendChild(host);
const root = host.attachShadow({ mode: 'open' });
// tot CSS-ul injectat ca <style> ÎN shadowRoot → nu scapă în pagină, nu intră din pagină
root.innerHTML = `<style>${WIDGET_CSS}</style>${WIDGET_HTML}`;
```
Reguli stricte: zero `<style>`/`<link>` în `document.head`, zero clase globale, fără `!important`
peste pagina gazdă. Font: stack de system-fonts (nu încărcăm webfont care ar putea bloca render-ul
gazdei). Singurul lucru din afara shadow DOM = host-ul gol + (opțional) un `z-index` mare pe host.

### Bootstrap config (token → tenant config)
La init, widgetul cere config-ul public de tenant de la gateway:
`GET /web/config?token=<pub_token>` → `{ theme, locales, default_locale, disclosure, brand_name }`.
`theme` derivă din `businesses.settings.web_widget` (vezi Database Changes — DOAR chei jsonb,
fără migrare): `{ primary_color, position: 'right'|'left', logo_url, header_title }`.
Tema se aplică prin CSS custom properties în shadowRoot (`--nx-primary`, `--nx-bg`, ...), nu prin
clase hardcodate. Fallback dacă lipsesc chei: paletă neutră default în widget.js.

### Transport: consumă NX-20 (SSE)
- **Trimitere mesaj user:** `POST /web/messages` cu body `{ token, session_id, text }` →
  gateway-ul împachetează un envelope NEUTRU (`channel_kind='web'`, `channel_account_id`=
  derivat din token, `sender_external_id`=`session_id`/`visitor_id`) pe stream-ul `inbound`.
  Widgetul NU construiește envelope — doar trimite text + session.
- **Primire răspuns bot:** `GET /web/stream` (EventSource SSE). Widgetul ascultă evenimente
  `message` (text bot) și `typing` (indicator). Reconectare nativă EventSource + `Last-Event-ID`
  pentru a relua de unde a rămas după drop de rețea (gateway-ul gestionează ID-urile — NX-20).
- **Sesiune:** `session_id` semnat de gateway la prima încărcare (cookie first-party gestionat de
  gateway / token de sesiune în memorie). Identitatea vizitatorului reală + merge cu WhatsApp =
  NX-23, NU acest task. Aici doar persistăm `session_id` în `localStorage` (cheie namespace-uită
  `nx_session_<token_prefix>`) ca să nu pierdem firul la refresh.

### i18n RO/HU/EN
Stringurile de UI (placeholder input, label „trimite", disclosure, mesaj eroare conexiune)
trăiesc într-un obiect `I18N = { ro: {...}, hu: {...}, en: {...} }` în widget.js. Limba aleasă:
(1) `default_locale` din config-ul de tenant; (2) override prin `data-locale` pe tag dacă e în
`supported_locales`; (3) altfel `ro`. **Conținutul mesajelor bot vine deja localizat din pipeline**
(ctx.language → locale pe faqs/cache/templates) — widgetul NU traduce mesaje, traduce DOAR chrome-ul
de UI.

### Disclosure AI permanent (art. 50 AI Act)
Un rând fix în header-ul widgetului, vizibil în ORICE stare (deschis), text din i18n:
`ro: "Asistent AI · răspunsuri automate"`. NU e un mesaj de sistem care se pierde în scroll —
e parte din chrome, mereu pe ecran. Implementat ca element separat în HTML-ul shadow, nu ca prim
mesaj din conversație (un mesaj ar dispărea la scroll → ne-conform).

### Câmpuri TurnContext scrise aici
**NICIUNUL.** Acesta e cod front-end static; nu rulează în worker, nu atinge `TurnContext`.
Maparea text web → `TurnContext` (channel_kind='web') se face în worker la NX-22.

### Securitate & PII
- Token PUBLIC în front-end (by design); secretul/HMAC-ul de sesiune stă DOAR la gateway (NX-20).
- Zero PII în `localStorage` în afară de `session_id` opac (NU telefon, NU nume) — PII trăiește în
  `channel_identities` server-side (P12). Telefonul, dacă vizitatorul îl dă, merge prin mesaj
  normal → pipeline → `channel_identities`, niciodată salvat în clientul JS.
- Loguri: widgetul NU loghează corpul mesajelor în `console` în producție (flag debug off).
- CORS/CSP + allowlist de domenii per tenant = NX-25 (gateway-side), NU aici. Widgetul doar
  presupune că gateway-ul respinge originile neautorizate cu 403.

### Performanță
- Bundle țintă < 25KB gzip, fără dependențe. `embed.js` `async`, nu blochează render-ul gazdei.
- `widget.js` lazy: UI greu (lista de mesaje) se montează la prima deschidere, nu la load.
- SSE = o singură conexiune long-lived per sesiune (vs polling). Reconectare cu backoff din NX-20.

## Principii CLAUDE.md aplicabile
- **P2 (LLM doar triaj+agent):** widgetul e front-end pur, ZERO apeluri LLM. Tot ce e inteligent
  rămâne în pipeline server-side. Aici e doar transport + render.
- **P5 (un singur punct de ieșire):** widgetul NU trimite direct nicăieri în afară de gateway-ul
  NX-20; răspunsurile vin pe SSE din `outbox`→dispatcher→sender web (NX-22). Niciun shortcut.
- **P11 (limba e parte din cheie):** UI-ul respectă `locale` (default din tenant / `data-locale`),
  iar conținutul bot e deja localizat upstream; widgetul nu „inventează" limba.
- **P12 (PII într-un singur loc):** zero PII în clientul JS / `localStorage` în afară de un
  `session_id` opac; telefonul ajunge în `channel_identities` server-side, niciodată în browser.
- **Cuplajul de canal stă DOAR la margini:** widgetul e marginea de INGESTIE/UI a canalului web;
  pipeline-ul (3-9) rămâne agnostic. Widgetul nu conține logică de business.

## Implementation Steps
1. `web/embed.js`: loader ~1KB — citește `data-token` (+ `data-locale` opțional) din
   `document.currentScript`, creează host-ul, `attachShadow({mode:'open'})`, injectează
   `widget.js` în shadowRoot. `async`, idempotent (nu montează de două ori).
2. `web/widget.js` — schelet: `WIDGET_HTML` (buton flotant + panou: header cu disclosure,
   listă mesaje, input, send), `WIDGET_CSS` (toate var-urile pe `--nx-*`), `I18N` (ro/hu/en).
3. Bootstrap: `GET /web/config?token=...` → aplică `theme` (set CSS custom properties pe
   `:host`), alege `locale`, randează disclosure-ul permanent în header.
4. Trimitere: la submit → `POST /web/messages {token, session_id, text}`; optimist render mesaj
   user; dezactivează input până la ACK.
5. Primire: `new EventSource('/web/stream?token=...&session_id=...')`; handler `message` →
   append bubble bot; handler `typing` → indicator; `onerror` → indicator „reconectare…" din i18n
   (EventSource reconectează singur + `Last-Event-ID` din NX-20).
6. Persistență sesiune: `localStorage` cheie `nx_session_<token_prefix>` (doar `session_id` opac).
7. Tema demo: secțiune `businesses.settings.web_widget` pe clientul demo (doc + exemplu jsonb).
8. **Test pe site terț:** pagină de test `web/test-host.html` cu CSS agresiv (reset global,
   `* { box-sizing: content-box }`, fonturi proprii) → confirmă zero scurgere/zero intrare CSS
   în/din shadow DOM. Test manual + checklist de izolare în PR (front-end static, fără pytest).
9. Lint front-end: `npx prettier --check web/` (+ eventual `eslint` dacă e configurat); fără
   erori în consolă pe pagina de test în Chrome/Firefox/Safari.

## Database Changes
**None (DDL).** Tema citită din `businesses.settings.web_widget` (jsonb existent, `default '{}'`)
— doar chei noi, fără migrare:
```jsonc
// businesses.settings (exemplu, clientul demo 6098812a-50fc-44bd-a1ba-bc77e6399158)
{ "web_widget": {
    "primary_color": "#1f6feb", "position": "right",
    "logo_url": "https://...", "header_title": "Sole Demo" } }
```
Citirea config-ului (server-side, în gateway NX-20) respectă tenant scoping:
`select settings->'web_widget', default_locale, supported_locales from businesses where id = $1`
(token public → `business_id` rezolvat la gateway, niciodată în front-end).

## API Changes
**None create aici.** Widgetul CONSUMĂ contractul NX-20:
- `GET /web/config?token=<pub>` → `200 {theme, default_locale, supported_locales, disclosure, brand_name}`;
  `403` token invalid/domeniu neautorizat (NX-25).
- `POST /web/messages` `{token, session_id, text}` → `202 {message_id}`; `429` rate-limit; `403`.
- `GET /web/stream?token=&session_id=` → SSE: evenimente `message`, `typing`; reconectare
  `Last-Event-ID`. Contractul exact = NX-20.

## Events de emis (analytics)
**None din front-end direct.** Botul are doar INSERT pe `analytics_events` server-side (P10:
observabilitatea vine din runner, nu din stagii — cu atât mai puțin din clientul JS). Evenimentele
relevante (`web_session_started`, `route`, etc.) le emite worker-ul când procesează mesajul web
(NX-22). Trimiterea de telemetrie din browser direct în `analytics_events` ar încălca modelul
append-only server-side și ar expune suprafață de abuz — deci intenționat NICIUNUL aici.

## Dependencies
- **NX-20** (Gateway SSE W1) — TREBUIE în main: `/web/config`, `/web/messages`, `/web/stream`,
  sesiune semnată cu token public, reconectare Last-Event-ID. Widgetul nu are ce consuma fără el.
- **NX-60** (abstracția de canal / envelope neutru) — în main: gateway-ul produce envelope `web`.
- Pentru ca răspunsurile să curgă efectiv înapoi în widget e nevoie și de **NX-22** (canal web în
  gates/sender — senderul scrie în stream-ul sesiunii), dar NX-22 poate veni DUPĂ acest card:
  widgetul se poate dezvolta și testa contra unui stream-mock al gateway-ului.

## Out of Scope
- **Canal web în gates/sender** (channel_kind='web', debounce 800ms, typing=eveniment SSE) — **NX-22**.
- **CORS/CSP + allowlist de domenii per tenant** (403 pe origine neautorizată) — **NX-25** (gateway-side).
- **Identitate vizitator + merge cu WhatsApp** (visitor_id semnat, audit_log) — **NX-23**.
- **Context de pagină** (page_url/product_id → hint de retrieval) — **NX-24**.
- **Golden web + load test 200 sesiuni** — **NX-26**.
- **Build pipeline / minificare avansată / npm publish** — v1 = fișier static servit ca atare.

## Test Cases
> Front-end static, ZERO LLM → fără apeluri reale în CI. Transportul (NX-20) se testează contra
> unui **stream-mock** servit local (EventSource pe un mic server de test sau fixture), nu contra
> gateway-ului real. Verificarea de izolare CSS e manuală + checklist în PR (browser real).

**Happy Path:**
1. Embed pe `test-host.html` → buton flotant apare; click deschide panoul; disclosure vizibil;
   tema (culoare primară din config-mock) aplicată.
2. Trimit „salut" → bubble user instant; mock SSE emite `message` „Bună!" → bubble bot randat;
   `typing` între ele afișează indicatorul.

**Edge Cases:**
1. `businesses.settings.web_widget` gol `{}` → widget randat cu paleta default, fără excepție.
2. `data-locale=hu` în `supported_locales` → chrome în maghiară; `data-locale=fr` (neacceptat)
   → cade pe `default_locale`.
3. Pagină gazdă cu `* { all: revert }` / reset agresiv → widgetul rămâne intact (shadow DOM);
   layout-ul gazdei nerafectat.
4. Refresh după conversație → `session_id` din `localStorage` reutilizat; firul continuă.

**Failure Cases:**
1. `/web/config` întoarce `403` (token invalid) → widgetul afișează discret „indisponibil" din i18n,
   NU crapă pagina gazdă, fără excepție în consolă.
2. SSE `onerror` (gateway jos) → indicator „reconectare…" din i18n; EventSource reîncearcă singur;
   la revenire reia cu `Last-Event-ID`, fără dublarea mesajelor.
3. `POST /web/messages` `429` (rate-limit) → mesaj de UI „prea multe mesaje, reîncearcă",
   input re-activat, fără pierderea textului tastat.

## Cost
**$0 LLM** — widgetul nu face niciun apel LLM/embeddings (front-end pur). Costul incremental real
e doar trafic/CDN static + o conexiune SSE long-lived per sesiune la gateway (NX-20). Costul de
inferență apare doar când mesajul web ajunge în pipeline (triaj nano + agent mini), identic cu
WhatsApp — bornat de aceleași limite (MAX 3 tool calls/tur, straturile gratuite 40-60%, cost guard
zilnic per business). Nimic specific de cost adăugat de acest card.
