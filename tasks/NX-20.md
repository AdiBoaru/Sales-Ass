# NX-20 — Web Widget W1 — Gateway SSE
**Owner:** S · **Faza:** P1 · **Zi/Ord:** Epic E26 (V1.5), după NX-60 (margini de canal) · **Branch:** `feat/NX-20-web-sse-gateway` · **Complexitate:** L · **Estimare:** 10h (umbrelă — spart mai jos în ≤4h)

## Goal
Al treilea canal: un **widget web** (chat pe site-ul clientului) care intră prin ACEEAȘI margine
neutră ca WhatsApp/Telegram (NX-60). Două endpointuri HTTP pe app-ul FastAPI: `POST /web/messages`
(client → bot, produce envelope-ul neutru pe stream-ul `inbound`) și `GET /web/stream`
(bot → client, **Server-Sent Events**). SSE, NU WebSocket, ca să treacă prin orice proxy/CDN
fără upgrade de protocol. Sesiunea e legată de un **token public per tenant** (nu PII, nu auth de
user) și de un `visitor_id` semnat în cookie/Local-Event-ID. Pipeline-ul (stagiile 3-9) rămâne
NEATINS: vede un `channel_kind='webchat'`, ca orice alt canal.

## Business Context
Mulți retaileri vor botul și pe site, nu doar pe WhatsApp — vizitatorul anonim care navighează
produse e exact lead-ul cel mai cald. Web-ul nu are fereastră 24h și nu cere număr de telefon →
intrare cu zero fricțiune, atribuire directă la conversie (același `checkout_links.ref_code`).
Reutilizăm tot pipeline-ul deja construit; costul marginal pe canal nou e doar transportul.
DoD comercial: 200 de sesiuni simultane stabile 10 minute pe staging (un magazin mediu).

## Technical Description

### De ce SSE și nu WebSocket
WebSocket cere upgrade de protocol (`Connection: Upgrade`) — multe CDN-uri/proxy-uri corporate îl
taie. SSE e un `GET` lung peste HTTP/1.1 obișnuit (`Content-Type: text/event-stream`), cu reconectare
nativă în browser (`EventSource`) și suport de `Last-Event-ID` din standard. Inbound-ul (client→bot)
NU stă pe acel stream — e un `POST` separat. Deci: două drumuri, simetrice cu webhook-ul Meta
(POST in / push out), dar push-ul e prin conexiunea SSE ținută deschisă.

### Marginea de INTRARE — `POST /web/messages` → envelope neutru (NX-60)
Identic ca pattern cu poller-ul Telegram (`src/channels/telegram/poller.py:_to_event`): parserul
web produce un `InboundEvent` și face `enqueue_inbound`. Worker-ul rezolvă tenantul cu
`resolve_channel('webchat', <public_token>)` — fără cod nou în consumer.

```python
# src/web/app.py (router montat în src/webhook/app.py)
@router.post("/web/messages")
async def web_message(req: WebMessageIn, redis: Redis = Depends(redis_dep)) -> dict:
    session = verify_web_session(req.token, req.visitor_id, req.sig)  # HMAC, vezi mai jos
    if session is None:
        raise HTTPException(403, "invalid session")
    if await web_rate_limited(redis, session.token, req.ip, session.visitor_id):
        raise HTTPException(429, "rate limited")           # IP + visitor, vezi mai jos
    body = (req.text or "").strip()
    if not (1 <= len(body) <= 2000):                        # buget de input dur (anti-abuz)
        raise HTTPException(400, "empty or too long")
    event = InboundEvent(
        channel_kind="webchat",
        channel_account_id=session.token,                  # public token = provider_account_id al canalului
        sender_external_id=session.visitor_id,             # vizitatorul = "userul pe canal"
        provider_msg_id=req.client_msg_id or str(uuid4()), # idempotent: clientul trimite un id
        content_type="text",
        body=body,
        sender_name=None,
    )
    await enqueue_inbound(redis, event.to_dict())
    return {"accepted": True, "msg_id": event.provider_msg_id}
```

`channel_account_id` = tokenul public al tenantului → `channels.provider_account_id` pentru rândul
`kind='webchat'`. Astfel `resolve_channel('webchat', token)` derivă `business_id` EXACT ca la
WhatsApp/Telegram, zero excepție de izolare (control plane, `admin_conn`). `sender_external_id` =
`visitor_id` → identity resolution scrie un `channel_identities(channel_kind='webchat',
external_id=visitor_id)`. **`visitor_id` e PII de canal → trăiește DOAR în `channel_identities`**
(ca wa_id/chat_id), niciodată în loguri (redactat ca celelalte external_id-uri).

### Sesiunea semnată — token public + visitor_id (FĂRĂ LLM, FĂRĂ DB pe calea fierbinte)
Tokenul public per tenant e un secret de site (nu PII), stocat în `channels.settings` la onboarding
(`{"public_token": "...", "session_secret": "..."}`). Widget-ul îl primește la `bootstrap` și-l
trimite la fiecare request. `visitor_id` e generat la primul contact și **semnat HMAC** cu
`session_secret` (per tenant) → clientul nu-și poate falsifica identitatea altui vizitator, dar nici
nu cerem login:

```python
# src/web/session.py — cod pur, zero LLM, zero DB pe hot path (secretul vine din channels cache)
def issue_visitor(token: str, session_secret: str) -> tuple[str, str]:
    visitor_id = f"web_{uuid4().hex}"
    sig = hmac.new(session_secret.encode(), f"{token}:{visitor_id}".encode(), "sha256").hexdigest()
    return visitor_id, sig

def verify_web_session(token, visitor_id, sig) -> WebSession | None:
    secret = _session_secret_for(token)          # lookup pe public_token → channels.settings (cache LRU+TTL)
    if secret is None:
        return None
    expected = hmac.new(secret.encode(), f"{token}:{visitor_id}".encode(), "sha256").hexdigest()
    if not hmac.compare_digest(expected, sig):
        return None
    return WebSession(token=token, visitor_id=visitor_id, business_id=_biz_for(token))
```

Lookup-ul `public_token → (business_id, session_secret)` e o operație de **control plane**
(`admin_conn`, ca `resolve_channel`) — derivă tenantul ÎNAINTE de a-l ști. Îl ținem într-un cache
LRU cu TTL scurt (60s) ca să nu lovim DB la fiecare mesaj/heartbeat. Acesta e singurul query nou pe
`admin_conn`, și e tot maparea canal→business — în spiritul excepției documentate.

### Marginea de IEȘIRE — `GET /web/stream` (SSE)
Aici e diferența reală față de WhatsApp/Telegram: acelea sunt **push** (dispatcher → API Meta).
Web-ul e **pull persistent**: clientul ține o conexiune SSE deschisă, iar răspunsurile trebuie
livrate PE acea conexiune. NU schimbăm contractul Sender (stagiul 9 scrie tot în `outbox`); adăugăm
un **transport de outbox specific web**: dispatcher-ul, pentru `channel_kind='webchat'`, în loc să
facă HTTP la o platformă, **publică pe un canal Redis Pub/Sub** `web:out:{visitor_id}`, iar handler-ul
SSE e abonat și retransmite ca eveniment SSE.

```python
# src/channels/web/sender.py — implementează ChannelSender, dar "trimite" prin Redis Pub/Sub
class WebSender:
    def __init__(self, redis: Redis) -> None:
        self._redis = redis
    async def send_text(self, account_id: str, to: str, text: str) -> str:
        msg_id = f"web_out_{uuid4().hex}"
        await self._redis.publish(f"web:out:{to}", json.dumps({"id": msg_id, "type": "text", "text": text}))
        return msg_id          # provider_msg_id sintetic (consistență cu contractul)
```

```python
# src/web/app.py — endpointul SSE
@router.get("/web/stream")
async def web_stream(token: str, visitor_id: str, sig: str, request: Request,
                     last_event_id: str | None = Header(default=None)) -> StreamingResponse:
    session = verify_web_session(token, visitor_id, sig)
    if session is None:
        raise HTTPException(403, "invalid session")
    async def gen():
        pubsub = (await get_redis()).pubsub()
        await pubsub.subscribe(f"web:out:{visitor_id}")
        # reconectare: replay mesajele lipsă din backlog Redis (LIST cu TTL) după Last-Event-ID
        for ev in await _replay_after(visitor_id, last_event_id):
            yield _sse(ev)
        try:
            while True:
                if await request.is_disconnected():
                    break
                msg = await pubsub.get_message(timeout=15, ignore_subscribe_messages=True)
                if msg is None:
                    yield ": keepalive\n\n"           # heartbeat la 15s (ține proxy-ul deschis)
                    continue
                yield _sse(json.loads(msg["data"]))
        finally:
            await pubsub.unsubscribe(f"web:out:{visitor_id}")
    return StreamingResponse(gen(), media_type="text/event-stream",
                             headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"})
```

`X-Accel-Buffering: no` + `Cache-Control: no-cache` opresc bufferizarea la nginx/CDN (altfel SSE nu
curge). Heartbeat-ul `: keepalive` la 15s ține conexiunea vie sub proxy-uri cu idle-timeout.

### Reconectare cu Last-Event-ID (backlog mic, NU re-livrare din outbox)
`EventSource` retrimite automat ultimul `id:` ca header `Last-Event-ID` la reconectare. Ca să nu
pierdem mesaje în fereastra de reconectare, dispatcher-ul scrie ÎN PLUS fiecare mesaj web într-un
backlog Redis per vizitator (`web:backlog:{visitor_id}`, LIST cu `LTRIM` la ultimele 20 + TTL 5 min).
La reconectare, `_replay_after(visitor_id, last_event_id)` redă din backlog evenimentele de după
id-ul confirmat. Pub/Sub nu persistă → backlog-ul e plasa; outbox-ul rămâne sursa durabilă de
adevăr (dacă livrarea SSE eșuează complet, rândul de outbox e marcat `sent` doar după publish reușit,
deci nimic nu se pierde tăcut — principiul 6).

### Rate limit pe DOUĂ chei (IP + visitor)
Web-ul e public anonim → abuz mai ușor decât WhatsApp. Două contoare Redis (refolosim
`rate_limit_count` din `src/worker/limits.py`, parametrizat pe cheie): `webrl:ip:{token}:{ip}` și
`webrl:visitor:{visitor_id}`. Oricare depășește pragul → `429`. IP-ul prinde un atacator care rotește
visitor_id; visitor_id prinde un client legit care spam-ează. Pragurile din config (default mai
strânse decât per-contact WhatsApp).

### Câmpuri TurnContext scrise aici
**NICIUNUL.** Web-ul e strict transport (margini NX-60). Pipeline-ul primește un `InboundEvent`
neutru și scrie aceleași câmpuri ca pentru orice canal (Triaj → `route`, Agent → `reply`/`retrieval`,
etc.). Acest task NU atinge contractul TurnContext — owner-ii câmpurilor rămân stagiile existente.

## Principii CLAUDE.md aplicabile
- **P5 (un singur punct de ieșire):** web-ul NU trimite direct din endpoint; `reply` iese prin
  `outbox` → dispatcher → `WebSender.publish`. SSE-ul doar retransmite ce-a publicat dispatcher-ul.
- **P7 (business_id pe tot):** tenantul se derivă din `public_token` via `resolve_channel('webchat', token)`
  (control plane); orice query de rundă rămâne pe `tenant_conn` cu `business_id`.
- **P12 (PII într-un loc):** `visitor_id` = id de canal → DOAR în `channel_identities`; logurile nu
  conțin token public, visitor_id sau IP în clar (IP redactat/hash în loguri, ca telefoanele).
- **P2 (LLM doar triaj+agent):** acest task e 100% cod determinist — sesiune HMAC, rate limit, SSE.
  Zero apeluri LLM/embeddings adăugate.
- **P1 + canal la margini:** cuplajul de transport stă DOAR în `src/web/` și `src/channels/web/`;
  pipeline-ul și consumer-ul rămân agnostice (consumer-ul rezolvă deja orice `channel_kind`).
- **P6 (niciodată tăcere):** reconectare cu backlog + heartbeat; outbox marcat `sent` doar la publish
  reușit → un client deconectat reprimește la reconectare, nu pierde răspunsul.

## API Changes
Endpointuri noi pe app-ul FastAPI (public, fără auth de user — autentificate prin token+HMAC):
- **`GET /web/bootstrap?token=<public_token>`** → `200 {public_token, visitor_id, sig, sse_url}`;
  `403` token necunoscut. (emite o sesiune nouă de vizitator)
- **`POST /web/messages`** body `{token, visitor_id, sig, text, client_msg_id?}` →
  `200 {accepted, msg_id}` · `403` sesiune invalidă · `400` text gol/>2000 · `429` rate limit.
- **`GET /web/stream?token=&visitor_id=&sig=`** (header opțional `Last-Event-ID`) →
  `200 text/event-stream` (evenimente `data:` + `id:`, heartbeat `:` la 15s) · `403` sesiune invalidă.
  Headere: `Cache-Control: no-cache`, `X-Accel-Buffering: no`.

## Events de emis (analytics)
Worker-ul emite deja evenimentele de pipeline (intent_detected/route/...) pentru orice canal —
`channel_kind='webchat'` apare natural în context. Adăugăm la marginea web, prin pipeline-ul de
analytics existent (NU din endpoint — endpointul e subțire):
- `web_session_started` {token_hash} — la `bootstrap` (token HASH-uit, niciodată în clar; FĂRĂ IP/PII).
- `web_rate_limited` {limiter: "ip"|"visitor"} — la 429 (fără IP/visitor_id în clar).

Justificare minimalism: evenimentele de conversație (handoff, cache_hit, order) sunt deja emise de
stagii pe `analytics_events` — nu le dublăm la transport.

## Test Cases
**Happy Path:**
1. `bootstrap` → `verify_web_session` pe (token, visitor_id, sig) emis = valid; `POST /web/messages`
   → envelope corect pe stream (assert pe argumentele `enqueue_inbound`).
2. Mesaj publicat de `WebSender` pe `web:out:{visitor_id}` → `GET /web/stream` îl emite ca
   `data: {...}\nid: web_out_...\n\n` (test cu pubsub fake, consum un singur eveniment + close).

**Edge Cases:**
1. Reconectare: client trimite `Last-Event-ID=web_out_3`; backlog are 3,4,5 → stream redă DOAR 4,5.
2. `text` gol sau 2001 caractere → `400` fără enqueue (buget de input dur).
3. Heartbeat: fără mesaje 15s → stream emite `: keepalive` (conexiunea rămâne deschisă).
4. `public_token` necunoscut la `bootstrap`/`stream` → `403` (canal inexistent/dezactivat).

**Failure Cases:**
1. `sig` falsificată (alt visitor_id semnat cu același token) → `403`, zero enqueue.
2. Rate limit: 21 mesaje în fereastră de la același IP (visitor_id rotit) → al 21-lea `429`;
   contorul pe IP prinde rotirea de visitor.
3. Redis Pub/Sub indisponibil la `WebSender.send_text` → excepție propagată → dispatcher `mark_failed`
   (retry/backoff), outbox NU marcat `sent` → mesajul nu se pierde (test: publish aruncă → status
   != 'sent').
4. Client SSE deconectat (`request.is_disconnected()`) → generatorul iese curat, `unsubscribe` rulat
   (fără leak de subscriber).

> consumat ca async generator. ZERO conexiuni reale în CI. Load test-ul (200 SSE) e manual pe staging,
> NU în CI.

## Cost
**$0 LLM/embeddings adăugat** — acest task e pur transport determinist (sesiune HMAC, rate limit, SSE,
Pub/Sub). Costul de inferență al unui mesaj web e IDENTIC cu WhatsApp/Telegram (același pipeline:
triaj nano + agent mini), guvernat de cost guard-ul existent (G2c) și de cache-ul semantic (40-60%
opresc înainte de LLM). Costul marginal al canalului = doar conexiuni SSE deschise (memorie/socket pe
proces API) — mărginit de rate limit (IP+visitor) și de heartbeat-ul care detectează și închide
conexiunile moarte.
