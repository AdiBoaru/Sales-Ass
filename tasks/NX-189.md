## Goal
Fă fațetele hard să **participe în retrieval** prin filtrare SQL **tri-state**, livrată **incremental, per fațetă, întâi în shadow**. NU e „ultimul pas de optimizare": e treapta de **candidate-recall** care deblochează enforcement-ul (NX-188) pe fiecare fațetă. Ordinea per fațetă: `NX-187 shadow/recall → NX-189 tri-state shadow → paritate + recall OK → NX-188 enforce`.

## Business Context
Match Gate post-retrieval (NX-187) vede doar pool-ul (`MAX_SEARCH_POOL=24`, models.py:138). Un produs conform aflat pe poziția 25 e invizibil → enforce fără participarea fațetei în retrieval = false-negative („nu există exact" când există). Mutarea fațetei în SQL aduce candidații conformi ÎN pool. Optimizarea de performanță e efect secundar, nu scopul.

## Semantica tri-state (obligatorie)
Un `WHERE fragrance_free = true` naiv elimină NULL-urile → rupe paritatea cu Match Gate (care păstrează UNKNOWN ca alternativă). SQL-ul trebuie să păstreze UNKNOWN:
- două pool-uri per fațetă enforce-uită: **MATCH** (`= true`) și **UNKNOWN** (`IS NULL` / atribut absent), conform politicii NX-188; **MISMATCH** (`= false`) exclus;
- **paritate pe `exact/alternatives/rejected`** cu evaluarea in-memory (NX-187), nu doar aceeași listă totală;
- **test dedicat** pentru valori NULL / atribute absente.

## Technical Description
- Registru (NX-186) → expresie SQL SIGURĂ per fațetă (fără interpolare de output LLM; cheile din allowlist).
- Doar fațete cu coverage peste prag (NX-186) + recall justificat (NX-187); tipuri stabile întâi (bool/number/enum), list/text ulterior.
- **Livrare per fațetă în SHADOW întâi:** dual-run (SQL tri-state vs in-memory) → raport paritate; abia apoi fațeta devine eligibilă pentru enforce în NX-188.
- Toate query-urile noi: `WHERE business_id = $1` explicit + test izolare.
- Degradare: fațetă imatură rămâne pe calea in-memory, fără regresie.
