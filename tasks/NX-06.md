# NX-06 — CTWA referral → atribuire ads
**Owner:** S · **Faza:** P2 · **Zi/Ord:** după T062 (motor proactiv/window) · **Branch:** `feat/NX-06-ctwa-referral` · **Complexitate:** M · **Estimare:** 6h

## Goal
Când un client intră în conversație dintr-o reclamă Click-to-WhatsApp (CTWA),
Meta atașează un bloc `referral` pe PRIMUL mesaj (source_id, ad_id, headline,
ctwa_clid). Taskul îl parsează la margine (webhook, fără DB), îl propagă pe
envelope-ul neutru, îl scrie ca eveniment `ctwa_entry` în `analytics_events`
și marchează fereastra gratuită Meta de **72h** (vs 24h normal) pe conversație.
Așa agenția poate raporta per campanie ce intrări/conversii vin din ce reclamă —
bucla de atribuire ads, capătul de sus al pâlniei (orders → ref_code e capătul
de jos, F2-2).

## Business Context
Clientul plătește reclame Meta care duc direct în WhatsApp. Fără atribuire,
agenția nu poate spune „reclama X a adus 40 de conversații și 3 comenzi" — adică
nu poate justifica bugetul de ads, nici optimiza. `referral.ad_id` + `ctwa_clid`
sunt exact ancora de atribuire. Bonus operațional: intrarea din CTWA deschide o
fereastră de mesagerie gratuită de 72h (nu 24h), deci proactivul (follow-up,
back-in-stock) poate trimite mesaje normale, fără cost de template, în acel
interval — dacă o știe.

## Technical Description

### Unde apare `referral` în payload-ul Meta
Blocul vine pe MESAJ (nu pe `value`), pe primul mesaj al unei conversații pornite
din reclamă:
```jsonc
// entry[].changes[].value.messages[0]
{
  "from": "40712345678", "id": "wamid.X", "timestamp": "...", "type": "text",
  "text": {"body": "Bună, despre crema din reclamă"},
  "referral": {
    "source_url": "https://fb.me/...",
    "source_id": "120209...",            // ad set / campaign id
    "source_type": "ad",
    "headline": "Cremă hidratantă -20%",
    "body": "...",
    "media_type": "image",
    "ctwa_clid": "ARBxc...."             // click id, ancora de atribuire
  }
}
```
(Meta mai trimite uneori `referral` și pe `value.referral` la `referral`-only
events — parserul citește ambele, prioritate pe cel de pe mesaj.)

### 1. Parser la margine (`src/webhook/meta.py`) — fără DB
`_extract_body` rămâne; adăugăm un `_extract_referral(msg, value)` care întoarce
un dict NORMALIZAT (doar câmpurile de care avem nevoie, nu payload brut redundant):
```python
_REFERRAL_FIELDS = ("source_id", "ad_id", "source_type", "headline", "ctwa_clid", "source_url")

def _extract_referral(msg: dict, value: dict) -> dict | None:
    ref = msg.get("referral") or value.get("referral") or {}
    if not ref:
        return None
    out = {k: ref.get(k) for k in _REFERRAL_FIELDS if ref.get(k)}
    # Meta nu trimite mereu un `ad_id` distinct de `source_id`; normalizăm.
    if "ad_id" not in out and out.get("source_id") and ref.get("source_type") == "ad":
        out["ad_id"] = out["source_id"]
    return out or None
```
`parse_webhook` umple noul câmp `referral` pe `InboundEvent`. Parsarea rămâne
defensivă (chei lipsă → None, nu crapă).

### 2. Envelope neutru (`src/channels/base.py`)
`InboundEvent` capătă `referral: dict[str, Any] | None = None` (serializabil JSON,
intră automat în `to_dict()`). Câmp opțional → canalele care nu au CTWA (Telegram)
îl lasă None. Marginea rămâne singurul loc care știe de formatul Meta (P + NX-60).

### 3. Worker: scrie eveniment + marchează fereastra 72h (`src/worker/processor.py`)
În `handle_turn`, DUPĂ `get_or_create_conversation` + `touch_last_inbound` și
ÎNAINTE de pipeline, un helper determinist (cod pur, NU LLM):
```python
async def _record_ctwa_entry(conn, business_id, conv_id, contact_id, event, ctx) -> None:
    ref = event.get("referral")
    if not ref:
        return
    # eveniment de atribuire (append-only, P10). FĂRĂ corpul mesajului, FĂRĂ telefon (P12).
    ctx.emit("ctwa_entry", ad_id=ref.get("ad_id"), source_id=ref.get("source_id"),
             headline=ref.get("headline"), ctwa_clid=ref.get("ctwa_clid"),
             source_type=ref.get("source_type"))
    # fereastra gratuită de 72h (vs 24h): tracker dedicat pe conversație, NU flag în state.
    await mark_ctwa_window(conn, business_id, conv_id)   # set ctwa_window_until = now()+72h
```
`ctx.emit("ctwa_entry", ...)` acumulează evenimentul; `_persist_events` (deja
existent) îl scrie în `analytics_events` la finalul turului, best-effort. Nu
adăugăm un INSERT separat — folosim canalul de evenimente existent.

`mark_ctwa_window` = query nou în `src/db/queries/conversations.py`:
```python
async def mark_ctwa_window(conn, business_id, conversation_id, *, hours: int = 72) -> None:
    await conn.execute(
        """update conversations
              set ctwa_window_until = now() + make_interval(hours => $3)
            where business_id = $1 and id = $2""",
        business_id, conversation_id, hours)
```
NU atinge `state`/`state_version` (ca `set_handoff`/`touch_last_inbound`) → zero
conflict cu patch-ul Sender-ului din același tur.

### 4. Fereastra liberă efectivă (helper SQL, citit de proactiv)
`in_24h_window(conv)` există deja. Adăugăm `in_free_window(conv)` care e adevărat
dacă suntem în 24h normal SAU în fereastra CTWA de 72h:
```sql
create or replace function in_free_window(conv conversations) returns boolean
language sql stable as $$
  select (conv.last_inbound_at  is not null and conv.last_inbound_at  > now() - interval '24 hours')
      or (conv.ctwa_window_until is not null and conv.ctwa_window_until > now());
$$;
```
Motorul proactiv (T062/T201, încă de scris) îl va folosi în locul lui
`in_24h_window` pentru a decide „mesaj normal vs template". NX-06 livrează helper-ul
+ tracker-ul; consumul lui proactiv e în taskul de proactiv (dependență, vezi mai jos).

### 5. Raport per campanie
Sursa = `analytics_events(event_type='ctwa_entry')` filtrat pe `business_id`,
agregat pe `properties->>'ad_id'`. NX-06 livrează:
- un view SQL `vw_ctwa_attribution` (read, tenant-filtrabil) — vezi Database Changes;
- joncțiunea cu conversii: `LEFT JOIN orders` pe contactul/conversația care a avut
  un `ctwa_entry`, ca raportul să arate intrări + comenzi atribuite per `ad_id`.
Dashboard-ul/Metabase citește view-ul; NU construim UI aici (out of scope).

### Câmpuri TurnContext scrise aici
Taskul scrie DOAR `ctx.events` (via `ctx.emit("ctwa_entry", ...)`) — owner: stagiile
emit, runner/processor scrie (P10). NU atinge `route`/`retrieval`/`reply`/`state`.
Fereastra 72h se scrie pe `conversations` (coloană dedicată), nu pe TurnContext.

### Estimare > 4h → [PROPUNERE] spargere
Seed-ul cere explicit „>4h → propune split parser vs window/raport". Umbrela NX-06
rămâne; livrare în două felii independente pe fișiere:

**[PROPUNERE] NX-06a — Parser referral + envelope + eveniment (S, ~2.5h)**

  `_record_ctwa_entry` în processor cu `ctx.emit("ctwa_entry")`.
- DoD: payload CTWA simulat → eveniment `ctwa_entry` cu `ad_id` corect în

- DoD: după un `ctwa_entry`, `in_free_window(conv)` = true 72h; view-ul arată o linie
  per `ad_id` cu nr. intrări (+ comenzi).

## Database Changes
Migrare `docs/00X_ctwa_window.sql` (aplicată cu rol admin, ca 003/004):
```sql
-- 1. tracker fereastră CTWA pe conversație (nu flag în state)
alter table conversations add column if not exists ctwa_window_until timestamptz;

-- 2. fereastra liberă efectivă = 24h normal SAU 72h CTWA
create or replace function in_free_window(conv conversations) returns boolean
language sql stable as $$
  select (conv.last_inbound_at  is not null and conv.last_inbound_at  > now() - interval '24 hours')
      or (conv.ctwa_window_until is not null and conv.ctwa_window_until > now());
$$;

-- 3. raport per campanie (read-only); filtrat pe business_id la interogare
create or replace view vw_ctwa_attribution as
select e.business_id,
       e.properties->>'ad_id'      as ad_id,
       e.properties->>'source_id'  as source_id,
       e.properties->>'headline'   as headline,
       count(*)                    as entries,
       count(distinct e.conversation_id) as conversations,
       date_trunc('day', e.created_at)   as day
from analytics_events e
where e.event_type = 'ctwa_entry'
group by 1,2,3,4,7;
```

UPDATE pe `conversations`. `mark_ctwa_window` folosește UPDATE existent. View-ul
e citit de admin/dashboard (nu de bot_runtime în runtime). Toate query-urile noi
de runtime conțin `business_id = $1`. (`faqs/semantic_cache/wa_templates` nu sunt
atinse → nu e nevoie de `locale` aici.)

## Test Cases
**Happy Path:**
1. `parse_webhook` cu mesaj ce conține `referral{source_id, ad_id, headline, ctwa_clid}`
   → `InboundEvent.referral` populat, serializabil prin `to_dict()`.
2. `_record_ctwa_entry` pe event cu referral → `ctx.emit("ctwa_entry")` cu `ad_id`
   așteptat + `mark_ctwa_window` apelat o dată; `_persist_events` (mock) primește evenimentul.

**Edge Cases:**
1. `referral` doar pe `value` (referral-only event), nu pe mesaj → parsat din `value.referral`.
2. `referral` fără `ad_id` dar cu `source_id` + `source_type='ad'` → `ad_id` normalizat = `source_id`.
3. Al doilea mesaj CTWA în aceeași conversație → re-armează fereastra (nou `ctwa_window_until`),
   un al doilea `ctwa_entry` (raportul numără 2 intrări) — comportament v1 acceptat.

**Failure Cases:**
1. `referral` gol/`{}` / malformat → `_extract_referral` întoarce None, zero eveniment,
   turul continuă normal (nicio excepție).
2. `mark_ctwa_window` aruncă (DB jos) → prins, logat, turul răspunde oricum (best-effort,
   ca `_persist_events`); evenimentul rămâne în `ctx.events` pentru persistare.
3. Telegram/canal fără CTWA (`referral=None`) → `_record_ctwa_entry` no-op, pipeline neatins.

> Teste fără apeluri externe: parserul e funcție pură pe dict; processorul cu `insert_events`
> mock + conexiune throwaway pentru `mark_ctwa_window`/`in_free_window`. ZERO OpenAI, ZERO Meta
> real în CI (P2 — taskul oricum nu cheamă niciun LLM).
