# NX-05 — Quality rating monitor + pauză automată pe proactiv
**Owner:** S · **Faza:** P1 · **Ord:** O1 · **Branch:** feat/NX-05-quality-monitor · **Complexitate:** S · **Estimare:** 6h

## Goal
Dacă Meta degradează quality rating-ul unui număr, afli în minute și mesajele proactive se opresc automat înainte ca numărul să fie restricționat sau banat.

## Business Context
Numărul de WhatsApp al clientului e activul lui comercial. Un ban = produsul mort pentru acel client + discuție foarte neplăcută pentru agenție. Rating-ul scade aproape întotdeauna din cauza mesajelor PROACTIVE (template-uri percepute ca spam), deci pauza automată pe proactiv e exact frâna corectă — conversațiile inbound continuă.

## Technical Description
Două surse, ambele implementate: (1) webhook field `phone_number_quality_update` (vine în același endpoint Meta — adaugi handler pe `entry[].changes[].field`); (2) polling zilnic Graph API pe phone number (`quality_rating`, `messaging_limit_tier`) ca plasă pentru evenimente pierdute. Starea se scrie în `channels.metadata` (jsonb: rating, tier, updated_at) + eveniment `analytics_events('quality_rating_changed')`. Reguli: rating `YELLOW` → alertă Slack; `RED`/`FLAGGED` → `businesses.settings.proactive_paused=true` + alertă cu instrucțiuni din runbook; revenire la `GREEN` → NU se reactivează automat (decizie umană — un singur flag de scos manual, dar documentat).

## Principii CLAUDE.md aplicabile
P3 (protejezi resursa limitată a clientului) · P6 (frână automată, repornire umană).

## Implementation Steps
1. Handler nou în webhook pentru `phone_number_quality_update` → update channels.metadata + event.
2. `src/jobs/quality_poll.py` (zilnic): GET pe Graph API per număr activ; diff → același update path.
3. Gardă în motorul proactiv (T201): verifică `proactive_paused` înainte de orice trimitere de template.
4. Alerte Slack: payload cu business, număr, rating vechi→nou, link runbook.
5. Secțiune runbook: cauze tipice de degradare + pașii Meta de remediere + cum repornești proactivul.
