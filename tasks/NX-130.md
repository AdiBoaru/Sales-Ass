# NX-130 — Legarea comenzilor de client prin external_customer_ref (login passthrough, half 2)

**Owner:** Claude · **Faza:** P2 · **Track/Wave:** Web-identity (3/3) · **Branch:** `feat/NX-130-orders-customer-ref` · **Complexitate:** M · **Estimare:** 4h

## Goal
Chiar cu o identitate web 100% de încredere (NX-129), `check_order` tot întoarce gol, fiindcă azi comenzile reale ale unui client nu se pot lega de el: `orders.contact_id` se populează DOAR prin atribuire de checkout-link (`src/webhook/orders.py:73-79`), iar `orders` n-are nicio coloană de client (`docs/schema_v2_production.sql:557-574`). Acest task adaugă `orders.external_customer_ref`, îl cară prin ingestie și-l folosește în `check_order` pentru clientul verificat — închizând „logat → îți caut comanda". E jumătatea de date (gaura reală a Opțiunii A).

## Business Context
Fără asta, login passthrough-ul (NX-129) e o ușă care se deschide spre o cameră goală. Legarea client↔comenzi face self-service real pe status/retur — scade tichetele de suport și crește încrederea, exact valoarea pe care o vindem managed. Generic: orice eshop care trimite un id de client în comenzi capătă verificare de comandă pe web, zero cod per-client.

## Technical Description

### Coloana + indexul
Migrare ordonată `docs/018_orders_external_customer_ref.sql` (runner `scripts/migrate.py`, NX-123):
```sql
alter table orders add column if not exists external_customer_ref text;
create index if not exists idx_orders_customer_ref
  on orders(business_id, external_customer_ref)
  where external_customer_ref is not null;
```
Grant: `bot_runtime` are deja SELECT/INSERT/UPDATE pe `orders` (coloană nouă acoperită). RLS neschimbat (politicile sunt pe `business_id`).

### PII — decizie de design (P12)
`external_customer_ref` = id-ul **OPAC** de client din eshop (ex. `cust_8842`), care NU e PII → poate sta în `orders`. **Dacă** singura cheie stabilă a magazinului e email/telefon → stocăm `sha256(email)` (coloană `external_customer_ref` ținând hash-ul, ca `channel_identities.external_id_hash`), NICIODATĂ email brut — `orders` rămâne fără PII (CONTRIBUTING #3, P12). `customer_ref` din JWT-ul NX-129 (`sub`) trebuie să fie ACEEAȘI cheie (id opac, sau pre-imaginea care se hash-uiește la fel). Constrângerea asta e contractul dintre half 1 și half 2.

### Ingestia
- `OrderIn` (`src/webhook/orders.py:48-57`): câmp opțional `customer_ref: str | None`.
- `process_order`/`upsert_order` (`src/db/queries/commerce.py:74-`): persistă `external_customer_ref` (la INSERT; la re-livrare NU-l downgradăm la NULL, ca atribuirea/contactul). Adaptorul platformă (Shopify/Woo → neutru) rămâne out of scope, ca în F2-2 — v1 ingestă forma neutră cu câmpul nou.

### Lookup-ul
- `get_orders_status` (`src/db/queries/commerce.py:221-`): parametru nou opțional `external_customer_ref: str | None` → `and ($N::text is null or o.external_customer_ref = $N)`. Mereu scoped pe `business_id`.
- `check_order_tool` (`src/tools/orders_tools.py`): cheia de căutare vine din CONTEXT, nu din args (izolare, ca azi):
  - canal identificat clasic (WhatsApp/Telegram) → `contact_id=ctx.contact.id` (neschimbat);
  - web VERIFICAT (NX-129) → `external_customer_ref = <customer_ref din sesiunea verificată>` (citit din `ctx`, NU din `order_ref`/model).
  `order_ref` rămâne doar îngustare. Un user web NU poate pasa un `customer_ref` arbitrar — vine din JWT-ul verificat server-side (oglindește principiul existent „scoped din ctx, nu din args").
- Opțional: la identificare (NX-129) backfill `orders.contact_id` pentru comenzile cu `external_customer_ref = customer_ref` → unifică și calea pe `contact_id`. Marcat opțional; nu blochează.

### Mesajul
Cu NX-128 deja în main: web verificat fără comenzi → `_no_orders_view` (acum adevărat: „nu găsesc comenzi pe contul tău"), nu mesajul de login.

## Principii CLAUDE.md aplicabile
- **P12 / PII:** `orders` rămâne fără PII — `external_customer_ref` e id opac (sau hash). Email/telefon brut NU intră în `orders`.
- **P7 (business_id pe tot):** coloană + index + query toate scoped pe `business_id`; indexul e compus `(business_id, external_customer_ref)`.
- **Izolare dură:** cheia de lookup pe web vine din sesiunea verificată (server-side), niciodată din argumentele tool-ului/model — un client nu poate cere comenzile altuia ghicind un `customer_ref`.
- **P2 (LLM doar la 2 puncte):** zero LLM — DDL + SQL + wiring.
- **Idempotență:** `upsert_order` rămâne idempotent pe `(business_id, external_id)`; câmpul nou nu se downgradează la re-livrare.

## Test Cases
**Happy Path**
- Ingestie comandă cu `customer_ref="cust_42"` → `orders.external_customer_ref="cust_42"`. Web verificat ca `cust_42` → `check_order` întoarce acea comandă (status/AWB/total).
- WhatsApp, contact cu comenzi pe `contact_id` → neschimbat, le găsește.

**Edge Cases**
- Comandă fără `customer_ref` (absent în payload) → `external_customer_ref` NULL; nu apare la căutarea pe `customer_ref` (corect).
- Re-livrarea aceleiași comenzi (idempotent) cu `customer_ref` absent → nu suprascrie valoarea existentă cu NULL.
- `customer_ref` derivat din email → stocat ca hash; match-ul se face pe hash.

**Failure Cases**
- Web verificat ca `cust_42` cere `order_ref` al unei comenzi a lui `cust_99` → `not_found` (izolare: `external_customer_ref` nu se potrivește), identic cu inexistent — fără oracol de existență.
- Model/tool args conțin un `customer_ref` injectat → IGNORAT; se folosește doar cel din sesiunea verificată.
- Migrarea rulată de două ori → `if not exists` o face idempotentă.
