# R2 — Carusel de produse pe Telegram (callback_query + editMessageMedia)
**Owner:** S · **Faza:** P2 · **Zi/Ord:** după hardening worker · **Branch:** `feat/R2-telegram-carousel` · **Complexitate:** M · **Estimare:** 4h

## Goal
Înlocuiește lista compactă (W1 v1) cu un **carusel**: UN singur card de produs
(poză + nume + preț) cu butoane inline `◀ 🛒 ▶`. `◀`/`▶` navighează prin produsele
recomandate editând ACELAȘI mesaj (`editMessageMedia`), `🛒` deschide pagina
produsului. Există ca să aducă afișarea pe Telegram la nivelul „pro" pe care îl
folosesc boții reali, fără retry storm-ul pozelor mari individuale (contextul R2).
Cheia arhitecturală: navigarea e un **drum de inbound NOU, NON-LLM** — `callback_query`
nu trece prin triaj/agent, e UI deterministă.

## Business Context
Prezentarea produselor e ultimul pas înainte de click → comandă. Un carusel curat
(o bulă, navigabil) crește rata de parcurgere a recomandărilor față de o listă de
linkuri. Canal de TEST (Telegram, `@solechat_bot`) → iterăm rapid pe UX înainte de
a-l porta pe WhatsApp (Multi-Product, alt task).

## Technical Description

### 1. `callback_query` → envelope neutru `kind="callback"` (ingestie)
Poller-ul ignoră acum `callback_query` ([poller.py:33](../src/channels/telegram/poller.py#L33)).
Îl adăugăm ca un al doilea tip de update mapat pe stream-ul `inbound`, pe modelul
deja existent al statusurilor (`kind="status"` în consumer):
- `answerCallbackQuery(callback_id)` **imediat în poller** (oprește spinner-ul
  Telegram, ~15s). E un ACK de transport, nu un mesaj către user → aparține marginii
  de ingestie, NU încalcă „un singur punct de ieșire" (acela e pentru mesaje).
- enqueue envelope cu `kind="callback"`, purtând: `channel_kind`, `channel_account_id`
  (bot id), `sender_external_id` = `callback.message.chat.id`, `provider_msg_id` =
  `callback.id` (idempotență), `card_message_id` = `callback.message.message_id`
  (ce mesaj edităm), `body` = `callback.data`.

**Protocol de `callback_data` — navigare STATELESS:** butoanele encodează indexul
ȚINTĂ, nu acțiunea relativă. Cardul care arată produsul `i` are:
`◀ → "car:nav:{i-1}"`, `▶ → "car:nav:{i+1}"` (omis dacă în afara limitelor).
Astfel handler-ul nu trebuie să persiste indexul curent — vine în buton. Produsele
vin din `state.displayed_products` (ordinea = setul de recomandare din W1). Prefix
`car:` ca să distingem de alte callback-uri viitoare.

### 2. Handler determinist în worker (NON-LLM)
`consumer.process_event` rutează azi `kind in {message, status}`. Adăugăm
`kind == "callback"` → `handle_callback` (modul nou, **fără** `run_pipeline`,
fără triaj/agent):
1. `tenant_conn(business_id)` (după `resolve_channel`, ca restul).
2. parse `body` → `("nav", idx)`; validează `0 <= idx < len(displayed_products)`
   (din `conversations.state`). Index invalid / set gol → no-op logat (cardul a
   expirat din state) — fără excepție, fără mesaj nou.
3. `product_id = displayed_products[idx].product_id` → **re-fetch** câmpurile de
   card (name, price, product_url, image) printr-un query determinist
   (`get_product_card(conn, business_id, product_id)`) — preț/stoc proaspăt,
   `state` rămâne ref-uri (principiul 8).
4. emite în `outbox` o acțiune `type="edit_media"` cu `{chat_id, card_message_id,
   image_url, caption, keyboard}` (keyboard recalculat pentru noul index). **NU**
   un mesaj nou — editează cardul existent.

### 3. Card inițial (Sender)
Când agentul întoarce produse (W1), pe Telegram emitem un reply `kind="carousel"`
(nou, pe lângă `products` compact din W1 — coexistă; `products` rămâne fallback).
Sender-ul scrie `outbox` `type="carousel"` cu primul produs + `displayed_products`
persistat în `state` (verifică în implementare că W1 chiar persistă
`displayed_products`; dacă nu, R2 îl adaugă — e necesar pentru navigare).
Dispatcher-ul → `sendPhoto(product[0], caption, keyboard ◀🛒▶)`.

### 4. `TelegramClient` — metode noi (httpx injectabil, mock în teste)
- `send_carousel_card(account_id, to, image_url, caption, index, total, url)` →
  `sendPhoto` cu `reply_markup` (◀ callback / 🛒 **url-button** spre `product_url`
  / ▶ callback). `🛒` ca url-button = reutilizează abordarea W1 (zero backend de
  coș). Întoarce `message_id`.
- `edit_message_media(account_id, chat_id, message_id, image_url, caption, keyboard)`
  → `editMessageMedia` cu `InputMediaPhoto` + `reply_markup`.
- `answer_callback_query(callback_id)` → oprește spinner-ul.

### 5. Dispatcher
`dispatch_row` rutează azi `ptype in ("text","products")`
([dispatcher.py:47](../src/worker/dispatcher.py#L47)). Adăugăm `"carousel"` →
`send_carousel_card` și `"edit_media"` → `edit_message_media`. Pe un canal fără
suport (ex. WhatsApp) → degradare la `send_text` cu lead-in-ul (principiul 6).
Idempotența pe `outbox.idempotency_key` rămâne; pentru `edit_media` cheia include
`(card_message_id, idx)` ca o re-livrare să nu strice nimic (edit idempotent).

## Principii CLAUDE.md aplicabile
- **P5 (un singur punct de ieșire):** caruselul ȘI editările ies DOAR prin
  `outbox` → dispatcher. Excepția controlată: `answerCallbackQuery` (ACK de
  transport, nu mesaj) în poller — documentat.
- **P8 (state = ref-uri):** `displayed_products` rămâne `{product_id, name, price}`;
  câmpurile de card se re-fetch-uiesc la navigare, nu se umflă state-ul.
- **P2/P7 (business_id pe tot):** `get_product_card` și orice citire în handler au
  `where business_id = $1`; `handle_callback` rulează pe `tenant_conn`.
- **P10 (LLM doar triaj+agent):** navigarea e 100% deterministă — ZERO LLM pe
  `callback`. Cardul inițial refolosește turul agentului existent (W1).

## Implementation Steps
1. `poller._to_event` / nou `_to_callback_event`: mapează `callback_query` →
   envelope `kind="callback"`; `answer_callback_query` imediat în `poll_once`.
2. `TelegramClient`: `send_carousel_card`, `edit_message_media`,
   `answer_callback_query`.
3. `db/queries/catalog.py`: `get_product_card(conn, business_id, product_id)` →
   {name, price, product_url, image} (1 produs, câmpuri de card).
4. `worker/callback.py`: `handle_callback` (parse, bounds, re-fetch, enqueue
   `edit_media`). `consumer.process_event`: rutează `kind=="callback"`.
5. Sender (`processor`): reply `kind="carousel"` → `outbox type="carousel"`;
   asigură `displayed_products` în `state`.
6. `dispatcher.dispatch_row`: rutează `carousel` + `edit_media`.
7. Helper comun de construit keyboard-ul (◀🛒▶ cu callback_data pe index) — o
   singură sursă, folosit de card inițial + edit.
8. Teste (vezi mai jos); test live manual pe `@solechat_bot`.

## Test Cases
**Happy Path:**
1. `handle_callback` cu `car:nav:1` pe un state cu 3 produse → enqueue `edit_media`
   pentru produsul index 1, keyboard cu ◀ și ▶.
2. `_to_callback_event` mapează un update `callback_query` real → envelope
   `kind="callback"` cu `card_message_id` + `data` corecte.

**Edge Cases:**
1. Index la capăt: `car:nav:0` → fără buton ◀; `car:nav:{last}` → fără ▶
   (wrap-around NU — clamp la capete, deterministic).
2. Pool de produse cu unul fără `image` → fallback la poză placeholder `.png`
   (ca W1), caruselul nu crapă.

**Failure Cases:**
1. `callback_data` cu index în afara `len(displayed_products)` (state schimbat
   între timp) → no-op + log, `answerCallbackQuery` tot trimis.
2. `editMessageMedia` întoarce eroare Telegram (mesaj prea vechi / șters) →
   `mark_failed` + backoff (ca orice outbox), bucla nu moare.
3. `callback` duplicat (Telegram re-livrează) → idempotent pe
   `outbox.idempotency_key` = `(card_message_id, idx)`, fără dublu-edit.

> Teste fără apeluri reale: `TelegramClient` primește httpx mock; `handle_callback`
> testat pe un `state` fixture. ZERO apeluri Telegram/LLM în CI (integration opțional
> pe `get_product_card`).
