# NX-61 — Telegram inbound via long polling (canal de TEST, fără HTTPS)
**Owner:** S · **Faza:** MVP (test multi-canal) · **Ord:** după NX-60 · **Branch:** feat/NX-61-telegram-inbound · **Complexitate:** M · **Estimare:** 5h

## Goal
Primește mesaje de pe Telegram și le bagă în ACELAȘI pipeline ca WhatsApp, fără
webhook public / HTTPS / verificare de semnătură — prin **long polling**
(`getUpdates`). Suficient ca să testăm conversația direct, pe VPS, cu un bot real.

## Business Context
Calea de test cea mai ieftină: un bot Telegram (BotFather, gratuit, instant) +
un proces care face long polling pe VPS. Zero birocrație Meta, zero tunel, zero
TLS. WhatsApp rămâne canalul primar de producție; Telegram e doar pentru iterare
rapidă pe comportamentul botului. (Webhook mode = opțiune de prod ulterioară.)

## Technical Description
Telegram Bot API: `getUpdates(offset, timeout=30)` întoarce update-uri noi;
confirmarea unui update = trimiterea următorului `offset = max(update_id)+1`.
Long-poll-ul ține conexiunea deschisă până la 30s → aproape realtime, fără webhook.

`src/channels/telegram/poller.py`: buclă care, pentru canalul Telegram al
tenantului, cheamă `getUpdates` și pentru fiecare `message` cu `text`:
construiește un `InboundEvent` NEUTRU (NX-60): `channel_kind='telegram'`,
`channel_account_id` = `provider_account_id` al canalului (bot id),
`sender_external_id = str(message.chat.id)`, `sender_name = from.first_name`,
`provider_msg_id = str(message.message_id)`, `body = text`, `payload = update`.
`enqueue_inbound` pe același stream → consumer-ul existent rezolvă tenantul
(`resolve_channel('telegram', bot_id)`) și rulează pipeline-ul. Echo iese prin
outbox; dispatcher-ul îl trimite cu TelegramClient (NX-62).

Offset-ul se ține în Redis (`tg:offset:{channel_id}`) ca să nu reprocesăm la
restart; `inbound_dedupe` (NX-51) rămâne plasa durabilă pe `(business, provider_msg_id)`.
Un poller per canal Telegram (pentru demo: unul singur, tokenul din config).
`src/channels/telegram/client.py`: wrapper httpx peste Bot API (`getUpdates`,
`send_message` pt NX-62).

## Implementation Steps
1. `src/channels/telegram/client.py`: `TelegramClient` (httpx injectabil) —
   `get_updates(offset, timeout)`; bază URL `https://api.telegram.org/bot{token}`.
2. `src/channels/telegram/poller.py`: încarcă canalul Telegram (admin_conn →
   `resolve`/listare), buclă getUpdates cu offset Redis, mapează update→InboundEvent,
   `enqueue_inbound`. Ignoră update-uri ne-text (deocamdată).
3. Filtrare: doar `update.message.text` (comenzi `/start` → tratate ca text simplu).
4. `docker-compose.yml`: serviciu `telegram-poller` (`python -m src.channels.telegram.poller`).
5. Test: parser update→InboundEvent (unit, fixture JSON Telegram); offset advance
   (mock client întoarce 2 update-uri → 2 enqueue, offset = last+1).
