# NX-79 — Tool-uri agent — cart_add + reorder
**Owner:** S · **Faza:** P1 · **Zi/Ord:** după G7 (toolset agent) · **Branch:** `feat/NX-79-cart-reorder` · **Complexitate:** M · **Estimare:** 4h

## Goal
Completează toolset-ul de vânzare cu cele două unelte care lipsesc din cele 12 documentate:
`cart_add(product_id, variant_id)` — acumulează un coș în `conversations.state` (ref-uri, nu
obiecte) ca pas intermediar înainte de `checkout_link`, și `reorder()` — citește ultimele
comenzi ale contactului și le propune ca recomandare de re-comandă. Ambele sunt cod determinist
scoped pe `business_id`, înregistrate în `TOOL_REGISTRY` + schemă OpenAI, sub capul dur de 3
apeluri/tur. Închid bucla „client vrea produsul X" → coș → checkout fără ca modelul să țină
starea coșului în prompt.

## Business Context
`checkout_link` cere deja un coș complet într-un singur apel — bun când clientul decide pe loc,
dar nefiresc într-o conversație unde adaugă produse unul câte unul („și crema aia", „pune și
serul"). `cart_add` lasă agentul să construiască coșul în pași, persistat în state, apoi
`checkout_link` îl transformă în link atribuibil. `reorder` exploatează cel mai ieftin lead:
un client care a mai cumpărat — o re-comandă cu un singur mesaj e conversie cu cost de retrieval
zero (fără embeddings, fără agent greu). Ambele cresc conversia pe coada caldă a bazei.

## Technical Description

### Starea actuală (citită din cod)
`src/tools/commerce_tools.py` are DOAR `checkout_link_tool` (validează produse contra catalog,
scrie `checkout_links`, întoarce `ToolResult` cu `links`+`prices` grounded). `src/tools/base.py`:
`TOOL_REGISTRY` + `register(name)`, `_SALES_TOOLS = (search_products, get_product_details,
compare_products, checkout_link)`, `_ORDER_TOOLS = (check_order,)`. `tool_definitions.py` are 5
scheme. `src/db/queries/commerce.py` are deja `get_orders_status(conn, business_id, *,
external_id, contact_id, limit)` (SELECT cu lateral pe `order_items`+`shipments`, filtrat dur pe
`business_id` ȘI `contact_id`) — `reorder` o REFOLOSEȘTE, nu scrie SQL nou. `ConversationState`
(`src/models.py`) are `displayed_products`, `active_search`, `pending_question`, `constraints` —
NU are `cart`.

### Problema de scriere a state-ului (cheia designului)
Tool-urile NU pot scrie `ctx.state` ca sursă persistată: state-ul se persistă în `processor.py`
(Sender), care pleacă de la `conv["state"]` din DB, NU de la `ctx.state`. Singura mutație de
state azi (`displayed_products`) curge prin `ctx.reply.products`. Deci `cart_add` are nevoie de
un canal explicit tool → state-patch. Adăugăm un câmp pe `ToolResult` și un purtător pe `ctx`:

```python
# src/tools/base.py — ToolResult
state_patch: dict[str, Any] = field(default_factory=dict)  # mutație de state cerută de tool (cart_add)

# src/models.py — TurnContext
# owner: stagiul Agent (acumulat din ToolResult.state_patch). Mutații de state cerute de tool-uri
# (ex. cart_add → {"cart": [...]}). Processor-ul îl merge-uiește în new_state la scriere.
state_patch: dict[str, Any] = field(default_factory=dict)
```

În `agent.py::execute` (callback-ul buclei) acumulăm: `ctx.state_patch.update(result.state_patch)`.
În `processor.py`, la construirea `new_state`, după branch-ul `displayed_products`:
`if ctx.state_patch: new_state = {**new_state, **ctx.state_patch}`. **Owner unic** (principiul 3):
`ctx.state_patch` e scris DOAR de stagiul Agent; processor-ul îl citește.

### `cart_add(product_id, variant_id)` — acumulează coșul (ref-uri, principiul 8)
```python
class CartAddArgs(BaseModel):
    product_id: str = Field(min_length=1)
    variant_id: str | None = None
    quantity: int = Field(default=1, ge=1, le=99)

@register("cart_add")
async def cart_add_tool(ctx, deps, args) -> ToolResult:
    a = CartAddArgs(**args)
    # validăm contra catalogului (scoped pe business) — nu acumulăm produse inexistente
    products = await get_products_by_ids(deps.conn, ctx.business.id, [a.product_id], limit=1)
    p = products[0] if products else None
    if p is None or p.get("price") is None:
        return ToolResult(ok=False, error="product_not_found",
                          llm_view="Produsul nu mai e în catalog.")
    # coșul curent din state (ref-uri: id, name, price, variant_id, quantity — NU obiectul complet)
    cart = list((ctx.state.constraints or {}).get("cart") or [])  # vezi nota de stocare mai jos
    # merge pe (product_id, variant_id): dacă există, crește quantity (cap 99); altfel append
    key = (a.product_id, a.variant_id)
    for line in cart:
        if (line["product_id"], line.get("variant_id")) == key:
            line["quantity"] = min(line["quantity"] + a.quantity, 99)
            break
    else:
        cart.append({"product_id": a.product_id, "variant_id": a.variant_id,
                     "name": p["name"], "price": round(float(p["price"]), 2),
                     "quantity": a.quantity})
    cart = cart[:10]  # cap dur la 10 linii (aliniat cu CheckoutArgs.max_length)
    total = round(sum(l["price"] * l["quantity"] for l in cart), 2)
    ctx.emit("cart_updated", lines=len(cart), value=total)
    lines = ", ".join(f"{l['name']} ×{l['quantity']}" for l in cart)
    return ToolResult(
        ok=True,
        products=[p],                       # complet → ctx.retrieval + validator
        prices=[total],                     # totalul coșului grounded → validator (P8)
        state_patch={"cart": cart},         # ref-uri compacte → persistate de processor
        llm_view=f"Coș actualizat ({len(cart)} produse): {lines} | total {total:.2f} lei",
    )
```
**Stocare în state:** coșul stă în `state.cart` (cheie nouă, top-level în jsonb), NU în
`constraints`. Adăugăm `cart: list[dict] = field(default_factory=list)` pe `ConversationState`
+ hidratare defensivă în `from_jsonb` (linii fără `product_id`/`name`/`price`/`quantity` → sărite,
exact ca `displayed_products`). `cart_add` îl citește din `ctx.state.cart`. Bugetul de 8KB rămâne
respectat (10 linii × ~5 câmpuri scalari ≪ 8KB; context builder taie oricum).

### `reorder()` — re-comandă din istoricul contactului
Fără argumente de la model (sau `limit` opțional): `contact_id` vine din `ctx`, NU din args (P7).
Refolosește `get_orders_status` cu `contact_id=ctx.contact.id`, ia ultimele comenzi (limit 3),
extrage liniile (`items`: name, quantity, unit_price) și le întoarce ca vedere compactă +
`prices` grounded (prețurile istorice din comenzi). NU scrie nimic; e citire.
```python
@register("reorder")
async def reorder_tool(ctx, deps, args) -> ToolResult:
    orders = await get_orders_status(deps.conn, ctx.business.id, contact_id=ctx.contact.id, limit=3)
    if not orders:
        return ToolResult(ok=False, error="no_orders",
                          llm_view="Nu găsesc comenzi anterioare pe contul tău.")
    last = orders[0]                          # cea mai recentă (deja order by placed_at desc)
    items = last.get("items") or []
    if not items:
        return ToolResult(ok=False, error="no_items", llm_view="Comanda anterioară n-are produse.")
    prices = [round(float(i["unit_price"]), 2) for i in items if i.get("unit_price") is not None]
    ctx.emit("reorder_suggested", order_id=last["id"], lines=len(items))
    lines = ", ".join(f"{i['name']} ×{i.get('quantity', 1)}" for i in items)
    return ToolResult(ok=True, prices=prices,
                      llm_view=f"Ultima comandă a clientului: {lines}. "
                               f"Sugerează re-comanda acestor produse.")
```
**Atenție grounding:** `reorder` întoarce nume + prețuri din comenzi, NU `products` din catalog
(produsele istorice pot fi inactive). Prețurile intră în `ToolResult.prices` → validatorul le
acceptă. Numele de produse din comandă NU sunt validate contra catalogului de validatorul de
produse — acceptabil: sunt date reale din `orders` ale tenantului, la fel ca vederile de
`check_order`. NU oferim link de checkout direct din `reorder` (clientul poate cere apoi
`cart_add`/`checkout_link`).

### Înregistrare + rutare
`cart_add` și `reorder` intră în `_SALES_TOOLS` (toolset-ul rutei `sales`). Ordinea în
`tool_definitions._SCHEMAS` rămâne stabilă (prefix static → prompt caching, principiul 6 din
tool_definitions). Capul dur de 3 apeluri/tur e impus deja în `run_tool_loop` (adaptor) — nu-l
atingem; doar adăugăm 2 unelte în set.

### Câmpuri TurnContext scrise de acest task
- `ctx.state_patch` (NOU) — owner: stagiul Agent (acumulat din `ToolResult.state_patch`).
- `ctx.retrieval.products` — deja owner Agent (cart_add adaugă produsul validat).
- `ctx.events` — `cart_updated` / `reorder_suggested` (via `ctx.emit`).
NU scriem `ctx.reply` (uneltele nu setează reply), NU `ctx.route`.

## Principii CLAUDE.md aplicabile
- **P2 (LLM doar triaj+agent):** ambele tool-uri sunt cod determinist; ZERO apeluri LLM/embeddings.
- **P3 (un singur owner per câmp):** `ctx.state_patch` e scris DOAR de stagiul Agent; processor-ul
  doar îl citește la merge. Coșul nu e scris din două locuri.
- **P5 (un singur punct de ieșire):** tool-urile NU trimit nimic; mutația de state ajunge la
  persistare prin Sender (processor) ca tot restul, nu printr-un side-channel.
- **P7 (business_id pe tot):** `get_products_by_ids` și `get_orders_status` filtrează pe
  `business_id = $1`; `contact_id`/`product_id` vin din `ctx`, NU din argumentele modelului.
- **P8 (state = ref-uri, nu obiecte):** `state.cart` ține {product_id, variant_id, name, price,
  quantity}, NU obiectul complet de produs. Cap 10 linii.

## Test Cases
> Toate testele folosesc stub-uri DB (fake `conn` cu `get_products_by_ids`/`get_orders_status`
> monkeypatch-uite) + `ctx`/`deps` fabricați. ZERO apeluri reale OpenAI/DB în CI; tool-urile sunt
> cod determinist, deci nici măcar LLM mock nu e necesar pentru ele (doar agent.py folosește
> `ScriptedLLM` dacă se testează integrarea).

**Happy Path:**
1. `cart_add(product_id=P1, quantity=2)` pe catalog cu P1 valid → `ok`, `state_patch["cart"]`
   are o linie `{product_id:P1, name, price, quantity:2}`, `prices=[price*2]`, `cart_updated`
   {lines:1, value} emis.
2. `reorder()` pe contact cu o comandă (2 linii) → `ok`, `prices` = unit_price-urile liniilor,
   `llm_view` listează produsele, `reorder_suggested` {order_id, lines:2} emis.

**Edge Cases:**
1. `cart_add` de două ori pe același (P1, variant=null) → o singură linie cu `quantity` cumulat
   (2+1=3), nu două linii.
2. `cart_add` peste 10 linii distincte → coșul taie la 10 (cap dur), fără excepție.
3. `cart_add` cu `quantity=99` apoi încă unul → `quantity` rămâne 99 (cap `min(..., 99)`).
4. `reorder` cu o comandă fără linii (items=[]) → `ok=False`, `error="no_items"`.

**Failure Cases:**
1. `cart_add(product_id=INVALID)` → `get_products_by_ids` întoarce [] → `ok=False`,
   `error="product_not_found"`, `state_patch` gol (coșul existent neatins).
2. `get_products_by_ids` aruncă (DB jos) → `run_tool` prinde excepția → `ToolResult(ok=False)`,
   turul nu se rupe (principiul 6), `state_patch` gol.
3. `reorder` pe contact fără comenzi → `ok=False`, `error="no_orders"`, fără excepție; agentul
   primește `llm_view` cu mesajul neutru și poate clarifica.
4. Izolare: `cart_add`/`reorder` cu un `ctx.business.id` diferit de produs/comandă → `[]` din DB
   (filtru `business_id = $1`) → `ok=False` (nu scurge date din alt tenant).

## Cost
**$0 LLM/embeddings adăugat.** Ambele tool-uri sunt cod determinist (un SELECT pe catalog, resp.
un SELECT pe orders) — NU cheamă OpenAI/embeddings. Costul marginal e o schemă în plus în
prefixul de tool-calling al agentului (tokeni de schemă, dar prefix STATIC → acoperit de prompt
caching OpenAI, 75-90% discount). Per 1000 mesaje: cost suplimentar neglijabil (sub-$0.01 din
tokenii de schemă cache-uiți). Mărginirea costului: cap dur 3 apeluri/tur (deja în
`run_tool_loop`) + `reorder`/`cart_add` nu declanșează retrieval semantic.
