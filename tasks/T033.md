# T033 — 002: RLS defense-in-depth + app.business_id
**Owner:** S · **Faza:** MVP · **Zi:** Z2 · **Branch:** `feat/T033-sql-rls` · **Complexitate:** M · **Estimare:** 1.5h

## Goal
Plasa de siguranță multi-tenant: RLS pe conv.* și catalog.* legat de GUC-ul de sesiune `app.business_id`. Mecanismul PRIMAR rămâne `WHERE business_id=$1` în cod (P7); RLS-ul transformă o greșeală de query din „datele altui client" în „zero rânduri" — categoric mai bine.

## Business Context
Un singur leak cross-tenant văzut de un client = reputația agenției terminată în piața mică din România. Costul plasei: ~3% overhead pe query. Trade corect.

## Technical Description
```sql
-- politică șablon, repetată per tabel tenant-scoped din conv și catalog:
alter table conv.messages enable row level security;
create policy tenant_isolation on conv.messages
  using (business_id = current_setting('app.business_id', true)::uuid);
-- '... , true' → NULL dacă GUC nesetat → niciun rând (fail closed), nu eroare.
```
Aplici pe: conv.{contacts, channel_identities, messages, conversation_state, short_memory, inbound_dedupe, orders, bookings, outbox, checkout_links, back_in_stock_subscriptions, gdpr_requests} și catalog.{products, product_variants, faq, faq_aliases, faq_alias_candidates, taxonomy, clarification_templates, response_cache, services, locations, knowledge_guides, product_embeddings, review_summaries}. NU pe analytics (botul doar inserează; insertul trece prin WITH CHECK — adaugă `with check (business_id = ...)` pe politicile tabelelor cu INSERT). Rolurile bot_runtime/catalog_sync/dashboard_read NU au bypassrls. Owner-ul migrărilor (postgres) bypass-uiește natural — corect pentru migrări. GUC-ul se setează per conexiune în pool (T044).
