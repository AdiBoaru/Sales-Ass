# T022 — 002: conv.channel_identities + migrare contacts
**Owner:** S · **Faza:** MVP · **Zi:** Z1-2 · **Branch:** `feat/T022-sql-identities` · **Complexitate:** S · **Estimare:** 1.5h

## Goal
Separă PERSOANA (contacts) de IDENTITATEA DE CANAL (telefon WhatsApp, id Telegram). Asta repară defectul structural găsit la audit: în 001, contactul ESTE canalul, deci „același client pe 2 canale" era imposibil. Tot aici se concentrează PII-ul (P12).

## Business Context
Identity resolution e pe roadmap (P2 multi-canal), dar schema trebuie corectată ACUM — migrarea contactelor cu istoric viu ar fi de 10x mai dureroasă.

## Technical Description
```sql
create table conv.channel_identities (
  id uuid primary key default gen_random_uuid(),
  business_id uuid not null references core.businesses(id),
  contact_id uuid not null references conv.contacts(id) on delete cascade,
  channel text not null check (channel in ('whatsapp','telegram','instagram','webchat')),
  external_user_id text not null,        -- E.164 pt WhatsApp; AICI stă PII-ul
  created_at timestamptz not null default now(),
  unique (business_id, channel, external_user_id)
);
create index idx_chid_contact on conv.channel_identities(contact_id);
-- migrare date existente (demo):
insert into conv.channel_identities (business_id, contact_id, channel, external_user_id)
select business_id, id, channel, external_user_id from conv.contacts
on conflict do nothing;
-- contacts: coloanele channel/external_user_id devin deprecated:
-- pas 1 (002): le lași pe loc dar codul nou NU le mai citește;
-- pas 2 (003, după M1): ALTER TABLE conv.contacts DROP COLUMN channel, DROP COLUMN external_user_id;
```
Drop-ul amânat = poți face rollback de cod fără rollback de date. get_or_create_contact (T045) lucrează DOAR prin channel_identities de acum.
