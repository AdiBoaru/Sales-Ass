## Goal
Ultimul pas al spargerii `agent.py`. Descoperirea din doc §2: **shaping-ul determinist POST-loop**
(`cheaper`/`attr_query`/cross-sell/rehidratare/checkout-fallback) împletit cu render-ul **ESTE
planner-ul implicit**. Îl facem explicit:
- **Faza E → `planner.py`**: `build_plan(ctx, deps, run) -> ResponsePlan` (mod + produse + offers).
- **Faza F → `finalize.py`**: `render(plan)` = dispatch renderer (comparison/rich/prose/order/fallback)
  → validează → retry → fallback.
- **Calitate (§5.7)**: îmbogățește `ResponsePlan` cu un ghid de conținut (must_answer/follow_up/tone)
  + response templates per intent + answer-completeness checks. Aici e SINGURA schimbare de comportament.

## Business Context
Intro-ul + follow-up-ul sunt „vocea de consultant" — diferența dintre un bot care listează și unul
care vinde (paritatea iZi). Structural, blocul post-loop de ~170 linii cu `if`-uri împletite e cel mai
greu de extins fără regresie. Un `ResponsePlan` explicit (a) desface „decide ce" de „randează cum" și
(b) devine locul natural pentru ghidul de calitate. Planul e determinist (fără cost); nano DOAR pe
cazuri complexe, în spatele unui kill-switch.

## Technical Description

### Felia 1 — STRUCTURĂ (byte-identic): `planner.py` + `finalize.py`
`ResponsePlan` (value-object, doar câmpuri structurale în această felie):
```
ResponsePlan(
    mode: Literal["comparison","rich","prose","order","fallback"],
    products: list[dict],            # set final: dedup / rehydrated / cheaper / cross-sell
    offers: PlanOffers,              # checkout_offer, cross_sell flag, commerce_note
    relevance: RelevanceMeta | None, # off-category signal pt compose
    reply_override: Reply | None,    # ramuri care răspund direct (cheapest_already/no_more/login)
)
```
- `src/agent/planner.py` — `build_plan(ctx, deps, run: ToolRun) -> ResponsePlan`: mută faza E
  (blocul post-loop din `agent_stage`) 1:1: order_gated→login, purchase→checkout-fallback,
  cross-sell, `_dedupe`, `attr_query`, `cheaper`, rehidratare, set `ctx.retrieval`, decide `mode`.
- `src/agent/finalize.py` — `render(ctx, deps, plan)`: mută faza F 1:1 (`_finalize`,
  `_finalize_grounded`, `_finalize_rich`, dispatch-ul comparison/rich/prose/fallback). Folosește
  `validator` (NX-142) + `compose` + `fallbacks` (NX-143).
- `stage.py` devine: A(admit) → `try_pre_intents` (NX-143) → C(prepare) → D(`ToolRun`+loop) →
  `plan = build_plan(...)` → `render(plan)`. Țintă < 120 linii.
- **Comportament byte-identic.** Aceleași branch-uri, doar reașezate în plan+render.

### Felia 2 — CALITATE (schimbare de comportament, kill-switched)
1. **Ghid de conținut pe `ResponsePlan`**: câmpuri noi `must_answer[]`, `must_not[]`, `tone`,
   `follow_up` derivate DETERMINIST din rută + `decision_axes` (NX-139, `compose.py`) +
   `response_style_profile` per business. Injectat ca bloc „Plan de răspuns: …" în user-ul de
   compunere (ca `decision_axes`). Ghid, NU conținut — grounding-ul rămâne la validator (P2).
2. **`response_style_profile` per business** (DomainPack/`businesses.settings`): `ton`,
   `nivel_detaliu`, `reguli_salut`, `reguli_upsell`, `disclaimere`. Citit de planner+prompt (P9).
   Default per vertical în `src/domain/defaults/*.json`.
3. **Response templates per intent** (`sales_recommendation`/`compare`/`no_results`/`order_status`/
   `price_objection`/`follow_up`): scheletul de secțiuni obligatorii, extinde `compose.py`.
4. **Answer-completeness checks** (cod pur, post-compunere, pre-livrare): `check_completeness(reply,
   plan) -> list[str]` — motivația? preț real? next step? întrebare utilă la intent vag? Lipsă →
   `completeness_gap` (event) + eventual retry ghidat (reutilizează bucla din `finalize`). NU
   blochează livrarea (P6).

### Felia 3 — opțional (kill-switch, al 3-lea LLM)
Nano generează ghidul de conținut pe cazuri complexe (compare multi-produs / obiecție).
