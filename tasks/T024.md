# T024 — 002: catalog.product_embeddings
**Owner:** S · **Faza:** MVP · **Zi:** Z1-2 · **Branch:** `feat/T024-sql-embeddings` · **Complexitate:** S · **Estimare:** 1h

## Goal
Stocarea vectorilor pentru ranking-ul semantic din search_products — în tabel SEPARAT de products, nu coloană pe el. Motivele: re-embed la schimbare de model fără lock pe tabelul fierbinte, content_hash previne re-embed inutil la fiecare sync, și poți ține două modele în paralel la o migrare de embeddings.

## Business Context
Căutarea semantică e diferența dintre „nu am găsit nimic" și o vânzare, pe query-uri umane gen „ceva pentru ten uscat iarna".

## Technical Description
```sql
create extension if not exists vector;  -- idempotent; verificat la T020
create table catalog.product_embeddings (
  product_id uuid primary key references catalog.products(id) on delete cascade,
  business_id uuid not null,
  model text not null,                       -- 'text-embedding-3-small'
  embedding vector(1536) not null,
  content_hash text not null,                -- sha256(ai_summary)
  updated_at timestamptz not null default now()
);
create index idx_pemb_hnsw on catalog.product_embeddings
  using hnsw (embedding vector_cosine_ops);
create index idx_pemb_business on catalog.product_embeddings(business_id);
```
Operatorul de distanță folosit ulterior: `embedding <=> $query` (cosine distance, mic = similar). HNSW peste subsetul filtrat: query-ul real (T112) filtrează ÎNTÂI pe products (categorie/preț/stoc/business) și ordonează pe distanță DOAR candidații — la 221 produse e oricum instant, dar pattern-ul scalează la 50k.
