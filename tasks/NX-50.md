# NX-50 — P0-A: Conexiunea workerului pe rol de login `bot_runtime`
**Owner:** S · **Faza:** MVP · **Zi:** Z2 · **Branch:** feat/NX-50-db-role · **Complexitate:** M · **Estimare:** 6h

## Goal
Izolarea multi-tenant nu mai depinde de starea de sesiune a unui pooler transaction-mode. Workerul nu se mai loghează NICIODATĂ ca `postgres`.

## Business Context
P0-A din audit (S5): azi workerul intră ca `postgres` prin poolerul Supabase (transaction-mode) și face `SET ROLE bot_runtime` + `SET app.business_id` de sesiune. Sub multiplexare, SET-urile se pot scurge la alt client sau reseta → trei moduri de eșec: query pe tenantul greșit, „date dispărute" aleator (GUC gol → RLS fail-closed), sau drum de cod care rulează ca superuser (bypass RLS total). Testul de izolare a trecut doar secvențial.

## Technical Description
**Varianta A (preferată):** pool asyncpg pe conexiunea DIRECTĂ (port 5432, sau session-mode pooler) cu rol de LOGIN `bot_runtime` (parolă proprie, nu postgres). La checkout: `SELECT set_config('app.business_id', $1, false)`; la release: `RESET app.business_id` (hook `reset` al pool-ului). **Varianta B (dacă rămâi pe transaction-pooler):** TOATE unitățile de lucru în tranzacție explicită, cu `SET LOCAL ROLE` interzis (rolul e deja de login) și `set_config('app.business_id', $1, true)` ca PRIM statement al tranzacției. Credențialele `postgres` rămân DOAR în tooling-ul de migrări, niciodată în `.env`-ul workerului.

## Principii CLAUDE.md aplicabile
P2 (business_id obligatoriu — acum garantat de infrastructură, nu de disciplină), P7 (secrete separate pe roluri).

## Implementation Steps
1. SQL (rulat ca admin, o dată): `ALTER ROLE bot_runtime LOGIN PASSWORD '<din vault>'; ` + verifică GRANT-urile existente (SELECT/INSERT/UPDATE pe tabelele runtime).
2. `.env`: `DATABASE_URL_BOT=postgresql://bot_runtime:...@db.<proj>.supabase.co:5432/postgres` (direct, sslmode=require). Șterge orice URL cu user postgres din configul workerului.
3. `src/db/pool.py`: `create_pool(dsn=settings.database_url_bot, min_size=2, max_size=10, init=_assert_role)`; context manager `tenant_connection(business_id)` care face checkout + `set_config(..., false)` + yield + `RESET` în finally.
4. Toate query-urile din `src/db/queries/*` primesc conexiunea din `tenant_connection` — nicio cale alternativă.
5. Verifică `statement_cache_size` implicit (pe conexiune directă rămâne activ = mai rapid; pe varianta B setează 0).
6. Documentează în `docs/db_connections.md`: cine se conectează cu ce rol, pe ce port, de ce.
