## Goal
Două semnale de upsell DETERMINISTE injectate în contextul agentului (stagiul 6),
ca modelul (mini) să poată propune natural complementarul corect și pragul de
transport gratuit — fără să inventeze nimic. (1) **Cross-sell map**: per categorie,
o listă `goes_with` de categorii complementare (curățător → cremă, fond de ten →
burete) trăiește în DB (groundată, principiul 9); context builder-ul, văzând ce
produse a arătat botul (`state.displayed_products`), adaugă blocul „Produse care
merg bine cu astea: <categorii>". (2) **Free-shipping gap**: dacă magazinul are un
prag de transport gratuit (`businesses.settings.free_shipping_threshold`) și coșul
(`state.cart`, NX-79) e SUB prag, builder-ul adaugă „Mai are nevoie de X lei până
la transport gratuit". Ambele sunt INPUT pentru agent (sugestii), nu reply-uri
separate — agentul decide mutarea de vânzare (CLAUDE.md stagiul 7), nu codul.

## Business Context
Cross-sell-ul și „mai pui 30 lei și ai transport gratis" sunt cele mai ieftine
pârghii de creștere a coșului mediu (AOV) — bani direcți pentru client, retainer
justificat pentru agenție. Dar trebuie GROUNDATE: dacă botul inventează
complementare aiurea („la rujul ăsta merge un aspirator") sau promite transport
gratuit la prag greșit, pierde încrederea și generează reclamații. NX-32 face
semnalul un FAPT din DB (harta de complementaritate + pragul per business), nu o
halucinație. Semnalul de free-shipping apare DOAR sub prag (peste prag e zgomot
care scade conversia — clientul deja are transport gratis).

## Technical Description

### Unde NU există `taxonomy` — decizia de stocare a `goes_with`
Schema_v2 **nu are tabel `taxonomy`** (decizie explicită, `schema_reference.md`
§Decizii pct.2: taxonomia bogată se adaugă aditiv DOAR la nevoie). Seed-ul vorbește
de „taxonomie", dar T130 (cardul de taxonomy din planul vechi) e obsolet — sursa
de adevăr e `categories`. **Deci `goes_with` se adaugă ADITIV pe `categories`**, ca
o coloană jsonb, nu printr-un tabel nou fictiv. E mapare categorie→categorii, deci
îi e locul natural pe rândul categoriei:
```sql
alter table categories
  add column if not exists goes_with jsonb not null default '[]'::jsonb;
  -- listă de slug-uri de categorii complementare, tenant-scoped implicit (categories e tenant-scoped)
  -- ex. pe categoria 'curatatoare': ["creme-fata","tonice"]
```
`goes_with` e o listă de **slug-uri** (nu UUID-uri) — stabile la re-seed, lizibile,
și matchable direct cu ce întoarce deja `list_category_slugs`/`search_products`.

### NX-32a — coloana `goes_with` + migrare + seed
Migrare în spiritul lui 003/004/005 (`docs/006_cross_sell.sql`, idempotentă,
re-rulabilă): `alter table ... add column if not exists`. `categories` e DEJA
tenant-scoped și în lista RLS din 003 → `bot_runtime` are deja `select` pe ea,
nu trebuie grant nou (botul DOAR citește `goes_with`; scrierea e a
dashboard-ului/seed-ului). Seed de test pe `nativex-demo` (beauty): câteva mapări
realiste pe categoriile seedate (curățător→cremă/tonic, machiaj→demachiant, etc.).

### NX-32b — query cross-sell în context builder (stagiul 6)
`ProductRef` din state poartă DOAR `{product_id, name, price}` (P8) — **fără
categorie**. Deci builder-ul are nevoie de un singur SELECT care, din id-urile
produselor arătate, derivă categoriile lor complementare:
```python
# src/db/queries/catalog.py
async def cross_sell_categories(
    conn: asyncpg.Connection, business_id: str, product_ids: list[str], *, limit: int = 4
) -> list[str]:
    """Slug-urile categoriilor complementare (`goes_with`) pentru produsele arătate.
    business_id = $1 (P7). Întoarce slug-uri DISTINCTE, exclude categoriile produselor
    deja arătate (nu sugerăm ce e deja pe ecran). Tenant-scoped + RLS ca plasă."""
    if not product_ids:
        return []
    rows = await conn.fetch(
        """
        with shown as (
            select distinct c.id as cat_id, c.slug as cat_slug, c.goes_with
            from products p
            join categories c on c.id = p.primary_category_id
            where p.business_id = $1 and p.id = any($2::uuid[])
        ),
        suggested as (
            select jsonb_array_elements_text(goes_with) as slug from shown
        )
        select distinct g.slug
        from suggested s
        join categories g on g.business_id = $1 and g.slug = s.slug
        where s.slug not in (select cat_slug from shown)   -- nu re-sugera ce e arătat
        limit $3
        """,
        business_id, product_ids, min(limit, 6),
    )
    return [r["slug"] for r in rows]
```
Bloc nou în `src/worker/context.py`, pe modelul `state_block`/`summary_block`
(pur, bugetat, gol→`""`). Deoarece blocul are nevoie de I/O (un SELECT), îl seedăm
**ca pe `ctx.summary`**: processor-ul îl pre-calculează și îl pune pe un câmp nou
`ctx.upsell`, iar `context.py` doar îl formatează (zero I/O în context builder, ca
acum). Asta păstrează `context_blocks(ctx)` pur:
```python
# src/worker/context.py
def cross_sell_block(ctx: TurnContext, *, max_chars: int = 160) -> str:
    """Categorii complementare produselor arătate (din ctx.upsell.cross_sell_slugs,
    seedat de processor). Gol → "". INPUT pentru agent, nu reply."""
    slugs = (ctx.upsell.cross_sell_slugs if ctx.upsell else [])
    if not slugs:
        return ""
    return ("Produse care merg bine cu cele arătate (categorii): " + ", ".join(slugs))[:max_chars]
```

### NX-32b — semnalul free-shipping gap
Pragul vine din `businesses.settings.free_shipping_threshold` (jsonb, lei, per
business — NU coloană nouă, settings e fix pentru feature flags). Coșul vine din
`state.cart` (NX-79). Calcul DETERMINIST în cod (zero LLM):
```python
# src/worker/context.py
def free_shipping_block(ctx: TurnContext, *, currency: str = "lei") -> str:
    """„Mai are nevoie de X lei până la transport gratuit" — DOAR sub prag.
    Prag din businesses.settings.free_shipping_threshold; coș din state.cart (NX-79).
    Peste prag / fără prag / coș gol → "" (semnalul e zgomot peste prag)."""
    threshold = ctx.business.settings.get("free_shipping_threshold")
    if not threshold or threshold <= 0:
        return ""
    cart_total = _cart_total(ctx.state)          # sumă din state.cart (ref-uri × preț, NX-79)
    if cart_total <= 0 or cart_total >= threshold:
        return ""                                # coș gol SAU deja peste prag → tăcere
    gap = threshold - cart_total
    return f"Coșul e la {cart_total:.2f} {currency}; mai are nevoie de {gap:.2f} {currency} până la transport gratuit ({threshold:.2f} {currency})."
```
`_cart_total` citește DOAR ref-urile din `state.cart` (P8 — `{product_id, name,
price, qty}`, nu obiecte complete). **NU** se cheamă DB pentru preț (prețul e deja
în snapshot-ul de coș, ca la `displayed_products`).

### Integrarea în `context_blocks` (ordine cronologică păstrată)
```python
def context_blocks(ctx: TurnContext) -> str:
    blocks = [
        summary_block(ctx),
        customer_profile_block(ctx.contact),
        state_block(ctx.state),
        cross_sell_block(ctx),        # NOU (NX-32b) — semnal de upsell
        free_shipping_block(ctx),     # NOU (NX-32b) — semnal de prag
    ]
    return "\n".join(b for b in blocks if b)
```
Stă în mesajul USER (dinamic), ca tot blocul de context — promptul system static
rămâne byte-identic (prompt caching neatins). Semnalele sunt SUGESTII; promptul
agentului (`prompt_builder`) capătă o frază scurtă „dacă apare un semnal de upsell,
poți să-l folosești natural, fără presiune; NU inventa complementare/praguri".

### Câmpuri TurnContext scrise aici
**Câmp nou: `ctx.upsell`** (dataclass `UpsellSignals { cross_sell_slugs: list[str] }`),
**owner = processor** (îl seedează cu `cross_sell_categories(...)` ÎNAINTE de
pipeline, ca `ctx.summary`). Context builder-ul DOAR citește `ctx.upsell` și
`ctx.state.cart`/`ctx.business.settings` — NU scrie niciun câmp (rămâne pur, ca
acum). Free-shipping gap-ul e calculat la fiecare formatare din state+settings, NU
stocat (derivat, nu flag — la fel ca `in_24h_window`). Niciun alt câmp atins.

### Error handling / degradare
- `cross_sell_categories` aruncă (DB) sau întoarce gol → `ctx.upsell` rămâne `None`
  / listă goală → `cross_sell_block` → `""` (fără semnal, turul continuă, P6).
  Processor-ul prinde excepția (best-effort, ca summarizer-ul) — un upsell ratat
  NU rupe răspunsul.
- Fără prag în settings / coș gol / peste prag → free-shipping block gol (tăcere
  pe semnal, NU pe răspuns).
- Semnalele NU produc niciodată reply singure — dacă agentul nu le folosește, nu se
  întâmplă nimic rău.

## Test Cases
Toate fără apeluri reale: blocurile sunt funcții pure; `cross_sell_categories` pe DB
de test (tranzacție rollback, ca `test_tenant_isolation`); golden cu ScriptedLLM
(`tests/test_golden.py`). ZERO OpenAI/DB-prod în CI.

**Happy Path:**
1. `displayed_products` = [o cremă din categoria `curatatoare` cu
   `goes_with=["creme-fata","tonice"]`] → `cross_sell_categories` →
   `["creme-fata","tonice"]` → `cross_sell_block` = „Produse care merg bine…:
   creme-fata, tonice". Golden: ScriptedLLM propune un tonic, nu un produs random.
2. Coș la 70 lei (2 ref-uri în `state.cart`), prag 150 → `free_shipping_block` =
   „…mai are nevoie de 80.00 lei până la transport gratuit (150.00 lei)". Golden:
   reply menționează 80 lei (grounded), nu o cifră inventată.

**Edge Cases:**
1. Categoria produsului arătat are `goes_with=[]` (fără mapare) →
   `cross_sell_categories` → `[]` → bloc gol; restul contextului intact.
2. `goes_with` conține un slug care NU mai există în `categories` (categorie ștearsă)
   → JOIN pe `categories` îl elimină → nu apare slug fantomă în sugestie.
3. Coș EXACT pe prag (150 == 150) → free-shipping gol (deja are transport gratuit,
   `>=` exclude pragul). Coș peste prag → gol.
4. `goes_with` sugerează o categorie care e DEJA printre cele arătate →
   exclusă (`not in shown`), nu re-sugerăm ce e pe ecran.
5. Fără `free_shipping_threshold` în settings (business care nu oferă) →
   free-shipping block mereu `""` (feature opt-in per business).

**Failure Cases:**
1. `cross_sell_categories` aruncă (DB jos) → processor prinde, `ctx.upsell=None` →
   `cross_sell_block`=`""`, pipeline-ul continuă, niciun reply ratat (P6).
2. `state.cart` malformat (preț lipsă pe un ref) → `_cart_total` sare ref-ul invalid
   (ca `ConversationState.from_jsonb` la `displayed_products`), nu crapă; calculează
   pe ce e valid.
3. `businesses.settings.free_shipping_threshold` non-numeric (string greșit din
