# T029 — 002: analytics.usage_daily
**Owner:** S · **Faza:** MVP · **Zi:** Z1-2 · **Branch:** `feat/T029-sql-usage` · **Complexitate:** XS · **Estimare:** 0.5h

## Goal
Rollup-ul zilnic per tenant: dashboard-ul, raportul lunar de factură și cost-tracking-ul citesc DE AICI, niciodată din evenimentele raw. Append-only events + rollup citibil = pattern-ul standard care ține analytics-ul ieftin la orice volum.

## Business Context
„Câte conversații, cât cost, cât revenue atribuit luna asta?" = întrebarea de la fiecare factură de retainer. Răspunsul trebuie să fie un SELECT simplu.

## Technical Description
```sql
create table analytics.usage_daily (
  business_id uuid not null references core.businesses(id),
  day date not null,
  conversations int not null default 0,
  messages_in int not null default 0,
  messages_out int not null default 0,
  templates_sent int not null default 0,
  tokens_in bigint not null default 0,
  tokens_out bigint not null default 0,
  cost_usd numeric(12,4) not null default 0,
  cache_hits int not null default 0,
  handoffs int not null default 0,
  orders_attributed int not null default 0,
  revenue_attributed numeric(14,2) not null default 0,
  intents jsonb not null default '{}'::jsonb,   -- {"sales": 41, "order": 12, ...}
  primary key (business_id, day)
);
```
Populat de jobul T195 (P1) cu upsert idempotent (`on conflict (business_id, day) do update`). Până la T195, query-urile manuale din T166 servesc aceleași răspunsuri direct din events — tabelul există de acum ca ținta să fie stabilă.
